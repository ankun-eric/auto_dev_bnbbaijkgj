# 后台菜单管理 - 图标选择器扩充优化 产品需求文档（PRD）

## 1. 需求概述

### 1.1 背景与目的

当前管理后台「首页配置 → 首页菜单管理」中已有一套 Emoji 图标选择器，支持 **7 大分类浏览**（医疗健康、运动健身、食物饮料、自然植物、活动物品、符号标志、动物）、**中文关键词搜索**（基于 260+ 条 Emoji 中文标签映射）以及**智能推荐**（根据菜单名称自动匹配推荐图标）。

然而，当前的 Emoji 库和中文标签映射仍存在覆盖不足的问题：

1. **Emoji 种类有限**：现有 7 个分类合计约 260 个 Emoji，许多常用 Emoji（如 🤖 机器人、🚗 汽车、🏢 办公楼、😀 表情等）未被收录
2. **搜索关键词覆盖不全**：搜索"机器人"、"汽车"、"办公"等常见关键词时无结果返回，影响管理员操作效率
3. **分类维度不够丰富**：缺少科技数码、人物表情、交通工具、建筑地标、天气时间、办公学习、金融商务等常用分类

本次优化旨在：大幅扩充 Emoji 库的数量和分类维度，同步扩充中文标签映射，让管理员在设置菜单图标时有更丰富、更全面的选择。

### 1.2 目标用户

- 管理后台运营人员（管理员角色）

### 1.3 核心价值

- 将可选 Emoji 数量从约 260 个扩充至 **500+**，覆盖绝大多数常见场景
- 新增 7 个图标分类，让分类浏览更直觉、更高效
- 同步扩充中文标签映射至 **500+ 条**，让关键词搜索覆盖率从约 60% 提升至 **95%+**
- 保持现有功能（搜索、推荐、分类浏览）不变，纯增量优化，零风险

#### F02: 每个新增分类的 Emoji 填充

为每个新增分类精选 15~30 个最常用、最直觉的 Emoji，确保：
- 每个分类内部 Emoji 主题一致，不混杂其他分类内容
- 优先收录在菜单图标场景下最常用的 Emoji
- 避免与现有分类中已有的 Emoji 大量重复（少量重复可接受，如 ⭐ 可同时出现在"天气时间"和"符号标志"中）

## 3. 页面/界面设计

### 3.1 页面结构与导航

本次优化不涉及新增页面，仅对现有**菜单编辑弹窗**中的 **Emoji 选择器 Modal**（`EmojiPickerModal` 组件）进行增量改造。

### 3.2 Emoji 选择器 Modal 界面变化

#### 变化 1：分类 Tab 区域

**现状**：7 个分类 Tab（医疗健康、运动健身、食物饮料、自然植物、活动物品、符号标志、动物）

**优化后**：14 个分类 Tab，新增：科技数码、人物表情、交通工具、建筑地标、天气时间、办公学习、金融商务

**UI 处理**：
- 分类 Tab 使用 `flexWrap: 'wrap'` 自动换行（现有实现已支持）
- 每个 Tab 保持现有样式（圆角按钮，选中蓝色高亮）
- Tab 排列顺序：医疗健康 → 运动健身 → 食物饮料 → 自然植物 → 活动物品 → 符号标志 → 动物 → 科技数码 → 人物表情 → 交通工具 → 建筑地标 → 天气时间 → 办公学习 → 金融商务

#### 变化 2：Emoji 网格区域

- 新增分类下的 Emoji 以与现有分类相同的网格布局展示（10 列，36x36px）
- 选中样式保持一致（蓝色边框高亮 + ✓ 标记）

#### 变化 3：搜索功能

- 搜索逻辑不变（基于 `EMOJI_LABEL_MAP` 中文关键词匹配，300ms debounce）
- 因标签库扩充（260+ → 500+），搜索结果自然更丰富
- 例如搜索"机器人"将能匹配到 🤖，搜索"汽车"将能匹配到 🚗🚕 等

#### 变化 4：智能推荐

- 推荐逻辑不变（基于 `EMOJI_KEYWORD_MAP` 关键词匹配，最多展示 8 个）
- 因关键词映射表扩充，推荐命中率进一步提升

## 4. 非功能性需求

### 4.1 性能要求

- Emoji 分类数据为静态常量（编译时确定），无运行时性能影响
- 中文标签映射扩充后搜索仍为内存遍历，500+ 条数据搜索延迟可忽略（< 1ms）
- Modal 首次打开渲染时间不超过 100ms

### 4.2 安全要求

- 本次为纯前端静态数据扩充，不涉及用户输入、接口调用、权限变更，无安全风险

### 4.3 兼容性要求

- 新增 Emoji 需确保在主流浏览器（Chrome 90+、Firefox 88+、Safari 14+、Edge 90+）中正常显示
- 部分较新的 Emoji（如 Emoji 14.0+）在旧浏览器中可能显示为方块，需尽量选择 Emoji 12.0 及以下版本的 Emoji 以确保最大兼容性
- 后端 `home_menu_items.icon_content` 字段已确认支持 UTF-8 4 字节 Emoji 字符存储，无需变更

## 5. 业务规则与约束

| 编号 | 规则 | 说明 |
|------|------|------|
| R01 | 已有菜单图标保持不变 | 本次优化不影响已保存的菜单图标数据 |
| R02 | 纯前端改造 | 不涉及后端接口变更、数据库迁移 |
| R03 | Emoji 去重 | 同一分类内不出现重复 Emoji；跨分类少量重复可接受 |
| R04 | 标签唯一性 | `EMOJI_LABEL_MAP` 中每个 Emoji 键唯一，不重复定义 |
| R05 | 现有功能不退化 | 搜索、推荐、分类浏览等现有功能行为不变，仅数据量增加 |

## 7. 异常处理与边界情况

| 场景 | 处理方式 |
|------|----------|
| 新增 Emoji 在某些浏览器中显示异常（方块/乱码） | 优先使用 Emoji 12.0 及以下版本；对于必须使用的新版 Emoji，在标签中标注"需新版浏览器" |
| 分类 Tab 过多导致换行后占用空间过大 | 14 个分类在 520px+ 宽度的 Modal 中可分 2~3 行展示，属于可接受范围 |
| 搜索无结果 | 保持现有行为——显示"未找到匹配的图标"提示 |
| 选中的 Emoji 来自新增分类，后续在旧版代码中打开 | 不受影响——Emoji 作为 Unicode 字符存储在数据库中，与前端分类无关 |
