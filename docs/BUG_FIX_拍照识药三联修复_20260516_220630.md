## 2. Bug 描述

### 2.1 错误现象

#### Bug #1：识别结果错乱（识别错药）

- **现象**：用户上传的是**阿司匹林**药品图片，AI 返回的识别结果却是「感冒药」（药名、成分、适应症全部对不上）
- **根因初判**：
  1. 视觉模型本身对药品包装的细节识别能力不稳定，存在「猜常见药」倾向
  2. 现有 Prompt 虽已写有「严禁虚构」，但**未要求模型必须以 OCR 提取到的药品文字为最高优先级证据**
  3. 后端**未对模型返回的药名与图片 OCR 文字做一致性二次校验**，模型一旦"幻觉"输出常见药，没有任何兜底拦截

#### Bug #2：识别内容啰嗦、格式混乱、多空行

- **现象**：
  - 识别卡片正文段落之间动辄出现 2~3 个连续空行
  - 同一信息块（如"注意事项"）内部又被拆成多段重复表达
  - 整体长度无上限，简单药品也输出长篇
  - 健康档案信息只用了 `gender / allergies / medications` 三项，没把慢病、年龄段、过敏严重度、在服药物剂量、体质等关键维度用上，个性化提醒形同虚设
- **根因初判**：
  1. Prompt 没有「行数上限 / 段落上限 / 禁止连续空行 / 禁止重复段落」的硬性格式约束
  2. `user_profile` 维度太单薄，导致 LLM 没有足够档案信息可结合，只能堆通用话术
  3. 输出未经任何后端格式清洗，模型空行直接透传到前端

#### Bug #3：结尾出现两次 `---disclaimer---`（重复免责声明）

- **现象**：识药卡片末尾连续渲染两段几乎一样的免责声明
- **根因初判**：
  1. Prompt 中要求模型自带一段免责声明，业务层在返回前又拼接了一段，前端再渲染一次
  2. **`call_ai_model` / `call_ai_model_stream` 层没有统一的输出清洗（sanitize）函数**，所有 AI 接口（拍照识药、AI 对话、健康自查、报告解读等）都各自拼装，难免重复
  3. 历史多次"打补丁"叠加，导致免责声明在多处插入

| 步骤 | 操作 | 预期结果 | 实际结果 |

| 3 | 观察识别卡片正文格式 | 段落紧凑、无连续空行、长度自适应 | 出现 2~3 个连续空行、重复段落、整体过长 |

| 维度 | 影响 |

## 3. 预期正确效果

### 3.1 Bug #1 修复后预期：识别准确 + OCR 优先 + 一致性二次校验

> 修复力度（用户选择）：**C. A + B 都做** —— 锁死 OCR 优先 + 后端做模型结果 vs OCR 一致性二次校验

**Prompt 改造（锁死 OCR 优先）：**

- 在 `identify_drug_from_image` 的系统 Prompt 中显式写入以下硬约束：
  - 「**药品名称的唯一可信来源**：图片中的 OCR 文字（药盒标签、说明书首页、品名条）。视觉判断与 OCR 冲突时，必须**优先采信 OCR**」
  - 「严禁基于"看起来像感冒药盒"这种视觉相似性凭空推测药名；OCR 没有捕获到清晰药名时，**必须**返回 `retake`（请求重拍）或 `pick_candidate`（候选列表）」
  - 「OCR 提取到的药名片段必须在最终输出的"药品名称"字段中显式出现，否则视为模型幻觉」

**后端二次校验（一致性兜底）：**

- 新增工具函数 `verify_drug_name_against_ocr(model_drug_name, ocr_text)`：
  - 使用编辑距离 + 关键字段拆分（通用名/商品名/规格）计算相似度
  - 阈值 **70%** 以下视为不一致
- 校验失败时的处理：
  1. 不直接返回错误结论
  2. **降级**为 `pick_candidate`（基于 OCR 文字给出候选药品列表），或返回 `retake` 让用户重拍
  3. 卡片上明显提示「图片中识别到的关键字与系统判断不一致，请确认或重新拍照」
- 校验成功（≥ 70%）后才正常输出完整识药卡片

### 3.2 Bug #2 修复后预期：格式硬约束 + 健康档案全维度结合

> 格式约束（用户选择）：**D. A + B（行数上限 15 行 + 每段最多 2 行 + 自适应空间保留）**
> 健康档案维度（用户选择）：**F. 全选（慢病 / 年龄段 / 过敏细化 / 在服药物 / 体质）**

**格式硬约束（Prompt + 后端双保险）：**

| 约束项 | 上限 | 实现方式 |
| --- | --- | --- |
| 整张卡片总行数 | ≤ 15 行（含标题） | Prompt 写入 + 后端 `sanitize_ai_output()` 行数截断 |
| 每个分段文字行数 | ≤ 2 行 | Prompt 写入 + 后端按段切分校验 |
| 连续空行 | ≤ 1 行（即最多 1 个空行作分隔） | 后端正则 `\n{3,}` → `\n\n` 强制压缩 |
| 重复段落 | 禁止 | 后端按段落 hash 去重 |
| 整体字数 | 软上限 ~400 字（自适应） | Prompt 提示软约束 |

**健康档案上下文升级：**

- 后端 `health_profile_service` 新增 `build_user_profile_for_drug_identify(user_id)`，向 Prompt 注入以下完整档案上下文（JSON 结构）：

```json
{
  "age_group": "老年人 / 成人 / 婴幼儿 / 孕产妇",
  "gender": "male/female",
  "chronic_diseases": ["高血压", "糖尿病", "中度肝功能不全"],
  "allergies": [
    { "name": "青霉素", "severity": "重度" },
    { "name": "海鲜", "severity": "轻度" }
  ],
  "current_medications": [
    { "name": "二甲双胍", "dose": "0.5g", "freq": "bid" },
    { "name": "氨氯地平", "dose": "5mg", "freq": "qd" }
  ],
  "tcm_constitution": "气虚质（如档案中存在）"
}
```

**输出结构（5 + 1 块）：**

1. 药品名称（OCR 优先 + 通用名/商品名）
2. 主要成分
3. 适应症
4. 用法用量
5. 注意事项
6. **结合您的健康档案**（新增独立块；用户选择 ① 推荐方案）
   - 仅在档案中存在相关风险点时输出；无风险点时**不强行增加**该块，避免空话
   - 风险点穷举对照：
     - 过敏史 ↔ 药品成分（含交叉过敏）
     - 慢病 ↔ 药品禁忌（肝肾功能、糖尿病、高血压等）
     - 当前在服药物 ↔ 药物相互作用
     - 年龄段 ↔ 剂量调整 / 禁用人群
     - 中医体质 ↔ 中成药寒热属性（如可用）

### 3.3 Bug #3 修复后预期：全局统一去重免责声明

> 实现位置（用户选择）：**C. 新增 `sanitize_ai_output()` 工具函数，所有 AI 接口在 return 前调用**

**实现方案：**

新增 `backend/app/utils/ai_output_sanitizer.py`，提供统一清洗函数：

```python
def sanitize_ai_output(text: str, *, max_lines: int = 15, max_paragraph_lines: int = 2,
                       dedup_disclaimer: bool = True) -> str:
    """
    AI 输出统一兜底清洗：
    1. 压缩连续空行（\n{3,} → \n\n）
    2. 段落 hash 去重（去掉完全重复段落）
    3. 免责声明去重（只保留最后一段 ---disclaimer--- ~ ---/disclaimer--- 之间内容，其余删除）
    4. 行数 / 每段行数硬截断
    """
```

**接入点（一次性全量改造，涵盖所有 AI 输出接口）：**

| 接入点 | 文件 | 改动 |
| --- | --- | --- |
| AI 模型主调用 | `ai_service.py::call_ai_model` | return 前调 `sanitize_ai_output(text)` |
| AI 模型流式 | `ai_service.py::call_ai_model_stream` | 流式末尾 flush 后调 sanitize |
| 拍照识药 | `identify_drug_from_image` | 返回结构化 JSON 前对各文本字段调 sanitize |
| AI 对话 | `ai_chat.py` 各回复接口 | return 前调 sanitize |
| 健康自查 | `HealthSelfCheckDrawer` 对应后端 | 同上 |
| 报告解读 | 报告解读接口 | 同上 |

**严禁**前端再拼接免责声明：免责声明字段统一由后端在卡片底部以单独字段返回，前端固定渲染 1 次，但**后端 sanitize 仍是兜底**，双层保险。

```
【药品识别 - 系统 Prompt 关键约束】

```

## 7. 【追加章节】链路差异修复 · 聊天内嵌识药引擎（方案 E）

> 追加时间：2026-05-16 23:xx
> 触发原因：用户反馈"`/drug` 专用页能识别成功，但 ai-home 拍照识药识别错乱"。经代码比对发现**两条链路完全不同**，原 3.1~3.3 节的所有 Prompt / 一致性校验 / 健康档案改造**只对 `/drug` 链路生效，ai-home 根本走不到**。本章节为最重要的根因补丁。

### 7.1 链路差异根因（核心发现）

| 维度 | `/drug` 专用链路（可识别成功） | ai-home 拍照识药（识别失败/错乱） |
|---|---|---|
| 后端入口 | `/api/drug/identify-v2` | `/api/chat/stream`（通用聊天接口） |
| OCR 提取 | ✅ 强制调用 `smart_ocr_recognize` | ❌ 完全没有 OCR |
| 图片送达 LLM 方式 | 图片 URL 作为多模态 `image_urls` 传给视觉模型 + OCR 文字作为文本上下文 | ⚠️ 仅把图片 URL **拼成文本字符串**（`[用户上传的图片 1 张]\n1. https://xxx.jpg`）发给聊天模型 |
| 识药专用 Prompt | ✅ 锁死 OCR 优先 + 严禁猜药 | ❌ 走通用对话 Prompt，无识药约束 |
| 结构化校验 | ✅ 返回 `recognized/confidence/medicines/...` 并写入 `DrugIdentifyDetail` | ❌ 让聊天 LLM 自由发挥（这是格式啰嗦 + 免责重复的真正源头） |
| 健康档案传入 | ✅ v2 已传 `gender/allergies/medications` | ⚠️ 由通用聊天系统 Prompt 拼装，与识药完全不同 |

**直白结论**：ai-home 的"拍照识药"实际上**根本没走识药专用链路**，而是走了**普通聊天 + 文本里贴一个图片 URL 字符串**。模型既看不到 OCR 文字、又没有"严禁猜药"硬约束、还没结构化校验，所以才会把阿司匹林识别成感冒药、格式乱、免责重复。

### 7.2 用户最终诉求（决定方案走向）

| 用户要求 | 翻译为技术约束 |
|---|---|
| ① ai-home 为主，菜单模式废弃，识药以 ai-home 方式为主 | **不能让前端跳走**，必须在聊天界面内完成识药 |
| ② 必须在主对话里继续聊天，不能跳到其他对话（对标豆包、阿福） | **不能跳 `drug_query` 会话**；识药结果就是当前会话的一条 AI 气泡；用户可继续追问"那能和布洛芬同服吗" |
| ③ 必须保证用户并发大、不影响返回速度 | OCR + 视觉识别必须**并行**；走**流式返回**；高频结果必须**缓存** |

**结论**：原"1-A 前端分流直调 v2 + 跳 drug_query 会话"方案 **直接否决**。采用 **方案 E：聊天内嵌识药引擎**——后端在聊天接口内做识药能力内嵌，对前端透明、对用户无感。

### 7.3 方案 E 架构图（聊天内嵌识药引擎）

```
用户在 ai-home 点【拍照识药】→ 上传图片 → 走原 chat/stream 接口（保持入口不变）
                                            ↓
            后端检测：button_type ∈ {photo_recognize_drug, drug_identify, medication_recognize}
                         且 message 含图片 URL  →  路由到 DrugIdentifyEngine
                                            ↓
                  ┌─────────────────────────┴─────────────────────────┐
                  ↓ (asyncio.gather 并行)                                ↓
            smart_ocr_recognize(image_url)                  vision_model_identify(image_url)
                  └─────────────────────────┬─────────────────────────┘
                                            ↓
              一致性校验 verify_drug_name_against_ocr  +  6 维健康档案 + 识药专用 Prompt
                                            ↓
                ┌───────────────────────────┴───────────────────────────┐
        置信度 ≥ 80% / 校验通过                          置信度 < 80% / 校验失败
                ↓                                                       ↓
       结构化结果 → SSE 流式推送                          retake / pick_candidate
       message_type = drug_identify_card                  message_type = drug_identify_retake
                ↓                                                       ↓
                └───────────────────────────┬───────────────────────────┘
                                            ↓
         同一会话入库为 assistant message，meta 字段含 medicines / family_member_id / image_hash
                                            ↓
         前端按 message_type 渲染：
           - drug_identify_card → DrugIdentifyCard（含"加入用药计划"按钮）
           - drug_identify_retake → 文本气泡 + 重新拍照按钮
                                            ↓
         用户在同一会话继续追问"能和布洛芬同服吗？"
                                            ↓
         后端 chat 接口检测到上一条 message 是 drug_identify_card，
         自动把卡片里的药品 JSON（通用名/成分/规格）作为**隐式 system context** 注入下一轮聊天
```

### 7.4 关键设计决策与对应实现点

#### 7.4.1 交互衔接：卡片 + 继续聊天框（豆包对标） · 用户选择 A

- 识药成功 → 会话内渲染 `DrugIdentifyCard`（与 `/drug` 页面完全等价）
- 卡片含「加入用药计划」按钮，复用现有 PRD-DRUG-CARD-V3 组件
- 卡片所在 assistant message 的 `meta` 字段持久化药品 JSON：
  ```json
  {
    "message_type": "drug_identify_card",
    "medicines": [{"name": "阿司匹林肠溶片", "spec": "100mg*30片", "generic_name": "阿司匹林", ...}],
    "image_hash": "sha256:xxx",
    "family_member_id": 123,
    "confidence": 0.92
  }
  ```
- 用户在同一输入框继续追问时，后端 chat handler 检测**最近 3 条 assistant message** 是否含 `drug_identify_card`，若是则将其 `medicines` 作为隐式 system context 拼入下一轮 Prompt（即"上文识别到的药品为：xxx，用户接下来的问题请围绕此药品回答"）

#### 7.4.2 性能 SLA：首字节 ≤ 2 秒 + P95 ≤ 10 秒（融合 B+C） · 用户选择 B/C

| 指标 | 目标 | 实现手段 |
|---|---|---|
| **首字节时间（TTFB）** | ≤ 2 秒 | SSE 先推送 `progress` 事件：「识别中…正在分析药盒文字」/「OCR 完成」/「视觉分析完成」/「正在结合您的健康档案…」 |
| **P95 总响应时间** | ≤ 10 秒（缓存命中 ≤ 1.5 秒） | OCR + 视觉并行（asyncio.gather）；模型连接池共享；Redis 缓存 |
| **100 并发不降级** | OCR/视觉总耗时不退化超过 50% | 信号量限流 + 任务队列 + 模型预热 + 健康档案预拉取 |
| **缓存命中** | 同图 7 天复用 | Redis Key = `drug_identify:{image_hash}:{family_member_id}`；命中直接返回结构化结果，TTL 7 天 |

#### 7.4.3 `family_member_id` 必传（小白主动加上） · 回应用户"其他·你给建议"

**强烈建议传 `family_member_id`，理由如下：**

| 对比项 | 不传（仅登录用户档案） | 传 family_member_id（咨询人档案） |
|---|---|---|
| 真实场景 | 妈妈给孩子拍药 → 基于妈妈档案 → 输出**成人剂量** → **医疗级错误** | 切换到"孩子"咨询人 → 输出**儿童剂量、儿童禁忌** → 正确 |
| 技术成本 | 0 | ai-home 调 chat 时本就有"当前咨询人 ID"，payload 多 1 个字段，后端透传，约 30 分钟 |
| 风险 | 给儿童按成人剂量 → 安全事故 | 无 |

**落地方式**：
- ai-home 前端发送消息时，payload 增加 `family_member_id`（来自当前选中的咨询人；若未选则传登录用户自身 ID）
- 后端 `DrugIdentifyEngine` 优先使用 `family_member_id` 拉取档案；不存在时降级为登录用户档案
- 当前咨询人切换功能若未上线，**代码预留入参**，零成本启用

#### 7.5.3 Prompt 关键调整摘要（追加给 DrugIdentifyEngine）

```
【DrugIdentifyEngine 系统 Prompt 关键约束 · 与 identify_drug_from_image 一致】
1. 药品名称的唯一可信来源 = OCR 文字。视觉与 OCR 冲突时优先 OCR。
2. OCR 未提取到清晰药名时，必须返回 retake / pick_candidate，严禁猜测常见药。
3. 输出整体 ≤ 15 行；每段 ≤ 2 行；段落之间最多 1 个空行；严禁重复段落；严禁多余免责声明。
4. 输出必须严格采用 5 块结构 + 仅在档案存在风险时增补「结合您的健康档案」第 6 块。
5. 「结合您的健康档案」素材来源限定于 user_profile JSON 中真实存在的字段；缺字段不推测。
6. 当前咨询人由 family_member_id 标识；剂量、禁忌、相互作用务必基于该咨询人档案（不是登录用户）。
```

### 7.7 方案 E 关键风险与监控

| 风险 | 缓解 |
|---|---|
| OCR + 视觉并行后 OCR 失败时仍要兜底 | OCR 失败时降级为纯视觉 + 强制 `pick_candidate`，不输出确定结论 |
| Redis 缓存命中错误结果 | 缓存 Key 含 `family_member_id`；档案变更后该用户缓存按需失效；用户在卡片上点"识别错了"可强制刷新（清缓存重识） |
| 隐式 context 注入污染后续无关聊天 | 注入窗口限制：仅最近 3 条 message 且 message_type=drug_identify_card；超出窗口或用户主动新话题后自动失效 |
| 聊天接口侵入复杂度上升 | 识药引擎独立在 `drug_identify_engine.py`，chat handler 仅做 1 行路由判断；新增/修改不污染聊天主逻辑 |
| 首字节 SLA 抖动 | 模型连接池预热 + 健康档案预拉取 + SSE 进度事件占位（即使后端真正识别仍在算，前端也能即时看到反馈） |

### 7.8 方案 E 与原 3.1~3.3 节的关系

- 原 3.1 OCR 优先 + 一致性校验 → **复用**到 `DrugIdentifyEngine` 中（B14 强约束共享同一函数）
- 原 3.2 格式硬约束 + 6 维健康档案 → **复用** Prompt 与 `build_user_profile_for_drug_identify()`
- 原 3.3 sanitize_ai_output → **仍然兜底**，所有 SSE 推送的文本片段在 flush 前一并清洗
- **方案 E 增加的核心价值** = 把上述能力**真正接到 ai-home 用户实际走的链路上**，并满足用户的 3 条新要求（不跳走 / 主对话内聊 / 高并发不降速）
