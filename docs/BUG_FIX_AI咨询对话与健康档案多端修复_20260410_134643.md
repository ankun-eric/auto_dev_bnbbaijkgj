# Bini Health — AI 咨询对话与健康档案 Bug 修复方案文档

## 1. Bug 发生背景

### 1.1 项目概述

Bini Health 是一款健康管理应用，涵盖 AI 健康咨询对话、健康档案管理、家庭成员管理、症状分析等功能模块。项目支持多端：H5 网页端、微信小程序端、Flutter App 端（Android/iOS）、管理后台（Admin Web）。

技术栈：
- **后端**：FastAPI + SQLAlchemy + MySQL
- **H5 前端**：Next.js + React + Ant Design Mobile
- **小程序**：原生微信小程序
- **Flutter App**：Flutter 跨平台客户端
- **管理后台**：Next.js + React + Ant Design

### 1.2 涉及功能模块

| 模块 | 说明 |
|------|------|
| AI 咨询对话 | `/chat/[sessionId]` 对话页面，包含文字输入、语音输入、咨询对象切换等功能 |
| 健康档案 | `/health-profile` 页面，展示和管理用户及家庭成员的健康信息 |
| 家庭成员管理 | 添加、编辑家庭成员信息，选择咨询对象 |
| 后端 API | `/api/family/members`、`/api/relation-types`、`/api/chat` 等接口 |

### 1.3 发现时间与发现方式

- **发现时间**：最近一次更新后
- **发现方式**：用户在 H5 端使用过程中发现，其他端（小程序、Flutter App）也需要同步检查并修复

## 2. Bug 描述

本次共发现 **9 个 Bug**，涉及 AI 咨询对话和健康档案两大模块，覆盖全部客户端（H5、小程序、Flutter App）。以下逐一详细描述。

---

### Bug 1：麦克风权限重复提示（全端）

#### 2.1.1 错误现象

在 AI 咨询对话页面，每次用户点击麦克风按钮时，无论之前是否已经授权过麦克风权限，系统都会再次弹出"允许访问麦克风"的权限请求弹窗。用户体验差，操作繁琐。

#### 2.1.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 进入 AI 咨询对话页面 | 正常显示对话界面 | 正常 |
| 2 | 第一次点击麦克风按钮 | 弹出权限请求弹窗 | 弹出弹窗（正确） |
| 3 | 允许麦克风权限 | 进入语音输入模式 | 进入语音模式（正确） |
| 4 | 再次点击麦克风按钮 | 直接进入语音模式，不再弹窗 | **再次弹出权限请求弹窗** |

#### 2.1.3 影响范围

- **全端受影响**：H5 网页端、微信小程序端、Flutter App 端
- H5 端：`checkMicPermission` 函数每次都弹 `Dialog.confirm`，没有先检查权限状态
- 小程序端：`_checkRecordAuth` 函数使用 `wx.authorize`，已授权时虽直接 success，但首次拒绝后逻辑可能重复弹窗
- Flutter 端：每次按住说话都会调用 `Permission.microphone.request()`

### Bug 2：小屏设备发送按钮消失 → 改为输入框内嵌发送按钮（全端）

#### 2.2.1 错误现象

在对话页面底部输入栏，当鼠标/手指点击输入框获得焦点后，在小屏手机上发送按钮（➤ 图标）会消失不见，用户无法发送消息。

#### 2.2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 在小屏设备上进入 AI 咨询对话页面 | 正常显示底部输入栏 | 正常 |
| 2 | 点击输入框使其获得焦点 | 输入框可输入，发送按钮可见 | **发送按钮从页面上消失** |

#### 2.2.3 影响范围

- **全端受影响**：H5、小程序、Flutter App
- 根因：底部输入栏包含 [切换对象按钮] + [输入框] + [麦克风按钮] + [发送按钮] 共 4 个元素并排，输入框 `flex-1` 撑满后，在窄屏设备上发送按钮（40px）被挤出可视区域

### Bug 4：新建家庭成员时关系选择列表不应包含"本人"（全端）

#### 2.4.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 在健康档案/对话页面点击"添加家庭成员" | 弹出关系选择弹窗 | 正常 |
| 2 | 查看关系选择列表 | 列表中不包含"本人"选项 | **列表中出现了"本人"选项** |

#### 2.4.3 影响范围

- **全端受影响**：H5、小程序、Flutter App
- 根因：前端从后端 `/api/relation-types` 接口拉回的列表包含"本人"，前端没有过滤掉

### Bug 5：点击"确认添加"保存家庭成员出错（全端）

#### 2.5.1 错误现象

在添加家庭成员弹窗中填写完信息后，点击"确认添加"按钮，保存失败。

#### 2.5.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 在添加家庭成员弹窗中选择关系 | 进入信息填写步骤 | 正常 |
| 2 | 填写姓名、性别、出生日期等信息 | 表单正常填写 | 正常 |
| 3 | 点击"确认添加"按钮 | 保存成功，弹窗关闭，成员列表更新 | **保存失败** |

#### 2.5.3 影响范围

- **全端受影响**：H5、小程序、Flutter App
- **根因已定位**：前端发送的请求体使用 `nickname` 字段，但后端 `family.py` 第 130 行校验的是 `data.name` 字段。前端没有传 `name` 字段，导致 `data.name` 为 `None`，后端返回 400 错误"姓名为必填项"

### Bug 7：数据库字段允许为空，页面层面要求必填（全端）

#### 2.7.1 错误现象

当前后端 API 对 `name`、`gender`、`birthday` 做了强制非空校验（返回 400 错误），但实际需求是：数据库表层面允许这些字段为空（方便老用户先建空记录后续补填），仅在前端页面层面做必填校验和友好提示。

#### 2.7.2 必填规则

| 场景 | 关系 | 姓名 | 性别 | 出生日期 |
|------|------|------|------|----------|
| "本人"档案编辑 | 固定为"本人"，不可修改 | 必填 | 必填 | 必填 |
| 新增其他家庭成员 | 必填 | 必填 | 必填 | 必填 |

- **数据库层面**：`nickname`、`gender`、`birthday` 字段允许为 NULL
- **前端层面**：提交时做必填校验，不满足则提示用户
- **后端层面**：去掉强制非空校验（接受空值），由前端保证数据完整性

#### 2.7.3 影响范围

- **全端受影响**：H5、小程序、Flutter App 前端校验 + 后端 API 修改

---

### Bug 8：存量老用户一次性脚本补"本人"记录

#### 2.8.1 错误现象

部分老用户（在本功能上线前注册的用户）没有"本人"这条家庭成员记录，导致进入健康档案页面后看不到"本人"选项。

#### 2.8.2 修复方案

| 项目 | 说明 |
|------|------|
| 一次性脚本 | 系统启动时执行一次全量脚本，为所有缺少"本人"记录的存量用户自动补一条 `relationship_type="本人"` 的家庭成员记录 |
| 记录内容 | 只创建关系为"本人"的空记录（姓名、性别、出生日期留空），用户后续自行补填 |
| 去掉兜底逻辑 | 删除 `GET /api/family/members` 接口中的自动创建兜底逻辑（查询接口不应做写入操作） |
| 新用户注册 | 保留注册流程中的 `ensure_self_family_member` 逻辑，新用户注册时正常创建"本人"记录 |

#### 2.8.3 影响范围

- 后端 API 修改 + 数据库一次性脚本
- 全端间接受益（确保所有用户都有"本人"记录）

|------|------|----------|----------|

#### 2.9.3 影响范围

## 4. 补充说明

### 4.1 修复优先级

| 优先级 | Bug 编号 | 说明 |
|--------|----------|------|
| P0（阻塞） | Bug 5、Bug 9 | 功能完全不可用（保存报错、弹窗空白），需最优先修复 |
| P1（严重） | Bug 1、Bug 2、Bug 8 | 严重影响用户体验或数据完整性 |

### 4.2 全端修复范围

- `/api/relation-types`：确保接口正常返回数据；考虑在返回数据中标记"本人"类型，方便前端过滤
