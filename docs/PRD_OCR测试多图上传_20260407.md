# OCR 识别测试 & 用户端体检报告 — 多图上传 产品需求文档（PRD）

## 1. 需求概述

### 1.1 背景与目的

当前 bini-health 系统存在两处多图上传需求：

**管理后台 OCR 测试**：管理后台的 **OCR 识别配置** 页面中，提供了「测试 OCR 识别」和「测试完整流程」两个测试功能，目前均**只支持上传一张图片**。实际使用中，体检报告往往有多页，需要一次性上传多张图片以验证多页文档的整体识别和 AI 智能合并能力。

**用户端体检报告上传**：H5 网页、Flutter App、微信小程序三端的体检报告上传页面，目前都只支持**一次上传一张图片（或一个 PDF）**。用户体检报告通常有多页，需要支持多张图片上传并合成一份完整的体检报告记录。

### 1.2 目标用户

| 模块 | 目标用户 |
|------|----------|
| 管理后台 OCR 测试 | 系统管理员（admin 角色） |
| 用户端体检报告上传 | 所有注册用户（H5/App/小程序） |

### 1.3 核心价值

- 管理员可一次性测试多页体检报告，验证 OCR + AI 智能合并的完整链路
- 用户可上传多页体检报告，获得完整的 AI 智能解读
- 多张图片合成一条报告记录，管理清晰
- 上传数量由管理后台全局配置统一控制，灵活可调

### 2.2 功能详细描述

#### 模块一：管理后台 OCR 测试多图上传

##### F01 / F04：多图逐张添加

**交互方式：**

- 弹窗内显示一个图片缩略图列表区域
- 每张已上传的图片显示为缩略图卡片，右上角有删除按钮（×），点击可移除该图片
- 列表末尾显示一个"+"添加按钮，点击触发文件选择器，每次选择一张图片
- 当已添加 5 张图片时，"+"按钮隐藏或禁用，并显示"已达上限"提示
- 图片按上传先后顺序排列，不支持拖拽排序

**图片限制：**

- 沿用当前的格式限制：`accept="image/*"`
- 沿用当前的大小限制（不变）

##### F02 / F05：串行识别

**处理方式：**

- 按图片在列表中的顺序（即上传先后顺序），逐张发送给后端 OCR 服务进行识别
- 每张识别完成后，将识别文本暂存，继续处理下一张
- 全部图片识别完成后，进入智能合并步骤

**单张兼容：**

- 当用户仅上传 1 张图片时，行为与当前完全一致，无需走智能合并

##### F03 / F06：智能合并结果

**合并逻辑：**

- 当上传 ≥ 2 张图片时，系统将所有图片的 OCR 识别文本汇总后，交给 AI 进行智能合并
- AI 默认将多张图片的内容视为**同一份报告的多页**
- AI 根据识别出的文字内容，自动按逻辑关系整合为一份完整的文本内容
- 合并后的结果一次性展示给用户

**结果展示：**

- 「测试 OCR 识别」弹窗：展示智能合并后的完整文本
- 「测试完整流程」弹窗：先展示合并后的 OCR 文本，再展示 AI 结构化解读结果

##### F05 特有逻辑：测试完整流程的 AI 解读

- 先串行 OCR 识别所有图片，获取全部文字
- 将全部文字合并后，一次性交给所选场景模板对应的 AI 进行解读
- 展示最终的 AI 结构化结果

##### F07：简单等待反馈

- 点击"开始识别"/"开始测试"按钮后：
  - 按钮变为 loading 状态
  - 弹窗内显示统一的加载动画（Spin）和文案"正在识别中..."
- 全部处理完成后，一次性展示最终结果

#### 模块二：用户端体检报告多图上传

##### F10：相册多选上传

**交互方式：**

- 用户点击"从相册选择"时，文件选择器/系统相册支持一次性多选图片
- 选中的图片立即出现在页面的缩略图列表中
- 已选图片数量达到上限时，系统提示"已达最大上传数量 N 张"
- 上限 N 由管理后台 OCR 全局设置中的 `max_batch_count` 配置控制（默认 5）

##### F11：拍照多次追加

**交互方式：**

- 用户可多次点击"拍照"按钮，每次拍摄一张照片，自动追加到缩略图列表中
- 缩略图列表中每张图片右上角显示删除按钮（×），可移除后重新选择
- PDF 上传仍为单文件模式，不支持多选

##### F12：COS 直传

**流程：**

1. 前端调用 `GET /api/cos/upload-token` 获取临时签名
2. 使用签名凭证将图片直接上传到腾讯云 COS
3. 获取每张图片的 COS URL
4. 全部图片上传完成后，将 URL 列表提交给 `upload-batch` 接口

**COS 不可用时（F17）：**

- 系统未配置 COS 时，直接报错提示"当前不支持多图上传"
- 单张上传仍可通过现有的 `/api/report/upload` 接口正常使用

##### F13：一站式识别接口 (`/api/report/upload-batch`)

**处理流程：**

1. 接口接收图片 URL 列表
2. 创建一条 `CheckupReport` 记录，`file_urls` 存储所有图片 URL，`file_url` 取第一张
3. 按顺序逐张调用 OCR 识别
4. 如果 ≥ 2 张图片，将全部 OCR 文本交给 AI 智能合并
5. 将合并后的文本交给 AI 进行结构化解读
6. 更新报告记录的 `ocr_result`、`ai_analysis_json` 等字段
7. 同步返回完整结果

##### F14：多图合成一条报告

- 多张图片合成一条 `CheckupReport` 记录
- 列表页缩略图显示第一张图片，与现有逻辑兼容
- 报告详情页可查看所有图片

##### F15：详情页缩略图网格

- 报告详情页顶部显示所有图片的缩略图网格
- 点击某张缩略图可放大查看原图
- 三端（H5、Flutter、小程序）统一交互

##### F16：上传限制动态获取

- 新增公开 API `/api/ocr/upload-limits`（无需 admin 权限）
- 返回 `max_batch_count` 和 `max_file_size_mb`
- 三端在页面加载时调用此接口获取限制值

### 3.3 用户端页面设计

#### 3.3.1 体检报告上传页（三端通用设计）

**改造前：**

- 三种上传方式按钮（相册/拍照/PDF），每次只能选择一张
- 选择后立即上传并跳转

**改造后：**

- 上传区域分为两部分：
  - **操作按钮区**：相册（支持多选）、拍照（多次追加）、PDF（仍为单文件）
  - **缩略图预览列表**：网格布局显示已选图片，每张右上角有删除按钮，底部显示"已选 M/N 张"
- 选完图片后，页面底部显示"开始解读"按钮
- 点击"开始解读"后：
  1. 前端逐张通过 COS 直传上传图片，获取 URL 列表
  2. 调用 `/api/report/upload-batch` 接口，传入 URL 列表
  3. 显示全屏 loading 动画 + "正在识别中..."
  4. 接口返回后跳转到报告详情/结果页

#### 3.3.2 报告详情页（改造后新增部分）

- 单张图片的报告与现有展示方式完全一致，不显示缩略图网格

## 4. 数据模型设计

### 4.1 数据实体与字段

#### 4.1.1 CheckupReport 表（修改）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| file_urls | JSON | nullable | **新增**。图片 URL 列表（JSON 数组），存储所有上传图片的 COS URL |
| file_url | String | nullable | 保留不变。单图时存唯一图片 URL，多图时存第一张图片 URL（兼容现有列表页缩略图、分享等逻辑） |
| thumbnail_url | String | nullable | 保留不变。取第一张图片的缩略图 URL |

**示例 `file_urls` 值：**

```json
[
  "https://xxx.cos.ap-guangzhou.myqcloud.com/reports/img1.jpg",
  "https://xxx.cos.ap-guangzhou.myqcloud.com/reports/img2.jpg",
  "https://xxx.cos.ap-guangzhou.myqcloud.com/reports/img3.jpg"
]
```

#### 4.1.2 OcrUploadConfig 表（已有，无修改）

| 字段 | 说明 |
|------|------|
| max_batch_count | 单次最多上传图片数量（默认 5） |
| max_file_size_mb | 单张图片最大文件大小（MB） |

### 4.2 实体关系

- `CheckupReport.file_urls` 为 JSON 数组字段，不需要额外关联表
- `CheckupReport.file_url` 始终等于 `file_urls[0]`（兼容现有逻辑）

## 5. 接口需求

### 5.1 接口清单

| 编号 | 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|------|
| **管理后台 OCR 测试** |
| A01 | POST | /api/admin/ocr/test-ocr | admin | 测试 OCR 识别（支持多图） |
| A02 | POST | /api/admin/ocr/test-full | admin | 测试完整流程（支持多图） |
| **用户端体检报告** |
| U01 | GET | /api/cos/upload-token | 登录用户 | **新增**。获取 COS 临时上传签名 |
| U02 | GET | /api/ocr/upload-limits | 公开 | **新增**。获取上传限制配置 |
| U03 | POST | /api/report/upload-batch | 登录用户 | **新增**。多图一站式上传+识别+解读 |
| U04 | GET | /api/report/detail/{id} | 登录用户 | 改造。返回 `file_urls` 字段 |

### 5.2 接口详细说明

#### A01: POST /api/admin/ocr/test-ocr

**请求：**

- Content-Type: `multipart/form-data`
- `files`: 1~5 个图片文件（保持向后兼容，也支持单个 `file` 字段）
- `provider`: 厂商名称（baidu / tencent / aliyun）

**处理逻辑：**

1. 逐张校验图片质量
2. 按顺序逐张调用 OCR 识别
3. 如果只有 1 张图片，直接返回识别结果（与当前行为一致）
4. 如果有 ≥ 2 张图片，将全部识别文本汇总后交给 AI 进行智能合并
5. 返回合并后的完整文本

**响应：**

```json
{
  "success": true,
  "provider_name": "baidu",
  "ocr_text": "合并后的完整文本...",
  "image_count": 3,
  "is_merged": true,
  "error": null
}
```

#### A02: POST /api/admin/ocr/test-full

**请求：**

- Content-Type: `multipart/form-data`
- `files`: 1~5 个图片文件
- `provider`: 厂商名称
- `scene_id`: 场景模板 ID

**处理逻辑：**

1. 逐张校验图片质量
2. 按顺序逐张调用 OCR 识别
3. 如果 ≥ 2 张图片，先将 OCR 文本交给 AI 智能合并
4. 将合并后的完整文本（或单张文本）交给所选场景模板的 AI 进行解读
5. 返回 OCR 文本和 AI 结构化结果

**响应：**

```json
{
  "success": true,
  "provider_name": "baidu",
  "ocr_text": "合并后的 OCR 文本...",
  "ai_result": { "...AI 解读结构化结果..." },
  "image_count": 3,
  "is_merged": true,
  "error": null
}
```

#### U01: GET /api/cos/upload-token（新增）

**权限**：登录用户

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| file_name | string | 否 | 文件名（用于生成 COS Key） |
| file_type | string | 否 | 文件 MIME 类型 |

**响应：**

```json
{
  "success": true,
  "cos_url": "https://xxx.cos.ap-guangzhou.myqcloud.com",
  "cos_key": "reports/2026/04/07/abc123.jpg",
  "authorization": "q-sign-algorithm=sha1&q-ak=xxx&...",
  "token": "xxx",
  "expires_at": "2026-04-07T12:30:00Z"
}
```

**说明：**

- 每次调用生成一个临时上传凭证（单次使用，有效期如 10 分钟）
- 前端使用 `authorization` + `token` 直接 PUT 到 COS
- 上传后图片 URL 为 `{cos_url}/{cos_key}`

#### U02: GET /api/ocr/upload-limits（新增）

**权限**：公开（无需登录）

**响应：**

```json
{
  "max_batch_count": 5,
  "max_file_size_mb": 10
}
```

**说明：**

- 数据来源为管理后台 OCR 全局设置中的 `OcrUploadConfig` 表
- 三端在页面加载时调用此接口获取限制值

#### U03: POST /api/report/upload-batch（新增）

**权限**：登录用户

**请求：**

```json
{
  "file_urls": [
    "https://xxx.cos.ap-guangzhou.myqcloud.com/reports/img1.jpg",
    "https://xxx.cos.ap-guangzhou.myqcloud.com/reports/img2.jpg",
    "https://xxx.cos.ap-guangzhou.myqcloud.com/reports/img3.jpg"
  ]
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| file_urls | string[] | 是 | 图片 URL 列表，长度 1~N（N 由 `max_batch_count` 决定） |

**处理逻辑：**

1. 校验 `file_urls` 数量不超过 `max_batch_count`
2. 创建 `CheckupReport` 记录：`file_urls` 存整个列表，`file_url` 取 `file_urls[0]`，`thumbnail_url` 取第一张
3. 按顺序逐张调用 OCR 识别（串行处理）
4. 如果 ≥ 2 张图片，将全部 OCR 文本交给 AI 智能合并（视为同一份报告的多页）
5. 将合并/单张文本交给 AI 进行结构化解读
6. 更新报告记录的 `ocr_result`、`ai_analysis_json`、`status` 等字段
7. 同步返回结果

**响应：**

```json
{
  "success": true,
  "report_id": 123,
  "ocr_text": "智能合并后的 OCR 文本...",
  "ai_result": { "...AI 结构化解读结果..." },
  "image_count": 3,
  "is_merged": true,
  "error": null
}
```

**超时说明：**

- 串行 OCR（每张 2~5 秒）+ AI 合并（3~10 秒）+ AI 解读（5~15 秒），总计可能 30~60 秒
- 接口超时建议设置为 120 秒
- 前端显示 loading 等待

#### U04: GET /api/report/detail/{id}（改造）

**响应新增字段：**

```json
{
  "id": 123,
  "file_url": "https://xxx.cos.xxx/reports/img1.jpg",
  "file_urls": [
    "https://xxx.cos.xxx/reports/img1.jpg",
    "https://xxx.cos.xxx/reports/img2.jpg",
    "https://xxx.cos.xxx/reports/img3.jpg"
  ],
  "thumbnail_url": "https://xxx.cos.xxx/reports/img1.jpg",
  "...其他现有字段..."
}
```

**说明：**

- 新增 `file_urls` 字段（JSON 数组），返回所有图片的 URL
- `file_url` 保持不变（第一张图片 URL），前端列表页兼容无需改动
- 当 `file_urls` 为 null 或长度 ≤ 1 时，为单图报告，前端按现有方式展示

## 6. 非功能性需求

### 6.1 性能要求

| 场景 | 预估耗时 |
|------|----------|
| 管理后台：5 张图片 OCR + 合并 | 10~35 秒 |
| 用户端：COS 直传 5 张 | 3~10 秒（取决于网络） |
| 用户端：upload-batch 处理 | 30~60 秒 |
| 后端接口超时 | 建议 120 秒 |

### 6.2 安全要求

- 管理后台测试接口：仅 admin 角色可访问
- 用户端上传接口：需登录鉴权
- COS 临时签名：单次使用、短有效期（10 分钟）
- COS 上传凭证限定上传目录（如 `reports/`）
- 上传数量和大小由后端校验，防止前端绕过

### 6.3 兼容性要求

- 管理后台：Chrome、Edge、Firefox 最新版本
- H5 网页：主流移动端浏览器（Chrome、Safari、微信内置浏览器）
- Flutter App：iOS 12+、Android 6+
- 微信小程序：基础库 2.21+
- 管理后台接口保持向后兼容（支持 `file` 单文件旧调用方式）
- 用户端现有的 `/api/report/upload` 单图接口保持不变

## 7. 业务规则与约束

| 编号 | 规则 | 说明 |
|------|------|------|
| R01 | 上传数量上限 | 由管理后台 `OcrUploadConfig.max_batch_count` 控制（默认 5） |
| R02 | 至少 1 张 | 未选择图片时按钮禁用 |
| R03 | 管理后台逐张添加 | 管理后台测试弹窗每次文件选择器只选 1 张 |
| R04 | 用户端灵活上传 | 用户端相册可多选、拍照可追加 |
| R05 | PDF 仍为单文件 | 用户端 PDF 上传不支持多选 |
| R06 | 按上传顺序 | 不支持拖拽排序，顺序固定 |
| R07 | 智能合并仅多图触发 | 仅上传 1 张时行为与现有完全一致 |
| R08 | 图片关系默认同一报告 | AI 默认将所有图片视为同一份体检报告的多页 |
| R09 | 格式与大小限制 | 管理后台沿用当前 `accept="image/*"`；用户端由 `max_file_size_mb` 控制 |
| R10 | COS 必须可用 | 用户端多图上传依赖 COS 直传，COS 不可用时报错 |
| R11 | 多图合成一条记录 | 用户端多张图片合成一条 `CheckupReport` 记录 |
| R12 | 列表页兼容 | 列表页缩略图显示第一张，无需改动 |

## 9. 异常处理与边界情况

### 9.1 管理后台 OCR 测试

| 场景 | 处理方式 |
|------|----------|
| 某张图片质量不合格 | 跳过该图，继续处理其他图片，结果中标注哪些图片被跳过 |
| 某张图片 OCR 识别失败 | 跳过该图，继续处理其他图片，结果中标注失败信息 |
| 所有图片均识别失败 | 返回 `success: false`，展示错误信息 |
| AI 智能合并失败 | 降级为简单拼接（按顺序换行连接各图文本），并提示"AI 合并失败，已使用简单拼接" |
| 测试完整流程中 AI 解读失败 | 仍展示 OCR 合并文本，AI 结果部分显示错误信息 |
| 厂商配置不存在 | 返回 `success: false`，提示"厂商配置不存在" |
| 接口超时 | 前端显示"请求超时，请减少图片数量重试" |

### 9.2 用户端体检报告上传

| 场景 | 处理方式 |
|------|----------|
| COS 未配置 | 多图上传按钮不可用，提示"当前不支持多图上传"；单图上传仍可通过原有接口正常使用 |
| COS 直传某张失败 | 前端提示"第 X 张图片上传失败"，允许重试或删除该图 |
| 全部 COS 直传失败 | 提示"图片上传失败，请检查网络后重试" |
| upload-batch 部分 OCR 失败 | 跳过失败图片，用成功图片的文本继续处理，结果中标注 |
| upload-batch 全部 OCR 失败 | 返回 `success: false`，报告状态标记为"识别失败" |
| AI 智能合并失败 | 降级为简单拼接，继续进行 AI 解读 |
| AI 解读失败 | 报告记录保留 OCR 文本，状态标记为"解读失败"，允许用户在详情页重试 |
| 上传数量超过限制 | 后端返回错误"上传图片数量超过限制（最多 N 张）" |
| 单张图片超过大小限制 | 前端校验拦截，提示"图片大小超过 Xmb 限制" |
| 接口超时（>120秒） | 前端显示"处理超时，请减少图片数量重试" |
| COS 签名过期 | 重新调用 `/api/cos/upload-token` 获取新签名后重试 |

## 10. 补充说明

### 10.1 AI 智能合并的提示词设计

当上传多张图片时，系统将所有 OCR 文本以如下格式拼接后发送给 AI：

```
以下是一份文档的多页扫描件的 OCR 识别文字，请将这些内容智能整合为一份完整、连贯的文本。
注意：这些内容来自同一份文档的不同页面，请按逻辑顺序整合，去除重复内容，保持信息完整。

--- 第 1 页 ---
{第1张图片的OCR文本}

--- 第 2 页 ---
{第2张图片的OCR文本}

... (以此类推)
```

### 10.2 管理后台前端实现建议

- 使用 Ant Design 的 `Upload` 组件，设置 `listType="picture-card"`，`maxCount={5}`
- 每张图片通过 `beforeUpload` 拦截本地预览，不自动上传
- 点击"开始识别"时，将所有图片打包为 `FormData`，通过 `files` 字段发送
- 多图时 loading 状态使用 `Spin` 组件包裹结果区域

### 10.3 用户端前端实现建议

**H5 网页：**

- 使用 Ant Design Mobile 的 `ImageUploader` 组件或自定义组件
- 相册多选通过 `<input type="file" accept="image/*" multiple>` 实现
- COS 直传使用 `fetch` + `PUT` 方法

**Flutter App：**

- 使用 `image_picker` 库的 `pickMultiImage()` 方法实现相册多选
- 拍照使用 `pickImage(source: ImageSource.camera)` 逐次追加
- COS 直传使用 `http` 或 `dio` 库 PUT 上传

**微信小程序：**

- 使用 `wx.chooseMedia({ count: N, mediaType: ['image'] })` 实现多选和拍照
- COS 直传使用 `wx.request` 的 PUT 方法
- 注意小程序并发请求限制（最多 10 个），需串行或分批上传

### 10.4 向后兼容

- 管理后台接口同时支持 `file`（单文件旧格式）和 `files`（多文件新格式）
- 用户端现有的 `/api/report/upload` 单图接口保持不变，不受影响
- `CheckupReport` 表新增 `file_urls` 字段为 nullable，现有单图记录该字段为 null
- 前端详情页需兼容 `file_urls` 为 null 的旧数据，按单图模式展示

### 10.5 数据库迁移

- `CheckupReport` 表新增 `file_urls` 列（JSON 类型，nullable）
- 通过 `schema_sync.py` 在应用启动时自动迁移，无需手动执行 SQL
- 现有数据不受影响（`file_urls` 默认为 null）
