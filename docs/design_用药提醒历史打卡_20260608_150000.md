## 二、DDL 设计（增量）

### 2.1 变更分析

当前 `medication_check_ins` 表通过比较 `check_in_date` 与 `check_in_time.date()` 可隐式区分正常打卡和补打卡，但存在边界问题（跨天打卡）。为确保数据清晰和查询高效，新增显式类型字段。

### 2.2 增量 DDL

```sql
-- [PRD-MEDICATION-HISTORY-V1 2026-06-08] 用药历史打卡：打卡类型字段
ALTER TABLE medication_check_ins
  ADD COLUMN check_in_type VARCHAR(16) NOT NULL DEFAULT 'normal'
  COMMENT '打卡类型：normal(正常打卡当天服用) / supplement(补打历史遗漏)';

-- 索引优化：日历查询需要按 user_id + check_in_date 联合查询
CREATE INDEX idx_checkins_user_date ON medication_check_ins (user_id, check_in_date);
```

### 2.3 数据回填语句

```sql
-- 将历史数据的 check_in_type 标记为 normal（默认值，无需回填）
-- 若存在 check_in_date < DATE(check_in_time) 的记录，标记为 supplement
UPDATE medication_check_ins
SET check_in_type = 'supplement'
WHERE check_in_date < DATE(check_in_time)
  AND check_in_type = 'normal';
```

### 2.4 后端模型同步

在 `backend/app/models/models.py` 的 `MedicationCheckIn` 类中新增字段：

```python
class MedicationCheckIn(Base):
    __tablename__ = "medication_check_ins"
    # ... 现有字段 ...
    check_in_type = mapped_column(
        String(16), nullable=False, default="normal",
        comment="打卡类型：normal(正常打卡) / supplement(补打卡)"
    )
```

> ⚠️ 注意：`created_at` 已作为默认字段存在，`check_in_type` 插入在上方，无需改动关系映射。

## 三、OpenAPI 3.0 接口设计

### 3.1 获取某月日历打卡状态

```
GET /api/medication/calendar?year=2026&month=6
```

**请求参数**：
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| year | int | 是 | 年份 |
| month | int | 是 | 月份（1-12） |

**响应**：
```json
{
  "year": 2026,
  "month": 6,
  "days": [
    {
      "date": "2026-06-07",
      "status": "partial",
      "total_count": 3,
      "done_count": 2
    },
    {
      "date": "2026-06-08",
      "status": "fully_done",
      "total_count": 3,
      "done_count": 3
    }
  ]
}
```

**status 枚举**：
| 值 | 含义 | 日历圆点 |
|----|------|----------|
| fully_done | 该天全部打卡完成 | 🟢 绿色 |
| partial | 该天部分打卡 | 🟠 橙色 |
| missed | 该天全部漏打 | 🔴 红色 |
| no_plan | 该天无用药计划 | 无圆点 |

**口径说明**：只统计 status=active、且该天在有效期内（start_date ≤ date ≤ end_date 或 long_term=true）的 reminder。每个 reminder 的 schedule 时间点数量为该天应打卡次数。

### 3.2 获取某天打卡详情

```
GET /api/medication/records?date=2026-06-07
```

**请求参数**：
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| date | string | 是 | 日期 YYYY-MM-DD |

**响应**：
```json
{
  "date": "2026-06-07",
  "date_label": "6月7日 周日",
  "summary": "partial",
  "records": [
    {
      "reminder_id": 1,
      "medicine_name": "阿司匹林",
      "dosage": "1 片",
      "dosage_value": "1",
      "dosage_unit": "片",
      "guidance": "餐后",
      "scheduled_time": "08:00",
      "check_in_id": 101,
      "check_in_type": "normal",
      "check_in_time": "2026-06-07 08:05:00",
      "status": "done",
      "is_today": false,
      "can_supplement": false
    },
    {
      "reminder_id": 1,
      "medicine_name": "阿司匹林",
      "dosage": "1 片",
      "dosage_value": "1",
      "dosage_unit": "片",
      "guidance": "餐后",
      "scheduled_time": "12:00",
      "check_in_id": null,
      "check_in_type": null,
      "check_in_time": null,
      "status": "missed",
      "is_today": false,
      "can_supplement": true
    }
  ]
}
```

**status 枚举**：
| 值 | 含义 |
|----|------|
| done | 正常打卡完成 |
| supplement | 已补打卡 |
| missed | 未打卡（可补，3天内） |
| expired | 未打卡（超3天不可补） |
| not_yet | 当天未来时间点，暂未到打卡时间 |

**can_supplement 规则**：
- 仅 `status = "missed"` 且 `date` 在 [today-3天, today-1天] 范围内为 true
- 若 `date == today`：can_supplement=false，前端提示「请在主页打卡」
- 若 `date < today-3天`：status="expired"，can_supplement=false

**is_today**：`date == today` 时为 true，前端据此展示提示「请在主页打卡」

### 3.3 补打卡

```
POST /api/medication/supplement
```

**请求体**：
```json
{
  "plan_id": 1,
  "check_in_date": "2026-06-06",
  "scheduled_time": "08:00"
}
```

**响应**：
```json
{
  "id": 102,
  "plan_id": 1,
  "check_in_date": "2026-06-06",
  "check_in_time": "2026-06-08 10:30:00",
  "check_in_type": "supplement",
  "message": "补打卡成功"
}
```

**业务校验**：
1. 校验 `plan_id` 属于当前用户
2. 校验 `check_in_date` 对应的 reminder 在该日期是否有效（start_date ≤ check_in_date ≤ end_date 或 long_term）
3. 校验 `scheduled_time` 是否在 reminder 的 schedule 列表内
4. 校验补打卡时限：`(today - check_in_date).days ≤ 3`
5. 校验 `check_in_date` 不能是今天（今天的在主页操作）
6. 幂等：同 reminder_id + check_in_date + scheduled_time 若已存在打卡记录，返回 409
7. 写入 `check_in_type = "supplement"`，`check_in_time = datetime.now()`

**错误码**：
| HTTP Status | detail | 说明 |
|-------------|--------|------|
| 400 | EXCEED_SUPPLEMENT_LIMIT | 超过 3 天补打卡时限 |
| 400 | TODAY_NOT_ALLOWED | 今天不能在历史页补打卡 |
| 400 | PLAN_NOT_ACTIVE | 用药计划已归档或不在有效期内 |
| 404 | PLAN_NOT_FOUND | 用药计划不存在 |
| 409 | ALREADY_CHECKED_IN | 该时间点已打卡（幂等拦截） |

## 四、业务流程说明

### 4.1 日历状态计算流程

```
用户进入页面
    │
    ▼
GET /api/medication/calendar?year=2026&month=6
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 1. 查询该月所有 status=active 的提醒             │
│    SELECT * FROM medication_reminders            │
│    WHERE user_id = {uid} AND status = 'active'   │
│      AND (long_term = 1 OR end_date >= month_start) │
│      AND start_date <= month_end                  │
│                                                  │
│ 2. 查询该月所有打卡记录                           │
│    SELECT * FROM medication_check_ins             │
│    WHERE user_id = {uid}                          │
│      AND check_in_date >= month_start             │
│      AND check_in_date <= month_end               │
│                                                  │
│ 3. 对每天 (day) 计算：                            │
│    ├── 展开该天有效的提醒时间点                    │
│    │   for each reminder:                          │
│    │     if start_date <= day <= end_date:          │
│    │        expected += len(schedule_of(r))          │
│    │                                               │
│    ├── 统计该天已打卡数                             │
│    │   done = count(check_ins where check_in_date == day) │
│    │                                               │
│    └── 判定状态                                    │
│        if expected == 0: status = "no_plan"        │
│        elif done == expected: status = "fully_done" │
│        elif done > 0: status = "partial"            │
│        else: status = "missed"                      │
└─────────────────────────────────────────────────┘
```

**性能优化**：
- 当月打卡记录一次查询，内存按日期分组
- 提醒计划使用 Python 内存计算，避免 N+1 查询
- 日历最多 31 天 × 用户最多几十条计划，内存开销可忽略

### 4.4 数据口径说明

**「某天有用药计划」口径**：
- 条件：reminder.status = 'active'
- 日期范围：start_date ≤ date ≤ end_date，或 long_term=true 且 start_date ≤ date
- 不区分今天/过去/未来

**「正常打卡 vs 补打卡」判定**：
- `check_in_type = 'normal'`：当天服务当天（check_in_date = 打卡当天日期）
- `check_in_type = 'supplement'`：补打历史遗漏（check_in_date < 打卡当天日期）
- 历史数据（迁移前）：`check_in_type = 'normal'`（默认值），不影响历史统计

**「可补打卡时限」判定**：
- 公式：`datetime.now().date() - check_in_date ≤ 2`（即仅限昨天、前天、大前天）
- 等效：最近 3 天，含昨天
- 今天不可在历史页补打卡（在主页操作）

**与今日主页的边界**：
- 今天（date=today）的打卡在主页 `/ai-home/medication-reminder` 操作
- 历史页选中今天时，未打卡记录不显示补打卡按钮，提示「请在主页打卡」
- 补打卡成功后 history 页数据刷新、calendar 圆点更新，但不会自动跳回主页

#### medication_history_v1.py 设计要点

```python
新增端点：
| 路径                                | 方法 | 用途                   |
|-------------------------------------|------|------------------------|
| /api/medication/calendar            | GET  | 某月日历打卡状态        |
| /api/medication/records             | GET  | 某天打卡记录详情        |
| /api/medication/supplement          | POST | 补打卡                  |

```

**关键实现逻辑**：

1. **calendar 端点**：遍历月内每天，对每天调用内部函数计算 expected/done，返回 status
2. **records 端点**：查询该日活跃提醒，匹配打卡记录，返回带 can_supplement 的详细列表
3. **supplement 端点**：校验时限 + 写库，`check_in_type='supplement'`

**临界情况处理**：
- 跨月查询 → 只返回请求月份的日期
- 同一 reminder 多个时间点 → 按 schedule 顺序匹配打卡记录
- 补打卡后同一时间点再次请求 → 返回 409（幂等保护）

## 六、关键交互细节

### 6.1 日历默认选中
- **默认选中日期：昨天**（`today - 1 day`）
- 原因：用户进历史页 90% 是为了补昨天的漏打卡
- 若昨天不在当前显示月份内，自动切换到上月

### 6.2 月份切换保持选中
```typescript
function changeMonth(delta: number) {
  const newMonth = currentMonth + delta;
  // 调整 year/month
  const newDate = new Date(currentYear, newMonth - 1, 1);
  setCurrentYear(newDate.getFullYear());
  setCurrentMonth(newDate.getMonth() + 1);
  
  // 检查选中日期是否在新月份内
  const selectedDay = parseInt(selectedDate.split('-')[2]);
  const daysInNewMonth = new Date(currentYear, newMonth, 0).getDate();
  if (selectedDay > daysInNewMonth) {
    // 选中日期不在新月份 → 选最后一天
    setSelectedDate(formatDate(new Date(currentYear, newMonth - 1, daysInNewMonth)));
  }
  // 请求新月度 calendar 数据
}
```
