## 1. Bug 发生背景

### 1.1 项目概述

bini-health 健康管理平台，基于 FastAPI + Next.js + MySQL 技术栈，支持 H5、管理后台、小程序、Flutter App 多端。

### 1.2 涉及功能模块

- **用户注册认证**（`backend/app/api/auth.py`）：手机号+密码注册（`POST /api/auth/register`）、短信验证码登录自动注册（`POST /api/auth/sms-login`）
- **家庭成员管理**（`backend/app/api/family.py`）：家庭成员 CRUD、"本人"成员自动创建
- **健康档案管理**（`backend/app/api/health_profile.py`）：健康档案 CRUD、引导状态
- **H5 健康引导页**（`h5-web/src/app/health-guide/page.tsx`）：新用户注册后的 4 步健康档案引导流程
- **H5 健康档案页**（`h5-web/src/app/health-profile/page.tsx`）：成员切换、档案编辑、添加成员弹窗
- **H5 健康自查页**（`h5-web/src/app/symptom/page.tsx`）：症状自查流程、选择咨询人弹窗、新建家庭成员表单
- **H5 聊天页**（`h5-web/src/app/chat/[sessionId]/page.tsx`）：AI 健康咨询对话、切换咨询对象弹窗、新建家庭成员入口
- **H5 家庭成员页**（`h5-web/src/app/family/page.tsx`）：家庭成员列表、添加成员弹窗（未完成的旧版页面，使用 mock 数据，**决定废弃**）

### 1.3 发现时间与发现方式

- 发现时间：2026 年 4 月 9 日
- 发现方式：
  - Bug 1：注册后进入健康档案页面（`/health-profile`），成员切换栏里没有"本人"

  - Bug 6：健康自查页和健康档案页新建成员时，成员关系选择使用下拉框，不够直观友好
  - Bug 7：4 个页面的"新建家庭成员"表单内容全部不对，交互方式、字段内容、数据来源全面不统一
  - Bug 8：聊天页"切换咨询对象"弹窗中"为自己"是硬编码的，与后端 `is_self` 成员重复（与 Bug 3 同类问题）；"新建家庭成员"按钮跳转到不存在的页面 `/family/add`（404）

### 2.2 各 Bug 详细描述

**Bug 1：新用户注册后未自动创建"本人"家庭成员**

注册成功后进入健康档案页面（`/health-profile`），顶部横向滚动成员切换栏中没有"本人"成员，只有一个"+"添加按钮。手机号+密码注册和短信验证码登录自动注册两种方式均存在此问题。

**Bug 2：引导流程第一步保存失败**

新用户注册成功后自动跳转到引导页 `/health-guide`，在第一步（基本信息）填写完毕后点击"下一步"，页面 Toast 弹出"保存失败"，无法继续到第二步。

**Bug 3：健康自查"选择咨询人"弹窗中出现重复的"本人"记录**

健康自查页面最后一步弹出"为谁咨询"弹窗时，列表中出现两条重复记录：
- 第一条：硬编码的"为自己"（`symptom/page.tsx` 第 502-523 行）
- 第二条：后端 `GET /api/family/members` 返回的 `is_self=true` 成员，显示为"本人·本人"

两条记录含义重叠，用户困惑。

**Bug 4：健康自查页"新建家庭成员"表单字段顺序和字段名不一致**

当前健康自查页（`symptom/page.tsx` 第 572-715 行）新建成员表单的字段顺序为：
`关系类型 → 昵称 → 出生日期 → 性别 → 身高(cm) → 体重(kg) → 既往病史 → 过敏史`

存在以下问题：
- "昵称"应统一为"姓名"（与健康档案页一致）
- 性别和出生日期的顺序颠倒了（期望：性别在前，出生日期在后）
- 4 个必填字段应排在最前面

**Bug 5：必填字段缺少红色 `*` 标记**

成员关系、姓名、性别、出生日期作为必填字段，在以下页面的标签后面缺少红色 `*` 提示：
1. H5 健康档案页（`/health-profile`）编辑/添加成员弹窗
2. 引导流程第一步（`/health-guide` Step1）
3. 健康自查页"选择咨询人"中新建成员表单（`/symptom`）
4. 聊天页"新建家庭成员"弹窗（`/chat/[sessionId]`）

**Bug 6：成员关系选择使用下拉框，交互不够直观**

健康自查页（`/symptom`）新建家庭成员时，成员关系采用下拉框（Picker）选择方式，用户需要点击展开 → 滚动找到选项 → 确认，操作成本较高。期望改为平铺按钮组布局，方便用户快速选择。

**Bug 7：4 个页面的"新建家庭成员"表单内容全部不对**

4 个入口的"新建家庭成员"在**交互方式、字段数量、字段名称、字段顺序、数据来源、视觉风格**上全面不统一：

| 页面 | 交互方式 | 字段内容 | 数据来源 | 核心问题 |
|------|----------|----------|----------|----------|
| 健康档案页（`/health-profile`） | 两步式 Popup：先选关系（平铺按钮组） → 展开绿色卡片 | 只有 3 个字段：姓名/性别/出生日期 | `GET /api/relation-types` 动态获取 ✅ | **缺少身高/体重/既往病史/过敏史 4 个选填字段** |
| 健康自查页（`/symptom`） | 一步式内嵌展开 | 8 个字段：关系类型(下拉框)/昵称/出生日期/性别/身高/体重/既往病史/过敏史 | 硬编码的 `relationshipOptions` ❌ | **不是两步式、下拉框非平铺、"昵称"应为"姓名"、字段顺序不对** |
| 聊天页（`/chat/[sessionId]`） | 点击后跳转页面 | — | — | **跳转到 `/family/add`（该路由不存在，404 白屏）** |
| 家庭成员页（`/family`） | Dialog 弹窗 | 只有 3 个文本输入框：姓名/关系手动输入/年龄 | `mockMembers` 硬编码假数据 ❌ | **未完成的旧版页面，完全未对接后端 API，决定废弃** |

**Bug 8：聊天页"新建家庭成员"跳转到不存在的页面 + 咨询对象列表重复"本人"**

聊天页（`/chat/[sessionId]`）底部"切换咨询对象"弹窗存在两个问题：
1. **"新建家庭成员"按钮点击后跳转到 `/family/add`，但该页面路由不存在（404 白屏）**
2. **弹窗中硬编码了"为自己"选项**，同时遍历了 `GET /api/family/members` 返回的所有成员，当后端返回 `is_self=true` 的"本人"成员时，"为自己"和"本人"重复出现

**Bug 9：家庭成员页（`/family`）使用 mock 假数据，完全未对接后端**

家庭成员页（`/family/page.tsx`）是一个**未完成的旧版页面**：
- 使用硬编码的 `mockMembers` 假数据（张大爷/李阿姨/小明），完全没有调用后端 API
- 添加成员弹窗（`Dialog`）只有 3 个简陋的**文本输入框**：姓名、关系（手动文本输入，非选择）、年龄
- 没有性别、出生日期、身高、体重、既往病史、过敏史等字段
- **决定直接废弃此页面**，所有"新建成员"入口统一到健康档案页的弹窗形式

## 3. 统一规格标准

### 3.1 "新建家庭成员"入口统一规格

**经确认，所有入口的"新建家庭成员"表单必须统一为以下规格：**

| 维度 | 统一标准 |
|------|----------|
| **交互方式** | 两步式 — 先选关系（平铺按钮组），选中后展开绿色卡片填写信息 |
| **弹窗形式** | 底部弹窗 Popup（以健康档案页为标准：圆角卡片式、绿色渐变背景） |
| **必填字段** | 成员关系 → 姓名 → 性别 → 出生日期（4 个，标签后加红色 `*`） |
| **选填字段** | 身高 → 体重 → 既往病史 → 过敏史（4 个，第二步展开后一屏呈现全部 8 个字段） |
| **字段名称** | 统一使用"姓名"（非"昵称"）、"成员关系"（非"关系类型"） |
| **关系类型选择** | 平铺按钮组布局，每行 4 个，圆角矩形带边框 |
| **关系类型数据来源** | `GET /api/relation-types` 动态获取管理后台配置（`is_active=True`，按 `sort_order` 排序，无"本人"） |
| **关系类型按钮样式** | 未选中：浅灰背景 `#f9f9f9` + 透明边框；选中：浅绿渐变背景 + 绿色边框 `#52c41a` + 绿色文字 |
| **按钮形状** | 圆角矩形（`rounded-xl`），带明确边框，有"按钮组/ButtonGroup"感 |
| **绿色卡片标题** | "填写{关系名}信息"（如"填写父亲信息"） |
| **既往病史/过敏史** | 标签多选（Tag）+ 手动输入"其他"，所有页面一致 |
| **必填校验** | 4 个必填字段必须全部填完才能提交，不满足时 Toast 提示"请填写完整的成员信息" |
| **保存 API** | `POST /api/family/members` |

## 5. Bug 原因分析与定位

### 5.1 根因分析

**Bug 1 & Bug 2 核心问题：`ensure_self_family_member()` 创建的 FamilyMember 记录缺少 `relation_type_id` 字段**

查看 `backend/app/api/auth.py` 第 58-65 行：

```python
db.add(FamilyMember(
    user_id=user_id,
    relationship_type="本人",
    nickname="本人",
    is_self=True,
    status="active",
))
await db.flush()
```

这段代码创建 FamilyMember 时**没有设置 `relation_type_id`**。

**根因链路：**

1. `ensure_self_family_member()` 在 `db.flush()` 阶段可能因 `is_self` 列不存在或约束冲突而失败 — 该函数**没有被 try/except 包裹**，若抛异常整个注册事务回滚
2. 即使成功执行，`ensure_self_health_profile()` 创建的是 `family_member_id=None` 的"主用户档案"，而引导页使用 `PUT /api/health/profile/member/{memberId}` 需要 `family_member_id=member.id` 的档案
3. 引导页初始化时在 `GET /api/family/members` 返回列表中找不到 `is_self=true` 的成员，`memberId` 设为 `null`
4. `saveCurrentStep()` 因 `if (!memberId) return` 直接跳过保存，导致"保存失败"

**Bug 3 根因：** `symptom/page.tsx` 硬编码了"为自己"选项，同时遍历了后端返回的所有成员（含 `is_self=True` 的"本人"），导致重复

**Bug 4 根因：** 健康自查页新建成员表单独立编写，未与健康档案页保持一致

**Bug 5 根因：** 前端标签未添加红色 `*` 必填标记

**Bug 7 根因：** 4 个页面独立开发，缺乏统一规范和公共组件

**Bug 8 根因：** 聊天页硬编码"为自己"（与 Bug 3 同源）+ `router.push('/family/add')` 对应的页面路由从未创建过

**Bug 9 根因：** `/family` 页面是早期开发的原型页面，未完成也未清理

### 5.2 相关代码定位

| 文件 | 关键位置 | 说明 |
|------|----------|------|
| `backend/app/api/auth.py` | 第 49-65 行 `ensure_self_family_member()` | "本人"成员创建逻辑，缺少 `relation_type_id` |
| `backend/app/api/auth.py` | 第 68-78 行 `ensure_self_health_profile()` | 创建的是 `family_member_id=None` 的主用户档案 |
| `backend/app/api/auth.py` | 第 183-189 行（register）和第 304-309 行（sms_login） | 两个注册入口都调用了上述函数 |

### 7.2 修复详细说明

#### FIX-01：补充 `relation_type_id`

在 `ensure_self_family_member()` 中，先查询 `relation_types` 表中 `name="本人"` 的记录，获取其 `id`，赋值到 FamilyMember 的 `relation_type_id` 字段。如果查不到（关系类型未初始化），则仍以 `relation_type_id=None` 创建。

#### FIX-02：添加异常日志和 try/except

将 `ensure_self_family_member` 的调用用 try/except 包裹，在 except 中**记录 logger.error** 而不是静默 pass。确保即使创建成员失败，注册本身仍然成功。

#### FIX-03：为"本人"成员创建关联档案

改造 `ensure_self_health_profile()`，在创建 `family_member_id=None` 的主用户档案之外，**同时也创建一条 `family_member_id=self_member.id` 的档案**。

#### FIX-04：后端兜底（核心修复）

在 `GET /api/family/members` 接口中增加逻辑：查询到成员列表后，如果其中没有 `is_self=True` 的成员，自动调用 `ensure_self_family_member()` 创建一个，再重新查询返回。所有端无需改动即可受益，同时覆盖存量老用户。

#### FIX-05：引导页 Step1 保存时同步更新 FamilyMember

当 `PUT /api/health/profile/member/{member_id}` 的 patch 数据包含 `name`、`gender`、`birthday` 时，同时查找对应的 `FamilyMember`（通过 `member_id`），将这 3 个字段写入 FamilyMember 记录。

#### FIX-06：手动添加成员 4 字段必填校验

在 `POST /api/family/members` 的业务逻辑中增加校验：

```python
if not data.name or not data.name.strip():
    raise HTTPException(status_code=400, detail="姓名为必填项")
if not data.gender or not data.gender.strip():
    raise HTTPException(status_code=400, detail="性别为必填项")
if not data.birthday:
    raise HTTPException(status_code=400, detail="出生日期为必填项")
if not data.relation_type_id and not data.relationship_type:
    raise HTTPException(status_code=400, detail="成员关系为必填项")
```

此校验仅对用户手动添加成员生效，注册自动创建"本人"不走此 API。

#### FIX-07：前端添加成员弹窗必填校验（健康档案页）

在用户点击"确认添加"前，前端校验 4 个字段：成员关系（不可为空）、姓名（不可为空字符串）、性别（必须选择男/女）、出生日期（必须选择日期）。缺失字段时 Toast 提示"请填写完整的成员信息"。
