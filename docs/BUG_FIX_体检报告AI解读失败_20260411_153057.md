# 体检报告AI解读"分析失败" Bug 修复方案文档

## 1. Bug 发生背景

### 1.1 项目概述

本项目为「宾尼小康」AI 健康管家系统，提供体检报告上传与 AI 解读功能。用户上传体检报告图片后，系统通过 OCR 识别文字内容，再调用 AI 模型对指标进行结构化解读，生成健康评分、分类指标详情及个性化建议。

### 1.2 涉及功能模块

| 模块 | 说明 |
|------|------|
| **H5 前端 - 体检报告结果页** | `h5-web/src/app/checkup/result/[id]/page.tsx` — 调用 `/api/report/analyze` 接口并展示 AI 解读结果 |
| **后端 - 报告上传接口** | `backend/app/api/report.py` — `/api/report/upload` 上传图片 + OCR |
| **后端 - AI 解读接口** | `backend/app/api/report.py` — `/api/report/analyze` 触发 AI 解读 |
| **后端 - AI 服务层** | `backend/app/services/ai_service.py` — `analyze_report_structured()` 调用 AI 模型并解析 JSON |
| **后端 - AI 模型配置** | 数据库 `AIModelConfig` 表 — 存储 AI 模型的 base_url、api_key、model_name 等配置 |
| **后端 - OCR 服务** | `backend/app/services/ocr_service.py` — 百度 OCR 文字识别 |

### 1.3 发现时间与发现方式

- **发现时间**：2026年4月10-11日（最近一两天）
- **发现方式**：用户在 H5 网页端使用体检报告上传功能时，发现 AI 解读步骤失败
- **之前状态**：该功能之前一直可以正常使用，近期才出现问题

## 2. Bug 描述

### 2.1 错误现象

用户在 H5 网页端（手机浏览器）的体检报告页面上传图片后：
1. **图片上传成功**，成功跳转到了结果页面 `/checkup/result/{id}`
2. 结果页展示 **"AI 正在分析..."** 的加载动画（progressive loading）
3. 等待一段时间后，弹出 Toast 提示：**"分析失败，请重试"**
4. 无论上传拍照版体检报告还是电子版截图，**均会失败**

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 打开 H5 网页端，进入首页 | 首页正常显示 | 正常 |
| 2 | 点击"体检报告"入口 | 进入体检报告页面 | 正常 |
| 3 | 上传一张体检报告图片（拍照或截图均可） | 图片上传成功，跳转到结果页 | 正常，图片上传成功 |
| 4 | 在结果页等待 AI 解读 | 展示 AI 解读结果（健康评分、分类指标、建议等） | **失败**：展示加载动画后弹出"分析失败，请重试" |

### 2.3 影响范围

- **受影响端**：H5 网页端（手机浏览器），其他端（Flutter App、微信小程序）如调用同一后端接口也可能受影响
- **受影响功能**：体检报告 AI 解读功能完全不可用
- **受影响用户**：所有使用体检报告功能的用户
- **数据影响**：已上传的报告图片和 OCR 文本正常保存，但 AI 解读结果（`ai_analysis`、`ai_analysis_json`）未生成，报告状态停留在 `failed`

## 3. 预期正确效果

用户上传体检报告图片后，系统应：
1. 成功上传图片并完成 OCR 文字识别
2. 跳转到结果页后，调用 `/api/report/analyze` 接口
3. 后端成功调用 AI 模型对 OCR 文本进行结构化解读
4. 前端展示完整的 AI 解读结果，包括：
   - 健康评分仪表盘（0-100分）
   - 检查项目统计（共检测项、异常项、优秀项）
   - 分类指标详情（血常规、肝功能等分类下各项指标的值、风险等级、详细建议）
   - 免责声明

## 4. 根因分析方向

根据已掌握的信息，问题发生在 **AI 解读步骤**（图片上传和 OCR 正常），且最近一两天才出现。近期代码提交未直接修改体检报告模块。可能的根因方向：

### 4.1 AI 模型服务端问题（最可能）

- **API Key 过期或额度耗尽**：数据库 `AIModelConfig` 中配置的 AI 模型 API Key 可能已过期或余额不足
- **模型服务不可用**：AI 模型的 `base_url` 对应的服务可能出现故障或变更
- **模型名称变更**：模型提供商可能下线或更名了当前配置的模型
- **请求超时**：`ai_service.py` 中 httpx 超时设置为 60 秒，如果模型响应变慢可能超时

### 4.2 AI 模型返回格式异常

- AI 模型返回的内容不是合法 JSON，导致解析失败
- 注意：当前代码在 JSON 解析失败时有 fallback（返回空分类 + score=0），不会抛出异常。但如果 AI 服务本身调用失败（网络错误、鉴权失败等），`call_ai_model` 会返回错误字符串而非抛出异常，后续 JSON 解析失败后走 fallback 路径，**不应该触发 500 错误**

### 4.3 后端异常传播

- 后端 `/api/report/analyze` 接口在调用 `analyze_report_structured()` 时，用 try-except 包裹，如果抛出异常会返回 HTTP 500 + `"AI解读失败: {error}"`
- 前端收到非 200 响应后走 catch 分支，显示"分析失败，请重试"

## 5. 修复方案

### 5.1 排查与定位（优先执行）

1. **检查后端日志**：查看 `/api/report/analyze` 接口的实际错误信息，确认是 AI 模型调用失败还是其他原因
2. **检查 AI 模型配置**：验证数据库 `AIModelConfig` 表中的 `base_url`、`api_key_encrypted`、`model_name` 是否有效
3. **手动调用 AI 接口**：用 curl/httpx 直接调用配置的 AI 模型接口，确认模型服务是否正常响应
4. **检查 OCR 结果**：确认已上传报告的 `ocr_result` 字段是否有内容（OCR 文本不为空）

### 5.2 代码层面优化

1. **增强错误信息透传**：后端 `/api/report/analyze` 接口在 AI 调用失败时，应记录详细错误日志（当前只在 HTTPException 中简单包含了 `str(e)`），增加 `logging.error` 记录完整堆栈
2. **前端错误提示优化**：前端 catch 分支可以尝试解析后端返回的 `detail` 字段，展示更具体的错误信息（如"AI 服务暂时不可用"而非笼统的"分析失败，请重试"）
3. **增加重试机制**：在 `call_ai_model` 中增加自动重试逻辑（如超时或网络错误时重试 1-2 次）
4. **超时时间调整**：当前 httpx 超时 60 秒，对于大量 OCR 文本的结构化解读可能不够，可适当增加到 120 秒

### 5.3 配置修复（如确认是配置问题）

- 更新 `AIModelConfig` 表中的 API Key 或模型配置
- 如 API Key 过期，更换为有效的 Key
- 如模型服务不可用，切换到备用模型

## 6. 补充说明

- 用户确认 AI 模型配置（API Key、接口地址等）最近**没有被手动修改过**
- 该功能之前**一直正常**，问题出现在最近一两天，指向外部依赖（AI 模型服务）变化的可能性较大
- 图片上传和 OCR 环节正常，问题**仅发生在 AI 解读步骤**
- 建议优先在服务器上查看后端日志，获取 `analyze_report_structured()` 的实际异常信息，这是定位根因的最快路径
