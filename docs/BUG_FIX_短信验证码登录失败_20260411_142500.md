# 短信验证码登录失败 Bug 修复方案文档

## 1. Bug 发生背景

### 1.1 项目概述

bini-health 是一个健康管理平台，包含 H5 Web 端、小程序端、Flutter App 等多个前端，后端基于 FastAPI + SQLAlchemy + MySQL 构建。用户可以通过手机号+短信验证码方式登录系统。

### 1.2 涉及功能模块

- **认证模块**：`/api/auth/sms-code`（发送验证码）、`/api/auth/sms-login`（短信验证码登录）
- **数据模型**：`VerificationCode`（验证码表）
- **前端登录页**：H5 端登录页面、小程序端登录页面

### 1.3 发现时间与发现方式

- 发现时间：最近一次系统部署更新之后
- 发现方式：客户反馈无法使用短信验证码登录

## 2. Bug 描述

### 2.1 错误现象

某客户使用短信验证码登录 H5 端时，出现以下两种表现：

| 场景 | 操作 | 实际提示信息 |
|------|------|-------------|
| 输入**正确**验证码 | 点击登录 | "登录失败，请检查验证码"（前端兜底提示） |
| 输入**错误**验证码 | 点击登录 | "验证码无效或已过期"（后端返回） |

关键特征：
- 该客户之前一直能正常登录，最近突然不行
- 目前仅该客户反馈此问题
- 短信能正常收到，验证码为正常的 6 位数字
- 账号未被禁用，状态正常
- 问题出现前做过系统更新/部署

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 打开 H5 端登录页面 | 正常显示登录表单 | 正常 |
| 2 | 输入手机号，点击"获取验证码" | 收到短信验证码 | 正常收到 |
| 3 | 输入收到的正确验证码，点击"登录" | 登录成功，进入首页 | 提示"登录失败，请检查验证码" |
| 4 | 输入一个错误的验证码，点击"登录" | 提示验证码错误 | 提示"验证码无效或已过期" |

### 2.3 影响范围

- **直接影响**：所有通过短信验证码登录的用户（H5 端、小程序端共用同一后端接口）
- **潜在影响**：小程序端尚未测试，但因共用 `/api/auth/sms-login` 接口，大概率存在同样问题
- **不受影响**：使用密码方式登录的用户不受影响（密码登录走 `/api/auth/login` 接口，不涉及时间比较逻辑）

## 3. 预期正确效果

1. 用户输入正确的手机号和验证码后，应当**成功登录**并跳转到首页
2. 用户输入错误的验证码后，应提示"验证码无效或已过期"
3. 验证码在 5 分钟有效期内应始终可用
4. 短信登录功能在所有端（H5、小程序、App）均应正常工作

## 4. 根因分析

### 4.1 根因定位：验证码时间处理中的时区混用（aware vs naive datetime）

后端代码中存在 **两种 datetime 写法混用** 的问题，导致验证码校验时时间比较行为异常。

#### 问题代码 1 — 验证码生成（`backend/app/api/auth.py` 第 296 行）

```python
expires_at = datetime.now(timezone.utc) + timedelta(minutes=5)
```

`datetime.now(timezone.utc)` 生成的是 **aware datetime**（带 UTC 时区信息的时间对象，如 `2026-04-11 06:00:00+00:00`）。

#### 问题代码 2 — 数据库模型（`backend/app/models/models.py` 第 242-243 行）

```python
expires_at = mapped_column(DateTime, nullable=False)           # 未启用 timezone=True
created_at = mapped_column(DateTime, default=datetime.utcnow)  # 使用 naive datetime
```

`DateTime` 字段未声明 `timezone=True`，MySQL 存储时会**丢弃时区信息**。存入 `2026-04-11 06:00:00+00:00`，取出变为 `2026-04-11 06:00:00`（naive datetime）。

#### 问题代码 3 — 验证码校验（`backend/app/api/auth.py` 第 324 行）

```python
VerificationCode.expires_at > datetime.now(timezone.utc)
```

用 **aware datetime** 去和数据库取出的 **naive datetime** 做 SQL 比较。

#### 故障机制

1. 验证码生成时，`expires_at` 带时区 → 存入数据库 → 时区被丢弃 → 变成 naive datetime
2. 校验时，Python 端用 `datetime.now(timezone.utc)`（aware）去比较数据库中的 naive datetime
3. 在某些 Python 版本 / SQLAlchemy 版本 / MySQL 驱动组合下，**aware vs naive 比较会引发 TypeError 或静默返回错误结果**
4. 最近的部署可能更新了依赖版本或运行环境，触发了这个潜在问题

#### 为什么短信能发但登录失败

防频率限制代码（第 287 行）使用的是 `datetime.utcnow()`（naive），和 `created_at` 的默认值一致（都是 naive），所以不会出错：

```python
VerificationCode.created_at > datetime.utcnow() - timedelta(seconds=60)
```

### 4.2 辅助佐证

| 观察点 | 说明 |
|--------|------|
| 两条不同的错误提示 | 正确验证码 → 后端查询异常 → 返回非预期错误 → 前端兜底提示；错误验证码 → 直接未匹配 → 后端明确返回"验证码无效或已过期" |
| 仅最近出现 | 与部署更新时间吻合，说明是环境/依赖变更触发 |
| 短信发送正常 | 发送逻辑不涉及 aware/naive 混用 |

## 5. 修复方案

### 5.1 修复策略：统一使用 naive UTC datetime

将所有涉及时间的地方统一使用 `datetime.utcnow()`（naive UTC），与数据库模型保持一致。

### 5.2 需修改的文件与修改内容

#### 文件 1：`backend/app/api/auth.py`

**修改点 A — 第 296 行（验证码生成）**

将：
```python
expires_at = datetime.now(timezone.utc) + timedelta(minutes=5)
```
改为：
```python
expires_at = datetime.utcnow() + timedelta(minutes=5)
```

**修改点 B — 第 324 行（验证码校验）**

将：
```python
VerificationCode.expires_at > datetime.now(timezone.utc),
```
改为：
```python
VerificationCode.expires_at > datetime.utcnow(),
```

### 5.3 修复后的验证要点

| 验证项 | 操作 | 预期结果 |
|--------|------|----------|
| 正确验证码登录 | 输入正确验证码点击登录 | 登录成功，进入首页 |
| 错误验证码登录 | 输入错误验证码点击登录 | 提示"验证码无效或已过期" |
| 验证码过期 | 等待 5 分钟后再使用验证码 | 提示"验证码无效或已过期" |
| 频率限制 | 60 秒内重复发送验证码 | 提示"发送过于频繁，请60秒后重试" |
| 多端验证 | H5 端和小程序端分别测试短信登录 | 均能正常登录 |

## 6. 补充说明

1. **此 Bug 影响全端**：`/api/auth/sms-login` 是后端统一接口，H5、小程序、App 端共用。修复后所有端的短信登录都将恢复正常。
2. **密码登录不受影响**：密码登录走 `/api/auth/login` 接口，不涉及时间比较，无此问题。
3. **建议后续优化**：全局审查项目中所有 `datetime.now(timezone.utc)` 的用法（如 `ensure_identity` 函数第 119 行的 `datetime.utcnow()` 已是正确写法），确保与数据库 `DateTime` 字段类型保持一致，避免类似问题再次出现。
4. **仅需修改 2 行代码**，修复简单明确，风险极低。
