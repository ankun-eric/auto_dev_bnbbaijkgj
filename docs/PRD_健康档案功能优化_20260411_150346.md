# 健康管理档案功能优化 产品需求文档（PRD）

## 1. 需求概述

### 1.1 背景与目的

当前「健康管理档案」模块中，慢性病史和遗传病史通过预设标签多选的方式填写，但缺少"其它"手工填写选项，无法满足用户个性化的病史记录需求。过敏史信息则采用三个独立文本输入框（药物过敏、食物过敏、其他过敏）的形式，操作不够便捷，也缺少常见过敏原的预设标签引导。同时后台管理端的预设标签管理范围有限，未覆盖过敏史和遗传病史分类。

此外，用户注册/登录后的健康档案引导页、健康自查"为谁咨询"面板中的健康信息填写，也存在同样的交互局限，需同步改造以保持全平台交互一致性。

本次优化旨在：
- 为慢性病史、过敏史、家族遗传病史三个板块统一增加"其它"手工填写能力
- 将过敏史改造为预设标签选择 + 手工填写的统一交互模式
- 在后台统一管理三类预设标签，提升运营效率
- 对历史过敏数据进行迁移适配
- 同步改造健康引导页和健康自查页面，确保所有涉及健康档案填写的入口交互一致
- 所有入口填写的健康信息同步回写到健康档案，保持数据一致性

### 1.2 目标用户

| 角色 | 使用场景 |
|------|----------|
| C 端用户（患者/家庭成员） | 在 H5、Flutter App、微信小程序端新建/编辑健康档案、注册引导补充档案、健康自查选择咨询对象时，通过预设标签 + 手工填写记录个人病史、过敏史、遗传病史 |
| 后台超级管理员 | 在 admin-web 后台维护慢性病史、过敏史、家族遗传病史的预设标签列表 |
| 后台普通管理员 | 在 admin-web 后台查看预设标签列表（只读） |

### 1.3 核心价值

- **信息完整性**：支持手工填写"其它"项，确保不在预设范围内的病史也能被记录
- **操作便捷性**：过敏史从纯文本输入改为预设标签点选，降低用户填写成本
- **管理灵活性**：后台统一管理三类预设标签，运营人员可动态调整常见选项
- **数据一致性**：历史过敏数据自动迁移，保证新旧数据平滑过渡；所有入口填写的健康信息同步回写到健康档案
- **交互统一性**：健康档案编辑页、注册引导页、健康自查页面三个场景的交互体验完全一致

### 2.2 功能详细描述

#### F01：慢性病史 — 增加"其它"手工填写项

**当前状态**：慢性病史（`chronic_diseases`）通过 `DiseasePreset` 表中 `category = 'chronic'` 的预设标签进行多选，数据以 JSON 数组存储。

**改造内容**：

1. 预设标签列表末尾固定显示一个「其它」选项（该选项不存储在 `DiseasePreset` 表中，由前端固定渲染）
2. 用户点击「其它」后，在标签列表下方直接展开一个文本输入框
3. 用户在输入框中填写内容后，点击「确认」按钮即可添加
4. 添加后的"其它"内容以标签形式显示在已选列表中，可点击删除
5. 支持多次添加"其它"内容（每次确认后输入框清空，可继续填写）

**数据存储**：

`chronic_diseases` 字段（JSON 数组）中，预设标签存储标签名称字符串，手工填写的"其它"内容以 `{"type": "custom", "value": "用户填写内容"}` 对象格式存储，以便前端区分展示。

示例：
```json
["高血压", "糖尿病", {"type": "custom", "value": "罕见病XXX"}]
```

#### F02：过敏史 — 预设标签选择 + 手工填写改造

**当前状态**：过敏信息分为三个独立文本字段：`drug_allergies`（药物过敏）、`food_allergies`（食物过敏）、`other_allergies`（其他过敏），均为纯文本输入。

**改造内容**：

1. 将三个独立文本字段合并为统一的「过敏史」板块
2. 过敏史使用预设标签多选方式，标签数据来源于 `DiseasePreset` 表中 `category = 'allergy'` 的记录
3. 预设标签列表末尾固定显示「其它」选项
4. 点击「其它」后，在标签列表下方直接展开文本输入框，填写确认后以标签形式添加
5. 已选标签可点击删除

**初始预设标签清单**（需在部署时通过数据迁移脚本写入 `DiseasePreset` 表）：

| 序号 | 标签名称 | category |
|------|----------|----------|
| 1 | 青霉素 | allergy |
| 2 | 头孢类 | allergy |
| 3 | 磺胺类 | allergy |
| 4 | 阿司匹林 | allergy |
| 5 | 海鲜/贝壳类 | allergy |
| 6 | 牛奶/乳制品 | allergy |
| 7 | 鸡蛋 | allergy |
| 8 | 花生/坚果类 | allergy |
| 9 | 小麦/面筋 | allergy |
| 10 | 花粉 | allergy |
| 11 | 尘螨 | allergy |
| 12 | 动物皮毛 | allergy |
| 13 | 乳胶 | allergy |
| 14 | 酒精 | allergy |

**数据存储**：

统一使用 `allergies` 字段（JSON 数组）存储，格式与慢性病史一致：
```json
["青霉素", "花粉", {"type": "custom", "value": "特殊过敏原XXX"}]
```

原 `drug_allergies`、`food_allergies`、`other_allergies` 三个字段保留但不再在前端使用（兼容历史数据查询），新数据统一写入 `allergies` 字段。

#### F03：家族遗传病史 — 增加"其它"手工填写项

**当前状态**：家族遗传病史（`genetic_diseases`）通过 `DiseasePreset` 表中 `category = 'genetic'` 的预设标签进行多选，数据以 JSON 数组存储。

**改造内容**：

与 F01（慢性病史）改造方式完全一致：
1. 预设标签列表末尾固定显示「其它」选项
2. 选中后在标签列表下方展开文本输入框
3. 确认后以标签形式添加，支持多次添加
4. 已选标签可删除

**数据存储**：

`genetic_diseases` 字段（JSON 数组），格式与慢性病史一致：
```json
["高血压", "糖尿病", {"type": "custom", "value": "其他遗传病XXX"}]
```

#### F04：后台预设标签管理 — 三类统一管理

**当前状态**：后台已有 `disease-presets` 管理页面，管理 `DiseasePreset` 表中的数据，`category` 字段目前包含 `chronic`（慢性病史）和 `genetic`（遗传病史）两个分类。

**改造内容**：

1. `DiseasePreset` 表的 `category` 字段新增 `allergy` 分类值
2. 后台管理页面顶部增加分类筛选下拉框，选项为：
   - 全部
   - 慢性病史（chronic）
   - 过敏史（allergy）
   - 家族遗传病史（genetic）
3. 新增/编辑预设标签时，`category` 为必填下拉选择字段
4. 支持标签的增删改查、排序调整、启用/停用操作（沿用现有功能）

#### F05：后台权限控制

**改造内容**：

1. **超级管理员**：对预设标签管理具有完整的增、删、改权限
2. **普通管理员**：只能查看预设标签列表，不显示新增/编辑/删除按钮，操作接口返回 403

**实现方式**：

- 后端 API 层根据当前用户的 `is_superuser` 字段判断权限
- 前端根据用户角色动态控制操作按钮的显示/隐藏

#### F06：历史过敏数据迁移

**迁移策略**：模糊匹配

1. 读取所有 `HealthProfile` 记录中的 `drug_allergies`、`food_allergies`、`other_allergies` 三个文本字段
2. 对每条记录，将文本内容按常见分隔符（逗号、顿号、分号、空格等）拆分为独立项
3. 对每个独立项，与过敏史预设标签列表进行模糊匹配（关键词包含判断）：
   - 匹配成功 → 加入 `allergies` JSON 数组（标签名称字符串）
   - 匹配失败 → 以 `{"type": "custom", "value": "原始文本"}` 格式加入 `allergies` 数组
4. 去重处理：同一标签不重复添加
5. 迁移完成后，原三个文本字段数据保留不删除（数据安全）

**迁移执行时机**：部署上线时通过数据库迁移脚本自动执行

#### F08：健康引导页 — 注册后引导页同步改造

**当前状态**：用户注册/登录后，系统通过引导流程分 4 步引导用户补充健康档案：
- 第 1 步：基本信息
- 第 2 步：慢性病史（预设标签多选）
- 第 3 步：过敏史（三个文本输入框：药物过敏、食物过敏、其他过敏）
- 第 4 步：遗传病史（预设标签多选）

**改造内容**：

1. 引导流程保持 **4 步**不变
2. **第 2 步（慢性病史）**：预设标签列表末尾增加"其它"手工填写，交互方式与健康档案编辑页完全一致（F01）
3. **第 3 步（过敏史）**：从三个文本输入框改为预设标签选择 + "其它"手工填写，交互方式与健康档案编辑页完全一致（F02）
4. **第 4 步（遗传病史）**：预设标签列表末尾增加"其它"手工填写，交互方式与健康档案编辑页完全一致（F03）
5. 引导页中填写的所有健康信息（包括"其它"手工内容）保存时同步写入用户的 `HealthProfile`

#### F09：健康自查页面 — "为谁咨询"面板同步改造

**当前状态**：健康自查（symptom）页面中，用户选中咨询对象（家庭成员）后，展开的健康档案编辑区域包含：
- 既往病史：页面内写死的 8 个标签（高血压、糖尿病等）+ "其他病史"文本输入框
- 过敏史：页面内写死的标签 + "其他过敏史"文本输入框
- 无「家族遗传病史」板块

**改造内容**：

1. **既往病史（慢性病史）**：预设标签改为从后台 `DiseasePreset` 表（`category = chronic`）动态获取，末尾保留"其它"手工填写，交互方式与健康档案编辑页一致（F01）
2. **过敏史**：预设标签改为从后台 `DiseasePreset` 表（`category = allergy`）动态获取，末尾保留"其它"手工填写，交互方式与健康档案编辑页一致（F02）
3. **新增「家族遗传病史」板块**：
   - 位置放在最后，顺序为：既往病史 → 过敏史 → 家族遗传病史
   - 预设标签从后台 `DiseasePreset` 表（`category = genetic`）动态获取，末尾保留"其它"手工填写
   - 交互方式与健康档案编辑页一致（F03）
4. 新建家庭成员时的健康信息填写同步改造

#### F10：数据同步回写 — 各入口健康信息同步到健康档案

**业务规则**：

1. 用户在**任何入口**（健康档案编辑页、健康引导页、健康自查页面）填写或修改的健康信息，都同步回写到该用户（或家庭成员）的 `HealthProfile` 记录中
2. 同步范围包括：慢性病史（`chronic_diseases`）、过敏史（`allergies`）、家族遗传病史（`genetic_diseases`）
3. 同步策略：以最新一次填写的内容为准，覆盖更新到 `HealthProfile`
4. 自查页面为家庭成员填写的信息，同步到该家庭成员的 `HealthProfile`

**实现方式**：

- 各入口的保存操作统一调用后端 `HealthProfile` 更新 API
- 如该用户/家庭成员尚无 `HealthProfile` 记录，则自动创建

## 4. 非功能性需求

### 4.1 性能要求

- 预设标签列表加载时间 ≤ 500ms
- 数据迁移脚本需支持批量处理，避免单条更新导致数据库压力过大
- 迁移脚本应具备幂等性，支持重复执行
- 健康引导页和自查页面的标签列表应使用缓存，避免重复请求

### 4.2 安全要求

- 后台预设标签管理接口需校验用户角色权限
- 用户手工填写的"其它"内容需进行 XSS 过滤和长度限制（建议最大 100 字符）

### 4.3 兼容性要求

- H5 端：兼容 iOS Safari 12+ / Android Chrome 70+ / 微信内置浏览器
- Flutter App：兼容 iOS 12+ / Android 6.0+
- 微信小程序：基础库 2.10.0+
- admin-web：兼容 Chrome 80+ / Edge 80+ / Firefox 75+

## 5. 业务规则与约束

| 编号 | 规则描述 |
|------|----------|
| BR01 | 用户手工填写的"其它"内容仅保存在该用户的健康档案中，不自动加入预设标签库 |
| BR02 | "其它"手工填写内容不允许为空，需前端校验 |
| BR03 | "其它"手工填写内容最大长度 100 字符 |
| BR04 | 预设标签和自定义标签去重：同名标签不可重复添加，提交时前端需做去重校验 |
| BR05 | 预设标签列表按 `sort_order` 升序排列，「其它」选项始终在末尾（前端固定渲染，不在数据库中） |
| BR06 | 后台停用的预设标签（`is_active = false`）不在 C 端展示，但已使用该标签的历史档案数据正常显示 |
| BR07 | 数据迁移使用模糊匹配策略：旧文本包含预设标签关键词即视为匹配成功 |
| BR08 | 迁移后原 `drug_allergies`、`food_allergies`、`other_allergies` 字段数据保留不删除 |
| BR09 | 用户在任何入口（健康档案编辑页、健康引导页、健康自查页面）填写/修改的健康信息，均同步回写到该用户或家庭成员的 `HealthProfile` |
| BR10 | 健康自查页面中，既往病史/过敏史/家族遗传病史的预设标签从后台 `DiseasePreset` 表动态获取，不再页面内写死 |
| BR11 | 健康自查页面"为谁咨询"面板中，三个板块的展示顺序为：既往病史 → 过敏史 → 家族遗传病史 |

## 7. 异常处理与边界情况

| 场景 | 处理方式 |
|------|----------|
| 预设标签列表接口请求失败 | 显示错误提示，提供重试按钮；同时允许用户通过"其它"手工填写 |
| 用户填写的"其它"内容包含特殊字符 | 前端进行 XSS 过滤，后端二次校验 |
| 用户填写的"其它"内容与现有预设标签重名 | 前端提示"该选项已存在于预设列表中，请直接选择"，不允许重复添加 |
| 数据迁移时某条记录格式异常 | 记录日志，将原始文本整体放入"其它"，不中断迁移流程 |
| 后台删除已被用户使用的预设标签 | 仅标记为停用（`is_active = false`），不物理删除；已使用该标签的用户档案数据不受影响 |
| 普通管理员尝试调用增删改接口 | 后端返回 403 Forbidden，前端提示"权限不足" |
| 健康引导页中用户跳过某步骤 | 该步骤的健康信息不写入 HealthProfile，不影响后续步骤 |
| 自查页面加载家庭成员的健康档案失败 | 显示空标签列表，允许用户手工填写，保存时自动创建 HealthProfile 记录 |
| 多入口并发修改同一健康档案 | 以最后一次提交为准（last-write-wins），后端按时间戳覆盖更新 |
