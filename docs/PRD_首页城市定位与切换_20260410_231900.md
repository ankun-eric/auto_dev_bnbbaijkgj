# 首页城市定位与切换 产品需求文档（PRD）

## 1. 需求概述

### 1.1 背景与目的

bini-health 是一个多端健康管理应用，目前已有首页、搜索、健康计划、健康档案、AI 咨询、家庭成员管理等功能模块。为提升用户体验和后续本地化服务能力，需要在首页搜索栏左侧增加城市定位显示，使用户能够查看当前所在城市，并支持手动切换城市。

该功能为后续基于地理位置的内容推荐、本地化医疗健康服务筛选等业务场景打下基础。

### 1.2 目标用户

- 所有使用 bini-health 的终端用户（H5、微信小程序、Flutter App）

### 1.3 核心价值

- 增强用户的位置感知体验，提升产品的本地化服务能力
- 为后续按城市筛选内容（医疗机构、健康服务、推荐内容等）预留完整的接口和架构
- 提升产品调性，向主流健康/生活服务类 App（美团、大众点评等）看齐

## 2. 功能需求

### 2.2 功能详细描述

#### F01：首页城市定位展示

**功能说明：**

在首页搜索栏左侧新增城市定位区域，整体布局为：`[城市名 ▼] | [搜索框]`，城市名与搜索框之间有竖线分隔（类似美团/大众点评风格）。

**业务规则：**

1. 用户首次进入首页时，请求位置授权
2. 授权成功：显示定位到的城市名（如"北京市"），城市名右侧带下拉小箭头 ▼
3. 授权失败/拒绝：显示"定位"二字，点击后引导用户重新开启位置权限
4. 用户曾手动选择过城市：优先显示用户选择的城市，而非 GPS 定位城市
5. 城市名显示为城市级别（如"北京市"、"深圳市"），不显示区县

**交互规则：**

- 点击城市区域 → 跳转至全屏城市选择页面（F03）
- 城市名过长时截断显示，最多显示 4 个汉字（如"乌鲁木齐"），超出部分省略

#### F02：GPS 定位与逆地理编码

**功能说明：**

各端获取用户 GPS 位置（经纬度），调用逆地理编码服务将经纬度转换为城市名称。

**技术方案（各端最佳实践）：**

| 端 | 定位方案 | 逆地理编码服务 |
|----|----------|----------------|
| H5 | 浏览器 Geolocation API / 高德 JS API | 高德地图 Web 服务 API |
| 微信小程序 | `wx.getLocation` | 腾讯地图逆地理编码 API（小程序生态天然适配） |
| Flutter App | `geolocator` 插件 | 高德地图 REST API |

**业务规则：**

1. 定位精度为城市级别，无需获取精确地址
2. 定位超时时间设置为 10 秒，超时后按定位失败处理
3. 定位结果缓存 30 分钟，避免频繁调用地图 API
4. 后端提供经纬度转城市接口作为统一兜底方案，前端也可直接调用地图服务

#### F03：城市选择页

**功能说明：**

点击首页城市区域后，进入全屏城市选择页面，用户可搜索或浏览并选择目标城市。

**页面结构（从上到下）：**

1. **顶部导航栏**：左侧返回按钮 + 标题"选择城市"
2. **搜索框**：支持输入城市名模糊搜索，实时过滤城市列表
3. **当前定位城市**：标签"GPS定位"+ 定位到的城市名，点击可直接选择；定位失败时显示"重新定位"按钮
4. **最近访问城市**：标签"最近访问"+ 最多 3 个最近选择过的城市标签，点击可直接选择；无历史记录时不显示此区域
5. **热门城市**：标签"热门城市"+ 网格排列的热门城市标签（由管理后台配置），点击可直接选择
6. **全国城市列表**：按拼音首字母 A-Z 分组，右侧有字母索引导航条，点击字母快速跳转到对应分组

**交互规则：**

- 选择任意城市后，立即返回首页，搜索栏左侧更新为所选城市名
- 搜索框输入时，实时过滤匹配的城市，以列表形式展示搜索结果
- 右侧字母索引条支持滑动快速定位

#### F04：城市持久化

**功能说明：**

用户手动切换的城市信息需持久化存储，下次打开应用时优先读取。

**业务规则：**

1. 用户手动选择城市后，将城市 ID 和城市名存储到本地（localStorage / SharedPreferences / wx.setStorageSync）
2. 用户打开应用时的城市展示优先级：用户手动选择的城市 > GPS 定位城市 > 显示"定位"
3. 最近访问城市列表同步持久化，最多保留 3 条，按访问时间倒序排列，重复选择同一城市时更新时间而非新增
4. 用户在城市选择页点击"GPS定位"城市时，视为手动选择，同时更新持久化记录

#### F05：后端城市数据管理

**功能说明：**

后端维护全国城市数据库，并提供 API 供前端调用。

**数据模型（`cities` 表）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| name | varchar(50) | 城市名称（如"北京市"） |
| short_name | varchar(20) | 城市简称（如"北京"） |
| pinyin | varchar(100) | 拼音（如"beijing"） |
| first_letter | char(1) | 拼音首字母（如"B"） |
| province | varchar(50) | 所属省份 |
| longitude | decimal(10,6) | 城市中心经度 |
| latitude | decimal(10,6) | 城市中心纬度 |
| is_hot | boolean | 是否为热门城市 |
| hot_sort | int | 热门城市排序（数字越小越靠前） |
| is_active | boolean | 是否启用 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

**API 设计（router 前缀 `/api/cities`）：**

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/list` | 获取全部启用城市列表（按首字母分组），支持 `keyword` 搜索参数 |
| GET | `/hot` | 获取热门城市列表（按 hot_sort 排序） |
| GET | `/locate` | 根据经纬度（`lng`, `lat` 参数）返回匹配的城市信息（逆地理编码兜底） |

**初始数据导入：**

- 使用国家统计局标准行政区划数据，导入地级市及以上城市（约 340+ 个）
- 默认热门城市：北京、上海、广州、深圳、杭州、成都、武汉、南京、重庆、西安、苏州、天津（可由管理后台调整）

#### F06：管理后台热门城市管理

**功能说明：**

管理后台提供热门城市管理页面，运营人员可配置首页城市选择器中显示的热门城市。

**页面功能：**

1. **热门城市列表**：展示当前所有标记为热门的城市，支持拖拽排序
2. **添加热门城市**：从全国城市库中搜索并添加为热门城市
3. **移除热门城市**：将城市从热门列表移除（不删除城市数据，仅取消热门标记）

**管理后台 API（router 前缀 `/api/admin/cities`）：**

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/list` | 获取全部城市列表（含分页、搜索） |
| GET | `/hot` | 获取热门城市列表 |
| POST | `/hot` | 批量设置热门城市（传入城市 ID 列表和排序） |
| DELETE | `/hot/{city_id}` | 移除某个热门城市 |

#### F07：城市筛选接口预留

**功能说明：**

在现有业务 API 中预留城市筛选参数，为后续按城市筛选内容做好架构准备。

**业务规则：**

1. 首页推荐、搜索等核心 API 增加可选参数 `city_id`
2. 当前阶段 `city_id` 参数传入后不做实际筛选逻辑，仅记录日志用于后续分析
3. 前端在调用这些 API 时携带当前选择的城市 ID
4. 后续需要启用城市筛选时，仅需在后端补充筛选逻辑，前端无需改动

## 3. 页面/界面设计

### 3.1 页面结构与导航

```
首页
├── 搜索栏（左侧新增城市定位区域）
│   ├── [城市名 ▼]（点击跳转城市选择页）
│   └── [搜索框]
└── ...原有内容区...

城市选择页（全屏）
├── 顶部导航栏（返回 + 标题）
├── 搜索框
├── GPS 定位城市
├── 最近访问城市（最多3个）
├── 热门城市（网格布局）
└── 全国城市列表（A-Z 字母索引）
```

### 3.2 各页面功能说明

#### 3.2.1 首页搜索栏改造

**布局：**

```
┌──────────────────────────────────────┐
│ [北京 ▼] │ 🔍 搜索健康服务...        │
└──────────────────────────────────────┘
```

- 左侧城市区域宽度自适应，最大宽度约 80px
- 城市名与搜索框之间有 1px 竖线分隔，竖线颜色为浅灰色
- 城市名字号略小于搜索框 placeholder 字号
- 下拉箭头 ▼ 使用图标，紧跟城市名后面

**状态变化：**

| 状态 | 城市区域显示 | 点击行为 |
|------|-------------|---------|
| 已授权+已定位 | "北京 ▼" | 跳转城市选择页 |
| 已授权+定位中 | "定位中..." | 不可点击 |
| 未授权/定位失败 | "定位 ▼" | 跳转城市选择页（页面内显示重新定位引导） |
| 已手动选择城市 | "上海 ▼" | 跳转城市选择页 |

#### 3.2.2 城市选择页

**视觉风格：** 白色背景全屏页面，简洁清爽

**各区域说明：**

1. **导航栏**：白色背景，左侧返回箭头，中间标题"选择城市"
2. **搜索框**：圆角矩形，灰色边框，placeholder "搜索城市"，有搜索图标
3. **GPS 定位城市区域**：
   - 标签"GPS定位"为灰色小字
   - 城市名为蓝色标签样式，带定位图标
   - 定位失败时显示"重新定位"按钮，蓝色文字
4. **最近访问城市区域**：
   - 标签"最近访问"为灰色小字
   - 城市名为圆角标签，横排排列，最多 3 个
   - 无历史时整个区域隐藏
5. **热门城市区域**：
   - 标签"热门城市"为灰色小字
   - 城市名为圆角标签，4 列网格布局
6. **全国城市列表**：
   - 按 A-Z 首字母分组，每组有灰色字母标题行
   - 城市名为列表行，点击整行可选择
   - 右侧固定字母索引条，竖直排列 A-Z，字号小，可滑动快速定位

### 3.3 管理后台城市管理页面

**路径：** 管理后台侧边栏 → 城市管理 → 热门城市配置

**页面功能：**

1. 热门城市列表表格：序号、城市名、省份、操作（移除）
2. 支持拖拽行调整热门排序
3. 顶部"添加热门城市"按钮，弹出搜索框从城市库中选择添加

## 4. 非功能性需求

### 4.1 性能要求

- 城市列表接口响应时间 < 500ms
- 城市列表数据客户端缓存，有效期 24 小时，减少重复请求
- GPS 定位超时时间 10 秒
- 定位结果本地缓存 30 分钟
- 字母索引滚动流畅，无卡顿

### 4.2 安全要求

- 用户位置信息仅用于城市定位，不记录精确坐标
- 位置权限为非强制获取，用户可拒绝且不影响其他功能使用
- 地图 API Key 后端管理，不暴露在前端代码中（H5 端需要前端 Key 的情况除外，做域名白名单限制）

### 4.3 兼容性要求

- H5 端：兼容主流浏览器（Chrome、Safari、微信内置浏览器等），HTTPS 环境下使用 Geolocation API
- 微信小程序：适配微信基础库 2.x+，正确处理 `scope.userLocation` 授权流程
- Flutter App：适配 Android 6.0+ 和 iOS 12+，处理运行时权限申请

## 5. 业务规则与约束

1. **城市数据唯一性**：城市名+省份为唯一标识，不可重复
2. **热门城市数量**：建议控制在 8-16 个，保证城市选择页视觉美观
3. **首字母分组**：城市按拼音首字母分组，多音字城市取最常用读音（如"重庆"取 C，不取 Z）
4. **城市级别**：仅包含地级市及以上行政区划，不包含县级市
5. **直辖市处理**：北京市、上海市、天津市、重庆市直接作为城市显示，不拆分区县
6. **特别行政区**：香港、澳门作为独立城市显示
7. **城市启停用**：管理员可在后台停用某城市（停用后前端列表不显示该城市，但已选择该城市的用户不受影响，直到重新选择）

## 6. 权限设计

| 角色 | 权限说明 |
|------|----------|
| 普通用户 | 查看城市列表、选择/切换城市、GPS 定位 |
| 管理员 | 热门城市配置（增加、移除、排序）、查看城市库 |

## 7. 异常处理与边界情况

| 场景 | 处理方案 |
|------|----------|
| 用户拒绝位置授权 | 搜索栏显示"定位"，城市选择页"GPS定位"区域显示"重新定位"按钮 |
| 定位超时（>10秒） | 等同定位失败处理，显示"定位"，城市选择页提示"定位超时，请重试" |
| 定位到的经纬度无法匹配任何城市 | 显示"定位"，引导用户手动选择城市 |
| 网络异常无法加载城市列表 | 使用本地缓存的城市数据（若有），否则显示加载失败+重试按钮 |
| 用户选择的城市被管理员停用 | 用户已持久化的城市继续展示，不强制清除；用户下次打开城市选择页时该城市不在列表中 |
| 搜索无匹配城市 | 搜索结果区域显示"未找到相关城市" |
| 多端城市数据不同步 | 城市选择为端本地持久化，各端独立，不做跨端同步 |
