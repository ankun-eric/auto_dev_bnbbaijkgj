# 家庭健康档案共管 产品需求文档（PRD）

## 1. 需求概述

### 1.1 背景与目的

bini-health 系统目前已有健康档案管理和家庭成员管理功能，用户可以在系统中手动为家人创建健康档案并维护基本信息。但目前的家庭成员管理仅支持单个用户独立操作，家人之间无法互相关联和协同管理。

本需求参考蚂蚁保（阿福）的家庭共管模式，引入"**家庭健康管家**"功能——主账号可以邀请家人关联，统一管理家人的健康档案、用药提醒、健康打卡、健康计划等，实现家庭成员之间的健康管理协作。

### 1.2 目标用户

- **主账号（发起邀请方）**：家庭中负责健康管理的核心角色，通常为子女或配偶，关注家人的用药和健康状况
- **被管理方（被邀请方）**：被邀请关联的家庭成员，通常为老人、孩子或需要健康关注的家人

### 1.3 核心价值

- 让家庭成员之间的健康管理不再孤立，形成"家庭健康管家"模式
- 主账号可以统一管理多位家人的健康事务（用药提醒、打卡、计划），减轻家人自行操作的负担
- 被管理方仅需同意授权即可，操作门槛极低
- 档案数据自动合并，减少重复录入

### 2.2 功能详细描述

#### F01 ~ F02：生成邀请二维码图片与分享

**前置条件**：主账号已在家庭成员切换栏中为该家人创建了一条健康档案记录。

**交互流程**：
1. 主账号在该家人的**档案详情页**中，看到一个明显的【邀请本人关联】按钮
2. 点击后，系统生成一张简洁的邀请图片，包含：
   - "XXX邀请您共同管理健康档案"文案
   - 小程序码/二维码
   - 小程序 logo
3. 生成的邀请图片下方提供三个操作按钮：
   - 【保存到本地】—— 将图片保存到手机相册
   - 【复制链接】—— 复制邀请链接到剪贴板
   - 【分享微信好友】—— 调用微信分享接口发送给指定好友

**业务规则**：
- 每次点击"邀请本人关联"都会生成一个新的邀请码
- 同一家人在同一时间只能有一个有效邀请
- 如果该家人已经关联（对方已同意），则按钮变为"已关联"状态，不可再次邀请

#### F03：扫码授权

**交互流程**：
1. 被邀请方在微信中使用**微信扫一扫**功能扫描邀请图片中的二维码
2. 自动跳转打开 bini-health 小程序
3. 系统判断登录状态：
   - **已登录**：直接弹出授权确认弹窗
   - **未登录/未注册**：引导用户先完成注册登录流程，登录成功后自动弹出授权确认弹窗
4. 授权确认弹窗内容："是否同意 **XX** 查看和管理您的健康档案？"
   - 【同意】—— 建立管理关系，跳转到首页
   - 【拒绝】—— 不建立关系，跳转到首页

**业务规则**：
- 如果邀请已过期（超过 24 小时），提示"邀请已过期，请联系对方重新邀请"
- 如果被邀请方已经被该主账号管理，提示"您已关联该用户"
- 如果被邀请方已达到被管理上限（3 人），提示"您已达到最大被管理人数限制"

#### F04：邀请有效期管理

- 生成的邀请二维码/链接在 **24 小时**后自动失效
- 过期后扫码提示"邀请已过期"
- 主账号可随时在档案详情页重新生成新的邀请

#### F05：主账号管理权限

主账号成功关联家人后，在健康档案页面的成员切换栏中切换到该家人，可进行以下操作：

| 管理范围 | 可执行操作 |
|---------|-----------|
| 健康档案 | 查看、编辑所有档案字段（基本信息、慢性病史、过敏史、家族遗传病史等） |
| 用药提醒 | 新建、编辑、删除用药提醒 |
| 健康打卡 | 新建、编辑、删除打卡任务 |
| 健康计划 | 新建、编辑、删除健康计划 |

#### F06：被管理方查看权限

被管理方同意授权后的权限极为有限：
- 在自己的**家庭成员列表**中看到"已关联"标记，知道有人在管理自己的档案
- **无法**查看主账号的任何具体信息（档案、提醒、计划等）
- 可以在"我的"页面和档案页顶部看到管理状态，并可取消授权

#### F07：人数限制

| 限制维度 | 上限 | 说明 |
|---------|------|------|
| 主账号邀请家人数 | 最多 **10 人** | 一个主账号最多邀请并关联 10 位家人 |
| 被管理方被管理数 | 最多 **3 人** | 一个用户最多同时被 3 个主账号管理 |

**边界情况处理**：
- 主账号尝试邀请第 11 位家人时，提示"您已达到最大邀请人数限制（10人）"
- 被邀请方尝试同意第 4 个主账号的邀请时，提示"您已达到最大被管理人数限制（3人）"

#### F08：档案合并

当被邀请方同意关联后，系统自动合并两份档案：

**合并规则**：
- 以**被邀请方（本人）**填写的数据为准
- 被邀请方未填写但主账号已填写的字段，用主账号的数据补充
- 合并完成后，双方看到的是**同一份实时数据**
- 合并是一次性操作，后续的修改直接作用于同一份数据

**合并示例**：

| 字段 | 主账号创建的值 | 被邀请方自己的值 | 合并结果 |
|------|-------------|----------------|---------|
| 姓名 | 张妈妈 | 张秀英 | 张秀英（以本人为准） |
| 身高 | 158cm | （空） | 158cm（补充） |
| 慢性病史 | 高血压 | 高血压、糖尿病 | 高血压、糖尿病（以本人为准） |

#### F09：合并后编辑权

- 主账号和被管理方**都可以编辑**同一份档案
- 最后修改的内容覆盖之前的（无冲突提示，简化实现）
- 所有修改都实时同步，双方看到的始终是最新数据

#### F10：操作记录

- 系统记录每次修改的操作来源："由 **XX**（主账号昵称）于 **YYYY-MM-DD HH:mm** 修改"
- 通知消息中也显示操作人信息
- 操作记录可在档案详情页的"修改记录"中查看

#### F11：取消授权

| 取消方 | 操作入口 | 操作后效果 |
|-------|---------|-----------|
| 被管理方 | "我的"页面 → 家庭关联 → 【取消授权】按钮 | 取消与该主账号的管理关系 |
| 被管理方 | 健康档案页面顶部提示条 → 点击 → 【取消授权】 | 同上 |
| 主账号 | 档案详情页 → 已关联状态 → 【解除关联】按钮 | 移除对该家人的管理关系 |

**取消后效果**：
- 管理关系解除，主账号无法再查看或管理该家人的信息
- 该家人从主账号的成员切换栏中消失（或恢复为"待邀请"状态）
- 被管理方的档案数据不受影响，仍保留合并后的完整数据
- 双方均收到通知

#### F12：成员切换栏关联状态展示

在主账号的健康档案页面成员切换栏中：

| 状态 | 展示方式 |
|------|---------|
| 已关联（对方已同意） | 家人头像右下角显示小绿点 + "已关联"标签 |
| 未关联（已创建档案但未邀请/对方未同意） | 家人头像旁显示"待邀请"灰色标签 |
| 本人 | 无标记，保持原样 |

#### F13：被管理方感知体验

被邀请方同意授权后，在自己的小程序中可看到以下变化：

**"我的"页面**：
- 新增"家庭关联"入口
- 点进去显示关联列表："XX 正在管理您的健康档案"
- 每条记录后有【取消授权】按钮

**健康档案页面**：
- 页面顶部显示提示条："您的档案正在被 **XX** 共同管理"
- 点击提示条可查看详情或取消授权

#### F14：通知机制

| 触发事件 | 通知对象 | 通知内容 | 通知渠道 |
|---------|---------|---------|---------|
| 被邀请方同意邀请 | 主账号 | "XX 已同意您的健康档案共管邀请" | 系统内消息 + 微信订阅消息 |
| 任一方取消授权 | 对方 | "XX 已取消健康档案共管关系" | 系统内消息 + 微信订阅消息 |
| 主账号为家人设置用药提醒 | 被管理的家人 | "XX 为您设置了用药提醒：[提醒内容]" | 系统内消息 + 微信订阅消息 |

#### F15：用药提醒通知对象

主账号为家人设置的用药提醒到时间后：
- **主账号**收到提醒通知
- **被管理的家人**也同时收到提醒通知
- 通知渠道：系统内消息 + 微信订阅消息（双渠道）

## 4. 非功能性需求

### 4.1 性能要求

- 邀请二维码图片生成时间 ≤ 2 秒
- 扫码后小程序打开到弹窗展示 ≤ 3 秒
- 档案合并操作 ≤ 1 秒
- 通知消息发送延迟 ≤ 5 秒

### 4.2 安全要求

- 邀请码使用随机 UUID，不可被猜测
- 邀请码单次使用后自动失效（防止被多人使用同一个邀请码）
- 授权操作需要用户主动确认，不可被自动完成
- 操作记录不可篡改

### 4.3 兼容性要求

- 微信小程序：微信版本 ≥ 7.0
- H5 端：主流浏览器（Chrome、Safari、微信内置浏览器）
- Flutter App：iOS 12+ / Android 6.0+

## 5. 业务规则与约束

| 编号 | 规则描述 |
|------|---------|
| BR01 | 必须先为家人创建健康档案记录，才能发起邀请 |
| BR02 | 主账号最多邀请 10 位家人 |
| BR03 | 一人最多同时被 3 个主账号管理 |
| BR04 | 邀请二维码/链接 24 小时有效 |
| BR05 | 同一家人在同一时间只能有一个有效邀请（未过期、未被使用） |

| BR08 | 双方都可编辑合并后的档案，最后修改覆盖之前的 |
| BR09 | 双方都可取消授权关系 |
| BR10 | 取消授权后，被管理方的档案数据保留不删除 |
| BR11 | 操作记录需包含操作人和操作时间 |
| BR12 | 用药提醒同时通知主账号和被管理家人 |

## 6. 权限设计

| 角色 | 权限说明 |
|------|----------|
| **主账号（发起邀请方）** | 为已关联家人创建/编辑/删除用药提醒、打卡任务、健康计划；查看/编辑家人的健康档案；发起邀请；解除关联关系 |
| **被管理方（被邀请方）** | 仅在家庭成员列表中看到"已关联"标记；在"我的"页面查看管理状态；取消授权关系；编辑自己的健康档案 |

**权限模型特点**：

- 这是一个**非对称权限**模型——主账号拥有完整的管理能力，被管理方几乎无感知
- 主账号对被管理家人的操作范围与对自己手动创建的家庭成员一致
- 被管理方无法反向查看主账号的任何信息

## 7. 异常处理与边界情况

| 场景 | 处理方式 |
|------|---------|
| 邀请码过期后扫码 | 提示"邀请已过期，请联系对方重新邀请" |
| 已关联用户再次扫同一主账号的邀请码 | 提示"您已与该用户建立关联" |
| 被管理方达到上限（3人）时扫码 | 提示"您已达到最大被管理人数限制（3人），无法接受新的邀请" |
| 主账号达到邀请上限（10人）时发起邀请 | 提示"您已达到最大邀请人数限制（10人），请先解除部分关联后再邀请" |
| 主账号为家人创建的档案与被邀请方自己的档案存在冲突 | 合并时以被邀请方本人数据为准，主账号数据仅补充缺失字段 |
| 取消授权后主账号再次邀请同一家人 | 允许重新发起邀请，被邀请方需重新同意 |
| 主账号被封禁/注销 | 自动解除所有管理关系，被管理方收到通知 |
| 被管理方被封禁/注销 | 自动解除与所有主账号的关系，主账号收到通知 |
| 网络异常导致合并失败 | 提示网络错误，保持原有数据不变，可重试 |
| 微信订阅消息发送失败 | 降级为仅系统内消息通知，不影响业务流程 |

| manager_user_id | Integer FK → User | 主账号（管理者）用户 ID |
| managed_user_id | Integer FK → User | 被管理方用户 ID |
| managed_member_id | Integer FK → FamilyMember | 关联的家庭成员记录 ID |
| status | Enum | 状态：pending / active / cancelled |
| created_at | DateTime | 创建时间 |
| cancelled_at | DateTime | 取消时间（可空） |
| cancelled_by | Integer FK → User | 取消操作人（可空） |

**邀请记录表（FamilyInvitation）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer PK | 主键 |
| invite_code | String UUID | 邀请码（唯一） |
