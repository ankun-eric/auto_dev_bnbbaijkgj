# 首页健康计划模块改造 产品需求文档（PRD）

## 1. 需求概述

### 1.1 背景与目的

bini-health 健康管理应用的首页"健康计划"模块及健康计划详情页，当前所有内容均为前端硬编码（mock 数据），未接入后端已有的 `HealthPlan`、`HealthTask` 等 API 接口。本次改造的目的是：

1. **将硬编码内容全面动态化**，前端对接后端 API，实现数据可配置、可管理
2. **建立三大健康计划分类体系**（用药提醒、健康打卡、自定义计划），覆盖用户的主要健康管理场景
3. **引入 AI 个性化计划生成能力**，结合用户健康档案自动生成专属健康计划
4. **实现推送通知提醒**，提升用户的健康管理依从性
5. **构建打卡与统计体系**，帮助用户量化健康管理效果

### 1.2 目标用户

- **普通用户**：使用 App / 小程序端进行日常健康管理、打卡、查看统计
- **后台管理员**：通过 admin-web 管理推荐计划模板、自定义计划模板分类、通用健康任务配置、用户打卡数据

### 1.3 核心价值

- 用户可获得**个性化**的健康管理方案（AI 生成 + 推荐模板 + 自定义）
- 通过**每日打卡 + 推送提醒**养成健康习惯
- 通过**数据统计**可视化健康管理效果，增强用户粘性
- 管理员可灵活配置内容，**无需开发人员介入**即可调整推荐计划和任务

### 2.2 功能详细描述

#### F01：健康计划分类体系

**三大分类定义：**

| 分类 | 图标 | 说明 |

| 用药提醒 | 💊 药品图标 | 设置每日用药时间和药品，到时推送提醒 |
| 健康打卡 | ✅ 打卡图标 | 自定义每日健康习惯打卡项，支持数值目标和提醒 |

**交互说明：**
- 健康计划详情页以三大分类为顶级导航，排版参考现有 AI 健康咨询页面的卡片风格
- 每个分类为独立的功能模块入口，点击进入对应的管理/列表页面

#### F02 & F03：用药提醒

**创建用药提醒流程：**

1. 用户点击"用药提醒"分类，进入用药提醒列表页
2. 点击"添加用药提醒"按钮
3. 填写药品信息表单：
   - **药品名称**（必填，文本输入）
   - **用药时段**（多选：早晨、中午、晚上、睡前）
   - **每个时段的具体时间**（时间选择器，如 08:00、12:30、18:00、22:00）
   - **用药剂量**（选填，文本输入，如"1 片""2 粒"）
   - **备注**（选填，如"饭后服用""需空腹"）
4. 保存后在列表页按时段分组展示

**用药提醒列表展示：**

```
┌─────────────────────────────────────┐
│ 用药提醒                    [+ 添加] │
├─────────────────────────────────────┤
│ 🌅 早晨 08:00                       │
│   · 阿莫西林 2 粒（饭后服用）        │
│   · 维生素 C 1 片                   │
├─────────────────────────────────────┤
│ 🌞 中午 12:30                       │
│   · 阿莫西林 2 粒（饭后服用）        │
├─────────────────────────────────────┤
│ 🌙 晚上 18:00                       │
│   · 降压药 1 片（饭前服用）          │
├─────────────────────────────────────┤
│ 😴 睡前 22:00                       │
│   · 褪黑素 1 粒                     │
└─────────────────────────────────────┘
```

**推送通知：**
- 到达设定的用药时间时，通过 **App 原生推送通知**（Flutter 端）和**微信订阅消息**（小程序端）提醒用户
- 通知内容示例："⏰ 用药提醒：现在该吃 阿莫西林 2 粒、维生素 C 1 片 了"
- 用户点击通知可直接跳转到用药提醒页面进行打卡确认
- H5 端不做推送通知（技术限制），仅做 App 内展示

**业务规则：**
- 同一药品可以出现在多个时段
- 已过时间的时段在当天标灰但不隐藏，用户仍可补打卡
- 每个用药项支持编辑和删除操作
- 用药提醒支持暂停/恢复功能（暂停后不发送推送通知）

#### F04 & F05：健康打卡

**创建健康打卡项流程：**

1. 用户点击"健康打卡"分类，进入打卡项列表页
2. 点击"添加打卡项"按钮
3. 填写打卡项表单：
   - **打卡名称**（必填，如"喝水""冥想""跑步"）
   - **目标值**（选填，数字输入，如 2000）
   - **目标单位**（选填，配合目标值使用，如"ml""分钟""步"）
   - **提醒时间**（选填，时间选择器）
   - **重复频率**（每天 / 工作日 / 自定义选择星期几）
4. 保存后在列表页展示

**打卡操作：**
- 无目标值的打卡项：只有"完成 / 未完成"两种状态，点击即完成打卡
- 有目标值的打卡项：用户打卡时可输入实际完成的数值（如实际喝水 1500ml），系统记录并与目标值对比
- 设置了提醒时间的打卡项：到时通过 App 推送 / 小程序订阅消息提醒

#### F06 ~ F10：自定义计划

**半结构化模板体系：**

预设 7 种模板分类（后台管理员可增删改）：

| 编号 | 模板名称 | 描述 | 预设每日任务示例 |
|------|----------|------|------------------|
| 1 | 饮食计划 | 控制饮食、营养搭配 | 记录三餐、控制热量摄入、多吃蔬菜 |
| 2 | 作息计划 | 早睡早起、午休规律 | 23:00 前入睡、7:00 起床、午休 30 分钟 |
| 3 | 康复计划 | 术后恢复、慢性病管理 | 按时服药、定期复查、适度运动 |
| 4 | 心理健康 | 冥想、减压、情绪管理 | 每日冥想 10 分钟、记录情绪日记、深呼吸练习 |
| 5 | 戒断计划 | 戒烟、戒酒、减少咖啡因 | 记录今日是否吸烟、替代活动打卡 |
| 6 | 健身计划 | 运动锻炼、体能提升 | 按运动类型自定义（跑步、力量训练、瑜伽等） |
| 7 | 其它 | 用户自由定义 | 无预设，用户完全自定义 |

**每种模板分类统一具备三种能力：**

1. **管理员推荐计划**：管理员在后台为每种模板分类配置推荐计划模板，用户可浏览并直接加入执行
2. **AI 定制计划**：用户点击"AI 为我定制"，AI 根据用户健康档案生成该分类下的个性化计划
3. **用户自定义计划**：用户在模板基础上自由创建自己的计划

**所有模板分类均使用相同的页面布局**——从上到下依次为：
1. "AI 为我定制"按钮区域
2. 管理员推荐计划列表
3. 我已创建/加入的计划列表
4. 创建新计划入口

**推荐计划详情与加入：**
- 每个推荐计划包含：计划名称、描述、适用人群标签、计划周期（天数）、每日任务列表
- 用户浏览推荐计划，点击查看详情，确认后加入执行
- 加入后该计划的每日任务自动出现在"今日待办"中

**AI 定制计划：**
- 用户在任一模板分类下点击"AI 为我定制"
- 系统读取用户健康档案，AI 根据分类主题和用户数据生成个性化计划
- 例如：在"饮食计划"下 AI 会根据用户体重、BMI、慢性病等生成专属饮食方案
- 用户查看 AI 生成结果，可确认加入或点击"重新生成"

**用户自定义计划：**
1. 用户点击"创建新计划"
2. 系统加载该模板的预设信息（名称、描述、建议任务）
3. 用户在此基础上自定义：
   - 修改计划名称和描述
   - 增删改每日任务列表
   - 设置计划周期（可选，无限期或指定天数）
4. 保存后计划生效，每日任务出现在"今日待办"中

**计划执行与打卡：**
- 加入/创建某个计划后，该计划的每日任务自动出现在"今日待办"中
- 用户按任务逐项打卡
- 有明确周期的计划展示进度条（当前第几天、总体完成率）

#### F11 & F12：AI 个性化计划生成

**自动生成触发条件：**
- 用户**首次完善健康档案**后（填写了身高、体重、慢性病史、过敏史等关键信息），系统自动调用 AI 生成一份个性化综合健康计划
- 生成结果以弹窗/页面形式展示给用户，用户可选择"采纳"或"稍后再看"

**手动重新生成：**
- 在健康计划页面提供"AI 为我生成计划"按钮（位于三大分类上方显著位置）
- 点击后 AI 重新读取用户最新健康档案，生成新的个性化计划
- 如果用户已有 AI 生成的计划，会提示"将替换当前 AI 生成的计划，是否继续？"

**AI 生成计划内容：**
- 根据用户档案中的健康数据（身高、体重、BMI、年龄、性别、慢性病史、过敏史、运动习惯等）综合分析
- 生成包含多个维度的综合健康计划：
  - 饮食建议与每日任务
  - 运动建议与每日任务
  - 作息建议
  - 针对慢性病的特殊提醒（如有）
- 每个任务附带具体的目标值和建议执行时间
- AI 生成的计划自动归入对应的模板分类中（饮食相关归入"饮食计划"、运动相关归入"健身计划"等）

- 首页健康计划模块改为"今日待办"汇总列表
- 汇总用户当天所有需要完成的任务，**不区分来源**（AI 生成的、推荐计划的、自定义的全部展示）
- 按**三大分类分组**展示：用药提醒 → 健康打卡 → 自定义计划
- **自定义计划组按具体模板类型再细分**，展示子分类小标题（如"饮食计划""健身计划""作息计划"等）

#### F14：打卡统计

**统计维度：**

1. **今日完成进度**：已完成 X / 总计 Y 项（含进度条/环形图）
2. **连续打卡天数**：展示用户连续完成每日任务的天数，中断则重新计算
3. **周完成率趋势图**：展示最近 7 天的每日完成率折线图/柱状图
4. **月完成率趋势图**：展示最近 30 天的完成率趋势
5. **各计划完成率排行**：按各个计划的完成率排序展示

**统计页面入口：**
- 首页今日待办区域点击"查看统计"可进入统计详情页
- 健康计划主页面也有统计入口

#### F15 ~ F18：后台管理

**F15：推荐计划模板管理**
- 管理员在 admin-web 中为**每种模板分类**管理推荐计划
- 管理结构：先选择模板分类（饮食计划/作息计划/健身计划等），再管理该分类下的推荐计划列表
- 支持操作：新增、编辑、删除、上下架（控制是否对用户可见）
- 计划模板字段：名称、描述、适用人群标签、计划周期（天数）、封面图（选填）、所属模板分类
- 每个计划下管理每日任务列表：任务名称、目标值、单位、排序

**F16：自定义计划模板分类管理**
- 管理员管理预设的模板分类（饮食计划、作息计划、康复计划、心理健康、戒断计划、健身计划、其它）
- 支持操作：新增分类、编辑分类名称和描述、删除分类、排序
- 每个分类可配置预设的建议任务列表（用户创建计划时作为参考）
- 每个分类可配置图标/Emoji

**F17：通用健康任务配置**
- 管理员配置首页"今日待办"中展示的通用/默认健康任务
- 适用场景：新用户注册后还没有创建任何计划时，首页展示管理员配置的默认任务作为引导
- 任务字段：名称、描述、目标值、单位、分类归属

**F18：用户打卡数据统计**
- 管理员可查看所有用户的打卡统计概览
- 支持按用户搜索、按时间范围筛选
- 展示：活跃用户数、平均完成率、各计划类型的参与人数、完成率排行

## 4. 非功能性需求

### 4.1 性能要求

- 首页今日待办数据加载时间 ≤ 1 秒
- 打卡操作响应时间 ≤ 500ms
- AI 计划生成时间 ≤ 10 秒（需展示加载动画）
- 推送通知延迟 ≤ 1 分钟（从设定时间到用户收到通知）

### 4.2 安全要求

- 用户只能查看和操作自己的健康计划与打卡数据
- 管理员操作需要管理员权限验证
- 用药数据属于敏感健康信息，传输和存储需加密处理

### 4.3 兼容性要求

- 三端一致体验：Flutter App、微信小程序、H5 功能和交互保持一致
- 推送通知仅覆盖 Flutter App（原生推送）和微信小程序（订阅消息）
- H5 端不做推送通知，仅做 App 内任务展示

## 5. 业务规则与约束

| 编号 | 规则描述 |
|------|----------|
| BR01 | 每个用户可以同时拥有多个不同分类的健康计划，数量不限 |
| BR02 | AI 生成的个性化计划每个用户同时只保留一份（重新生成会替换旧的） |
| BR03 | 推荐计划可以被多个用户同时加入，每个用户有独立的打卡进度 |
| BR04 | 已完成的打卡项当天不可重复打卡（有目标值的可以多次更新数值） |
| BR05 | 连续打卡天数的计算规则：当天完成至少 1 项任务即算打卡成功，连续中断 1 天即重置 |
| BR06 | 用药提醒的推送通知需要用户在 App 端授权通知权限；小程序端需要用户订阅模板消息 |
| BR07 | 用户删除某个计划后，该计划下的历史打卡数据保留（统计中仍可见），但不再生成每日任务 |
| BR08 | 管理员下架某个推荐计划后，已加入该计划的用户可以继续执行直到完成，新用户不可见 |
| BR09 | 自定义计划模板分类被管理员删除后，已基于该模板创建的用户计划不受影响 |
| BR10 | 新用户注册后，首页今日待办展示管理员配置的通用健康任务，引导用户创建自己的计划 |
| BR11 | 所有模板分类的能力统一：均支持管理员推荐 + AI 定制 + 用户自定义三种计划来源 |
| BR12 | 首页今日待办中，自定义计划按模板类型细分展示子标题；无待办的类型显示但置灰并标注"今日无待办" |
| BR13 | 所有进行中计划的任务全部汇总展示在首页今日待办中，不区分来源 |

## 7. 异常处理与边界情况

| 场景 | 处理方案 |
|------|----------|
| AI 生成计划失败（网络错误/模型异常） | 展示友好错误提示"AI 暂时开小差了，请稍后再试"，提供重试按钮 |
| 用户健康档案信息不完整时触发 AI 生成 | 提示用户"请先完善健康档案"，引导跳转到健康档案编辑页 |
| 推送通知权限未授权 | 创建用药提醒时提示"开启通知权限后才能按时提醒您"，引导到系统设置页 |
| 小程序订阅消息额度用完 | 提示用户重新授权订阅消息 |
| 用户当天无任何待办任务 | 首页展示"今天没有待办任务 🎉"+ 引导"去创建新的健康计划" |
| 网络异常导致打卡失败 | 本地暂存打卡记录，网络恢复后自动同步 |
| 跨天打卡（如 23:59 打卡到 00:01 提交） | 以用户点击打卡时的日期为准 |
| 管理员删除模板分类时存在用户关联数据 | 提示"该分类下有 N 个用户计划正在使用，删除后不影响已创建的计划，确认删除？" |
| 某模板分类下无推荐计划 | 推荐计划区域展示"暂无推荐计划"，用户仍可通过 AI 定制或自定义创建 |
| 用户未完善健康档案时点击"AI 为我定制" | 提示"请先完善健康档案以获得更精准的定制方案"，引导跳转到健康档案编辑页 |

## 8. 补充说明

### 8.1 与现有功能的关联

- **健康档案**：AI 生成计划需要读取用户的健康档案数据，确保健康档案模块的数据接口可供调用
- **AI 健康咨询**：健康计划页面的 UI 排版参考 AI 健康咨询页面的卡片风格，保持视觉一致性
- **后端 API**：后端已有 `HealthPlan` 和 `HealthTask` 数据表及部分 API，需扩展以支持三大分类体系、用药提醒、推送通知等新功能
- **首页**：原有的硬编码"每日健康任务"模块需完全替换为动态的"今日待办"模块

### 8.2 数据模型扩展建议

需要在现有数据库基础上扩展以下核心表：

- **medication_reminder**：用药提醒表（用户 ID、药品名称、剂量、时段、提醒时间、状态等）
- **health_checkin_item**：健康打卡项配置表（用户 ID、名称、目标值、单位、提醒时间、重复频率等）
- **health_checkin_record**：打卡记录表（用户 ID、打卡项 ID、日期、实际完成值、打卡时间等）
- **plan_template_category**：自定义计划模板分类表（名称、描述、图标、排序，管理员可配置）
- **recommended_plan**：推荐计划模板表（管理员配置，关联模板分类）
- **recommended_plan_task**：推荐计划每日任务表（关联推荐计划）
- **user_plan**：用户计划表（用户 ID、来源类型[推荐/AI/自定义]、所属模板分类、计划名称、周期等）
- **user_plan_task**：用户计划每日任务表（关联用户计划）
- **user_plan_task_record**：用户计划任务打卡记录表（用户 ID、任务 ID、日期、完成值、打卡时间）
- **push_notification_config**：推送通知配置表（设备 token、订阅状态等）

### 8.3 推送通知技术方案

| 端 | 技术方案 |
|----|----------|
| Flutter App | Firebase Cloud Messaging (FCM) 或极光推送等第三方推送服务 |
| 微信小程序 | 微信订阅消息 API（需用户主动授权订阅） |
| H5 | 不做推送通知，仅 App 内展示 |

后端需新增定时任务服务（如 Celery / APScheduler），在到达用药/打卡提醒时间时触发推送。
