# bini-health 健康打卡全部失败 Bug 修复方案文档

## 1. Bug 发生背景

### 1.1 项目概述

bini-health 是一套健康管理系统，支持 H5 网页端、Flutter App、微信小程序和后台管理端（Admin-web）。后端基于 FastAPI + SQLAlchemy + MySQL 构建。

### 1.2 涉及功能模块

- **健康习惯打卡**：用户自定义健康习惯（饮水、运动等），每日打卡记录
- **用药提醒打卡**：到时间吃药后点击确认打卡
- **健康计划任务打卡**：医生或系统分配的健康计划任务，完成后打卡
- **今日待办打卡**：汇总所有待办打卡项，统一打卡入口
- **打卡送积分**（最新上线功能）：上述所有打卡行为均集成积分发放逻辑

### 1.3 发现时间与发现方式

- **发现时间**：2026 年 4 月 11 日，"打卡送积分"功能上线后立即出现
- **发现方式**：用户反馈所有打卡操作均提示"打卡失败"

---

## 2. Bug 描述

### 2.1 错误现象

"打卡送积分"功能上线后，系统中**所有类型的打卡操作**（健康习惯、用药提醒、健康计划任务、今日待办）在**所有客户端**（H5、Flutter App、微信小程序）上对**所有用户**均提示"打卡失败"，无更详细的错误信息。

打卡功能在此次更新前一直正常运作，除"打卡送积分"外近期无其他代码改动。

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 登录任意用户账号，进入健康打卡页面 | 正常显示打卡项列表 | 页面正常显示 |
| 2 | 点击任意一项健康习惯的"打卡"按钮 | 打卡成功，显示积分获得反馈 | 提示"打卡失败" |
| 3 | 切换到用药提醒页面，点击"确认服药"打卡 | 打卡成功 | 提示"打卡失败" |
| 4 | 切换到健康计划任务页面，点击任务打卡 | 打卡成功 | 提示"打卡失败" |
| 5 | 在今日待办页面点击任意待办项打卡 | 打卡成功 | 提示"打卡失败" |
| 6 | 换一台设备/换一个端（H5/App/小程序）重复上述操作 | 打卡成功 | 同样提示"打卡失败" |

### 2.3 影响范围

| 维度 | 影响详情 |
|------|----------|
| **打卡类型** | 全部：健康习惯打卡、用药提醒打卡、健康计划任务打卡、今日待办打卡 |
| **客户端** | 全部：H5 网页端、Flutter App、微信小程序 |
| **用户范围** | 所有用户，无一例外 |
| **业务影响** | 核心打卡功能完全不可用，用户无法记录任何健康行为，健康数据中断 |

## 3. 预期正确效果

修复后，系统应恢复以下正确行为：

1. **打卡功能正常**：所有类型的打卡（健康习惯、用药提醒、健康计划任务、今日待办）均能正常完成，不再报错
2. **积分发放正常**：
   - 打卡成功后，若积分规则已启用，自动按配置的"每次积分值"发放积分
   - 达到"每日上限"后，打卡仍然成功，但不再发放额外积分，并提示已达上限
   - 若积分规则未启用，打卡正常完成，不发放积分，不报错
3. **积分发放容错**：积分发放过程中若出现任何异常（如配置缺失、数据库写入失败等），不应影响打卡本身的成功，打卡数据应正常记录
4. **前端反馈正确**：打卡成功后，各端正确显示积分获得反馈（Toast/SnackBar），积分进度条正确更新

## 4. 根因分析

### 4.1 变更范围

本次"打卡送积分"更新对以下 4 个打卡接口均集成了积分发放逻辑：

| 接口路径 | 方法 | 打卡类型 |
|----------|------|----------|
| `/api/health-plan/medications/{id}/checkin` | POST | 用药提醒打卡 |
| `/api/health-plan/checkin-items/{id}/checkin` | POST | 健康习惯打卡 |
| `/api/health-plan/user-plans/{id}/tasks/{id}/checkin` | POST | 健康计划任务打卡 |
| `/api/health-plan/today-todos/{id}/check` | POST | 今日待办打卡 |

每个接口在原有打卡逻辑之后，新增了以下代码：

```python
from app.services.checkin_points_service import award_checkin_points
points_result = await award_checkin_points(db, current_user, "打卡类型")
await db.flush()
```

### 4.2 疑似根因（需代码验证）

由于所有打卡类型、所有端、所有用户同时失败，且唯一的变更就是积分集成，根因锁定在 `checkin_points_service.py` 的 `award_checkin_points` 函数。以下是最可能的三个原因：

**原因 A：`PointsType.checkin` 枚举值未在数据库中同步**

代码中 `PointsType` 枚举新增了 `checkin` 值，但如果数据库的枚举列（MySQL ENUM 类型）未通过 migration 更新，插入 `PointsRecord` 时会因枚举值不匹配而抛出数据库错误，导致整个事务回滚（包括打卡记录本身）。

**原因 B：`award_checkin_points` 函数抛出未捕获异常**

`award_checkin_points` 函数内部有多步操作（读取配置、查询今日积分、创建积分记录、更新用户积分），任何一步异常都会向上抛出，导致打卡接口返回 500 错误。当前打卡接口代码中**没有对积分发放做 try-except 保护**，积分异常会直接中断打卡流程。

**原因 C：`SystemConfig` 配置读取异常**

如果 `SystemConfig` 表中的打卡积分配置行不存在或数据格式异常（如 `config_value` 为 NULL），`int()` 转换可能失败。

### 4.3 修复方向

1. **检查数据库 migration**：确认 `PointsType` 枚举的 `checkin` 值已正确同步到 MySQL 数据库的 ENUM 列
2. **增加容错保护**：在所有打卡接口中，对 `award_checkin_points` 调用包裹 try-except，确保积分发放失败不影响打卡本身
3. **验证配置数据**：确认 `SystemConfig` 表中三条打卡积分配置记录存在且格式正确
4. **增加日志**：在积分发放失败时记录详细错误日志，便于后续排查

## 6. 修复方案

### 6.1 数据库层修复

- 检查并确认 MySQL 数据库中 `points_records` 表的 `type` 列 ENUM 定义已包含 `checkin` 值
- 如未包含，执行 ALTER TABLE 添加枚举值
- 检查 `system_configs` 表中打卡积分三条配置是否存在且值合法

### 6.2 后端代码修复

- 在 `health_plan_v2.py` 的 4 个打卡接口中，对 `award_checkin_points` 调用添加 try-except 保护
- 积分发放异常时：打卡本身仍然成功，积分默认返回 `{"points_earned": 0, "points_limit_reached": False}`，并记录错误日志
- 在 `checkin_points_service.py` 中增强防御性编程，对配置值转换添加异常处理

### 6.3 验证要点

- 修复后逐一测试 4 个打卡接口，确认打卡成功
- 验证积分正常发放并正确计算每日上限
- 模拟积分服务异常，确认打卡不受影响
- 在 H5、Flutter App、微信小程序三端验证打卡功能恢复正常
