## 1. Bug 发生背景

### 1.1 项目概述

bini-health 是面向 C 端的 AI 健康助手系统，核心包含：

- **后端**：FastAPI + SQLAlchemy + MySQL
- **H5 端**：Next.js + React（移动端 H5，主要承载 AI 对话、健康档案、用药计划等核心交互）
- **小程序端**：微信小程序原生

本次 Bug 集中在 H5 端的 **AI 首页（`ai-home`）** 这一核心入口页面，涉及顶部布局密度、消息铃铛角标、铃铛弹层中的"用药提醒"数据这三处。

### 1.2 涉及功能模块

## 2. Bug 描述

### 2.1 错误现象

#### 现象 1：AI 首页顶部布局太松，整体空间不好看

- **页面**：AI 首页 `/ai-home`
- **现象**：顶部 LOGO + "欢迎您"问候 + 用户头像 + 四宫格快捷入口（健康计划 / 今日用药 / 病例管理 / 健康档案）这块区域**整体偏下、留白过多**，纵向占据约 240px，信息密度低，视觉上"空荡"。
- **根因初判**：欢迎区上下 `padding` 偏大（16/16），头像偏大（56），主标题字号偏大（18px），宫格图标偏大（44×44）且行距偏大（14）。

#### 现象 2：右上角消息铃铛的红色数字角标消失

- **页面**：AI 首页 `/ai-home` 顶栏右上角
- **现象**：用户**之前**能在铃铛右上角看到红色数字角标（例如显示未读条数 `3`），但**现在该角标完全消失**，只剩一个铃铛图标。
- **根因初判**：在上一期"AI 首页最终版优化"（PRD-AI-HOME-OPTIM-FINAL-V1）中，铃铛按钮组件的 `badgeCount` 入参被改为"传入任意值都不会影响铃铛视觉"，红点提示被整体迁移到了汉堡菜单按钮上，导致铃铛上的红色数字角标被废弃。这是上一期变更的**副作用**。

#### 现象 3：点击铃铛后弹出的抽屉里"用药提醒"数据为空

- **页面**：AI 首页点击右上角铃铛 → 弹出"提醒"抽屉
- **现象**：抽屉中"用药提醒"分区**标题仍在**，但**下方一直显示"暂无用药提醒"**，即使当前咨询人**今天确实有用药计划**（在健康档案 Hero 卡的「今日用药 · N ›」中能看到有 N 条数据）。
- **根因初判**（结合用户补充信息：「需要获取，健康档案，档案人，今日用药，在里面内容里获取」）：
  - 铃铛抽屉**当前调用的是旧接口** `/api/medication-reminder/today`，该接口的数据源、状态过滤规则、咨询人切换逻辑与健康档案 Hero「今日用药」的**同源接口** `/api/medication/today` **并不一致**，存在以下任一情况：
    - 旧接口未跟随"咨询人切换"刷新，导致返回他人或空数据
    - 旧接口的"今日"过滤条件、状态机与新版用药计划体系不匹配，过滤后剩余为 0 条
    - 旧接口未与 `MedicationLog` / `MedicationPlan` 的最新数据模型对齐
  - 而**健康档案 Hero 与首页四宫格「今日用药」按钮均已经统一使用** `/api/medication/today` 同源接口（见接口模块注释：「ai-home 今日用药按钮红点 / 健康档案 Hero 今日用药徽标 / 今日用药抽屉内容渲染」），唯独**铃铛抽屉这一处遗漏了未切换**，造成"健康档案能看到、铃铛抽屉看不到"的数据割裂现象。

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 用户登录 H5 后进入 AI 首页 `/ai-home` | 顶部欢迎区 + 四宫格紧凑、视觉舒适 | 整体偏下、留白过多、信息密度低（**现象 1**） |
| 2 | 系统中存在未读提醒/未读消息 | 铃铛右上角出现红色数字角标显示具体未读数 | 铃铛右上角无任何角标（**现象 2**） |
| 3 | 切换到一个今天有用药计划的咨询人，进入健康档案 Hero | Hero 卡显示「今日用药 · N ›」（N > 0） | 正常显示，例如 N=3（作为对照参照） |
| 4 | 返回 AI 首页，点击右上角铃铛 → 打开提醒抽屉 | "用药提醒"分区列出该咨询人今天 3 条用药 | 标题在，下方显示"暂无用药提醒"（**现象 3**） |

### 2.3 影响范围

- **功能影响**：
  - AI 首页是用户每次进入 App 的核心入口，顶部布局直接决定**第一眼观感**与功能可达性
  - 铃铛红点角标承担"是否有未读提醒"的关键提示作用，消失后用户**无法在首页一眼感知**是否需要查看提醒
  - 铃铛抽屉"用药提醒"为空，会让用户误以为今天没有需要服用的药物，**存在潜在用药风险**
- **用户影响**：所有使用 AI 首页的用户（H5 端），尤其是有用药计划的中老年慢病用户
- **数据影响**：仅展示层影响，未涉及数据库数据损坏

## 3. 预期正确效果

### 3.1 顶部布局优化（采用方案 B · 中度压缩）

按本次产品确认的方案 B 调整 AI 首页顶部，整体压缩约 55px 纵向空间：

| 维度 | 修复后目标值（方案 B） |
|---|---|
| 欢迎区上下 padding | 6 / 4 |
| 头像尺寸 | 44 × 44 |
| 主标题字号（欢迎语） | 13px |
| 四宫格图标尺寸 | 40 × 40 |
| 四宫格行距 | 8 |
| 顶部整段高度 | ≈ 185px |

> **注**：产品同学保留了"一会再出几个设计图选"的可能。本次修复**先按方案 B 实施**；若后续产品基于实际效果想再微调（如"头像保留 56"等），属于增量微调，复用同一组样式变量直接改值即可，不影响本期主修复。

设计示意：

```
┌────────────────────────────────────────────┐
│  [Logo]   早上好，张先生            🔔 ≡  │  ← 顶栏紧凑
│  [头像44] 今天也要好好照顾自己哦         │  ← padding 6/4
│                                            │
│  💊         🩺         👨‍👩‍👧         📋  │
│ 今日用药   我的设备   家庭成员   健康档案  │  ← 图标 40，行距 8
└────────────────────────────────────────────┘
```

### 3.2 铃铛红色数字角标恢复

- 在铃铛按钮组件中**恢复 `badgeCount` 的视觉效果**：
  - 当 `badgeCount > 0` 时，在铃铛右上角渲染**红色圆形角标**，显示具体数字
  - 当 `badgeCount > 99` 时，显示 `99+`
  - 当 `badgeCount = 0` 或未传时，不渲染角标
- **样式**：背景 `#FF3B30`（iOS 风险红），白色文字，字号 10px，圆角 50%，最小宽高 16×16，绝对定位于铃铛右上角并轻微外溢以保证视觉醒目
- **数据源**：保持现有"未读提醒数"聚合口径不变，即包含【用药提醒未完成数 + 系统通知未读数】之和（与原版本一致）
- **汉堡菜单红点**：现在汉堡菜单上的红点提示**保留不变**（不影响），形成"铃铛带数字 + 汉堡带红点"双通道提醒，与用户选项 2-A 一致

### 3.3 铃铛抽屉「用药提醒」数据修复（统一到同源接口）

将铃铛抽屉的用药提醒**改造为与健康档案 Hero「今日用药」完全同源**：

- **前端改造**：铃铛抽屉组件 `ReminderDrawer` 拉取用药数据的接口由：
  - ❌ 旧：`/api/medication-reminder/today?consultant_id=xxx`
  - ✅ 新：`/api/medication/today?consultant_id=xxx`
- **数据展示口径**：与健康档案 Hero「今日用药」**完全一致**——即"档案人今日所有用药项"，包含每一条药品的：
  - 药品名称
  - 服用时间（早 / 中 / 晚 / 睡前等时段）
  - 剂量
  - 服用状态（待服 / 已服 / 已跳过）
- **咨询人切换联动**：抽屉打开时**必须**携带当前已选咨询人的 `consultant_id`，切换咨询人后再次打开抽屉时数据要**全量随切换**（与健康档案 Hero 行为一致）
- **空态判断**：仅当同源接口返回 `items.length === 0` 才显示"暂无用药提醒"
- **铃铛红点角标联动**：同源接口返回的 `hasTodayMedication = true` 且存在"未完成"状态的项时，计入铃铛红色数字角标的"用药提醒未完成数"，确保**铃铛红点数字 = 抽屉实际可见的未完成项数**，做到所见即所得

### 3.4 修复后验收口径

修复完毕、上线后，从产品角度需满足以下验收点：

1. ✅ AI 首页打开后，顶部欢迎区 + 四宫格整段视觉高度 ≈ 185px，比原版本明显紧凑
2. ✅ 当存在未读提醒时，铃铛右上角显示红色数字角标（如 `3`）；无未读时不显示
3. ✅ 同一咨询人在健康档案 Hero 看到的「今日用药 · N」与在铃铛抽屉里看到的"用药提醒"条数**完全一致**
4. ✅ 切换咨询人后，铃铛抽屉的用药提醒**同步切换**为新咨询人的今日用药
5. ✅ 铃铛红色数字角标里的"未完成数"与抽屉内"待服"项数量**完全一致**

## 4. 补充说明

### 4.1 改动范围一览（仅说明改动点，不写代码）

| # | 改动对象（功能名） | 改动类型 | 关键改动点 |
|---|---|---|---|
| 1 | AI 首页主页面（欢迎区 + 四宫格容器） | **样式** | 调整 padding / 头像尺寸 / 字号 / 宫格图标与行距，使其符合方案 B 的数值 |
| 2 | 铃铛按钮组件 `ReminderBellButton` | **逻辑 + 样式** | 恢复 `badgeCount` 的视觉渲染，新增红色数字角标 DOM（含 `99+` 兼容） |
| 3 | 铃铛抽屉组件 `ReminderDrawer` | **接口切换** | 将用药提醒数据源由旧的 `/api/medication-reminder/today` 切换到同源接口 `/api/medication/today` |
| 4 | 铃铛红点未读数聚合逻辑（首页拉取处） | **数据对齐** | 用药未完成数改用同源接口的 `items` 中"待服状态"项的条数，与抽屉口径完全一致 |
| 5 | 旧接口 `/api/medication-reminder/today` | **降级保留 / 待清理** | 本期保留不动，避免影响其他端（小程序）的潜在调用；后续如确认无其他依赖，可在下一次清理期统一下线 |

### 4.2 关键设计决策（已与产品对齐）

- **采用方案 B 中度压缩**：顶部约节省 55px，对中老年用户字号仍保持在可读范围（13px 主标题、40 宫格图标），不会过密
- **铃铛 + 汉堡双通道提醒**：用户明确选择 `2-A`（完全恢复成之前的红色数字），同时不要求拆除汉堡红点，因此并存
- **用药提醒一律以"健康档案 → 档案人 → 今日用药"为唯一真相源**：与用户补充信息完全一致，从架构层面解决"多个入口数据割裂"的隐患
- **统一为同源接口**：本次修复彻底封堵"两个接口数据不一致"这一类问题的复发可能

### 4.3 排查与验证手段（修复期内 AI 自行执行，无需用户参与）

- 后端：直接读取数据库与服务端日志，确认 `/api/medication/today` 在指定 `consultant_id` 下返回的 `items` 列表内容
- 前端：在铃铛抽屉打开时检查请求 URL、`consultant_id` 参数、响应体；对比健康档案 Hero 同时刻的请求结果
- 视觉：修改后对 AI 首页顶部进行像素级量测，确保整段高度落在 ≈ 185px

如需用户配合验证，仅会请用户在**浏览器按 F12 打开控制台**查看页面是否还有红色错误（不会请用户去看任何服务端日志）。

### 4.4 上线方式

- 所有改动作为**一期工程一次性全量修复并上线**到小白 AI 云服务器
- 不分版本、不分批次
- 上线后立即按本文 §3.4 的验收口径回归
