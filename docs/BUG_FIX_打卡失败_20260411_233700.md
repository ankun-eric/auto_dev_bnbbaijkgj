# bini-health 健康打卡失败 Bug 修复方案文档

## 1. Bug 发生背景

### 1.1 项目概述

bini-health 是一个健康管理系统，包含健康习惯打卡、用药提醒打卡、健康计划任务打卡等核心功能，覆盖 H5 网页端、Flutter App、微信小程序多端。

### 1.2 涉及功能模块

- 健康习惯打卡（`/api/health-plan/checkin-items/{id}/checkin`）
- 用药提醒打卡（`/api/health-plan/medications/{id}/checkin`）
- 健康计划任务打卡（`/api/health-plan/user-plans/{id}/tasks/{id}/checkin`）
- 今日待办打卡（`/api/health-plan/today-todos/{id}/check`）
- 打卡积分发放服务（`checkin_points_service.py`）

### 1.3 发现时间与发现方式

"打卡送积分"功能上线后立即发现，所有用户在所有端上打卡均失败。

## 2. Bug 描述

### 2.1 错误现象

所有类型的健康打卡（健康习惯、用药提醒、健康计划任务）在所有端（H5、Flutter App、微信小程序）上均提示"打卡失败"，无更多详细错误信息。

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 用户登录任意端（H5/App/小程序） | 正常登录 | 正常登录 |
| 2 | 进入健康打卡页面（任意类型） | 页面正常显示 | 页面正常显示 |
| 3 | 点击打卡按钮执行打卡操作 | 打卡成功，显示积分奖励反馈 | 提示"打卡失败" |

### 2.3 影响范围

- **影响用户**：所有用户（100%）
- **影响端**：H5 网页端、Flutter App、微信小程序（全部）
- **影响功能**：健康习惯打卡、用药提醒打卡、健康计划任务打卡、今日待办打卡（全部打卡功能瘫痪）
- **影响严重度**：P0 — 核心功能完全不可用

## 3. 根因分析

### 3.1 直接原因

"打卡送积分"功能上线后，4 个打卡接口均集成了 `checkin_points_service.py` 的积分发放调用。积分发放逻辑抛出异常时，没有做异常保护（try-except），导致整个打卡事务被回滚，打卡操作失败。

### 3.2 根本原因

1. **数据库枚举未同步**：`PointsType` 枚举新增了 `checkin` 值，但 MySQL 数据库的 ENUM 列可能未执行 ALTER TABLE 同步更新，导致写入积分记录时数据库报错
2. **缺乏异常容错**：打卡接口中调用积分发放服务时，没有将积分逻辑包裹在 try-except 中，积分异常直接导致打卡主流程失败
3. **积分配置依赖未验证**：积分发放服务读取 `SystemConfig` 表中的三条打卡积分配置（`checkin_points_enabled`、`checkin_points_per_action`、`checkin_points_daily_limit`），如果配置记录不存在或值异常，会导致服务层报错

## 4. 修复方案

### 4.1 修复策略

| 序号 | 修复项 | 说明 |
|------|--------|------|
| 1 | 数据库枚举同步 | 确认 `PointsType` 枚举的 `checkin` 值已在 MySQL ENUM 列中正确添加，必要时执行 ALTER TABLE |
| 2 | 积分发放异常容错 | 在 4 个打卡接口中，将积分发放调用包裹在 try-except 中，确保积分异常**永远不会影响打卡本身** |
| 3 | 配置数据完整性验证 | 积分发放服务在读取配置时增加默认值保护，配置不存在时使用安全默认值（enabled=False, per_action=0, daily_limit=0） |
| 4 | 日志增强 | 积分发放异常时记录详细日志，便于后续排查 |

### 4.2 涉及修改文件

| 文件 | 修改内容 |
|------|----------|
| `backend/app/services/checkin_points_service.py` | 增加配置读取默认值保护，增强异常日志 |
| `backend/app/api/health_plan_v2.py` | 4 个打卡接口的积分发放调用增加 try-except 容错 |
| `backend/app/models/models.py` | 确认 PointsType 枚举定义正确 |
| 数据库迁移脚本 | 确保 ENUM 列包含 `checkin` 值 |

### 4.3 验证方案

1. 修复后在所有端（H5、App、小程序）执行所有类型打卡，确认打卡成功
2. 验证打卡成功后积分正确发放
3. 模拟积分服务异常场景，确认打卡本身不受影响
4. 运行已有的 10 个打卡积分自动化测试用例
