## 1. 文档概述

### 1.1 需求背景

**业务铁律**：一个注册用户只能有一条"本人健康档案"（`family_member_id` 标记为本人）。当前通过 `family_member_id IS NULL` 标识，同 user_id 下该条件的记录有且只能有一条。

**现状问题**：数据库中已出现脏数据（如用户 18 有 2 条本人档案），原因：
1. `health_profiles` 表没有唯一约束保证 `(user_id, family_member_id)` 唯一性，且 MySQL UNIQUE 索引不把多个 `NULL` 视为相等
2. 多个接口在查不到本人档案时采用"查不到就新建"的兜底逻辑

### 1.2 修复方案总览

| 层面 | 措施 |
|------|------|
| 数据库 | `family_member_id` 由 `NULL` 改为 `0` + 联合唯一索引 `(user_id, family_member_id)` |
| 代码 - 接口 | 5 处"查不到就新建"改为"查不到就报错"（其中 guide-status 已删除） |
| 代码 - 查询 | `scalar_one_or_none()` 改为 `scalars().first()` 防御重复 |
| 代码 - 替换 | 全局 `NULL → 0` 替换所有本人档案判定逻辑 |
| 前端 | 适配新增的 404 / 400 错误码展示对应文案 |
| 数据 | 脏数据清理 + 执行顺序保障 |

## 3. 数据模型设计（DDL）

### 3.1 health_profiles 表变更

**变更前**:
```sql
family_member_id INT DEFAULT NULL  -- ForeignKey → family_members.id, nullable=True
```

**变更后**:
```sql
family_member_id BIGINT NOT NULL DEFAULT 0  -- 0 表示本人档案，>0 表示关联 FamilyMember
```

**DDL 执行顺序（必须按序执行）**:

```sql
-- Step 1: 脏数据清理（先删后改，保证数据一致性）
-- 见 5.3 节

-- Step 2: 将现存本人档案的 family_member_id 从 NULL 改为 0
UPDATE health_profiles SET family_member_id = 0 WHERE family_member_id IS NULL;

-- Step 3: 修改字段定义
ALTER TABLE health_profiles 
  MODIFY COLUMN family_member_id BIGINT NOT NULL DEFAULT 0;

-- Step 4: 添加联合唯一索引
-- 注意：执行前必须确保 Step 2 已完成，否则唯一索引创建会因重复 NULL 失败
ALTER TABLE health_profiles 
  ADD UNIQUE INDEX idx_user_family_member (user_id, family_member_id);
```

**设计说明**:
- 使用 `BIGINT` 而非原 `Integer`：与 `family_members.id`（BIGINT）类型一致，避免外键类型不匹配
- `NOT NULL DEFAULT 0`：彻底消除 NULL 语义，使 MySQL UNIQUE 索引正确生效
- 联合唯一索引 `(user_id, family_member_id)`：保证每个用户下每条 family_member 关联唯一

### 3.2 ORM 模型变更

**文件**: `backend/app/models/models.py`，`HealthProfile` 类

```python
# 变更前
family_member_id = mapped_column(Integer, ForeignKey("family_members.id"), nullable=True)

# 变更后
family_member_id = mapped_column(BigInteger, ForeignKey("family_members.id"), nullable=False, default=0)
```

> 注意：需同步 import `BigInteger` from sqlalchemy。

## 5. 业务流程说明

### 5.1 全局 NULL → 0 替换规则

| `ChatSession.family_member_id.is_(None)` | `ChatSession.family_member_id == 0` |

### 5.2 查询侧防御改造

**原则**: 所有查询本人档案的语句，将 `scalar_one_or_none()` 改为 `scalars().first()`。

`scalar_one_or_none()` 在有多条匹配记录时会抛出 `MultipleResultsFound` 异常（SQLAlchemy 2.x 行为），导致 500 错误。改为 `scalars().first()` 后即使存在脏数据也只会静默取第一条，不会中断业务流程。

**需要改造的位置**（仅 HealthProfile 相关）:

| 文件 | 行号 | 说明 |
|------|------|------|
| `health_profile.py` | 61 | `GET /api/health/profile` |
| `health_profile.py` | 79 | `POST /api/health/profile` 创建查重 |
| `health_profile.py` | 113 | `PUT /api/health/profile` |
| `health_profile.py` | 519 | 上传体检报告 |
| `health_profile_self.py` | 194 | `_get_self_profile()` |
| `admin.py` | 2136 | 管理后台导出 |

**注意**: `family_management.py` 第 310、382、542 行已经使用 `scalars().first()`，无需改造。

### 5.3 脏数据清理策略

**原则**: 在数据库加唯一索引之前必须完成清理，否则 ALTER 会报错。

**查询脏数据**:
```sql
SELECT user_id, COUNT(*) AS cnt
FROM health_profiles
WHERE family_member_id IS NULL
GROUP BY user_id
HAVING COUNT(*) > 1;
```

**清理策略**: 保留最早创建的（`MIN(id)`），删除多余的：
```sql
DELETE hp FROM health_profiles hp
INNER JOIN (
    SELECT user_id, MIN(id) AS keep_id
    FROM health_profiles
    WHERE family_member_id IS NULL
    GROUP BY user_id
    HAVING COUNT(*) > 1
) keeper ON hp.user_id = keeper.user_id
WHERE hp.family_member_id IS NULL
  AND hp.id != keeper.keep_id;
```

**执行后验证**:
```sql
-- 确认无重复
SELECT user_id, COUNT(*) AS cnt
FROM health_profiles
WHERE family_member_id IS NULL
GROUP BY user_id
HAVING COUNT(*) > 1;
-- 期望结果：Empty set
```

## 6. 执行顺序与回滚方案

### 6.1 执行顺序

```
第 1 步: 脏数据清理（SQL）           ← 必须在加唯一索引前
第 2 步: 查询侧改造                  ← scalar_one_or_none → scalars().first
第 3 步: 4 处接口改造                ← 查不到就报错（含前端适配）
第 4 步: auth.py bug 修复            ← ensure_self_health_profile
第 5 步: 全局 NULL → 0 替换          ← 所有 family_member_id.is_(None)
第 6 步: 数据库加唯一索引             ← DDL: UPDATE + ALTER + ADD INDEX
第 7 步: 前端适配                    ← 404/400 错误文案
第 8 步: 测试                        ← pytest + 手工回归
```

**依赖关系**:
- 第 1 步必须最先执行（否则第 6 步 DDL 失败）
- 第 2-5 步可并行执行（无依赖），但建议按顺序逐步验证
- 第 6 步依赖第 1 步完成后执行
- 第 7 步可与第 2-5 步并行

### 6.2 回滚方案

如需回滚到 NULL 语义（不推荐）：

```sql
-- 1. 删除唯一索引
ALTER TABLE health_profiles DROP INDEX idx_user_family_member;

-- 2. 恢复字段为可空
ALTER TABLE health_profiles MODIFY COLUMN family_member_id INT DEFAULT NULL;

-- 3. 将 0 恢复为 NULL
UPDATE health_profiles SET family_member_id = NULL WHERE family_member_id = 0;
```

回滚后需同步恢复代码中的 `== 0` → `is_(None)` 替换。

### 6.3 风险点与注意事项

| 风险点 | 缓解措施 |
|--------|----------|
| 新增 profile 时 `family_member_id=0` 但无对应 FamilyMember 记录 | `ensure_self_family_member` 在注册流程中先于 `ensure_self_health_profile` 执行 |
| MedicationReminder 表也有 `family_member_id IS NULL` 需要同步 | P1 优先级统一替换，且该表无需唯一索引 |
| 外键约束：`family_member_id=0` 不指向任何真实 FamilyMember | `family_members` 表 id 从 1 开始自增，0 作为特殊哨兵值，但需确认 MySQL 外键 ON DELETE/UPDATE 行为不会报错。建议在 DDL 中暂时移除外键约束或使用 `SET FOREIGN_KEY_CHECKS=0` |

> **外键处理建议**: 在执行 `MODIFY COLUMN` 时，如果 MySQL 严格检查外键引用（0 不指向任何 family_members 行），可使用 `SET FOREIGN_KEY_CHECKS=0` 临时关闭，修改完成后再恢复。生产环境若使用腾讯云 MySQL，需确认该参数可设置。
