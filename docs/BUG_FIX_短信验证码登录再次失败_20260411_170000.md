# 宾尼小康（bini-health）短信验证码登录再次失败 — Bug 修复方案文档

## 1. Bug 发生背景

### 1.1 项目概述

**宾尼小康（bini-health）** 是一个 AI 健康管理项目，包含 H5 网页端、微信小程序端、Flutter App 端和管理后台。项目使用 FastAPI + MySQL 8.0 作为后端技术栈，前端 H5 使用 Next.js + Ant Design Mobile。

### 1.2 涉及功能模块

- **短信验证码登录**（`/api/auth/sms-login`）— 核心认证模块
- **验证码发送**（`/api/auth/sms-code`）— 验证码生成与下发模块
- **验证码数据表**（`verification_codes`）— 验证码存储

### 1.3 发现时间与发现方式

- **发现时间**：最近，部署新功能（家庭健康档案共管等）之后
- **发现方式**：用户反馈，所有用户（包括测试号 13800138000）均无法通过短信验证码登录
- **历史修复记录**：此前已修复过一次"datetime aware vs naive 混用"导致的验证码校验失败问题，将 `datetime.now(timezone.utc)` 统一改为 `datetime.utcnow()`。当前代码中已是修复后的状态。

### 1.4 近期变更

最近部署了新功能，且**做了数据库变更（迁移/重建）**，这是一个关键线索。

## 2. Bug 描述

### 2.1 错误现象

用户在 H5 网页端输入正确的手机验证码后，系统提示 **"登录失败，请检查验证码"**。该提示是前端的兜底文案（`detail || '登录失败，请检查验证码'`），说明后端要么返回了非预期的错误，要么返回的 `detail` 字段为空。

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 打开 H5 登录页面 | 正常显示登录界面 | 正常 |
| 2 | 输入手机号（如 13800138000） | 手机号输入框正常 | 正常 |
| 3 | 点击"获取验证码" | 显示"验证码已发送" | 正常（测试号不发真实短信） |
| 4 | 输入验证码（如测试号固定 123456） | 登录成功，跳转首页 | **显示"登录失败，请检查验证码"** |

### 2.3 影响范围

- **影响端**：H5 网页端（其他端共用同一后端接口，预计也受影响）
- **影响用户**：**所有用户**均无法登录，包括测试号（13800138000 + 123456）
- **严重程度**：**P0 / 致命级**——系统核心登录功能完全不可用
- **短信接收**：用户可以正常收到短信验证码，问题出在后端校验环节

## 3. 预期正确效果

1. 用户输入正确的手机号和验证码后，**成功登录**并跳转到首页
2. 新用户验证码登录后，**自动注册**并跳转到健康引导页
3. 输入错误验证码时，提示 **"验证码无效或已过期"**（后端返回的 detail）
4. 测试号（13800138000 + 123456）**始终可以正常登录**

## 4. 根因分析

### 4.1 关键线索汇总

1. **所有用户都不行**（包括测试号）→ 问题出在通用的验证码校验逻辑
2. **测试号也不行** → 问题不在短信发送，而在验证码比对或数据库查询
3. **最近部署了新功能，且做了数据库变更** → 高度怀疑数据库迁移导致了问题
4. **上次已修复过 datetime aware vs naive 问题** → 当前代码已统一使用 `datetime.utcnow()`

### 4.2 可能的根因（按可能性排序）

#### 根因 A（最可能）：数据库迁移导致 `verification_codes` 表结构异常或数据丢失

最近做了数据库变更/重建，可能导致：
- `verification_codes` 表被删除或重建，之前保存的验证码记录全部丢失
- 表结构发生变化（如 `expires_at` 字段类型变更），导致时间比较查询行为异常
- 数据库重建后，MySQL 的 `time_zone` 配置与 Python 端的 UTC 时间产生偏差

**验证码校验的核心 SQL 逻辑**（`auth.py` 第 319~328 行）：
```python
result = await db.execute(
    select(VerificationCode)
    .where(
        VerificationCode.phone == data.phone,
        VerificationCode.code == data.code,
        VerificationCode.expires_at > datetime.utcnow(),
    )
    .order_by(VerificationCode.created_at.desc())
    .limit(1)
)
```

如果 MySQL 的 `time_zone` 设置为东八区（`+08:00`），而 Python 使用 `datetime.utcnow()`（UTC 时间），那么 `expires_at` 存入的时间是 UTC，但 MySQL 进行 `>` 比较时可能使用服务器本地时间，造成验证码"看起来已过期"。

#### 根因 B：数据库迁移引入了新的 model 字段，导致 `VerificationCode` ORM 查询异常

部署新功能时可能修改了 `models.py`，如果 ORM 模型定义与数据库实际表结构不匹配（缺字段或类型不一致），SQLAlchemy 查询可能抛出异常，被 FastAPI 捕获后返回 500 错误，前端拿不到 `detail` 就显示兜底文案。

#### 根因 C：`sms-code` 接口生成验证码时没有正确写入数据库

如果数据库连接或事务提交出现问题，`sms-code` 接口虽然返回了"验证码已发送"，但验证码记录实际没有持久化到数据库。后续 `sms-login` 查询时自然找不到匹配记录。

## 5. 修复方案

### 5.1 排查步骤（需查看服务器日志确认根因）

1. **检查后端容器日志**：查看 `sms-login` 接口在登录失败时是否有异常堆栈输出
2. **检查 MySQL 时区配置**：执行 `SELECT @@global.time_zone, @@session.time_zone;` 确认数据库时区
3. **检查 `verification_codes` 表**：确认表是否存在、结构是否正确、是否有近期写入的验证码记录
4. **检查数据库迁移日志**：确认本次部署是否重建了数据库表

### 5.2 针对各根因的修复措施

#### 针对根因 A：MySQL 时区不一致

**修复方式**：在数据库连接 URL 中强制指定 UTC 时区，或在 MySQL 容器中显式设置 `time_zone='+00:00'`

- 在 `docker-compose.yml` 的 MySQL 服务中增加时区配置
- 或在 SQLAlchemy 的 `connect_args` 中指定 `init_command: "SET time_zone='+00:00'"`

#### 针对根因 B：ORM 模型与数据库不匹配

**修复方式**：执行数据库 schema 同步，确保 `verification_codes` 表结构与 `VerificationCode` 模型一致。

#### 针对根因 C：验证码未持久化

**修复方式**：检查 `get_db()` 中的事务提交逻辑，确保 `sms-code` 接口在正常返回前已经 commit 了事务。当前代码中 `get_db()` 使用的是 `yield + commit` 模式，理论上应该没问题，但需确认是否有中间件或异常处理拦截了提交。

### 5.3 额外加固措施

1. **在 `sms-login` 接口中增加详细的错误日志**：当验证码查询为空时，记录当前 `datetime.utcnow()` 的值，以及数据库中该手机号最近的验证码记录和 `expires_at` 值，方便后续排查
2. **在 `sms-code` 接口返回前，增加一次查询确认**：确保验证码已成功写入数据库
3. **统一 MySQL 容器时区**：在 `docker-compose.yml` 中为 MySQL 容器设置 `TZ=UTC` 环境变量和 `command: --default-time-zone='+00:00'`

## 6. 补充说明

1. **此 Bug 为 P0 级**：登录功能完全不可用，影响所有用户，需要立即修复
2. **上次修复仅解决了 Python 代码层面的 aware/naive 问题**，但没有统一 MySQL 容器的时区配置。如果本次部署重建了数据库容器，MySQL 默认使用系统时区（可能是 CST/东八区），与 Python 的 `datetime.utcnow()` 产生 8 小时偏差
3. **预计修复时间**：确认根因后，十几分钟内即可完成修复和重新部署
4. **未做任何临时处理**：用户尚未做任何重启或清理操作，数据库中应该保留着完整的问题现场，有利于排查
