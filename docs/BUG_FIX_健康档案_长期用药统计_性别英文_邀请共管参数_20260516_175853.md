# 健康档案页面 Bug 修复方案文档

> 涉及 H5 健康档案页（`/health-profile`）的 3 个 Bug：长期用药数量与列表不一致、家庭成员卡片性别显示英文、共管 Tab 点击「邀请共管」跳转后报"缺少成员参数"。

---

## 1. Bug 发生背景

### 1.1 项目概述

bini-health 是一款面向家庭健康管理的 AI 助手产品，提供 **AI 健康咨询、家庭成员档案、用药计划、共管与提醒、健康事件追踪**等模块。本次涉及的健康档案页位于 H5 端，是用户管理本人 / 家庭成员所有健康相关数据的核心入口。

### 1.2 涉及功能模块

本次 3 个 Bug 集中在健康档案页的以下模块：

| 模块 | 子功能 | 受影响层 |
|------|--------|----------|
| Hero 卡片（健康摘要） | 四格摘要指标 —— 长期用药数量 | 后端统计接口 + 前端文案 |
| 家庭成员卡片（顶部横向列表） | 成员资料展示 —— 性别字段 | 前端展示层 |
| 共管与提醒 Tab | 邀请共管入口 | 前端跳转逻辑 + 入口显示条件 |

### 1.3 发现时间与发现方式

| 项 | 内容 |
|----|------|
| 发现时间 | 2026-05-16 |
| 发现方式 | 产品 / 用户在 H5 健康档案页人工测试时发现 |
| 复现环境 | H5 端，本人 + 家庭成员两类档案均可复现 |

## 2. Bug 描述

### 2.1 错误现象

#### Bug 1：Hero 卡显示「长期用药 9 种」，但用药计划 Tab 全空

- 在 **本人健康档案** 的 Hero 卡四格摘要中显示「长期用药 **9 种**」。
- 切到「用药计划」Tab，**今日用药** 与 **全部用药计划** 两个分段都是 **0 条**，列表完全空白。
- 用户在用药计划 Tab 内新增一条用药，保存成功后列表中仍然看不到，Hero 卡数字也没变化。
- 根本矛盾在于：**Hero 卡的"长期用药"口径**与**用药计划列表**两边逻辑对不齐，导致用户在视觉上出现"9 vs 0"的强烈冲突。
- 业务上正确的口径应当是「**在用药品**」（当前正在服用的药品总数，含按起止日期管理的非长期药），而不是仅统计 `long_term=true` 的那部分。

#### Bug 2：家庭成员卡片中性别显示为英文

- 在健康档案页顶部"家庭成员横向卡片列表"中，**非本人**的成员卡显示了 `male` / `female` 等英文原文，没有翻译成中文「男 / 女」。
- 该家庭成员资料是用户在 H5 健康档案页通过「新建家庭成员」弹层创建的，弹层选择的是中文「男 / 女」，但展示层落入了原始 DB 字段值（实际存储是英文枚举）的渲染分支。

#### Bug 3：点击「邀请共管」跳转到新页，页面提示「缺少成员参数」

- 在「共管与提醒」Tab 的"共管家人"区块，无论是 **本人卡** 还是 **家庭成员卡**，都显示了 `+ 邀请共管 ›` 入口。
- 点击该入口后，跳转到新的邀请页（`/family-invite`），整页只显示一行红字：**"缺少成员参数"**，无法继续操作。
- 产品上的正确逻辑应当是：**邀请共管只在"非本人"的家庭成员卡片中显示**（因为"邀请共管"的语义是"邀请别人来共同管理这位家庭成员"，不存在本人邀请自己），并且跳转时必须把当前选中的成员 ID 透传给邀请页。

| 2 | 切到家庭成员档案 → 共管与提醒 Tab，点击「+ 邀请共管」 | 跳转到邀请页并自动生成邀请二维码 / 链接 | 跳转后页面显示「缺少成员参数」红字 |

### 2.3 影响范围

| 影响维度 | 说明 |
|----------|------|
| 功能影响 | ① 用药模块在 Hero 卡的统计完全不可信；② 家庭成员性别显示不专业、影响品牌观感；③ 共管邀请流程在 H5 端整条链路断掉，用户无法发起共管邀请 |

| 数据影响 | 不涉及数据丢失或数据污染，纯展示层 + 跳转参数缺失问题，无需做数据迁移或补救 |
| 关联功能 | 用药计划 Tab、家庭成员管理、共管与提醒 Tab、家庭邀请页（`/family-invite`） |

## 3. 预期正确效果

### 3.1 Bug 1：Hero 卡口径改为「在用药品」

#### 文案修改

- 将四格摘要中第 4 格的文案从 **「长期用药 X 种」** 改为 **「在用药品 X 种」**。
- 单位仍保留为「种」。
- 全端涉及该指标展示的位置统一改为「在用药品」（包括 PRD-469 Hero 摘要、可能复用同一数据的其他卡片）。

#### 统计口径

- "在用药品"的定义统一为：**当前用户所有「正在生效」的用药记录条数**。
- 「正在生效」的判定规则：
  - `long_term = TRUE`：始终计入。
  - `long_term = FALSE` 或 `NULL`：当且仅当（`start_date` 为空 或 `start_date <= 今天`）且（`end_date` 为空 或 `end_date >= 今天`）时计入。
- 该口径必须与「用药计划」Tab 的「全部用药计划」分段**展示的条数完全一致**，做到"Hero 看到几条，列表里就有几条"。

#### 行为一致性

- 用户在"用药计划"Tab 新增 / 编辑 / 删除任意用药记录后：
  - "用药计划"列表立即刷新。
  - Hero 卡的"在用药品"数字同步刷新（页面切回时不需要硬刷，Tab 切换 / 数据回写后自动同步即可）。

#### 后端接口

- 修改 Hero 摘要接口 `/api/prd469/summary/{profile_id}` 中"长期用药"字段的统计 SQL：
  - 原：`SELECT COUNT(*) FROM medication_reminders WHERE user_id = ? AND long_term = TRUE`
  - 新："在用药品"口径，按上述判定规则统计。
- 返回的 `hero_metrics[3]` 改为：

```json
{ "label": "在用药品", "count": <在用药品数>, "unit": "种" }
```

#### 前端兜底

- 前端的兜底文案数组（接口失败时使用）也同步从「长期用药」改为「在用药品」，避免接口异常时仍显示老文案造成混乱。

### 3.2 Bug 2：性别中文映射

- 在家庭成员卡片、Hero 卡副标题等所有展示成员性别的位置，统一对原始字段值做映射：
  - `male` / `M` / `1` / `男` → **「男」**
  - `female` / `F` / `2` / `女` → **「女」**
  - `other` / `unknown` / 空值 / 其它无法识别值 → **「未设置」**（或不展示该项）
- 编辑成员资料保存时，前端写回后端的性别值统一为中文「男 / 女」（与后端 DB 现存数据风格保持一致）；
- 同时在展示层增加映射函数，保证即便后端历史数据混入英文枚举，前端也能稳定显示为中文。

### 3.3 Bug 3：共管邀请按钮与跳转修复

#### 入口显示条件

- 仅当**当前选中的成员是"非本人"（`is_self === false`）的家庭成员**时，才在「共管与提醒」Tab 的"共管家人"区块顶部显示「+ 邀请共管 ›」入口。
- 本人卡片下不显示此按钮（业务语义：本人不能邀请自己来共管自己）。
- 该按钮的露出还应同时满足"当前用户对该成员有邀请权限"（已是创建者 / 所有者），无权限时也不显示。

#### 跳转参数

- 点击「+ 邀请共管」时，必须将当前选中的家庭成员 ID 拼接到跳转 URL：

```
/family-invite?member_id={selectedMemberId}
```

- 不再出现无参数跳转的情况，从根上杜绝邀请页的「缺少成员参数」错误。

#### 邀请页兜底

- 邀请页 `/family-invite` 仍保留对 `member_id` 缺失的兜底校验（防御性编程），但兜底文案优化：
  - 不再仅展示一行裸红字「缺少成员参数」，而是给出友好提示并提供「返回上一页」按钮：「未指定要邀请共管的家庭成员，请返回健康档案页，在对应家人卡片中点击「邀请共管」入口。」
- 这样即使后续有其他路径误进入此页，用户也能自助回到正确流程。

## 4. 补充说明

### 4.1 修复涉及的关键文件（说明性）

- 后端 Hero 摘要接口实现位置：PRD-469 健康档案后端模块
- 前端健康档案页：H5 端 `health-profile` 页面
- 前端共管 Tab：H5 端 PRD-469 共管与提醒区块组件
- 前端邀请页：H5 端 `family-invite` 页面
- 性别映射函数：建议放在 H5 端公共 `utils/format.ts` 之类的格式化工具中统一收口，便于全端复用

### 4.2 测试要点

#### Hero 卡 / 用药计划口径一致性测试

| 用例 | 操作 | 预期 |
|------|------|------|
| 仅长期药 | 用户只有 3 条 `long_term=true` 的用药 | Hero 卡显示「在用药品 3 种」，列表 3 条 |
| 仅短期药且当前有效 | 用户只有 2 条 `start≤today≤end` 的短期药 | Hero 卡显示「在用药品 2 种」，列表 2 条 |
| 短期药已过期 | 用户有 1 条 `end < today` 的短期药 | Hero 卡不计入，列表"全部用药计划"分段可见但应标记"已结束" |
| 混合 | 2 条长期 + 1 条当前有效短期 + 1 条已过期 | Hero 卡显示「在用药品 3 种」 |
| 新增同步 | 在用药计划 Tab 新增 1 条长期用药 | 列表立即出现，返回 Hero Tab 后数字 +1 |

#### 性别映射测试

| 用例 | 输入 | 预期展示 |
|------|------|----------|
| 英文 male | DB 值 `male` | 卡片显示「男」 |
| 英文 female | DB 值 `female` | 卡片显示「女」 |
| 中文男 | DB 值 `男` | 卡片显示「男」 |
| 空 / 未知 | DB 值为空 | 卡片显示「未设置」或隐藏该项 |

#### 共管入口测试

| 用例 | 操作 | 预期 |
|------|------|------|
| 本人档案 | 进入本人健康档案 → 共管与提醒 Tab | 不出现「+ 邀请共管」按钮 |
| 家庭成员档案 | 进入家庭成员档案 → 共管与提醒 Tab | 出现按钮 |
| 正常跳转 | 点击按钮 | URL = `/family-invite?member_id=<该成员id>`，页面正常生成邀请码与二维码 |
| 异常直达 | 手动访问 `/family-invite`（无参数） | 显示友好兜底提示与"返回"按钮，而非裸红字 |

### 4.3 工作量与上线

得益于自动化开发能力，本次三个 Bug 将作为**一期工程一次性全量修复并上线**，预计在半天内完成"修复 → 自测 → 部署"全流程，无需分批发布。

### 4.4 风险与回归

- 修改 Hero 摘要接口的统计 SQL 时，必须同步检查是否还有其他模块（如 AI 咨询对象档案卡 `consultant_profile_card`）也使用了"长期用药"口径；如果它们仍需保持"仅 `long_term=true`"的语义，则要避免误改它们的口径，仅修改 Hero 摘要相关返回。
- 性别映射放在公共 utils 后，需要回归排查 H5 端是否有其他地方"裸打印 gender"，统一替换为映射函数调用。
- 共管入口的"是否显示"判断要兼容"成员加载中"的中间态，避免短暂闪烁。
