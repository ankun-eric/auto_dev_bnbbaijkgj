# bini-health 积分规则配置优化与完善健康档案送积分 产品需求文档（PRD）

## 1. 需求概述

### 1.1 背景与目的

bini-health 系统的后台管理端"积分规则配置"页面，在上线"打卡送积分"功能后，出现了两处重复的健康打卡积分配置（通用规则 Tab 和打卡规则 Tab），用户使用时容易混淆。同时，通用规则中的"完善个人资料"配置项名称不够准确，且后端从未实现积分发放逻辑。

本次优化包含两部分：
1. **积分配置页面合并**：删除打卡规则 Tab，将健康打卡配置统一到通用规则 Tab，新增"健康打卡每日上限"字段
2. **完善健康档案送积分**：将"完善个人资料"改名为"完善健康档案"，并实现后端积分发放逻辑——用户首次完善健康档案 4 个核心字段后自动发放一次性积分

### 1.2 目标用户

- 后台管理员（配置积分规则的运营人员）
- 普通用户（完善健康档案获取积分的终端用户）

### 1.3 核心价值

- 消除配置入口的重复和歧义，降低运营人员的理解成本
- 精简页面结构，提升后台管理体验
- 激励用户完善健康档案，提升健康数据的完整度
- 配置项名称与实际功能一致，避免运营误解

## 2. 功能需求

### 2.1 功能清单总览

| 编号 | 功能模块 | 功能点 | 优先级 | 说明 |
|------|----------|--------|--------|------|
| 1 | 前端-积分规则页面 | 删除打卡规则 Tab | 高 | 移除整个 Tabs 组件和打卡规则 Tab 页 |
| 2 | 前端-积分规则页面 | 通用规则 Tab 新增"健康打卡每日上限"字段 | 高 | 在现有"健康打卡"字段下方新增 |
| 3 | 前端-积分规则页面 | "完善个人资料"改名为"完善健康档案" | 高 | 修改标签文字和字段 key |
| 4 | 后端-积分规则 API | 新增 `healthCheckInDailyLimit` 字段 | 高 | GET/POST 接口支持读写新字段 |
| 5 | 后端-积分发放服务 | 改读通用规则配置 | 高 | 废弃原打卡规则三字段，统一读取通用规则 |
| 6 | 后端-健康档案积分 | 实现"完善健康档案"积分发放 | 高 | 首次完善 4 个核心字段后自动发放一次性积分 |

### 2.2 功能详细描述

#### 2.2.1 前端 — 积分规则页面改造

**现状：**
- 积分规则页面使用 Tabs 组件，有"通用规则"和"打卡规则"两个 Tab
- 通用规则 Tab 的"任务奖励"卡片下有"健康打卡：XX 积分/次"字段
- 通用规则 Tab 的"任务奖励"卡片下有"完善个人资料：XX 积分/次"字段
- 打卡规则 Tab 有三个字段：启用开关、每次打卡积分、每日上限

**改造后：**
- **删除 Tabs 组件**：页面不再有 Tab 切换，直接展示原"通用规则"的内容
- **删除打卡规则 Tab 的全部内容**
- **"完善个人资料"改名为"完善健康档案"**：修改显示标签文字
- **在"任务奖励"卡片的"健康打卡"下方**，新增一行：
  - 字段名：**健康打卡每日上限**
  - 输入框：数字输入，单位为"积分"
  - 占位文字：请输入每日积分上限
  - 校验：必须为非负整数

**配置项最终效果（任务奖励卡片）：**

| 配置项 | 输入类型 | 单位 | 说明 |
|--------|----------|------|------|
| 分享文章 | 数字 | 积分/次 | 已有，保留不动 |
| **完善健康档案** | 数字 | 积分/次 | 原"完善个人资料"，**改名** |
| 评价服务 | 数字 | 积分/次 | 已有，保留不动 |
| 健康打卡 | 数字 | 积分/次 | 已有，保留（现在要真正生效） |
| **健康打卡每日上限** | **数字** | **积分** | **新增** |

**业务规则：**
- 不需要单独的"启用/禁用"开关
- "健康打卡"积分值设为 0 即表示关闭打卡送积分功能
- "健康打卡每日上限"设为 0 表示不限制每日积分上限

#### 2.2.2 后端 — 积分规则 API 改造

**GET `/api/admin/points/rules`**
- 返回值中新增 `healthCheckInDailyLimit` 字段（整数，默认 0）
- 将 `completeProfile` 字段的前端显示名改为"完善健康档案"（后端字段 key 可保持不变以保证兼容）
- 废弃原有三个打卡规则字段（`checkin_points_enabled`、`checkin_points_per_action`、`checkin_points_daily_limit`），可继续返回但标记为 deprecated

**POST `/api/admin/points/rules`**
- 请求体支持接收 `healthCheckInDailyLimit` 字段
- 将值写入 `SystemConfig` 表

#### 2.2.3 后端 — 打卡积分发放服务改造

`checkin_points_service.py` 中的积分发放逻辑：
- **每次打卡积分**：改为读取通用规则中的 `healthCheckIn` 字段值
- **每日上限**：改为读取通用规则中新增的 `healthCheckInDailyLimit` 字段值
- **是否启用**：不再依赖 `checkin_points_enabled` 开关，改为判断 `healthCheckIn > 0` 即为启用
- 废弃对 `checkin_points_enabled`、`checkin_points_per_action`、`checkin_points_daily_limit` 三个配置的读取

#### 2.2.4 后端 — "完善健康档案"积分发放（新增功能）

**触发条件：**
- 用户通过健康档案编辑接口保存"本人记录"时
- 系统检测到以下 **4 个核心字段全部已填写**：性别、生日、身高、体重
- 且该用户此前 **从未获得过** "完善健康档案"类型的积分奖励

**发放规则：**
- 积分数量 = 通用规则中"完善健康档案"（`completeProfile`）字段配置的值
- 每个用户 **终身只发放一次**（首次完善时触发）
- 如果配置的积分值为 0，则不发放
- 积分发放必须做异常容错（try-except），发放失败不影响健康档案保存

**实现要点：**
- 在保存健康档案的接口中（保存"本人记录"时），增加完善度判定逻辑
- 判定 4 个字段（性别 `gender`、生日 `birthday`、身高 `height`、体重 `weight`）是否全部非空
- 查询 `PointsRecord` 表，检查该用户是否已有 `completeProfile` 类型的积分记录
- 如果 4 字段全部填写 + 此前无该类型记录 → 发放积分，写入积分记录
- `PointsType` 枚举需确认包含 `completeProfile` 或等效值

#### 2.2.5 数据迁移

- 如果 `checkin_points_per_action` 有值且 `healthCheckIn` 无值或为 0，则将 `checkin_points_per_action` 的值迁移到 `healthCheckIn`
- 如果 `checkin_points_daily_limit` 有值，则将其值迁移到 `healthCheckInDailyLimit`
- 迁移完成后，保留旧字段数据但不再使用

## 4. 非功能性需求

### 4.1 性能要求

- 配置页面加载时间 < 1 秒
- 保存操作响应时间 < 500ms
- "完善健康档案"积分判定和发放不增加健康档案保存接口的响应延迟（异步或快速同步）

### 4.2 安全要求

- 积分规则配置仅管理员角色可操作
- 配置变更需记录操作日志
- "完善健康档案"积分必须通过后端校验防刷（数据库唯一约束 + 查重）

### 4.3 兼容性要求

- 前端兼容现有后台管理系统的 UI 框架
- 后端 API 保持向后兼容，旧字段保留但标记为 deprecated

## 5. 业务规则与约束

| 规则编号 | 规则描述 |
|----------|----------|
| R1 | "健康打卡"积分值为 0 时，打卡不发放积分（等同于关闭功能） |
| R2 | "健康打卡每日上限"为 0 时，表示不限制每日积分上限 |
| R3 | 每次打卡获得的积分 = "健康打卡"字段值（积分/次） |
| R4 | 用户当日已获得的打卡积分 >= 每日上限时，打卡仍然成功，但不再发放积分 |
| R5 | 积分发放异常时，不影响打卡本身的成功 |
| R6 | "完善健康档案"积分值为 0 时，完善档案不发放积分 |
| R7 | "完善健康档案"积分每个用户终身只发放一次（首次完善时触发） |
| R8 | "完善健康档案"的判定条件：性别、生日、身高、体重 4 个字段全部非空 |
| R9 | 用户后续修改已完善的健康档案，不再重复发放积分 |
| R10 | 积分发放异常时，不影响健康档案保存本身的成功 |

## 6. 权限设计

| 角色 | 权限说明 |
|------|----------|
| 超级管理员 | 可查看、编辑积分规则配置 |
| 普通管理员 | 可查看积分规则配置（根据现有权限体系） |
| 普通用户 | 无权访问后台积分规则页面；可通过完善健康档案和健康打卡获取积分 |

## 7. 异常处理与边界情况

| 场景 | 处理方式 |
|------|----------|
| 健康打卡积分值未配置或为空 | 默认为 0，不发放积分 |
| 每日上限未配置或为空 | 默认为 0，不限制 |
| 输入非数字或负数 | 前端校验拦截，提示"请输入非负整数" |
| 旧配置与新配置同时存在 | 以新配置（通用规则）为准，旧配置不再读取 |
| 数据迁移失败 | 记录日志，使用默认值，不影响系统运行 |
| 用户重复触发"完善健康档案"积分 | 后端查重，已发放过则跳过，确保终身只发一次 |
| 用户清空字段后重新填写 | 不再发放积分（只认首次完善） |
| "完善健康档案"积分发放异常 | 捕获异常并记录日志，健康档案保存正常完成 |
| 4 个字段只填了部分 | 不触发积分发放，等全部填完后才发放 |

## 8. 补充说明

- 本次需求与"健康打卡失败"Bug 修复一并处理
- 修复后需确保所有打卡接口正常工作，积分按通用规则配置正确发放
- 打卡规则 Tab 删除后，原有的三个后端配置字段（`checkin_points_enabled`、`checkin_points_per_action`、`checkin_points_daily_limit`）保留在数据库中但不再使用，避免数据丢失
- "完善个人资料"字段的后端 key `completeProfile` 可保持不变以保证 API 兼容性，仅修改前端显示文字为"完善健康档案"
- "完善健康档案"的积分类型在 `PointsType` 枚举中使用 `completeProfile`，无需新增枚举值
