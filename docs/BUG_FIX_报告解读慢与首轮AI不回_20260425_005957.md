# 体检报告解读功能优化 Bug 修复方案文档

> **涉及 Bug**：① 上传图片 + 选择咨询人后"解读中"页面过慢（≥ 1 分钟）；② 进入 AI 咨询页后默认首问以用户气泡显示且 AI 无回复；③ `type=report_interpret` 模式下左下角仍可切换咨询人。

---

## 1. Bug 发生背景

### 1.1 项目概述

bini-health 健康管理应用，提供**体检报告 AI 解读**能力：用户上传体检报告图片 → 选择咨询人 → 系统自动 OCR 并调用大模型生成结构化解读 → 进入 AI 咨询会话继续追问。

报告解读功能于早前完成"对话化升级"，将专属的 `/h5/checkup/chat/[sessionId]` 页面**统一迁移**到通用 AI 咨询页 `/h5/chat/[sessionId]`，通过 URL query `type=report_interpret&auto_start=1` 区分场景。小程序 / Flutter 侧同步新增 `report_interpret / report_compare` 分支。

### 1.2 涉及功能模块

| 端 | 涉及页面 / 模块 |
|---|---|
| 后端 | `report_interpret.py`（interpret/start、session/chat、session/stream）、`checkup_api_v2.py`、OCR 服务、AI 大模型调用、提示词模板 `checkup_report_interpret` |
| H5 | "上传报告 → 选咨询人 → 解读中"过渡页、`/h5/chat/[sessionId]` 聊天页 |
| 小程序 | `miniprogram/pages/chat/index` 的 `report_interpret / report_compare` 分支 |
| Flutter App | `flutter_app/lib/screens/ai/chat_screen.dart` 的 `_processRouteArguments` / `_reportCards` |

### 1.3 发现时间与发现方式

- 发现时间：近期（稳定复现）
- 发现方式：业务侧在真实使用中反馈"解读太慢 + 进入聊天页 AI 首轮不说话"
- 复现率：
  - Bug A（慢）：**100%**，每次都慢
  - Bug B（首轮 AI 不回）：**100%**，每次都不回
  - Bug C（仍可切咨询人）：**100%**

## 2. Bug 描述

### 2.1 错误现象

#### Bug A — "解读中"页面过慢（≥ 1 分钟）

用户点击「AI 开始解读」后，页面进入"解读中…"的加载态，进度条不断转圈，**整体耗时超过 1 分钟**才跳转到 AI 咨询页。此问题自该功能上线以来**一直存在**，并非近期才变慢。

#### Bug B — 默认首问显示为用户气泡 + AI 首轮不回

进入 AI 咨询页后：

1. 聊天区先出现一条**用户气泡**，内容为默认首问（类似"请帮我解读这份报告"等带报告内容的 prompt）
2. 紧接着出现一条 assistant 气泡，展示**预先生成的报告解读内容**（即后端在 `/start` 时已写入的首条 AI 消息）
3. 但用户气泡对应的"当前这轮提问"**AI 不再给出任何回复**——既没有新的 AI 气泡，也没有"输入中…"动效
4. 用户**后续**再主动发问，AI 能正常回复（所以非整个会话死掉，仅首轮静音）

#### Bug C — `type=report_interpret` 模式下仍可切咨询人

AI 咨询页左下角保留了「切换咨询人」入口（以及顶部可能存在的相关入口），在报告解读场景下点击可以把会话切换到其他家庭成员，**与当前报告的主体不一致**，造成数据语义错乱。

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 进入"体检报告解读"入口，上传 1 张体检报告图片 | 上传成功 | 上传成功 |
| 2 | 选择咨询人（家庭成员），点击「AI 开始解读」 | 数秒内跳转至 AI 咨询页 | 卡在"解读中…"页面 **≥ 1 分钟** |
| 3 | 等待跳转到 `/h5/chat/[sid]?type=report_interpret&auto_start=1` | 聊天区静默展示 AI 流式解读，无用户气泡 | 聊天区**先显示用户气泡（默认首问）**，再显示预存的 AI 解读，**但当前这轮 AI 不再回复** |
| 4 | 观察聊天页左下角 | `report_interpret` 模式下**无**切换咨询人入口 | 左下角切换咨询人按钮仍然显示，可点击切换 |
| 5 | 用户手动再发一条消息 | AI 正常回复 | AI 正常回复（说明仅首轮"静音"） |

### 2.3 影响范围

- **受影响端**：H5 / 微信小程序 / Flutter（iOS & Android）三端**全部**
- **受影响功能**：体检报告解读（`type=report_interpret`），以及与之同架构的报告对比（`type=report_compare`）需一并排查
- **受影响用户**：所有使用报告解读的用户（全量用户）
- **业务影响**：
  - 首次使用体验极差（等待 1 分钟 + 首问看起来"没人搭理"）
  - 咨询人切错可能导致**错配家庭成员档案**，AI 追问时带入错误的个人信息，存在**数据隐私 / 医疗语义错误**风险

## 3. 根因分析

### 3.1 Bug A（过慢）根因

后端 `/api/report/interpret/start` 接口在**单次 HTTP 请求**内**同步串行**完成以下全部工作：

```text
[上传入库] → [OCR 识别所有指标] → [拼接报告内容 + 提示词模板] → [调用大模型生成首次解读] → [落库 assistant message] → [返回 session_id]
```

前端必须等该请求完全返回后才能拿到 `session_id` 并跳转到聊天页，因此用户感知的"解读中"时长 = OCR 耗时 + AI 首次生成耗时，实测 ≥ 1 分钟。

**核心问题**：把"必须同步给出的结果（session_id）"和"耗时的异步计算结果（AI 首次解读）"塞进同一个同步接口里。

### 3.2 Bug B（首轮 AI 不回 + 默认气泡显示）根因

结合 3.1 的实现，首次解读**已经在 `/start` 阶段被写入会话**作为首条 assistant message。进入聊天页后：

1. 前端因 URL 带有 `auto_start=1`，再次触发"自动发送默认首问"的逻辑，把默认问句作为**一条用户气泡**塞到 UI 里并调用 `POST /api/report/interpret/session/:sid/chat`
2. 后端判断"本会话已存在首条 AI 解读消息"，走了**幂等/去重分支**，**直接返回**而未生成新的 assistant 消息，也未通过 SSE 推流
3. 前端因此只看到用户气泡 + 已有的历史 AI 解读，当前这轮不再有新 AI 气泡 → 表现为"AI 不回"

**次要问题**：默认首问"应该隐式"的约束在前端被漏实现，变成显式的用户气泡。

### 3.3 Bug C（仍可切咨询人）根因

`/h5/chat/[sessionId]` 聊天页的左下角"切换咨询人"组件，当前渲染条件**未判断 `type`**，在 `type=report_interpret / report_compare` 场景下没有隐藏；小程序、Flutter 端同理。

## 4. 预期正确效果

### 4.1 Bug A — 解读中页面几乎不停留

- 点击「AI 开始解读」后，**≤ 3 秒**拿到 `session_id` 并跳转到 AI 咨询页
- AI 咨询页进入后**立即**显示一条 AI 气泡，展示"正在解读报告…"的占位状态，随后**流式**输出完整解读内容
- OCR 与首次 AI 解读全部在**后台异步任务**中完成，前端通过 SSE 流接收

### 4.2 Bug B — 默认首问完全隐式 + AI 稳定首轮回复

- 用户进入聊天页后，**不再出现任何用户气泡形式的默认首问**
- 默认首问作为**隐藏 prompt**由后端在异步任务中与报告内容一并发给 AI
- AI 以**正常 AI 气泡 + 流式输出**的方式给出首次解读回复
- 若后台任务异常 / SSE 掉线，前端具备**超时重试**机制，确保首轮一定能拿到回复

### 4.3 Bug C — 报告解读场景下彻底去掉切咨询人入口

- `type=report_interpret` 与 `type=report_compare` 两种模式下：
  - H5 聊天页：左下角切换咨询人按钮 / 下拉**整体不渲染**
  - 小程序：`report_interpret / report_compare` 分支下**不渲染**相关入口
  - Flutter：`chat_screen.dart` 在对应 mode 下**不渲染**相关入口
- 其他 `type`（如普通 AI 咨询、用药识别等）维持原有行为不变

## 5. 修复方案（一次性三端全量修复）

### 5.1 后端改造

#### 5.1.1 `/api/report/interpret/start` 接口改为"秒回 + 后台异步"

拆分职责：

```text
同步部分（接口即时返回）：
  1) 创建 ChatSession（type=report_interpret），拿到 session_id
  2) 保存上传的图片 URL 到 CheckupReport / session 关联
  3) 入队后台任务（Celery / BackgroundTasks / 自建任务队列）
  4) 立刻返回 { session_id, status: "processing" }

异步部分（后台任务）：
  1) OCR 识别所有指标，入库 checkup_report_details
  2) 拼接报告内容 + `checkup_report_interpret` 提示词模板 + 咨询人档案
  3) 调用大模型，以 SSE 的形式把 assistant message **流式**写入会话
  4) 任务结束后更新会话/报告状态为 ready
```

关键点：

- **session_id 必须在同步阶段就生成并返回**，这是解决 Bug A 的根本
- 异步任务的输出通过 `/api/report/interpret/session/:sid/stream`（SSE）推送；前端聊天页进入后立即 `EventSource` 订阅
- **不再**在 `/start` 阶段预先写入首条 assistant message（避免 Bug B 的幂等分支误命中）

#### 5.1.2 默认首问作为"隐藏 system / user prompt" 由后端持有

- 前端**不再**发送默认首问文本；`/start` 返回后，**前端只订阅 SSE**，不再调用一次 `/session/:sid/chat` 去"触发"首问
- 后端在异步任务中，把默认首问（含报告内容、咨询人档案、场景要求）以 **system + hidden user prompt** 的方式组装，但**不落库为 user message**（或落库后在查询接口中过滤 `is_hidden=true`，不下发给前端渲染）
- 数据层建议：`ChatMessage` 增加 `is_hidden: bool`（默认 false）字段；`GET /session/:sid/messages` 接口默认过滤 `is_hidden=true` 的消息

#### 5.1.3 幂等/去重分支修正

原"会话已存在首条 AI 消息则不再生成"的分支需要重新梳理：

- 针对 `auto_start` 触发的自动首问，改为"**仅当后台任务已成功生成 assistant 消息**时才视为完成"，而不是基于消息数量判断
- 引入 `ChatSession.interpret_status` 字段（`pending / processing / ready / failed`），前端据此判断是否需要等待 SSE

#### 5.1.4 失败兜底与重试

- 后台任务异常时，向 SSE 流发送一条标准错误事件（`event: error`），前端可显示友好提示并提供"重新解读"按钮
- 提供 `POST /api/report/interpret/session/:sid/retry` 接口，允许重新触发异步解读任务

### 5.2 前端改造（三端一致行为）

#### 5.2.1 "解读中"过渡页极简化

- 上传图片 + 选咨询人 + 调用 `/start` 这段 loading 缩短为**真实等待 `/start` 响应**的时间（预期 ≤ 3 秒）
- `/start` 返回 `session_id` 后立刻跳转至聊天页，**不再**在过渡页做"假进度条"

#### 5.2.2 聊天页进入时的行为变更

在 `type=report_interpret` / `type=report_compare` 下：

1. **不再**发送自动首问（移除原先读取 `auto_start=1` 后调用 `/session/:sid/chat` 的逻辑，或仅在 `type` 非报告类时保留该行为）
2. 一进入页面立即建立 SSE 订阅到 `/session/:sid/stream`
3. 若 `interpret_status=ready`，直接拉取 messages 渲染；若 `processing`，显示一个**占位 AI 气泡**（"AI 正在解读报告…"），接到 SSE chunk 实时拼接
4. **超时保护**：进入聊天页 X 秒（建议 60s）仍未收到任何 SSE chunk 且 status 仍非 ready，自动调用一次 `/retry` 并提示用户

#### 5.2.3 默认首问彻底不渲染为用户气泡

- 移除前端在 `auto_start` 场景下往 message list 里**本地插入 user 气泡**的代码
- 消息渲染层增加对 `is_hidden` 字段的兜底过滤（即便后端漏过滤，前端也不展示）

#### 5.2.4 去掉左下角"切换咨询人"入口（重点）

| 端 | 改造点 |
|---|---|
| H5 | `h5-web/src/app/chat/[sessionId]/page.tsx`：聊天页底部工具栏渲染咨询人切换组件时，判断 `type in ['report_interpret','report_compare']` 则不渲染 |
| 小程序 | `miniprogram/pages/chat/index.{wxml,js}`：在 `report_interpret / report_compare` 分支下，`wx:if` 条件里排除切换咨询人相关 UI |
| Flutter | `flutter_app/lib/screens/ai/chat_screen.dart`：`_processRouteArguments` 已识别 type，在构建底部工具栏时根据 type 隐藏切换组件 |

同时建议：顶部（如有）的咨询人展示改为**只读**展示（展示当前报告对应的咨询人姓名，不可点击切换）。

### 5.3 数据库与索引

- `ChatMessage` 新增字段：`is_hidden BOOLEAN DEFAULT FALSE`（用于隐藏的默认首问 user message，若选择落库方案）
- `ChatSession` 新增字段：`interpret_status VARCHAR(16) DEFAULT 'ready'`（`pending / processing / ready / failed`）
- 迁移脚本：`backend/app/services/report_interpret_async_migration.py`（幂等 ALTER，兼容历史会话默认 `ready`）

### 5.4 提示词模板

- `checkup_report_interpret` 模板保持现有结构，确认变量占位符支持"报告文本 + 咨询人档案 + 场景指令"
- 在提示词里强化**"请以结构化解读形式直接输出首次分析，不要等待用户追问"**，避免 AI 只回一句"好的，请稍等"

### 5.5 SSE 稳定性

- 后端 SSE 端点补全心跳事件（`event: ping`，每 15s 一次），防止反向代理（nginx / CloudFront）的空闲超时踢连接
- 前端三端统一实现：断线自动重连（最多 3 次），重连后基于 `last_event_id` 续传

## 7. 风险与回滚

### 7.1 风险点

- 异步化后若 SSE 被反向代理掐断，体验会比现在更差 → 必须在上线前验证 nginx / 网关的长连接超时配置
- 历史会话中已预先写入的 assistant 首条消息，需要**兼容渲染**（不影响老数据）

### 7.2 回滚策略

- 保留 feature flag：`REPORT_INTERPRET_ASYNC_ENABLED`（默认 on）
- 若上线后出现严重问题，后端可一键切回同步模式（保留原 `/start` 同步实现作为兜底分支）

## 8. 补充说明

- 本次修复将"体检报告解读"的首轮体验从"硬等 1 分钟 + 首轮静音"改为"秒跳 + 流式回复"，是影响核心体验的一次关键优化
- 所有 Bug 由 AI 一次性全部修复，三端同步交付，不拆分多期
- 上线后建议在"咨询人切换"隐藏位置保留一个**入口说明 tooltip**（如首次进入时短暂提示"本次会话针对 XX 的体检报告，如需给其他人解读，请返回重新上传"），避免用户疑惑"怎么切不了人了"
