# 宾尼小康 AI 健康管家 — 部署腾讯云 COS 后全局 500 错误修复方案

## 1. Bug 发生背景

### 1.1 项目概述

宾尼小康 AI 健康管家（bini-health）是一个全栈健康管理平台，技术栈为 FastAPI + SQLAlchemy + MySQL（后端）、Next.js + Ant Design（管理后台与 H5 前端）、Flutter（移动端）、微信小程序。近期完成了腾讯云 COS 对象存储功能的全量接入，新增了 COS 基础配置管理、CDN 加速、上传限制、数据迁移、统计概览等管理后台功能，以及 H5/小程序/Flutter 三端的上传进度条和文件大小限制校验。

### 1.2 涉及功能模块

- **COS 基础配置保存**（`PUT /api/admin/cos/config`）— 主要触发入口
- **COS 连接测试**（`POST /api/admin/cos/test`）
- **上传限制配置保存**（`PUT /api/admin/cos/upload-limits`）
- **系统其他模块的保存操作** — 同样受到影响
- 涉及后端文件：`backend/app/api/cos.py`、`backend/app/models/models.py`、`backend/app/schemas/cos.py`、`backend/app/utils/cos_helper.py`
- 涉及前端文件：`admin-web/src/app/(admin)/cos-config/page.tsx`

### 1.3 发现时间与发现方式

- **发现时间**：部署腾讯云 COS 对象存储功能之后立即出现
- **发现方式**：用户在管理后台操作存储配置保存按钮时触发，前端弹出错误提示 `Request failed with status code 500`

## 2. Bug 描述

### 2.1 错误现象

1. **主要现象**：管理后台「存储配置」页面，修改配置内容后点击「保存」按钮，前端提示 `Request failed with status code 500`（HTTP 500 Internal Server Error）
2. **波及范围扩大**：不仅 COS 基础配置保存报 500，**测试连接按钮（A）和上传限制保存（B）也均报 500 错误**
3. **读取正常**：打开存储配置页面时，之前填写的配置信息能正常显示，说明 GET 接口工作正常
4. **Secret Key**：用户确认有填写 Secret Key
5. **所有报错均为 HTTP 500**，错误码一致

### 2.2 重现步骤

| 步骤 | 操作 | 预期结果 | 实际结果 |
|------|------|----------|----------|
| 1 | 部署腾讯云 COS 存储功能到服务器 | 系统正常运行 | 部署完成后出现全局性异常 |
| 2 | 打开管理后台 → 存储配置页面 | 页面正常加载，显示已有配置 | ✅ 页面正常加载，配置数据正常显示 |
| 3 | 修改任意配置字段（如 Secret Key），点击「保存」按钮 | 提示"COS配置更新成功" | ❌ 前端弹出 `Request failed with status code 500` |
| 4 | 点击「测试连接」按钮 | 返回连接测试结果 | ❌ 同样报 500 错误 |
| 5 | 修改上传限制配置，点击保存 | 提示"上传限制配置已更新" | ❌ 同样报 500 错误 |

### 2.3 影响范围

- **直接影响**：管理后台所有涉及 COS 的写操作（保存配置、测试连接、更新上传限制）全部不可用
- **间接影响**：系统其他模块的保存操作也出现 500 错误，说明问题具有全局性
- **不影响**：GET 读取类接口仍正常工作（页面能正常展示数据）

## 3. 预期正确效果

1. 点击「保存」按钮后，COS 基础配置成功写入数据库，前端提示"COS配置更新成功"
2. 点击「测试连接」按钮后，系统正常调用腾讯云 COS API 进行连通性测试并返回结果
3. 修改上传限制配置后点击保存，成功更新各业务模块的文件大小限制
4. 系统其他模块的保存操作恢复正常

## 4. 根因分析（基于代码审查）

根据以下关键线索：
- **时间线**：刚部署完 COS 功能后立即出现
- **范围**：不仅 COS 接口，系统全局写操作均报 500
- **GET 正常、写操作异常**：说明数据库连接本身没问题，但写操作时触发了异常

**最大可能原因（按概率排序）：**

### 原因 1：数据库 Schema 不同步（概率最高）

本次 COS 功能部署新增了多个数据库模型和字段：
- 新表：`cos_upload_limits`、`cos_migration_tasks`、`cos_migration_details`
- `cos_configs` 表新增字段：`cdn_domain`、`cdn_protocol`、`path_prefix`、`test_passed`

虽然后端启动时会执行 `Base.metadata.create_all` 自动建表，但 SQLAlchemy 的 `create_all` **不会自动为已存在的表添加新字段**。如果 `cos_configs` 表在之前就已经创建过，那么新增的 `cdn_domain`、`cdn_protocol`、`path_prefix`、`test_passed` 四个字段可能不存在于实际数据库中，导致任何涉及这些字段的写操作抛出 `OperationalError`（Unknown column），触发 500。

由于 COS 配置保存接口 `update_cos_config` 中会操作这些新字段，且全局异常处理不够完善，可能导致数据库 session 进入错误状态，波及其他接口。

### 原因 2：Docker 容器重建后数据库连接池异常

参考项目历史中的 "502 Bad Gateway 修复" 记录，Docker 网络在容器重建时可能出现连接问题。类似地，后端容器重建后，数据库连接池可能存在失效连接，导致写操作时连接异常。

### 原因 3：后端异常处理缺失导致的连锁反应

当前 COS 接口中使用的是 `await db.flush()` 而不是显式 `await db.commit()`，依赖 FastAPI 的 session 中间件自动提交。如果 flush 过程中因 Schema 不匹配抛出异常但未被正确捕获和回滚，可能导致数据库 session 处于不一致状态，进而影响后续所有请求。

## 5. 修复方案

### 5.1 修复数据库 Schema 不同步问题

1. **手动执行 ALTER TABLE 补齐缺失字段**：连接到生产数据库，为 `cos_configs` 表添加缺失的四个字段：
   - `cdn_domain VARCHAR(300) NULL`
   - `cdn_protocol VARCHAR(10) DEFAULT 'https'`
   - `path_prefix VARCHAR(200) DEFAULT ''`
   - `test_passed BOOLEAN DEFAULT FALSE`

2. **或者**：删除 `cos_configs` 表后重启后端容器，让 `create_all` 重新创建完整的表结构（如果表中没有重要数据）

### 5.2 增强后端接口的异常处理

1. 为 COS 配置保存接口添加 try-except 包裹，捕获数据库异常并返回有意义的错误信息
2. 确保异常发生时正确回滚 session，避免污染后续请求

### 5.3 验证 Docker 网络与数据库连接

1. 检查后端容器与 MySQL 容器之间的网络连通性
2. 确认数据库连接池配置是否包含连接回收和健康检查机制

### 5.4 添加数据库 Schema 自动同步机制

1. 在后端启动流程中（`lifespan` 函数）增加 Schema 检查逻辑：对比 ORM 模型定义与实际数据库表结构，自动执行 ALTER TABLE 补齐缺失的字段
2. 避免未来新增字段时再次出现此类问题

## 6. 补充说明

- 用户未查看过浏览器开发者工具的详细响应体或服务器日志，确切的异常堆栈需在修复过程中通过查看 Docker 容器日志获取
- 问题具有**全局性**（不限于 COS 模块），修复时需优先排查数据库层面的共性问题
- 由于 GET 接口正常，说明数据库连接本身没问题，高度指向**写操作时触碰了不存在的字段或表结构不匹配**
- 建议修复后对所有管理后台的保存操作进行逐一验证，确认 500 错误全部消除
