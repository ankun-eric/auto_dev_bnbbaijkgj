# 健康档案页面优化 产品需求文档（PRD）

> 文档版本：v1.0  
> 编制日期：2026-05-18  
> 所属模块：bini-health · 健康档案（ai-home 主页）

---

## 1. 需求概述

### 1.1 背景与目的

当前"健康档案"是 bini-health 用户进入 App 后承载咨询人画像、用药、提醒、共管邀请等信息的核心页面。随着「家庭共管/守护」模式的强化，原有页面在以下几个方面已无法满足实际使用诉求：

- **咨询人切换体验不连续**：滚动后切换条被滚走，操作成本高
- **切换后数据不彻底**：部分模块（如用药计划）仍展示当前登录人的数据，而非选中咨询人的数据
- **画像信息冗余**：Hero 卡承载了过多次要字段（既往病史 / 过敏史 / 家族遗传等），核心信息被稀释
- **共管入口缺失**：共管/守护邀请入口隐藏过深，与"被守护"状态可视性不足
- **提醒区域定位错乱**：站内/微信消息已是默认通道，却仍以开关形式占用主页面位置
- **设备入口对非本人不适用**：右上角设备 Logo 对"非本人咨询人"在业务上不成立

本次优化的核心目的是：

1. **强化以"咨询人"为主轴的页面结构**，所有数据严格随选中咨询人切换
2. **重塑 Hero 卡片信息密度**，突出当下最高频的"今日用药"与"共管关系"
3. **理顺共管/守护体系入口**，让本人态、被守护态、未被守护态在视觉上一目了然
4. **将"AI 外呼提醒"从用药计划层抽到被守护人层**，并迁移至「家庭守护列表」管理
5. **下线对非本人咨询人不适用的设备入口**，并对设备管理体系做配套重构

### 1.2 目标用户

| 用户角色 | 描述 |
|---------|------|
| 健康主账号用户（本人） | 在 App 中维护自己的健康档案、用药、设备的主用户 |
| 共管发起者 | 想要把家人加为咨询人，并发起共管邀请的用户 |
| 共管守护者 | 已经成功守护一个或多个家人健康的用户 |
| 被守护人 | 自己的健康档案被家人共管的用户 |

### 1.3 核心价值

- **一致性**：切换咨询人 = 切换"整页世界"，绝不留任何"本人数据残影"
- **清晰性**：用户一眼看出"我正在看谁的档案"、"这个人是不是被守护"、"今天他还要吃几次药"
- **聚焦性**：移除高频区被低频开关挤占的问题，把宝贵的首屏面积留给关键信息
- **闭环性**：邀请共管、查看守护、管理外呼提醒等流程在「家庭守护列表」里形成闭环

## 2. 功能需求

### 2.1 功能清单总览

| 编号 | 功能模块 | 功能点 | 说明 |
|------|---------|--------|------|
| F1 | 顶部双层吸顶 | 标题栏 + 咨询人切换条整体吸顶 | 下方内容滚动时，标题栏与咨询人切换条同时固定在屏幕顶部 |
| F2 | 咨询人切换数据联动 | 切换咨询人后全页面数据一刀切随之切换 | 包括 Hero 卡、用药计划、今日用药、提醒、共管区域等所有人维度数据 |
| F3 | 被守护角标 | 头像右上角胶囊角标"被守护" | 当主账号已守护该咨询人时，在咨询人头像右上角显示带背景色的圆角胶囊 |
| F4 | Hero 卡重构 | 关系标签 / 字段瘦身 / 今日用药入口 | 弱化关系标签、移除三类历史病史字段、把"在用药物"改为"今日用药 + 数字 + 箭头" |
| F5 | 共管区域 | Hero 卡下方新增"邀请共管"区域 | 根据本人/非本人 + 守护状态，呈现三种不同形态 |
| F6 | 漏打卡提醒区域改造 | 原区域大幅简化为入口 | 原"站内 / 微信 / AI 外呼"开关全部移除；保留一个"打卡提醒设置"入口跳转到家庭守护列表 |
| F7 | AI 外呼提醒迁移 | 从 ai-home 移到家庭守护列表，单层结构 | 在被守护人维度统一配置开关 + 免打扰时段 + 外呼对象，所有用药计划共用一份 |
| F8 | 顶部设备 Logo 下线 | 右上角设备入口图标移除 | 健康档案页右上角不再保留设备入口图标 |
| F9 | 设备管理体系重构 | 我的设备页重做 + 非本人态隐藏入口 | 健康档案页非本人态完全不出现设备入口；本人态保留完整设备管理；非本人设备只在家庭守护列表里以只读视图出现 |

### 2.2 功能详细描述

#### F1 顶部双层吸顶

- **吸顶范围**：「健康档案」标题栏 + 咨询人切换条整体作为一个吸顶组合
- **触发时机**：用户开始向下滚动时立即生效，向上回滚仍保持
- **视觉规则**：
  - 吸顶状态下，底部加一条 1px 的细分割线或轻投影，与下方内容形成层级分隔
  - 吸顶状态下保持与非吸顶时一致的背景色，不做颜色突变
- **交互规则**：
  - 标题栏左侧按原有规则保留（返回 / 标题），右侧不再保留设备 Logo（详见 F8）
  - 咨询人切换条横向滑动选择，被选中咨询人居中（或贴左）并高亮
  - 被守护角标（F3）在吸顶状态下同样可见

#### F2 咨询人切换数据联动（一刀切口径）

- **核心原则**：选中咨询人 ID 变化后，Hero 卡及其下方所有"与人相关"的数据区域，**全部按当前选中咨询人重新拉取**，不再保留任何"本人数据兜底"
- **明确需要联动的区域**（包括但不限于）：
  - Hero 卡（姓名、关系标签、年龄、性别、基础健康指标等）
  - 今日用药数量
  - 用药计划（**当前明确未联动，必须修复**）
  - 提醒/打卡入口对应的被守护人
  - 邀请共管 / 已守护 / 管理 等共管区域
  - 其他健康指标卡片、最近一次测量、健康任务等所有按人维度组织的内容
- **业务规则**：
  - 主账号未守护的"非本人咨询人"，敏感数据按权限读取（详见第 6 章权限设计）
  - 切换过程中应有轻量 loading（骨架屏或局部刷新指示），避免出现"瞬时看到上一个人的数据"
  - 切换后，页面滚动位置回到顶部
- **异常处理**：
  - 选中咨询人的某些数据拉取失败时，对应区域显示"加载失败，点击重试"，其他成功区域正常展示，不阻塞整页

#### F3 被守护角标

- **位置**：咨询人头像的右上角
- **样式**：圆角矩形胶囊（背景色填充 + "被守护"3 个字，字号小于正文）
- **可见范围**：**在主账号视角下，凡是被主账号本人当前已守护的咨询人，其头像右上角都显示"被守护"角标**
  - 例：A 守护了 B 和 C，则在 A 的账号里，咨询人切换条中 B 和 C 的头像上都显示"被守护"角标
- **不可见的情况**：
  - 该咨询人是"本人"自己（不显示，本人不存在"被守护"概念）
  - 该咨询人当前未被主账号守护
- **联动**：Hero 卡内的大头像同步显示此角标（在大头像右上角同样形式呈现）

#### F4 Hero 卡片重构

##### F4-1 关系标签

- **位置**：放在咨询人姓名的**后面**，与姓名同行
- **字号**：明显小于姓名（建议为姓名字号的 60%~70%）
- **显示规则**：
  - 本人：显示文字"本人"
  - 其他咨询人：显示具体关系（"妈妈"、"爸爸"、"儿子"、"女儿"、"配偶"、"奶奶"等，跟随该咨询人在家庭关系中维护的关系字段）
- **样式**：可使用浅灰底圆角小标签或纯灰色小字，避免抢占姓名视觉重心

##### F4-2 字段瘦身

- **移除**：既往病史、过敏史、家族遗传 这 3 项字段在 Hero 卡上**完全不显示**
  - 它们作为档案完整字段仍保留在更深层的"完整档案详情页"里，不在 Hero 卡上展示
- **保留**：基础人口学信息（姓名、关系、年龄/性别）+ 关键即时健康摘要

##### F4-3 今日用药入口（替换原"在用药物"）

- **位置**：原"在用药物"模块所在位置
- **呈现形态**：
  - 标题文案：今日用药
  - 主体：**数字**（今日待打卡用药条数）+ 跳转箭头（如「›」或右向箭头图标）
  - 示例：`今日用药 · 5 ›`
- **点击行为**：直接跳转到打卡页（当前选中咨询人的打卡页）
- **数据规则**：
  - 数字定义：当天所有用药计划中应打卡次数总和（含已打、漏打、待打），具体口径与现有打卡页一致
  - 当数字为 0 时：仍显示 "今日用药 · 0 ›"，点击仍可进入打卡页（不隐藏入口）

#### F5 共管区域（Hero 下方）

在 Hero 卡正下方新增一行"共管/守护"区域，**根据当前选中咨询人的身份与守护状态呈现三种不同形态**：

| 场景 | 当前咨询人 | 守护状态 | 区域呈现 | 点击行为 |
|------|-----------|---------|----------|----------|
| 场景 A | 本人 | — | "已守护 N 人" + 进入箭头 | 点击进入「家庭守护列表」 |
| 场景 B | 非本人 | 当前主账号尚未守护此人 | 显示按钮「+ 邀请共管」 | 点击进入"邀请共管"页面（详见 F5-2） |
| 场景 C | 非本人 | 当前主账号已守护此人 | 显示按钮「管理」 | 点击进入「家庭守护列表」中此被守护人的详情/设置 |

##### F5-1 场景 A：本人态

- 文案模板：`已守护 N 人 ›`（N = 当前主账号正在守护的家人数量；当 N=0 时显示"暂未守护任何人"，并提供一个次级"去添加家人 / 去邀请家人"入口）
- 整行可点击，进入「家庭守护列表」

##### F5-2 场景 B：非本人 + 未共守护

- 主体为一个明显的按钮：`+ 邀请共管`
- 点击进入**"邀请共管"页面**，该页内容包含：
  - 该咨询人基本信息卡（头像 + 姓名 + 关系）
  - 邀请方式说明
  - 邀请操作：生成"邀请该咨询人本人加入守护"的专属邀请链接 / 小程序卡片 / 二维码，发送给该咨询人本人去授权
  - 邀请说明文案：告知用户"对方需要在自己微信里点击邀请并完成授权后，您才正式成为该家人的守护者"
  - 历史邀请记录（若有，按时间倒序列出邀请状态：待对方接受 / 已接受 / 已拒绝 / 已过期）

##### F5-3 场景 C：非本人 + 已共守护

- 主体为一个按钮：`管理`
- 点击进入「家庭守护列表」中**该被守护人的详情页**，详情页包含：
  - 守护关系基本信息
  - AI 外呼提醒设置（详见 F7）
  - TA 的设备只读视图（详见 F9）
  - 解除守护入口（带二次确认）

#### F6 漏打卡提醒区域改造

- **当前状态（待优化）**：包含站内消息开关、微信消息开关、AI 外呼提醒开关 + 时段
- **优化后**：
  - **完全移除**站内消息、微信消息这两个开关（站内/微信通知作为系统默认通道，永远开启，不暴露给用户配置）
  - **AI 外呼提醒整体迁出**到「家庭守护列表」（详见 F7）
  - 原区域**整块移除**，但保留一个**纯入口链接**：「打卡提醒设置 ›」
- **入口跳转行为**：
  - 当选中咨询人为"本人"：点击跳转到家庭守护列表中"对自己"的提醒设置项（即把本人也视作一个被守护人节点来管理外呼）
  - 当选中咨询人为"非本人 + 已守护"：点击跳转到家庭守护列表中"该被守护人"的提醒设置项
  - 当选中咨询人为"非本人 + 未守护"：该入口**隐藏**（无权限管理对方提醒）

#### F7 AI 外呼提醒迁移（单层结构）

##### F7-1 归属位置

- 从健康档案页（ai-home）**整体迁移**到「家庭守护列表」中**每个被守护人**的详情设置项里
- 健康档案页不再保留任何 AI 外呼相关的设置控件

##### F7-2 配置结构（简化为单层）

针对每一个被守护人（含本人对自己），配置一份**唯一**的 AI 外呼提醒设置，**该被守护人名下的所有用药计划共用这一份设置**：

| 字段 | 类型 | 说明 |
|------|------|------|
| AI 外呼总开关 | 开 / 关 | 关闭后，针对该被守护人的所有用药计划都不再发起 AI 外呼 |
| 免打扰时段（起始） | 时间 (HH:mm) | 该时段内不外呼 |
| 免打扰时段（结束） | 时间 (HH:mm) | 该时段内不外呼 |
| 外呼对象 | 单选：被守护人本人 / 守护者 | 选择外呼电话拨给谁 |

##### F7-3 与现有数据的迁移

- 原来挂在"每条用药计划"上的 AI 外呼字段（`ai_call_enabled` / `ai_call_dnd_start` / `ai_call_dnd_end` / `ai_call_target_user_id`）整体停用，不再作为可配置项暴露给用户
- 原挂在"人"维度的总开关 `medication_ai_call_enabled` 升级为本次单层结构的总开关
- **数据迁移策略**：
  - 对每个用户，取其名下所有用药计划中"AI 外呼开关为开"的占多数的状态，作为新结构的总开关初始值；若不存在历史用药计划，则默认"关闭"
  - 免打扰时段：取该用户最近一次有效配置的时段；无历史则默认"22:00 - 07:00"
  - 外呼对象：取最近一次有效配置；无历史则默认"被守护人本人"
- 原有"每条用药计划单独配 AI 外呼"的能力被舍弃，新结构下不再支持单计划差异化配置（如未来需要差异化可再行评估）

##### F7-4 触发逻辑

- 当某条用药计划到达打卡提醒时间，并触发 AI 外呼时：
  - 取该计划所属被守护人的"单层 AI 外呼配置"
  - 若总开关关闭，则不外呼
  - 若当前时间落在免打扰时段内，则不外呼
  - 否则向"外呼对象"发起电话

#### F8 顶部设备 Logo 下线

- 健康档案页右上角原有的"设备 Logo"图标**直接移除**
- "我的设备"入口在健康档案页**暂时不重新挂载**（暂时无主入口）
- 设备相关功能改由以下两条路径承载：
  - **本人维护自己的设备**：通过"我的"/个人中心相关入口（具体入口与原我的设备入口位置一致，本 PRD 不重复定义其位置）
  - **查看被守护人设备**：通过「家庭守护列表」→ 该被守护人详情页 → "TA 的设备"模块（详见 F9）

#### F9 设备管理体系重构

##### F9-1 健康档案页处理

- **本人态**：健康档案页面上**不再出现**设备入口图标（原 Logo 已删除）
- **非本人咨询人态**：健康档案页**完全不出现**任何设备入口

##### F9-2 我的设备页（本人）重做

- 入口：从"我的/个人中心"进入
- 范围：仅本人名下的设备（一个登录账号绑定的设备）
- 功能：
  - 设备列表（图片 + 设备名 + 型号 + 绑定时间 + 当前状态）
  - 新增绑定设备（扫码 / 蓝牙搜索 / 手动输入序列号）
  - 解绑设备（带二次确认）
  - 查看设备实时数据 / 历史上报数据
  - 设备固件升级提醒（如有）

##### F9-3 TA 的设备（非本人，只读）

- 入口：「家庭守护列表」→ 被守护人详情页 → "TA 的设备"模块
- 业务约束（重要）：
  - 一台设备只能绑定一个登录账号；被守护人的设备绑定在被守护人自己的微信账号上
  - 主账号无法直接增/删/解绑被守护人名下的设备
- 功能：
  - **只读**展示被守护人名下设备列表（设备名 + 状态 + 最近一次上报时间 / 数值）
  - **只读**查看最近一段时间的测量数据
  - 提供"提醒 TA 绑定设备"入口（当被守护人尚无设备时显示）：点击后向被守护人发送一条提醒消息（站内/微信）

## 3. 页面/界面设计

### 3.1 页面结构总览

```
健康档案页（ai-home）
├─ [吸顶层] 顶部标题栏（左：返回，中：健康档案，右：— 已删除设备 Logo）
├─ [吸顶层] 咨询人切换条（横滑头像列表，被守护者头像带"被守护"角标）
├─ Hero 卡（姓名 + 小号关系标签 / 年龄性别 / 关键摘要 / 今日用药 N ›）
├─ 共管区域（三态切换：已守护 N 人 ›  /  + 邀请共管  /  管理）
├─ 其他健康卡片（按人维度的指标、任务、记录等，全部随选中咨询人切换）
└─ [入口] 打卡提醒设置 ›（点击跳转家庭守护列表对应项）
```

### 3.2 主要页面说明

#### 3.2.1 健康档案页（重构后）

## 4. 非功能性需求

### 4.1 性能要求

- 咨询人切换后，主屏数据应在 800ms 内完成可视化更新（含骨架屏 → 真实数据）
- 吸顶切换条滚动应保持 60fps，无卡顿
- 邀请链接 / 二维码生成应在 500ms 内完成

### 4.2 安全要求

- 邀请链接需为一次性、带时效（建议 7 天）且与被邀请人手机号 / 微信 OpenID 绑定校验
- 非本人咨询人的敏感健康数据仅在已建立守护关系时可读
- TA 的设备数据只读，主账号无任何写操作权限

### 4.3 兼容性要求

- 微信小程序：兼容主流机型与系统版本（与现有产品基线一致）
- 与既有家庭关系、用药计划、打卡、设备、家庭守护列表等模块的接口完全向后兼容

## 5. 业务规则与约束

1. **数据归属唯一性**：所有按"人"维度展示的数据，必须严格属于当前选中咨询人，不允许出现"本人数据兜底"或"上一个人数据闪现"
2. **被守护角标可见性**：只在主账号视角下、且主账号本人正在守护该咨询人时显示
3. **三种共管形态互斥**：本人态 / 非本人未守护 / 非本人已守护，同一时间仅展示其中一种
4. **AI 外呼配置唯一性**：每个被守护人只有一份 AI 外呼配置，所有用药计划共用，不再支持单计划差异化
5. **设备绑定单账号性**：一台设备同一时间只能绑定一个登录账号，主账号无法跨账号绑/解被守护人的设备
6. **历史病史等敏感字段不在 Hero 卡呈现**，但仍可在完整档案详情页查看
7. **站内 / 微信消息为默认通道**，不向用户暴露开关
8. **本期所有功能一次性完整上线**，不拆分批次

## 7. 异常处理与边界情况

1. **切换咨询人时网络异常**：整页保留上一份骨架屏 + 顶部提示"网络异常，点击重试"，禁止显示上一个人的数据
2. **今日用药数为 0**：显示 `今日用药 · 0 ›`，仍可点击进入打卡页
3. **已守护 0 人**：本人态共管区域显示"暂未守护任何人，去邀请家人 ›"
4. **被守护人解除守护**：原"管理"按钮自动变为"+ 邀请共管"，"被守护"角标消失，相关数据按未守护权限重新拉取
5. **邀请链接过期**：被邀请人点击后跳转过期提示页，主账号侧历史邀请列表显示"已过期"
6. **被守护人未绑定任何设备**：TA 的设备模块显示"TA 暂未绑定设备，提醒 TA 绑定 ›"
7. **AI 外呼免打扰时段跨天**（如 22:00 - 07:00）：按跨天逻辑判断，避免遗漏
8. **AI 外呼外呼对象选择"守护者"但当前不存在守护关系**：外呼对象自动回退为"被守护人本人"
