## 功能简介

本次更新重点解决 **「支付宝 H5 测试报错『未安装 python-alipay-sdk』」复发问题**，并将其升级为长期治理：

1. **真正修复**：服务器上的 `alipay_service.py` 已更新为最新版（移除 `AliPayCert` 顶层硬依赖，改为 try/except lazy import 兼容 python-alipay-sdk 3.3.0/4.x），并通过 `--no-cache` 强制重建 backend 镜像，确保依赖真正装入容器。
2. **启动期分级自检**：后端启动时自动检查 10 个核心 + 三方 SDK 是否可 import；**核心依赖（FastAPI/SQLAlchemy/aiomysql/Pydantic）缺失会让容器直接退出**，可选依赖（支付宝/微信/腾讯云/阿里云/OSS/COS 等）缺失只打 CRITICAL 日志，不阻塞启动。
3. **后台 SDK 健康看板**：
   - 支付配置页**顶部红绿灯卡片**：全绿一行通过，任意红展开 banner + 一键复制安装命令；
   - 系统管理新增**「环境健康检查」菜单**：完整 4 分组（核心运行时 / 支付 SDK / 短信 SDK / 对象存储 SDK）+ 总览 Statistic + 重新检测按钮；缺失项默认展开、正常项默认折叠。

## 使用说明

### 一、查看支付配置页顶部 SDK 自检卡片（最常用）

1. 用管理员账号登录管理后台
2. 左侧菜单点击 **支付配置**
3. 页面**最顶部**会展示一行（或一块）SDK 自检结果：
   - **全绿一行**：✅ 支付环境自检通过：N 项依赖均正常 — 此时支付宝/微信支付的 SDK 都装好了，可以放心点「测试连接」
   - **红色 banner**：列出所有缺失的支付 SDK 名称、用途、安装命令；右侧「复制」按钮一键复制安装命令，粘贴到容器执行即可
4. 修复后点击 banner 上的 **「重新检测」** 按钮，红绿灯实时刷新

### 二、进入完整环境健康检查菜单（运维巡检）

1. 用管理员账号登录管理后台
2. 左侧菜单：**系统管理 → 环境健康检查**
3. 顶部卡片显示总览：
   - 总计：当前注册的 SDK 总数（默认 10 项）
   - 正常：已装且 import 成功的项数
   - 核心缺失：核心依赖缺失数（理论上始终为 0，否则容器根本起不来）
   - 可选缺失：三方 SDK 缺失数（如 wechatpy/oss2 未集成，会显示在这里）
   - 最近检测时间
4. 下方四个分组卡片（各自独立折叠）：
   - **核心运行时**：FastAPI / SQLAlchemy / aiomysql / Pydantic
   - **支付 SDK**：python-alipay-sdk / wechatpy
   - **短信 SDK**：tencentcloud-sdk-python / aliyun-python-sdk-core
   - **对象存储 SDK**：oss2 / cos-python-sdk-v5
5. 每张卡片右上角红色徽章数字 = 该分组缺失项数。缺失项默认展开，每一项展示：图标（绿✓/红✗）+ 模块名 + 等级标签（核心/可选）+ 用途说明 + 版本号
6. 缺失项下方还会带上：`pip install xxx` 安装命令 + 一键复制按钮 + 详细错误原文（如 `No module named 'oss2'`）
7. 顶部右侧 **「重新检测」** 按钮：在容器内 `pip install` 后无需重启，点击该按钮即可立即看到最新状态

### 三、回归验证支付宝 H5「测试连接」按钮

1. 进入 **支付配置 → 支付宝 H5** Tab
2. 点击右上角 **「测试连接」** 按钮
3. **预期表现**：
   - ✅ **不再返回**：「测试失败：支付宝 SDK 未安装：未安装 python-alipay-sdk...」
   - ✅ 如果通道已正确配置：返回「测试通过：参数正确、网络连通、签名/验签正常」
   - ✅ 如果 DB 中私钥/公钥仍是占位值：返回业务错（如「调用支付宝异常：RSA key format is not supported」），这是配置问题、不是 SDK 问题
4. 同步抽测**支付宝 APP / 微信小程序 / 微信 APP** 三个 Tab 的「测试连接」按钮，确保都不再报 SDK 未安装

## 注意事项

1. **菜单仅管理员可见**：「环境健康检查」与「支付配置页」均要求 `admin` 角色，普通用户/商家用户访问会跳登录页或 401。
2. **SDK 安装清单是敏感信息**：接口 `/api/admin/health/sdk` 已挂在 admin 鉴权后，避免暴露系统依赖明细。
3. **「重新检测」只能反映安装情况，不校验连通性**：本期不做支付宝沙箱 ping、不做 OSS 桶可达性等连通性自检；如需上线连通性自检，将作为 V2 范围。
4. **「可选 SDK 缺失」不是 Bug**：本项目目前只集成了 `python-alipay-sdk` 与 `tencentcloud-sdk-python`，因此 `wechatpy/aliyunsdkcore/oss2/qcloud_cos` 显示缺失是**预期表现**，不影响业务运行。如未来需要使用这些 SDK，安装后红绿灯会自动变绿，无需改代码。
5. **核心依赖缺失永远不会到这个页面**：FastAPI/SQLAlchemy 等核心包缺失，容器在启动期就会因 `RuntimeError("[SDK-HEALTH] 核心依赖缺失")` 直接退出，根本来不及打开后台。
6. **本次 alipay 测试报错的真正根因**：服务器代码版本未及时更新到含修复 commit 的版本（旧版 `alipay_service.py` 顶层硬 import `AliPayCert`，但 `python-alipay-sdk 3.3.0` 中无此类）。本次部署使用 `docker compose build --no-cache backend admin-web` 强制重建，并新增 SDK 健康看板作为长期可见性保障。
