## 功能简介

### 修复了什么 Bug

在管理后台「功能按钮管理」中，把按钮类型选成「页面跳转（page_navigate）」、填上跳转地址（如 `/health-profile`）保存后：

- **现象 A**：再次点击「编辑」，发现"跳转地址"输入框是**空的**——保存按钮其实只是"看着成功了"，链接根本没写进数据库。
- **现象 B**：到 H5 端 AI 对话首页，点击该按钮（功能宫格或下方胶囊条），**不会**跳转到目标页面，反而会在对话区"以用户身份"发出一条消息（消息内容是按钮名，比如「健康档案」三个字），对话区被污染。

### 修复后预期

- 选「页面跳转」类型保存的跳转地址，**真的写入数据库**，下次编辑回填正常。
- H5 用户在宫格或胶囊条点击「健康档案」这类页面跳转按钮，**浏览器直接跳转**到配置的页面（如 `/health-profile`），对话区不再"突然冒出一条消息"。
- 即便后台真的配错了（链接为空），H5 端也只会弹一个轻提示「该按钮跳转地址异常，请联系管理员」，不会污染对话流。

### 不影响什么

- 「先弹卡片再跳转 = 是」的流程不变（先弹卡片 → 用户点卡片按钮 → 再跳转）。
- 老的 `external_link` 类型按钮、各种 AI 功能（问卷 / 拍照识药 / 文件上传 / 快捷提问）行为完全不变。
- 后端校验规则、数据库结构、所有 API 接口签名均无任何变化。

## 使用说明

### 场景 1：运营在管理后台配置一个"页面跳转"按钮

1. 访问 [管理后台 - 功能按钮管理](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/function-buttons) 并登录。
2. 点击「新增按钮」（或在列表里找到一个已有按钮，点击「编辑」）。
3. 弹窗中：
   - 「名称」：例如 `健康档案`
   - 「图标 Emoji」或「图标 URL」：选一个
   - **「按钮类型」选「页面跳转」**
   - **「跳转地址」填 `/health-profile`**（也可以是 `https://...` 外链，或 `pages/...` 小程序页面）
   - 「先弹卡片再跳转」：保持默认「否」（直接跳转）
   - 其它字段按需配置
4. 点击「保存」。
5. **验证**：不刷新页面，再次点击同一行的「编辑」，检查"跳转地址"是否回填为 `/health-profile`（修复前是空，修复后回填正确）。

### 场景 2：用户在 H5 端使用"页面跳转"按钮

1. 在手机浏览器或 PC 浏览器中打开 [AI 对话首页](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/ai-home)。
2. 登录后，在首页中部「功能宫格」中找到刚才配置的按钮（如「健康档案」），点击。
3. **预期**：浏览器**直接跳转**到 `/health-profile` 页面，对话区不会出现"突然多一条用户消息"的情况。
4. 同样地，在对话输入框上方的「胶囊条」中点击同一个按钮，行为也是**直接跳转**。

### 场景 3：选「先弹卡片再跳转 = 是」的流程

1. 在管理后台编辑该按钮，把「先弹卡片再跳转」切换为「是」。
2. 保存后，到 H5 端点击该按钮：
   - 对话区会先出现一张"说明卡片"。
   - 用户点击卡片上的按钮后才真正跳转。

### 场景 4：修复后历史数据的处理（运营必须了解）

> ⚠️ **重要**：受这个 Bug 影响的按钮（已选过「页面跳转」类型并保存过的按钮），其在数据库中的"跳转地址"字段已经被擦写成了空。修复发布后**不会自动恢复**历史值。

**运营需要做的**：

1. 在管理后台「功能按钮管理」中，逐一打开每个按钮类型为「页面跳转」的按钮。
2. 把对应的「跳转地址」**重新填一遍**（如「健康档案」按钮 → `/health-profile`）。
3. 点击「保存」。
4. 验证一次：再次编辑该按钮，"跳转地址"已正确回填。
5. 到 H5 端实际点击一次按钮，确认跳转生效。

预计需要处理的按钮数量很少（按目前业务只有"健康档案"等少数几个），人工处理 1–2 分钟即可。

## 注意事项

1. **支持的跳转地址格式**：
   - `https://...` / `http://...`：外链，浏览器整页跳转
   - `/xxx`：H5 站内路径，使用 router push 跳转（如 `/health-profile`、`/services`）
   - `pages/xxx`：小程序页面路径（保留兼容）
2. **跳转地址校验**：后端校验非空时必须以 `http(s)://` 或 `/` 或 `pages/` 开头，否则 400 报错。如：`javascript:alert(1)` 这种会被拒绝。
3. **空跳转地址的兜底**：理论上不会再出现，但万一出现（如运营失误保存了空值），H5 端会弹"该按钮跳转地址异常，请联系管理员"，并打前端埋点日志 `page_navigate_empty_url`，便于排查。
4. **是否影响"先弹卡片再跳转"流程**：不影响。两种模式都能正确读到跳转地址。
5. **是否需要重新登录 / 清缓存**：不需要。管理后台与 H5 端在阶段 2 部署时已经重建镜像，刷新页面即可看到新行为。
