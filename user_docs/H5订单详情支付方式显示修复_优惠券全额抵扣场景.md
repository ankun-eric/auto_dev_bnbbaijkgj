## 功能简介

### 修复了什么 Bug？

在本次修复之前，当用户使用「优惠券全额抵扣」下单（实付 ¥0.00）后，
**H5 端订单详情页**展示的「支付方式」字段会**错误地显示为「支付宝（H5）」**（中文括号），
实际上正确的展示应该是「优惠券全额抵扣」。

同样的隐患也存在于：

- 余额全额支付 0 元单 → 错误显示成预选通道文案
- 积分兑换 0 元单 → 错误显示成预选通道文案
- 商家端订单列表的「支付方式」列在上述场景中也会错误展示

### 本次修复后的效果

修复完成后，「支付方式」字段会按「以实付方式为准、预选通道仅作通道补充」的优先级展示：

| 场景 | 实付金额 | 修复前 H5 显示 | 修复后 H5 显示 |
| --- | --- | --- | --- |
| 优惠券全额抵扣 0 元单 | 0 | 支付宝（H5） ❌ | **优惠券全额抵扣** ✅ |
| 余额全额支付 0 元单 | 0 | 预选通道文案 ❌ | **余额支付** ✅ |
| 积分兑换 0 元单 | 0 | 预选通道文案 ❌ | **积分兑换** ✅ |
| 真实支付宝 H5 支付 | > 0 | 支付宝（H5） ✅ | **支付宝（H5）** ✅（保持不变） |
| 真实微信小程序支付 | > 0 | 微信支付（小程序） ✅ | **微信支付（小程序）** ✅（保持不变） |

### 修复涉及范围

- **后端代码**：`backend/app/api/unified_orders.py` 中的 `_build_payment_method_text` 函数（核心优先级调整）
- **后端代码**：`backend/app/api/merchant.py` 商家端订单列表 `payment_method_text` 改为复用统一构造器
- **后端测试**：新增 `backend/tests/test_payment_method_text_priority_v10.py`，覆盖修复方案 §四 的 TC-01 ~ TC-10
- **客户端无任何代码变更**：H5 / 小程序 / Flutter App / admin 后台均已统一从后端 `payment_method_text` 字段取值，本次仅后端修复即可全端一致生效，**无需重新下载任何客户端**

## 使用说明

### 一、H5 用户端体验步骤

1. **下单一张优惠券全额抵扣 0 元单**
   - 用手机浏览器打开 [H5 主页面](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/)
   - 登录用户账号
   - 选择任意一个商品，进入下单流程
   - 选择已领取的「全额抵扣券」（让实付金额变为 ¥0.00）
   - 提交订单完成下单
2. **查看订单详情**
   - 进入「我的 → 我的订单」，点击刚下的 0 元单
   - 在订单详情页向下滚动，找到「支付方式」一行
3. **核对显示文案**
   - 修复后此处应显示「**优惠券全额抵扣**」
   - 不应再出现「支付宝（H5）」、「微信支付（小程序）」等预选通道文案

### 二、商家端体验步骤

1. 用商家账号登录 [商家端订单列表](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/orders)
2. 找到一张「优惠券全额抵扣」0 元订单
3. 「支付方式」列应展示「**优惠券全额抵扣**」，与 H5 端一致

### 三、admin 管理后台体验步骤（回归确认）

1. 登录 [管理后台](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/)
2. 进入「订单管理 → 统一订单」
3. 0 元单的「支付方式」列应展示「**优惠券全额抵扣**」（修复前 admin 后台已正确，此为回归确认）

## 注意事项

1. **本次修复仅涉及后端**：H5 / 小程序 / Flutter App 等客户端**无需重新下载或安装**；用户刷新订单详情页面即可看到正确文案
2. **历史订单自动修正**：因为「支付方式」文案是后端响应时实时构造的，**之前已生成的所有 0 元订单，访问详情时也会自动展示为正确的中文文案**，无需做数据迁移
3. **不影响交易、不影响金额、不影响订单状态**：本次仅是文案展示优先级修复，订单的 `payment_method` 数据库落库本身一直是正确的（`coupon_deduction` / `balance` / `points`）
4. **统一取数口径**：建议各端订单展示「支付方式」时**只信任**后端返回的 `payment_method_text` 字段，不要在前端做本地拼装兜底，避免后续再出现文案不一致
