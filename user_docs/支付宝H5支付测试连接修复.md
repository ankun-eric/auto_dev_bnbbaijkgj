## 功能简介

本次修复让「支付宝 H5 / APP」配置在以下四种私钥粘贴姿势下都能直接通过测试，
**用户不再需要去甄别支付宝密钥工具同时生成的两个文件谁是谁**：

1. **PKCS#8 + PEM 头**（`-----BEGIN PRIVATE KEY-----` ... `-----END PRIVATE KEY-----`）— 推荐
2. **PKCS#8 裸 base64**（`应用私钥PKCS8.txt` 文件内容）— 推荐
3. **PKCS#1 + PEM 头**（`-----BEGIN RSA PRIVATE KEY-----` ... `-----END RSA PRIVATE KEY-----`）
4. **PKCS#1 裸 base64**（`应用私钥RSA2048.txt` 文件内容）

系统会在保存时**自动统一转换为 PKCS#8 PEM 格式**后再加密落库；同时对**完全无法识别**的内容
（乱码、误粘其他文件等）会在「保存」按钮处直接拦截并给出友好的中文报错文案，
不再像以前那样要等到点击「测试」按钮才暴露错误。

## 操作步骤

### 路径一：用户已遇到 Bug — 立即修复（5 分钟）

如果您之前已经在「支付宝 H5 / APP」中保存过私钥并踩了这个坑，最快恢复方式：

1. 打开 [管理后台 → 支付配置](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/payment-config)
   并使用管理员账号登录
2. 切到「支付宝 H5」（或「支付宝 APP」）标签页 → 点「编辑」按钮
3. 在「应用私钥」输入框下方现在会看到一行**灰色小字提示**：

   > 请粘贴支付宝密钥工具生成的「应用私钥PKCS8.txt」文件内容
   > （**注意不是** 应用私钥RSA2048.txt）

4. 用文本编辑器打开支付宝密钥工具同目录下的 **`应用私钥PKCS8.txt`** 文件，**全选 + 复制**
5. 把「应用私钥」输入框**清空**后粘贴 PKCS#8 内容，点「保存」
6. 点「测试连接」按钮 → 应当返回：

   > 测试通过：参数正确、网络连通、签名/验签正常

> 备注：测试环境 + 正式环境共用同一把密钥时，两套环境都重新粘贴一次即可，
> 无需手动删除旧记录，PUT 接口会用新值覆盖旧密文。

### 路径二：新接入支付宝商户

1. 在支付宝开放平台「密钥工具」生成 RSA2048 密钥对（同时输出两个 txt 文件）
2. 直接复制 **`应用私钥PKCS8.txt`** 的内容（推荐），或复制 `应用私钥RSA2048.txt` 也可以
   （系统会自动识别并转换）
3. 在「应用私钥」输入框中粘贴，按页面提示填好其他字段，点「保存」即可
4. 点「测试连接」自检是否通过

## 修复后系统的关键行为变化

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 粘贴 `应用私钥RSA2048.txt`（PKCS#1 裸 base64） | 静默保存 → 测试时报 `RSA key format is not supported` | 自动识别并转换为 PKCS#8 落库 → 测试通过 |
| 粘贴 `应用私钥PKCS8.txt`（PKCS#8 裸 base64） | 测试通过 | 测试通过（行为不变） |
| 粘贴含 `-----BEGIN RSA PRIVATE KEY-----` 头 | 静默保存 → 测试时报错 | 自动识别 PKCS#1 PEM → 转 PKCS#8 → 测试通过 |
| 粘贴乱码 / 非 base64 字符 | 静默保存 → 测试时报底层异常 | 「保存」按钮直接拦截，返回 422 + 友好中文文案 |
| 历史脏数据（库内已有 PKCS#1 私钥） | 每次测试都失败 | 运行时自动兜底转换 PKCS#8，测试通过 |
| 真的拿到了底层 SDK 抛 RSA 错 | 弹「调用支付宝异常：RSA key format is not supported」 | 弹「应用私钥格式不被支持。请使用 应用私钥PKCS8.txt 文件…」 |

## 注意事项

1. 仅 `super_admin` / `admin` 角色可访问支付配置页（其他角色看不到该菜单）
2. 数据库存储的私钥**始终是 AES-256-GCM 加密**形态（密文以 `ENC::AES256::` 为前缀），
   本次修复**未更换** `PAYMENT_CONFIG_ENCRYPTION_KEY`，所有历史密文继续可用，**零迁移**
3. 「应用私钥」字段在编辑场景下**留空 = 保留旧值**（不会因为留空而清空已保存的私钥）
4. 修复同时覆盖「支付宝 H5」和「支付宝 APP」两个通道（两者共享同一套底层逻辑）
5. 如果您已经在数据库中存了 PKCS#1 旧私钥，可以选择：
   - **方案 A**（推荐）：去支付配置页重新粘贴 `应用私钥PKCS8.txt` 内容并保存，让库内数据归一
   - **方案 B**：什么都不做，运行时兜底会自动转换，但日志中会保留为 PKCS#1 形态
6. 修复**未改动**：数据库 schema、AES 加密密钥、支付宝网关地址、SDK 版本

## 常见疑问

### Q1：为什么支付宝官方会同时给两个私钥文件？

- **PKCS#1** 是 RSA 早期标准，仅描述 RSA 算法本身（裸 base64 无算法标识）
- **PKCS#8** 是更通用的私钥封装标准，外层加了算法标识（适配多种算法）
- 现代密码学库（Java JCE / Python cryptography / Go crypto）**默认偏好 PKCS#8**
- 支付宝为兼容老商户两个都给，但**新接入方一律推荐 PKCS#8**

### Q2：以前能用、现在不行，是什么原因？

- 本项目的 `python-alipay-sdk` 在不同 `cryptography` 库版本下行为略有差异
- 旧版 `cryptography` 对裸 base64 的 PKCS#1 容忍度较高
- 当前线上 `cryptography` 收紧了校验，导致同样的旧操作不再能跑通
- 本次修复把识别/转换逻辑**收口在系统内部**，未来无论 SDK / `cryptography` 怎么升级，
  业务层都不再受影响

### Q3：测试环境通过后，正式环境是否也要做？

是。由于「测试环境」和「正式环境」复用同一把密钥，建议在两套环境都点击一次「测试连接」按钮。
本次修复同时部署到了两套环境，行为完全一致。
