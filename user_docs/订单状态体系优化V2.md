## 二、功能简介

本次按 PRD V2「核销订单状态体系优化」实施，将原本「7 状态 + 退款双轴」升级为「**12 状态单轴 + 核销码 5 态独立机 + 三端分层映射**」，对标 6 大主流到店服务平台（美团 / 大众点评 / 抖音生活服务 / 快手本地生活 / 京东到店 / 小红书本地生活）的最佳实践。

### 1. 订单 12 状态枚举

| # | 枚举值 | 中文 | 终态 | 说明 |
|---|---|---|---|---|
| 1 | `pending_payment` | 待付款 | ❌ | 订单已创建未支付 |
| 2 | `pending_shipment` | 待发货 | ❌ | 实物商品已支付待发货 |
| 3 | `pending_receipt` | 待收货 | ❌ | 实物已发货待签收 |
| 4 | `pending_appointment` | 待预约 | ❌ | 已支付但未设预约时间（**新增**） |
| 5 | `appointed` | 已预约 | ❌ | 已设预约时间，等待赴约（**新增**） |
| 6 | `pending_use` | 待核销 | ❌ | 已支付/预约日到，等待到店核销 |
| 7 | `partial_used` | 部分核销 | ❌ | 多次核销商品，已使用部分次数（**新增**） |
| 8 | `completed` | 已完成 | ✅ | 全部履约完成 |
| 9 | `expired` | 已过期 | ✅ | 到期未核销，履约失败（**新增**） |
| 10 | `refunding` | 退款中 | ❌ | 退款流程进行中（**新增**） |
| 11 | `refunded` | 已退款 | ✅ | 退款完成（**新增**） |
| 12 | `cancelled` | 已取消 | ✅ | 用户取消/超时未付 |

> 旧的 `pending_review` 已不再产生（数据迁移自动改为 `completed`），但 ENUM 中保留以兼容历史数据。"待评价"在客户端按 `completed AND has_reviewed=false` 动态计算显示。

### 2. 核销码 5 态独立状态机

| 枚举 | 中文 | 说明 |
|---|---|---|
| `active` | 可核销 | 正常可核销 |
| `locked` | 已锁定 | 核销中 / 双确认中 |
| `used` | 已核销 | 已完成核销 |
| `expired` | 已过期 | 已过期作废 |
| `refunded` | 已退款 | 已退款作废 |

### 3. 客户端 5 Tab + 全部

H5 我的订单页 Tab 顺序：**全部 / 待付款 / 待收货 / 待使用 / 已完成 / 退货售后**。

- 取消"待评价"独立 Tab，已完成 Tab 中未评价订单卡片右下角显示"去评价"按钮。
- "全部"Tab 取消二级筛选，纯时间倒序，卡片右上角直接显示状态文字。
- "待使用"Tab 包含 `pending_appointment / appointed / pending_use / partial_used` 4 种状态。
- "已完成"Tab 包含 `completed + expired` 合并展示。
- "退货售后"Tab 子筛选：**审核中 / 退款中 / 已退款 / 已拒绝**（文案对标主流平台）。

### 4. 管理后台

- 状态筛选下拉展开为 12 个完整枚举 + 中文标签。
- 新增"核销码状态"筛选维度（5 态独立）。
- 退款流程辅助筛选文案改为：审核中 / 已批准 / 退款中 / 已拒绝 / 已退款。
- Admin Tab 也按 12 状态完整展示，外加"全部"和"退款申请"工作流入口。

### 5. 后端核心接口（V2 新增）

- `GET /api/orders/unified?tab=<all|pending_payment|pending_receipt|pending_use|completed|refund_aftersales>&sub_tab=<all|reviewing|refunding|refunded|rejected>` — 客户端 5 Tab 列表。
- `GET /api/orders/unified/counts` — 返回 8 个旧字段 + 5 个 V2 字段（`v2_pending_payment / v2_pending_receipt / v2_pending_use / v2_completed / v2_refund_aftersales`）。
- `POST /api/orders/unified/{id}/appointment` — 用户设置预约时间，订单从 `pending_appointment → appointed`。
- `GET /api/admin/orders/unified?status=<...>&redemption_code_status=<...>` — 支持 12 状态完整筛选 + 核销码状态独立筛选；status 支持逗号分隔多值。
- `GET /api/admin/orders/v2/enums` — 12 状态 + 5 核销码 + 7 退款标签。
- `GET /api/admin/orders/v2/stats` — GMV / refunded_amount / net_revenue / fulfillment_rate / status_breakdown / redemption_code_breakdown。

订单响应（`GET /api/orders/unified`、`GET /api/orders/unified/{id}`）新增 4 个客户端展示字段：

- `display_status`：动态计算的中文展示文本（含"待评价"特殊处理）。
- `display_status_color`：建议的展示色值。
- `action_buttons`：可显示的按钮 key 列表（`pay / cancel / confirm_receipt / set_appointment / show_qrcode / review / view_refund / rebuy`）。
- `badges`：徽章文本（如 `可核销 / 部分已核销 / 已预约`）。

## 三、数据库变更

| 表 | 变更 |
|---|---|
| `unified_orders` | `status` ENUM 扩列到 13 值（12 + pending_review 兼容） |
| `order_items` | 新增 `redemption_code_status VARCHAR(16) DEFAULT 'active'` + 索引 |
| `order_items` | 新增 `redemption_code_expires_at DATETIME NULL` |
| `system_configs` | 写入 `orders_status_v2_migration_version=1`（防重复迁移） |

### 一次性数据迁移规则（幂等）

1. `pending_review → completed`
2. `refund_status=refund_success AND status<>cancelled → refunded`
3. `refund_status IN (applied/reviewing/returning) → refunding`
4. `pending_use AND order_items.redemption_code_expires_at < NOW() → expired`

迁移仅在首次启动时执行，由 `system_configs.orders_status_v2_migration_version` 标志位控制，重复部署不会重复迁移。

## 四、用户使用说明

### A. H5 用户：我的订单页

1. 打开 [我的订单](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/unified-orders)。
2. 顶部 Tab 选择查看维度：全部 / 待付款 / 待收货 / 待使用 / 已完成 / 退货售后。
3. **待付款** Tab：每个订单卡片右下角显示「取消订单」「去支付」按钮。
4. **待收货** Tab：包含未发货 + 已发货待签收的订单，已发货可点「确认收货」。
5. **待使用** Tab：包含待预约/已预约/待核销/部分核销的订单。  
   - 待预约（`pending_appointment`）：右下角显示「去预约」紫色按钮。  
   - 待核销/部分核销：显示「核销码」按钮调出二维码。  
   - 卡片底部时间右侧出现绿色「可核销」徽章（醒目提示用户尽快赴约）。
6. **已完成** Tab：包含 `completed`（已正常完成）+ `expired`（到期未核销）两类。  
   - 已完成且**未评价**：状态文字显示"待评价"，按钮区显示「去评价」绿色按钮。  
   - 已完成且已评价 / 已过期：按钮区显示「再来一单」。
7. **退货售后** Tab：二级筛选条按 审核中 / 退款中 / 已退款 / 已拒绝 切换。
8. **全部** Tab：纯时间倒序，每张卡片右上角直接用色块显示当前状态文字。

### B. 管理员：订单管理后台

1. 打开 [订单管理](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/product-system/orders)。
2. 顶部 Tab 可一键切换到 12 个状态中的任一个 + 全部 + 退款申请。
3. 筛选条新增三类：
   - **状态下拉**：列出 12 个枚举完整中文标签 + "全部状态"。
   - **核销码状态**：active / locked / used / expired / refunded — 与订单主状态正交。
   - **退款流程**：审核中 / 已批准 / 退款中 / 已拒绝 / 已退款。
4. 列表 Tag 颜色按状态自动着色。
5. 操作按钮（发货 / 退款审批 / 退款详情）逻辑与原版本一致，未修改。
6. 数据接口：可直接调用 `GET /api/admin/orders/v2/stats` 获取整体 GMV / 履约率等指标，方便对接 BI。

## 五、注意事项

1. **旧接口保留**：`GET /api/orders/unified`（不带 tab 参数）、老的 `status=<7 旧值>` 完全兼容；旧的小程序、Flutter App、verify-miniprogram 不需要改造即可继续运行。
2. **`pending_review` 不再产生**：数据迁移已把所有存量 pending_review 改为 completed；新逻辑下，订单不会再进入 pending_review 状态。但若有旧数据走漏网，仍能正确显示（卡片显示"待评价"）。
3. **状态机推进规则**：
   - 实物订单（fulfillment_type=delivery）：支付后 → `pending_shipment`。
   - 到店且需预约（appointment_mode 非 none 且未设时间）：支付后 → `pending_appointment`。
   - 到店且已设预约时间：支付后 → `appointed`。
   - 其它到店：支付后 → `pending_use`。
4. **退款融合**：发起退款时主状态 `status` 直接进入 `refunding`；撤回退款时回到 `pending_use`（实物订单可由商家手动改回 `pending_receipt`）；`refund_status` 字段保留作详细审计。
5. **核销码状态机**：本期仅在数据库层落地字段 + admin 筛选维度；`active → locked → used / expired / refunded` 完整流转将在下个版本由商户端核销 API 全面接入，本期保持手动改库测试可用。
6. **GMV / 履约率口径**：
   - GMV = SUM(paid_amount) WHERE status='completed'
   - 履约率 = completed / (completed + expired + cancelled)
   - 净收入 = GMV - 已退款总额
7. **不在本期范围**：卡功能购卡链路、营销/优惠券与订单状态的耦合优化、订单结算/财务报表链路重构、Flutter App / 微信小程序 / verify-miniprogram 的 UI 改造（这些客户端继续走老接口）。
8. **灰度方案**：项目部署即更新即生效，未实现 status_v2 双写或前后端版本号灰度，部署后立刻全量启用 V2 体系。
