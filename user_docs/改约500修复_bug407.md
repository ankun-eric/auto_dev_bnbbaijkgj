## 问题概述（一句话）

**手机号同时是顾客 + 店主（双重身份）的账号，在用户端 H5 / 小程序 / App 进入"统一订单详情 → 改约"，点【确认改期】后后端返回 HTTP 500，前端弹出「改约失败，请稍后再试或联系客服」。**

本次 v3 修复已彻底解决此 500 故障。

---

## 真实根因（与"双重身份"实际无关）

排查到最深的服务器堆栈后定位到的真实根因，**与 phone 反查、身份判定、改约次数无关**：

1. 改约接口 `/api/orders/unified/{id}/appointment` 在写库后会调用 `notify_order_rescheduled` 下发改期通知。
2. 通知函数内部用 `getattr(order, "store", None)` / `getattr(order, "user", None)` 取门店和用户对象。
3. 在 SQLAlchemy **异步 session** 中，未预加载的 relationship 一旦被这种"惰性方式"读取，会立即抛出 `MissingGreenlet: greenlet_spawn has not been called; can't call await_only() here`。
4. 该异常一旦抛出，会把当前 session 标记为「已损坏」状态。
5. 主流程外层虽然有 `try/except`，但 catch 之后 `return {...}` 表达式里又对 ORM 对象做了一次属性读取（`order.reschedule_count`），**第二次再触发 lazy-load**，再次抛出同一个 `MissingGreenlet`，最终 FastAPI 兜底为 500 抛给前端。

> 正因为这条链路在「商家身份号」上更容易触发（其订单关联了 store_id），所以现象上看起来"只有双重身份号才会 500"——**但其实任何只要触发该 lazy-load 路径的订单都会爆**。

---

## 本次修复要点

### 后端：`backend/app/api/unified_orders.py`

1. **`_set_order_appointment_impl` 的 `select(UnifiedOrder)` 显式 `selectinload`**：把 `items.product`、`store`、`user` 全部预加载，从根上消除 lazy-load 触发点。
2. **return 字段在 commit 前先抓成 local 变量**：避免 commit 后 ORM 对象 expire，再次访问标量列又回查数据库。
3. **notify 之前先 `await db.commit()`**：把订单的"改约写入"先落库，让通知通道是否成功不影响主交易。
4. **notify 出错时 `await db.rollback()` 复位 session**：防止 session 被毒化影响下一次请求。
5. **外层 except 兜底也加 `db.rollback()`**：异常路径下 session 必复位，并仍以结构化错误码 `RESCHEDULE_INTERNAL_ERROR` 返回。

### 后端：`backend/app/services/reschedule_notification.py`

1. **门店信息显式 `SELECT MerchantStore WHERE id = order.store_id`**：不再依赖 `order.store` lazy-load。
2. **用户信息显式 `SELECT User WHERE id = order.user_id`**：不再依赖 `order.user` lazy-load。
3. **商品名优先用 `OrderItem.product_name` 冗余字段**：不再 lazy-load `OrderItem.product`。
4. **任一显式查询失败均降级为空字符串**：通知文案做兜底，永不向上抛。

> ⚠️ 本次修复仅涉及 **后端 Python 代码**，**未改动**任何 H5 / 小程序 / Flutter App / Admin Web 端代码，因此**无需重新打包客户端**。原有版本的客户端（已经在使用的安装包/版本）会立即受益于服务端修复。

## 用户操作指引（如何验证修复）

### 顾客端 H5 改约流程

1. 用任意已支付未核销的订单的归属手机号登录用户端 H5（双重身份号、纯顾客号皆可）。
2. 进入 [我的订单](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/orders/) → 找到订单 → 点详情。
3. 点【改约】或【修改预约时间】按钮。
4. 选择明天起 90 天内任意可用时段，点【确认改期】。
5. 应看到「预约已确认」成功提示并自动刷新订单详情页。
6. 失败场景（如选了过去时段、超出 90 天、达到改约上限）现在会以**结构化错误码**返回（`RESCHEDULE_TIME_EXPIRED` / `RESCHEDULE_TIME_OUT_OF_RANGE` / `RESCHEDULE_LIMIT_EXCEEDED` 等），前端按 code 给出更精准的提示，不再一律弹"改约失败"。

## 注意事项

1. **测试环境通知通道未配置凭证**：`notify_result.channels` 中 `wechat_subscribe` / `app_push` / `sms` 均会返回 `ok=false` 并附说明（"未配置 RESCHEDULE_SMS_TEMPLATE_ID"等）。这是测试环境配置缺省的预期行为，**不影响改约本身的成功状态**。生产环境需配置对应模板/凭证。
2. **"改约成功但通知发送失败"不再是问题**：本次修复后，通知通道异常会被独立 try/except 接住，并立即 `db.rollback()` 复位 session；改约的写库与状态翻转早已在 commit 阶段完成，一定可以 200 返回。
3. **501/502/网关层错误**：如果遇到 502 Bad Gateway，那是 Nginx → 容器 链路问题，与本次代码无关，请刷新或联系运维。
