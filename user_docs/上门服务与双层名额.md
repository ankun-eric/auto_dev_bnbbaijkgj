## 功能简介

本次新增"**上门服务履约**"功能 + 完善"**双层名额控制**"机制，覆盖后端、admin-web、H5、微信小程序、Flutter 安卓 App 五端。

### 1. 履约类型新增 `on_site`（上门服务）

`FulfillmentType` 枚举新增第 4 种类型：

| 类型 | 含义 | 角标颜色 |
|------|------|----------|
| `in_store` | 到店服务 | 暖橙 #FF8A3D |
| `delivery` | 快递配送 | 科技蓝 #3B82F6 |
| `virtual` | 虚拟商品 | 尊贵紫 #8B5CF6 |
| **`on_site`** | **上门服务**（家政、维修、上门按摩等） | **生机绿 #10B981** |

数据库中 `products.fulfillment_type` 与 `order_items.fulfillment_type` 的 ENUM 已通过 `schema_sync` 幂等扩展，老订单与老商品完全兼容。

### 2. 商品级名额恢复编辑（F2）

- 按时段预约：`time_slots[].capacity` 每段名额可填 0 ~ 9999，**留空或 0 表示该时段不限量**。
- 按天预约：`daily_quota`（PRD 中称 `daily_capacity`，**复用原字段含义对齐**）可填 0 ~ 9999，留空或 0 表示当天不限量。
- 后端 Pydantic 验证层 `TimeSlotItem.capacity` 已改为 `Optional[int]`，并新增 0~9999 范围校验。

### 3. 门店级名额文案升级（F3）

`MerchantStore.slot_capacity` 文案统一为「**门店总接待名额**」，含义：门店在同一日期同一时段可接待的最大订单数（**所有商品共享**）。
- 旧文案"单时段最大接单数"已替换为新名称
- 校验下限改为 0（0 = 不限），上限保持 9999

### 4. 双层名额校验（F4 · 核心）

下单接口 `/api/orders/unified` 在创建订单前执行**双层名额校验**：

```
某日某时段可接收订单数 = MIN(
    商品级剩余 = 商品级名额 - 已占用商品级订单数,
    门店级剩余 = 门店级总名额 - 该门店所有上门/到店订单已占用数
)
```

- **商品级**：检查 `time_slots[].capacity` 或 `daily_quota`
- **门店级**：检查 `MerchantStore.slot_capacity`
- **取最严**：两者皆为 0 / null 时不限；任一已满即拒绝下单
- **统计口径**：`pending_payment / pending_shipment / pending_receipt / pending_appointment / appointed / pending_use / partial_used / pending_review / completed` 计入占用；`cancelled / refunded` 不计入；待支付订单超 15 分钟自动放弃占用
- **并发安全**：使用 `SELECT ... FOR UPDATE` 行锁防超卖

### 5. 上门服务必填地址（F5）

C 端三端（H5、小程序、Flutter App）下单时，若购买的商品 `fulfillment_type=on_site`：
- **必须**从用户地址簿选一个收件人地址作为服务地址
- 若用户未选择，前端按钮禁用并提示"请选择上门服务地址"
- 后端校验 `service_address_id` 必传 + 必须属于当前下单用户

`UnifiedOrder` 表新增 2 个字段：
- `service_address_id` (int, FK → `user_addresses.id`)：上门地址 ID
- `service_address_snapshot` (json)：下单时刻地址快照，**后续地址簿修改不影响此订单**，包含 `name / phone / province / city / district / street / detail / postal_code / address_id / snapshot_at`

### 6. B 端订单详情显示完整地址（F6）

商家管理后台 → 订单详情：
- 若订单包含 `on_site` 商品 + 有 `service_address_snapshot`，自动显示「上门服务地址」卡片
- 显示**完整、不脱敏**的联系人、电话、省市区、详细地址（仅订单关联商家与平台管理员可见，C 端不暴露此接口）

### 7. 历史数据兼容（F7）

- 老商品的 `time_slots[].capacity` 值**保留**，可继续在 admin 编辑
- 老订单不会因新增 `service_address_*` 字段而失败（字段允许 NULL）
- 老门店的 `slot_capacity` 数值不变，仅文案更新

### 商家端体验（5 分钟）

1. 打开 [Admin 后台](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/) 登录

## 数据库变更摘要

```sql
-- products / order_items 表 ENUM 扩展
ALTER TABLE products MODIFY fulfillment_type ENUM('in_store','delivery','virtual','on_site') NOT NULL;
ALTER TABLE order_items MODIFY fulfillment_type ENUM('in_store','delivery','virtual','on_site') NOT NULL;

-- unified_orders 新增上门地址字段
ALTER TABLE unified_orders
  ADD COLUMN service_address_id INT NULL COMMENT '上门服务地址 ID',
  ADD COLUMN service_address_snapshot JSON NULL COMMENT '上门地址快照',
  ADD INDEX idx_service_address_id (service_address_id);
```

> 上述变更由 `backend/app/services/schema_sync.py::_sync_on_site_fulfillment` **幂等执行**，无需手动迁移。
