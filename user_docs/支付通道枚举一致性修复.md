## 功能简介

本次发布是一次 **支付链路稳健性修复**，专门解决以下长期存在的 Bug：

> H5 / 小程序 / Flutter App 三端在调用「创建订单」接口时，错误地把**通道编码**（`alipay_h5` / `wechat_h5` / `wechat_miniprogram` / `alipay_app`）当作**支付方式**字段传给后端，导致：
>
> 1. 订单表的 `payment_method` 字段被通道级值污染
> 2. 管理后台按"支付方式 = 微信"过滤时漏查所有 `wechat_*` 订单
> 3. 财务月报、支付方式饼图统计口径错乱
> 4. 后端无强校验，前端传什么就存什么

修复后达到以下效果：

- ✅ **三端下单时**：`payment_method` 字段只可能是 `wechat` 或 `alipay`（provider 级别）
- ✅ **支付路由不变**：`channel_code` 字段仍由 `/pay` 接口使用，路由到具体支付通道（H5 / 小程序 / APP）
- ✅ **后端兜底校验**：即使老版本 App 用户继续传 `wechat_h5` 等老格式，后端会自动归一化为 `wechat`，**保持向前兼容**
- ✅ **非法值拒绝**：后端收到 `paypal` 等无法识别的支付方式时返回 400，避免脏数据继续污染

## 注意事项

1. **历史订单字段值不会自动修改**：本次修复后，新下的订单 `payment_method` 字段会写入 `wechat` / `alipay`；但**修复之前**已存在的订单中 `wechat_h5` / `alipay_app` 等脏数据**不会自动改变**，需要技术团队执行一次性 SQL 迁移脚本归一化（不影响用户使用）。
2. **老版本 App 用户向前兼容**：即使用户暂未升级 App，老版本 App 继续传 `wechat_app` / `alipay_h5` 时，后端会自动归一化为 `wechat` / `alipay`，保证下单可用，**不会立即崩溃**。
3. **0 元订单不受影响**：0 元订单的确认走 `confirm-free` 接口（仅使用 `channel_code`），与本次修复无冲突。
4. **支付路由保持不变**：`channel_code`（用于路由到 H5 / 小程序 / APP 通道配置）的逻辑完全没变，仅订单主表的 `payment_method` 字段语义被纠正。
