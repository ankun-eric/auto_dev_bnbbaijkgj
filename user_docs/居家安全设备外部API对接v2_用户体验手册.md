## 一、功能介绍

居家安全设备模块本期打通了"**最后一公里**"——让系统管理员能在管理后台一键把"我方接收回调的地址"推送给设备厂商云端；设备触发时，厂商云端会把告警实时推送到我方系统，并完成全链路通知（站内推送 + 短信 + 小程序订阅消息）。

### 1. 适用设备

| 类型编号 | 设备名称 | 颜色 |
|---|---|---|
| 1 | 宾尼紧急呼叫器 | 红 |
| 2 | 宾尼烟雾报警器 | 橙 |
| 7 | 宾尼水位报警器 | 黄 |

### 2. 角色与场景

| 角色 | 使用场景 |
|---|---|
| 系统管理员 | 在管理后台填写厂商对接参数、把回调地址推送到厂商云端 |
| 被守护者（老人） | 触发紧急呼叫器、烟雾/水位报警器 |
| 家人（守护人） | 通过 App/小程序站内推送、短信接收告警 |
| 客服/运维 | 通过推送历史排查对接异常 |

### 3. 核心价值

- **实时**：设备触发 → 家人收到通知，端到端 1～5 秒
- **可观测**：所有上行推送结果、所有回调接收记录均可在后台审计
- **可演进**：留好签名钩子、AI 外呼标记，便于后续扩展

## 三、操作步骤

### 步骤 1：进入回调地址配置 Tab

登录管理后台后，依次点击：**菜单 → 居家安全设备管理 → Tab4 "回调地址配置"**。

页面分为 4 个区块：
1. 上游接口参数（基础地址 + 接口路径 + deptId + Token）
2. 我方回调地址（回调域名 + 固定路径）
3. 操作按钮（保存配置 / 推送到上游）
4. 最近一次推送结果 + 推送历史（最近 3 次）

### 步骤 2：填写上游接口参数

| 字段 | 填写示例 | 说明 |
|---|---|---|
| 上游基础地址 | `http://119.3.169.29` | 自动去除末尾 `/` |
| 上游接口路径 | `/treatment/api/setMsgCallBackUrl` | 自动补开头 `/` |
| 完整上游 URL | （只读，自动拼接） | = 基础地址 + 接口路径 |
| 机构 ID（deptId） | 厂商分配，例 `your_dept_id` | 必填 |
| 鉴权 Token | 厂商分配 | 默认密文，点 👁 切换明文 |

> **拆分基础地址 + 接口路径的好处**：厂商接口路径变更时只改路径不动基础地址；测试/生产环境的基础地址可独立切换；长 URL 拆开后肉眼可对得上。

### 步骤 3：填写我方回调地址

| 字段 | 内容 | 说明 |
|---|---|---|
| 回调域名 | `https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27` | 含 `https://`，自动去末尾 `/` |
| 回调路径 | `/api/home_safety/callback/alarm` | 固定只读 |
| 完整回调 URL | （只读，自动拼接） | = 回调域名 + 回调路径 |

### 步骤 4：保存配置

点击 **"💾 保存配置"** 按钮：

- 仅写入本地数据库，**不**调用任何外部接口
- 适合"先填好草稿、二次核对后再推送"
- 保存后表单恢复"已保存"态，不再触发离开提示

### 步骤 5：推送到上游

点击 **"🚀 推送到上游"** 按钮：

1. 弹出二次确认弹窗：

   > 即将向上游 `{完整上游URL}` 推送回调地址 `{完整回调URL}`，厂商对同一机构只保留 1 个回调地址，本次操作将**覆盖**上一次的设置。是否继续？
   >
   > [取消]　[确认推送]

2. 点击"确认推送"后，系统真实发起 `POST {完整上游URL}`：
   ```http
   POST http://119.3.169.29/treatment/api/setMsgCallBackUrl
   Authorization: Bearer eyJhbGci0iJIUzI...
   Content-Type: application/json

   {
     "deptId": "your_dept_id",
     "url": "https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/api/home_safety/callback/alarm"
   }
   ```
3. 推送结果即时回填到下方 **"最近一次推送结果"** 区块，并追加到 **"推送历史"** 表格

### 步骤 6：查看最近一次推送结果

| 字段 | 含义 |
|---|---|
| 推送时间 | `YYYY-MM-DD HH:mm:ss` |
| 推送状态 | ✅ 成功 / ❌ 失败 |
| 推送 URL | 本次实际推送给上游的回调 URL |
| 上游返回码 | 上游 JSON 中的 `code` |
| 上游消息 | 上游 JSON 中的 `message` |
| 上游原始响应 | 完整 JSON 字符串（折叠展示） |

### 步骤 7：查看推送历史（最近 3 次）

| 列 | 内容 |
|---|---|
| 时间 | 推送时间 |
| 推送 URL | 本次推送的回调 URL |
| 操作人 | 后台登录用户的用户名（如 `admin`、`张三`） |
| 状态 | ✅ 成功 / ❌ 失败 |
| 上游消息 | 上游返回的 message |

> 数据库照常落库所有历史记录，**前端只展示最近 3 条**。

## 五、厂商真实报文示例

```json
{
  "param": {
    "devId": "00084bd7",
    "devType": "1",
    "occurTime": 1547100617645,
    "gwId": "AC000397",
    "devName": null,
    "callType": 0
  },
  "dataType": "call-msg",
  "msgId": "201901111103150010853592"
}
```

我方落库映射：

| 厂商字段 | 我方字段 | 处理逻辑 |
|---|---|---|
| `param.devId` | `device_sn` | 直接落库 |
| `param.devType`（字符串） | `type`（int） | 转 int，1=紧急呼叫器、2=烟雾报警器、7=水位报警器 |
| `param.occurTime`（毫秒戳） | `alarm_time` | 转 `datetime` |
| `param.gwId` | `gw_id` | 仅落库不校验 |
| `param.devName` | `dev_name` | 可为空 |
| `param.callType` | `call_type` | 0=新消息 |
| `dataType` | `data_type` | 当前固定 `call-msg`，其它类型忽略并日志 |
| `msgId` | `vendor_msg_id` | **唯一索引，永久幂等** |

响应固定：`{"code":0,"message":"ok"}`

## 六、注意事项

1. **永久幂等**：同一 `msgId` 重复推送只入库一次，第二次直接返回 `200 ok`。
2. **5 分钟同设备去重**：保留原有"同 device_sn 在 5 分钟内只触发一条新告警"机制作为第二道防线。
3. **覆盖式推送**：厂商对同一 `deptId` 只保留 1 个回调地址，再次推送即覆盖。请务必在二次确认弹窗中核对完整 URL。
4. **保存 ≠ 推送**：保存只写本地，推送才会真正影响线上对接配置。
5. **gwId 仅记录不校验**：避免拦截真实告警。
6. **本期不接 AI 外呼**：告警表统一标记为 `failed`，原因为"本期未对接外呼通道"，**不影响**站内/短信/小程序订阅消息正常发送。
7. **本期不验签**：信任厂商域名白名单，预留 `verify_signature(headers, body) -> bool` 钩子函数，未来扩展即可。
8. **历史展示 3 条**：数据库全量落库，前端只展示最近 3 条，超出部分本期不提供"查看更多"入口。
9. **来源 IP 仅日志记录**：每次回调的来源 IP（`X-Real-IP` / `X-Forwarded-For` / `REMOTE_ADDR`）全量落到日志/审计表，**不**参与拦截判断。
10. **未保存离开提示**：表单修改未保存时切换 Tab/关闭页面会弹出"您有未保存的修改，是否离开？"确认框，避免误丢配置。
11. **Token 密文展示**：默认显示 `eyJh****IUzI`，点 👁 才切换明文，防止屏幕被偷拍。
12. **推送按钮置 loading**：推送过程中按钮自动 disable，防重复点击。

## 七、典型异常场景与现象

### 回调接收的 8 大异常兜底（除内部异常外统一返回 200，避免厂商重试雪崩）

| # | 场景 | 现象 |
|---|---|---|
| 1 | `msgId` 重复 | 返回 `200 ok`，不重复入库 |
| 2 | `dataType` 不支持（非 `call-msg`） | 落日志，跳过业务处理 |
| 3 | `devType` 不在 {1,2,7} 之内 | 落异常日志 |
| 4 | `devId` 未在系统绑定 | 落异常表，不触发通知 |
| 5 | `occurTime` 解析失败 | 用服务器当前时间兜底 + 日志告警 |
| 6 | 关键字段缺失（`devId`/`devType`/`msgId`） | 落异常日志 |
| 7 | 内部异常（DB 不可用等） | 详细日志，触发运维告警，HTTP 500 |
| 8 | 5 分钟同设备重发 | 累计 `repeat_count`，不重新通知 |

### push_upstream 异常展示

| # | 场景 | 在"最近一次推送结果"中展示 |
|---|---|---|
| 1 | 网络超时（>10s） | 状态 ❌，消息"请求超时" |
| 2 | 上游 5xx | 状态 ❌，消息"上游服务异常" |
| 3 | 上游 4xx（如 Token 过期） | 状态 ❌，消息为上游返回的 `message` |
| 4 | 上游返回非 JSON | 状态 ❌，消息"上游响应解析失败"，原始响应可查看 |
| 5 | 上游 code 非成功 | 状态 ❌，按上游 `code/message` 展示 |
| 6 | 配置字段缺失 | 前端表单校验拦截，不发起调用 |

## 八、常见问题（FAQ）

**Q1：保存了配置但厂商还没生效？**
A：保存只写本地，必须再点"🚀 推送到上游"才真正影响线上对接。

**Q2：推送失败显示"请求超时"怎么办？**
A：先检查上游基础地址、网络可达性、Token 是否过期；确认无误后管理员手动再次点击推送即可，本期不自动重试。

**Q3：厂商重复发送同一 `msgId` 会重复通知家人吗？**
A：不会。`vendor_msg_id` 是唯一索引，第二次回调会被识别为重复，直接返回 `200 ok`，不重复入库、不重复通知。

**Q4：设备触发后，家人能在多久内收到通知？**
A：端到端 1～5 秒，其中回调接收 → 落库 < 200ms，落库 → 家人站内推送 < 1s。

**Q5：为什么"AI 外呼状态"全是 `failed`？**
A：本期未对接外呼通道，统一标记 `failed`，鼠标悬停可见原因。不影响站内/短信/小程序通知正常发送。

**Q6：客服角色为什么不能改配置？**
A：按权限设计，客服仅查看告警和推送历史；修改/推送需要"超级管理员"或"居家安全运营管理员"角色。
