## 功能简介

本次修复彻底重做了**商家 H5 端 · 订单详情**中【调整预约时间】按钮的交互流程，让它与客户端「改约」抽屉完全对齐：

- ❌ **修复前**：无论商品预约模式是什么，统一弹出**两个对话框**——先选日期，再被强制弹出写死的「上午（09:00）/ 下午（13:00）/ 晚间（18:00）」三选一时段，与商品实际配置毫无关系。
- ✅ **修复后**：**从底部弹出抽屉**，按订单对应商品的预约模式自动分支：
  - **按日期**模式：抽屉里仅显示日期选择，**不再弹出无意义的时段选择**。
  - **按时段**模式：抽屉里同时显示日期选择 + 9 段固定时段网格（与客户端「改约」一致），并**按服务器时间自动隐藏已过去的时段**。

修复要点：

1. 移除了写死的「上午/下午/晚间」二次时段弹窗。
2. 抽屉交互完全对齐客户端「改约」（底部弹出 + 横向日期 + 时段网格 + 确认改约按钮）。
3. 时段固定为 9 段切片（06:00-08:00 / 08:00-10:00 / ... / 22:00-24:00），按服务器时间过滤当日已过的整段时段。
4. 后端 `PUT /api/merchant/orders/{id}/appointment-time` 接口加固：
   - 终态订单（已取消 / 已退款 / 已完成 / 退款中）拒绝改约。
   - 过去日期拒绝。
   - 按时段模式：必须传时段，且当日已过的时段拒绝；按日期模式：忽略前端误传的时段值，避免脏数据写入。
   - 改约成功后触发 PRD-04 三通道改期通知（微信订阅消息 / APP push / 短信）。
5. 入口按钮文案保持「调整预约时间」不变（按方案文档 §3.2 要求，不改为「改约」）。

## 使用说明

### 一、入口

1. 用商家账号登录商家 H5 端 [移动端入口](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/m/)
2. 进入「订单」列表，点击任意一条**未完成状态**（待服务 / 已确认 / 待核销 等）的预约订单
3. 在订单详情页底部找到【调整预约时间】按钮（终态订单按钮不可见）
4. 点击【调整预约时间】

### 二、按日期模式（商品 = 按日期预约）

1. 屏幕底部弹出**抽屉**，标题显示「调整预约时间」
2. 抽屉里只有一行「预约日期」输入框（点击展开日期选择器）
3. 日期选择器：
   - 不能选**已过去的日期**（自动置灰）
   - 可选范围：今天起 90 天
   - 默认定位到订单当前预约日期（方便微调）
4. 选好日期 → 点击底部蓝色「确认改约」按钮
5. Toast 提示「改约成功」→ 抽屉自动关闭 → 订单详情自动刷新
6. **注意**：抽屉里**不会**出现任何时段选择控件（修复前的"上午/下午/晚间"已彻底移除）

### 三、按时段模式（商品 = 按时段预约）

1. 屏幕底部弹出**抽屉**，标题显示「调整预约时间」
2. 抽屉里依次显示：
   - 「预约日期」选择按钮（同上）
   - 「预约时段」9 段网格（每行 3 个，例如 06:00-08:00 / 08:00-10:00 / 10:00-12:00 / ... / 22:00-24:00）
3. 时段网格按"所选日期"动态过滤：
   - 选「今天」：已过去的整段时段自动隐藏（按服务器时间，不依赖手机本地时间）
   - 选「今天且无可选时段」：网格区域显示「今日剩余时段已过，请选择明天起的日期」+「一键切到明天」按钮
   - 选「明天及以后」：9 段全部展示
4. 默认选中订单当前预约日期 + 时段（在原时间附近做微调最方便）
5. 选好日期 + 时段 → 底部蓝色「确认改约」按钮变为可点击状态
6. 点击「确认改约」→ Toast 提示「改约成功」→ 抽屉关闭 → 订单详情刷新

### 四、改约成功后

- 顶部 Toast 提示「改约成功」
- 抽屉自动关闭
- 订单详情页自动刷新（预约时间、订单状态、改期通知状态卡片均同步更新）
- **客户会收到 PRD-04 三通道改期通知**：
  - 微信订阅消息（小程序登录的客户）
  - APP push（Flutter App 登录的客户）
  - 短信（兜底通道，所有客户都收到）
- 如果三通道全部失败，运营会收到企业微信群机器人告警，详情卡片会在订单详情页"改期通知状态"区域展示红字提示「⚠ 三通道全部失败，运营已收到企业微信告警，请人工电话联系客户兜底」

### 五、异常分支

| 场景 | 提示 | 后续动作 |
|------|------|----------|
| 终态订单（已取消 / 已退款 / 已完成 / 退款中） | 按钮不可见或返回「订单状态已变更，无法改约」 | 关闭抽屉，刷新列表 |
| 选了已过去的日期 | Toast「所选日期已过期，请重新选择」 | 抽屉保留，重新选 |
| 按时段模式但未选时段 | Toast「请选择预约时段」 | 抽屉保留 |
| 选了今天但所选时段已过 | Toast「所选时段已过期，请重新选择」 | 抽屉保留，建议改用「一键切到明天」 |
| 网络异常 / 接口超时 | Toast「网络异常，请重试」 | 抽屉保留已选项 |
| 服务器时间拉取失败 | 抽屉顶部红字「网络异常，时段以服务器为准；如改约失败请重试」 | 仍可继续操作，提交时由后端兜底再校验 |

## 注意事项

1. **入口按钮文案保持「调整预约时间」**，不会改为「改约」，避免与客户端"改约"产生概念混淆——商家是替客户调整，客户自己操作叫"改约"。
2. **9 段时段固定，不取自商品自身的"上午/下午/晚间"配置**：与客户端「改约」保持完全一致（PRD-01 全平台固定时段切片体系）。这避免了修复前那种"商家选了上午（09:00）但商品其实开放的是 14:00 起"的不匹配情况。
3. **历史脏数据自动清理**：按日期商品的订单如果之前被错误写入了"上午/下午/晚间"等时段值，本次改约后会自动清掉 `appointment_data.time_slot` 字段，新数据干净规整。
4. **时间统一以服务器时间（UTC+8）为准**：避免商家手机被人为调快/调慢导致改约请求绕过校验。前端按服务器时间隐藏过去时段（用户体验层），后端二次校验 `appt_end <= now_local`（数据安全层），双保险。
5. **改约不消耗用户的"改期次数"**：本次接口走的是商家维度的"调整预约时间"，与客户自助"改约"独立计数，不会影响 PRD-03 的 `reschedule_limit`。
6. **改期通知是异步触发**：改约成功后，三通道通知在后端并行下发，不会阻塞抽屉关闭。如客户端长时间未收到通知，可在订单详情页"改期通知状态"卡片查看每个通道的下发结果（成功/失败 + 失败原因）。
7. **跨端体验已对齐**：
   - 商家 H5 端 → 本次修复
   - 商家 PC 后台 / 商家小程序 / 商家 App：本次不动（如需对齐请另立修复事项）
   - 客户端（H5 / 小程序 / Flutter App）"改约"功能：本次未动，作为参考标准

## 验收要点（用户自测清单）

- [ ] 按日期商品订单 → 点击【调整预约时间】 → 抽屉只有日期 → 选日期 → "确认改约" → 成功 → 不会出现任何时段相关弹层
- [ ] 按时段商品订单 → 点击【调整预约时间】 → 抽屉有日期 + 9 段时段 → 默认定位到当前预约日期/时段 → 切换日期时时段网格按所选日期重新过滤 → "确认改约" → 成功
- [ ] 保存成功后 Toast 出现 → 抽屉关闭 → 订单详情刷新（预约时间字段已更新）
- [ ] 客户端收到 PRD-04 三通道之一（或多个）改期通知
- [ ] 终态订单（已完成/已取消）按钮不可见
- [ ] 选过去日期被拒绝
- [ ] 选今天且时段都已过 → 网格区域显示空状态卡片 + 一键跳明天
