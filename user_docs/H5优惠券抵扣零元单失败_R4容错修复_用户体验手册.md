## 功能简介

本次修复解决了 H5 端用户使用「全额抵扣优惠券」下 0 元单时弹出"下单失败"的 P0 级 Bug。

### Bug 现象（修复前）

- 用户在 H5 商品详情页选择商品 → 选择时段 → 在结算页勾选「全额抵扣的优惠券」→ 点击「立即下单」
- 屏幕弹出 Toast「下单失败」，订单创建失败
- 后端实际返回 `HTTP 500 Internal Server Error`
- 同一用户对同一张优惠券多次领取时，后端也会 500（重复领取检查失败）

### 修复后效果

- 0 元单可正常下单进入「免支付确认页」
- 即使数据库内同一用户对同一张券存在多张 unused 记录（历史脏数据），也能正常下单
- 重复领取同一张券时正确返回 409 Conflict（"您已领取过该优惠券"），不再 500
- 仅核销「最早领取的那张」券，其余 unused 记录保持不变

## 真正根因（v1.2 现场新发现）

历史 v1.0 / v1.1 文档曾怀疑过两个根因：

| 编号 | 怀疑项 | 实际状态 |
|------|--------|----------|
| R1 | 前端 0 元场景把 payment_method 误传 alipay/wechat | ✅ v1.0 已修，已上线生效 |
| R2 | 后端 Schema 白名单不含 coupon_deduction | ✅ v1.0 已修，已上线生效 |
| R3 | MySQL ENUM 列 `unified_orders.payment_method` 未含 coupon_deduction | ✅ 现场实测 ENUM 列已含，**不是问题** |
| **R4** | **`scalar_one_or_none()` 在用户拥有多张同一 unused 券时抛 MultipleResultsFound** | **🎯 真正根因，本次 v1.2 修复** |

### R4 根因细节

后端 `backend/app/api/unified_orders.py` 在创建订单时执行以下查询：

```python
uc_result = await db.execute(
    select(UserCoupon).where(
        UserCoupon.user_id == current_user.id,
        UserCoupon.coupon_id == data.coupon_id,
        UserCoupon.status == UserCouponStatus.unused,
    )
)
user_coupon = uc_result.scalar_one_or_none()  # ← 多于一条时抛 MultipleResultsFound
```

而生产数据库 `user_coupons` 表实际存在脏数据：

```
id=4   user_id=2 coupon_id=3 status=unused expire_at=2026-05-19
id=16  user_id=2 coupon_id=3 status=unused expire_at=2026-06-03
id=17  user_id=2 coupon_id=3 status=unused expire_at=2026-06-03
```

`scalar_one_or_none()` 期望最多 1 行；返回 3 行时抛 `sqlalchemy.exc.MultipleResultsFound`，
异常未被捕获，FastAPI 中间件兜底为 500 Internal Server Error。

### 本次修复方案

| 修复点 | 文件 | 修改方式 |
|--------|------|----------|
| **R4-A** 下单时校验优惠券 | `backend/app/api/unified_orders.py` ~541-580 | `scalar_one_or_none()` → `scalars().first()` + `order_by(UserCoupon.id.asc())`，并对多条命中打 warning 日志 |
| **R4-B** 下单时核销优惠券 | `backend/app/api/unified_orders.py` ~995-1015 | 同上口径，确保校验/核销命中同一条 |
| **R4-C** 领取时重复检查 | `backend/app/api/coupons.py` ~176-188 | `scalar_one_or_none()` → `scalars().first()`，重复时返回 409 而非 500 |
| **回归测试** | `backend/tests/test_h5_zero_order_dup_coupon_r4_v12.py` | 新增 3 用例（R4-A 多条 unused 不再 500、R4-B 仅核销最早领的那张、R4-C 重复领取返 409） |

> ⚠️ **MySQL 兼容性提醒**：原本想用 `expire_at.asc().nullslast() + id.asc()` 排序（最早过期优先，对用户更有利），但 MySQL 不支持 PostgreSQL 的 `NULLS LAST` 语法，会报 1064 SQL 语法错误。最终改为仅按 `id ASC`（最早领的优先），简单稳定且兼容。

### 测试结果

| 测试集 | 结果 |
|--------|------|
| 新增 R4 专项 | **3 / 3 PASS**（R4-A 核心、R4-B 核销、R4-C 领取） |
| 既有 H5 0 元单 v1.0 | **5 / 5 PASS** |
| 既有 0 元单 v2.2 | **7 / 7 PASS** |
| **合计** | **15 / 15 PASS** |

## 用户操作步骤（复测）

### 场景 1：使用全额优惠券下 0 元单

1. **打开 H5 用户端** → [https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/)
2. 登录账号
3. 进入任意商品详情页 → 点击「立即下单」
4. 在结算页勾选可用的「全额抵扣优惠券」
5. 实付金额变为 ¥0.00
6. 点击「立即下单」按钮
7. **预期**：跳转到「免支付确认页」（`/pay/confirm-free`）或「支付成功页」，文案显示「优惠券抵扣」
8. 进入「我的订单」可看到该订单状态正常

### 场景 2：曾经持有多张同一优惠券的用户下单

1. 后台登录 → 优惠券管理 → 给同一用户重复发放同一张券（模拟脏数据，仅测试需要）
2. 该用户登录 H5 → 选商品 → 使用此券下单
3. **预期**：正常下单成功，不再报「下单失败」
4. 后台查看该用户的优惠券列表：仅有 1 张 used，其他保持 unused（系统选择最早领取的那张去核销）

### 场景 3：重复领取同一张优惠券

1. 用户已持有某张券（已领取过）
2. 再次调用领取该券
3. **预期**：弹出 Toast「您已领取过该优惠券」（HTTP 409 Conflict）
4. 修复前：可能因脏数据触发 500 「领取失败」

## 注意事项

1. **本次仅服务端代码变更**：H5 / 小程序 / Flutter / 后台四端代码均未改动，**无需重新发布客户端版本**。用户当前持有的 H5 页面、小程序版本、APP 版本均可直接体验修复。

2. **数据库脏数据未清理**：本次修复仅做代码层容错，未对 `user_coupons` 表内已存在的重复记录做物理清理。系统会自动选择最早领取的那张去核销，不影响业务逻辑。后续如需彻底清理可执行：
   ```sql
   -- 仅供 DBA 在低峰期手动执行
   DELETE u1 FROM user_coupons u1
   INNER JOIN user_coupons u2
     ON u1.user_id = u2.user_id
    AND u1.coupon_id = u2.coupon_id
    AND u1.status = 'unused' AND u2.status = 'unused'
    AND u1.id > u2.id;
   ```

3. **建议数据约束加固（后续 PRD）**：在 `user_coupons` 表对 `(user_id, coupon_id, status)` 添加部分唯一索引（仅当 status='unused' 时唯一），从源头杜绝脏数据。本次未做以保持向后兼容，需评估对历史数据的影响。

4. **历史文档（v1.1）的判断已被本次实测推翻**：v1.1 文档以 95% 置信度怀疑 R3（DB ENUM 列未扩列），但本次实测显示 ENUM 列在更早的 v2.2（commit `cb7d53f`）就已通过 `schema_sync._sync_payment_config` 启动钩子扩容到位。真正根因是 R4，与 ENUM 无关。

5. **生产部署节奏**：服务器（newbb.test.bangbangvip.com）已通过 `git fetch + reset --hard origin/master + docker compose build backend + docker compose up -d backend` 完成镜像重建并启动新容器，无需再手动操作。

