## 功能简介

本次修复解决了「健康档案首页」上「守护人」双卡片入口的三个口径问题：

1. **「我守护的人」卡片点击后跳转旧页 Bug 修复**：之前用户点击卡片会跳到已废弃的 `/guardian-system` 页面（实际是服务器上的旧构建产物缓存所致），修复后将稳定跳转到 `/health-profile/i-guard` 新版页面。
2. **「我守护的人」统计口径优化 — 本人也计入**：之前数字只包含"我守护的其他人"（如 2 人），现在数字包含**本人**（如 3 人 = 本人 + 2 个被守护人），符合健康类产品「本人即家庭档案第一人」的主流做法。在「我守护的人」详情页的列表头部会显示"本人"卡片，带有醒目的"本人"蓝色徽章。
3. **「守护我的人」统计口径优化 — 拆为双数字**：之前只显示一个数字（已守护者数量），现在改为「**已守护 X 人 / 待确认 Y 人**」的双数字 UI，让用户一眼看到"我发出的反向邀请有几个对方还未确认"。

## 使用说明

### 1. 进入健康档案首页

在 H5 中打开「健康档案」Tab（路径 `/health-profile`），登录后即可看到首页。

### 2. 查看「守护人」双卡片入口

在「本人」Tab 下，页面中部会有两张并排的卡片：

- **左边：我守护的人**（👨‍👩‍👧 蓝色图标）
- **右边：守护我的人**（💚 绿色图标）

#### 2.1 「我守护的人」卡片

- 副标题数字显示为「**N 人**」，**包含本人**（即 1 + 已建立的被守护人数）。
- 点击整张卡片，进入「我守护的人」详情页（路径 `/health-profile/i-guard`）。
- 详情页顶部第一张卡片就是「本人」，名字旁带蓝色「本人」徽章，下方提供三个操作：
  - 「编辑档案」：进入本人健康档案
  - 「邀请记录」：查看我发起的邀请和别人邀请我的记录
  - 「我的 AI 外呼额度」：查看本月剩余 AI 外呼额度
- 本人卡片下方依次列出真正的被守护人，按"守护中"和"待守护 / 已解除"两组展示。

#### 2.2 「守护我的人」卡片

- 副标题显示为「**已守护 X 人 / 待确认 Y 人**」：
  - **X = 已守护**：已经接受您邀请并建立守护关系的人数。
  - **Y = 待确认**：您发出但尚未被使用、且未过期的反向邀请数量。
- 点击卡片进入「守护我的人」详情页。

### 3. 非本人 Tab 视角

切换到家人 Tab 时，只显示「守护 TA 的人」卡片，点击弹出只读抽屉（不可操作），符合 PRD 「家人 Tab 不展示『我守护的人』」的设计。

## 注意事项

1. **缓存清理**：如您在修复前已经访问过此页面，请使用浏览器**无痕窗口**或在浏览器设置中清空缓存后再重新访问，以确保下载到最新前端代码。
2. **旧 `/guardian-system` 链接已下线**：如您历史收藏了 `/guardian-system` 链接，该路径现在会返回 404；请改为收藏 `/health-profile/i-guard`。
3. **「本人」项不可删除**：「我守护的人」详情页中本人项是虚拟项，不参与「守护人配额」计算，也不显示"解除""代付"等按钮，点击「编辑档案」可进入本人健康档案。
4. **「待确认」数字的统计口径**：仅统计您发出的、未达使用上限（`used_count < max_uses`）、未过期（`expires_at > 当前时间`）、且状态为 `pending` 的反向邀请；邀请被接受、过期或您主动取消后，该数字会自动减少。
5. **接口兼容**：后端 `/api/reverse-guardian/guardian-count` 返回结构升级为 `{count, active_count, pending_count, total_count}`，其中 `count` 字段保留兼容旧前端，等于 `active_count`，老版本 H5 不会因此报错。

## 体验步骤（建议测试场景）

1. **场景 A：单纯查看本人**
   - 用一个没有任何守护关系的账号登录 H5 → 进入健康档案 → 看到「我守护的人」显示 `1 人`（仅本人）→ 点击卡片 → 进入详情页，第一张卡片是本人。

2. **场景 B：有 2 个被守护人**
   - 已守护 2 人的账号登录 → 看到「我守护的人」显示 `3 人`（本人 + 2 人）→ 点击进入详情页 → 顶部本人 + 「守护中（2）」分组。

3. **场景 C：有人主动邀请您当守护人但尚未接受**
   - 您调用「生成邀请二维码」生成 1 张有效邀请 → 回到健康档案 → 看到「守护我的人」显示 `已守护 0 人 / 待确认 1 人`。
   - 当对方扫码接受后，再刷新页面，应显示 `已守护 1 人 / 待确认 0 人`。

4. **场景 D：旧链接 404 验证**
   - 浏览器访问 `https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/guardian-system` → 应返回 404 页面。

## 兼容性与回滚说明

- **后端**：`/api/guardian/v12/i-guard` 接口的 `total_count` 在原值基础上 +1，并在 `items[0]` 注入本人虚拟项（`is_self: true`、`role_badge: "self"`、`relation_label: "本人"`）。前端已做幂等保护（顶部本人行单独渲染、列表中过滤 `is_self`）。
- **回滚预案**：如生产出现异常，可仅回滚后端两个文件（`guardian_system_v12.py`、`reverse_guardian.py`），前端已对 `pending_count` 字段缺失做 `?? 0` 兜底，不会报错。
