## 功能简介

本次针对「我的档案」页面共修复了 **4 个体验问题**，让"选择人 Tab"徽标更醒目、"我守护的人"卡片统计口径与运营预期完全对齐、邀请额度数字前后端统一、本人卡片按钮排版整齐美观：

| 序号 | 问题 | 优化前 | 优化后 |
|------|------|--------|--------|
| 1 | 选择人 Tab 状态徽标 | 已绑定显示「守护中」，灰/蓝色 | 已绑定显示「**已绑定**」，**橙色 `#FF7A1A`** 加粗，配橙底 `#FFF3E8` |
| 2 | 「我守护的人」卡片统计 | 显示「档案记录数 N」（口径混乱） | 统一为「**已守护 X/Y**」，X = 已绑定的非本人档案数，Y = 会员可守护人上限 |
| 3 | 详情页「还可邀请 X 位」数字 | 前后端双口径不一致，已过期/失效邀请也被算占额，数字会算错 | 后端 `quota_used` 仅计 `bound` + `inviting(未过期)`；前端直接读 `can_invite_count`，数字 100% 一致 |
| 4 | 本人卡片 3 个按钮高度 | 因 primary/secondary padding 不同导致高度不齐 | 3 个按钮统一高度 **28px**，视觉协调 |

## 使用说明

### 1. 选择人 Tab 徽标查看（修复 1）

1. 打开 [我的档案](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/health-profile)
2. 页面顶部是横向滑动的"选择人 Tab"区域（每个圆形头像即一位家人档案）
3. **已绑定（已建立守护关系）的家人**：头像右上角会出现 **橙色「已绑定」徽标**（`#FF7A1A`，配 `#FFF3E8` 浅橙底，加粗）
4. 未绑定/孤儿档案/邀请中等其他状态：不显示该徽标

### 2. 「我守护的人」卡片统计（修复 2）

1. 切换到 **本人 Tab**（带"本人"标签的那个头像）
2. 页面下半区出现两个并排卡片，左侧即「我守护的人」入口卡片
3. 卡片副文案统一为 **「已守护 X/Y」**：
   - **X**：已绑定（建立了守护关系）的非本人档案数（不含本人）
   - **Y**：会员等级允许守护的人数上限（如超级 VIP 会显示 `∞`）

### 3. 详情页「还可邀请 X 位」（修复 3）

1. 点击「我守护的人」卡片，进入详情页（路径 `/health-profile/i-guard`）
2. 顶部蓝底统计区显示 **「守护人：X 人，还可邀请 Y 位，共 M 位」**
3. **新的占额口径**：仅"已绑定"和"邀请中（pending 且未过期）"占名额；已过期、已失效、未响应、未邀请 **不占名额**
4. **示例**：会员上限 5 人 → 已绑定 2 人 + 邀请中（未过期）1 个 → 占额 = 3，还可邀请 = `5 - 3 = 2` 位
5. 把那条邀请置为"已过期"后立即刷新页面 → 还可邀请显示 **3** 位（过期不再占额）

### 4. 本人卡片 3 个按钮（修复 4）

在「我守护的人」详情页（同上路径），页面正文第一张卡片是 **本人卡片**，含 3 个按钮：

- 「编辑档案」（次按钮，白底蓝边）
- 「邀请记录」（次按钮，白底蓝边）
- 「AI 外呼额度」（主按钮，黄底白字）

本次修复后，**3 个按钮高度统一为 28px**，左右间距由 padding 控制，垂直视觉完全对齐，不再出现"主按钮比次按钮高一截"的错位。

## 口径定义（统一对齐，避免歧义）

| 口径名 | 定义 | 备注 |
|--------|------|------|
| **X（已守护）** | 已绑定（`bind_status=bound`）且非本人的档案数 | **不含本人**（方案 A） |
| **Y（可守护上限）** | `summary.max_guardians`（按会员等级动态读取） | 超级 VIP = ∞ |
| **占用配额（quota_used）** | 已绑定 + 邀请中（pending 且未过期） | **不含已过期/已失效/未响应/未邀请** |
| **还可邀请（can_invite_count）** | `max_guardians - quota_used`（≥ 0） | 后端权威计算，前端直接消费 |

## 注意事项

1. **缓存刷新**：如果浏览器看不到新效果，请下拉刷新或清除站点缓存（手机微信内可在右上角菜单选"刷新"）
2. **会员等级**：守护人上限 Y 与您的会员等级挂钩，可在「会员中心 → 套餐」查看与升级
3. **本人不占名额**：您本人始终不计入 X、不占用配额
4. **过期邀请刷新延迟**：邀请过期判定基于服务端时间，前端刷新页面后立即生效
5. **共享额度 / 解除守护**：按钮逻辑与之前保持一致，本次未变动

## 风险与回滚

| 风险点 | 影响 | 应对 |
|--------|------|------|
| 后端 `quota_used` 口径变更 | 已绑定但邀请已过期的用户：从"占配额"变为"不占" → `can_invite` 数字 **会变大** | 这是预期效果，符合 PRD |
| 前端废弃自算 `maxGuard - guardCount` | 接口异常时 `can_invite_count` 为 `null` | 已兜底为 `0`，不会出现 NaN |
| 徽标橙色 `#FF7A1A` | 与全局橙色一致 | 无新增色值 |

如需回滚：在服务器执行 `git revert <本次提交哈希>` 后重新构建 H5 与 backend 容器即可。
