## 功能简介

本次修复彻底解决了「健康档案首页 → 档案列表」中出现的**空姓名档案**问题：

- **现象**：档案列表中存在姓名（昵称）为空白、NULL 或纯空格的脏档案，用户在切换/识别家庭成员档案时一头雾水。
- **根因**：数据库历史上未对 `family_members.nickname` 设置非空约束；老版本前后端缺失校验；本人档默认值不统一（部分填 "本人"，部分填 "用户XXXX"）。
- **后果**：经清点存在 16 条空姓名脏档案（其中 2 条 active 前端可见，14 条历史残留），均为非本人档（零误删风险）。

本次修复完成 4 件事，让档案列表干净、可信、不再产生空姓名档案：

1. **后端数据库层加固**：`family_members.nickname` 列加上 `NOT NULL` 约束，从根上禁止空姓名落库。
2. **后端 API 层强校验**：新建/编辑档案接口对 `nickname` 进行 `trim` 校验，留空或纯空格统一返回 400「姓名不能为空」。
3. **后端默认值规则统一**：注册时自动建本人档的 `nickname` 由 "本人" 改为「用户{手机号后4位}」（无手机号时退化为「用户{user_id}」），与用户主表口径完全一致。
4. **后端自动清理 + 前端兜底**：服务启动时一键级联清理 16 条历史脏档案及其下挂数据；H5 列表渲染前再加一道过滤，双重保险。

## 使用说明

### 入口路径

打开 H5 → 顶部菜单或首页快捷入口进入「健康管理」→「健康档案」→「档案列表」。

### 你能感受到的变化

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 打开档案列表 | 列表中混着姓名为空白的档案，看着像 BUG | 列表只展示有姓名的档案，整洁清晰 |
| 新建家庭成员档案 | 姓名留空也能保存，生成"无名档案" | 留空提交直接报错「姓名不能为空」 |
| 编辑档案，把姓名清空 | 旧接口默默接受，落库变成空姓名 | 后端拒绝并提示「姓名不能为空」 |
| 新手机号注册后看本人档 | 姓名固定为 "本人" | 姓名自动为「用户{后4位}」，更具辨识度 |
| 后续可改名 | — | 进入「个人资料」可随时改本人档姓名 |

### 手动验证步骤（可选）

1. **看列表**：进入「档案列表」，确认所有卡片都有姓名（无任何"未命名"或空白卡）。
2. **试新建**：点击「邀请家人」/「+ 新增」打开新建抽屉，将「姓名」留空或只填空格，点击「保存档案」→ 应弹出红色提示「姓名不能为空」，无法保存。
3. **试编辑**：进入任意一个家庭成员档案的编辑入口，将姓名清空保存 → 应被后端拒绝。
4. **新手机号注册**：用一个新手机号注册并登录，回到「档案列表」→ 本人卡片姓名为「用户XXXX」（X 为手机号后 4 位），不会显示「本人」字样。

## 注意事项

1. **本次修复完全无损存量数据**：被清理的 16 条档案全部为 `is_self=0` 的空姓名档案，零本人档案受影响；体检报告、血压、血糖等核心健康数据不受影响（这些表中也确认无脏档案下挂）。
2. **服务重启自动迁移**：本次清理脚本已嵌入后端启动流程，幂等可重复执行。若后续不慎再次出现空姓名档案，下次重启同样会自动清理。
3. **姓名字段必填规则永久生效**：今后无论从 H5、小程序还是后台批量导入，姓名都必须非空（数据库层 + API 层双保险）。
4. **首字母头像**：档案卡片的头像底色与姓名首字符相关，因此修改姓名后头像可能变化，属正常现象。
5. **若你之前看到过空姓名档案**：现在刷新页面即可消失（无需重新登录）。
