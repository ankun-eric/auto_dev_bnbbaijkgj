## 详细操作说明

### R1. 商家角色统一为 4 角色

#### 变更前后对比

| 分量 | 变更前 | 变更后 |
|---|---|---|
| 角色数量 | 5 个英文 code（owner/manager/verifier/finance/clerk/staff…） | 仅 4 个：`boss / store_manager / finance / clerk` |
| 中文显示 | admin 端把 `clerk` 显示成"核销员"，H5 端却显示"店员"，同一字符串两种含义 | 全端统一：`boss=老板、store_manager=店长、finance=财务、clerk=店员` |
| `verifier`（核销员） | 独立角色 | **已并入"店员"**（核销职责由店员承担） |
| 历史 `owner / manager / verifier / staff` | 直接报错或显示混乱 | **过渡期自动归一化**（一个版本兼容） |

> 您不需要做任何手动操作。系统启动时会自动把数据库里历史的 `verifier / staff / owner / manager` 全部归一化为新的 4 角色之一。如果您之前的账号是"核销员"（如手机号 6366 的用户 18），登录后会自动显示为"店员"。

### R2. admin 员工列表"账号"列合并

**操作路径**：admin 后台 → 商家管理 → 商家账号 → 点击某行的「员工列表」按钮 → 弹出右侧抽屉

**变更点**：

- ❌ 移除：抽屉表格中"账号"列（因为账号本质就是手机号，与"手机号"列完全重复）
- ✅ 保留：姓名 / 手机号 / 角色 / 状态 / 创建时间 / 最后登录时间 / 绑定门店 / 操作
- ✅ 调整：商家账号顶部搜索框 placeholder 改为 **"请输入手机号搜索"**

### R3. 员工启停权限收紧（仅老板可启停）

#### 权限矩阵

| 操作者 | 启用/冻结按钮可见 | 后端是否放行 | 备注 |
|---|---|---|---|
| 老板 boss | ✅ 可见（除自己行） | ✅ 放行 | 仍禁止启停自己 |
| 店长 store_manager | ❌ **完全不可见**（所有行） | ❌ 拦截，返回 `无权限操作` | **本轮新增收紧** |
| 财务 finance | ❌ 看不到员工管理菜单 | - | 维持现状 |
| 店员 clerk | ❌ 看不到员工管理菜单 | - | 维持现状 |

#### 老板使用步骤

1. 用老板账号登录 [商家 H5 PC 员工管理](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/staff)（或 [移动端](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/m/staff)）
2. 在某个员工行点击 **「停用」/「启用」** 按钮（PC）或拨动 **开关**（移动端）
3. 弹窗确认后，即时生效
4. 您**自己那一行不会出现**任何启停按钮 — 这是为了防止误操作锁死自己

#### 店长访问体验

- 店长进入员工管理页面：能看到员工列表，但**所有行的启停按钮都已隐藏**
- 顶部说明栏会提示："**店长可查看员工列表与权限；启停员工与重置密码请联系老板（仅老板可操作）**"
- 即使店长用工具绕过前端调接口，后端也会返回 403 + `无权限操作` 兜底拦截

#### 后端兜底文案

| 场景 | 文案 |
|---|---|
| 非老板调用启停接口 | `无权限操作` |
| 老板启停自己 | `无权限操作` |
| 目标用户不在管辖范围 | `员工不存在` |

### B1. 商家"我的"页个人信息空白 Bug 修复

#### 现象

升级前：用任何角色登录商家 H5，进入「我的 → 个人信息」页面，姓名 / 手机号 / 角色 / 所属门店 / 商家名 / 账号状态 / 创建时间 / 最后登录时间 8 个字段**全部显示为 `—`**。

#### 根因

后端注册了**两个完全相同路径**的 `GET /api/merchant/profile` 接口：
- 老接口（`merchant.py`）只返回 `nickname / avatar` 2 个字段
- 新接口（`account_security.py`）返回完整 8 字段

FastAPI 按注册顺序，老接口覆盖了新接口，导致前端拿到的 8 字段全部是 `undefined`。

#### 修复

直接**删除** `merchant.py` 中的老接口，新接口成为唯一实现。

#### 验收方式

| # | 测试场景 | 预期 |
|---|---|---|
| 1 | 老板账号登录 H5 PC，进入"我的"→个人信息 | 8 字段全部正确 |
| 2 | 店长账号登录 H5 PC，同上 | 8 字段全部正确 |
| 3 | 财务账号登录 H5 PC，同上 | 8 字段全部正确（角色显示"财务"） |
| 4 | 原核销员账号（用户 18，手机号尾号 6366）登录 H5 PC | 角色显示"店员"，8 字段全部正确 |
| 5 | 4 角色账号在 H5 移动版重复 1-4 | 同上 |

> ✅ 部署验证已通过：后端 `GET /api/merchant/profile` 唯一实现 = `app.api.account_security.merchant_get_profile`。

## 注意事项

1. **角色码兼容期**：本版本对历史角色码 `verifier / staff / owner / manager` 仍做映射兼容；**下一个版本会移除**这些兼容代码。请前端/三方调用方尽快迁移到 4 角色（boss / store_manager / finance / clerk）。
2. **数据迁移自动执行**：后端启动时会自动把 `merchant_store_memberships.role_code` 中的历史值归一化。本次部署后 DB 实际分布如下：
   ```
   role_code | member_role | 行数
   ----------+-------------+------
   boss      | owner       | 2
   store_manager | store_manager | 1
   clerk     | verifier    | 1
   ```
   说明：物理 ENUM `member_role` 仍保留 5 值（不破坏存量），但业务上只看 `role_code` 4 值。
3. **手动回滚**：如需回退到迁移前文本，在服务器执行 `backend/migrations/v1_0_merchant_role_unify_rollback.sql`（建议优先用迁移前 mysqldump 恢复）。
4. **路由冲突全局扫描报告**：本次执行的扫描发现 8 处与本 PRD 无关的历史冲突（积分商城、优惠券模块），已落盘到后端容器的 `/tmp/route_conflicts.json`。本次只修复了 PRD §B1 提到的 `/api/merchant/profile` 这一冲突，其余由 PM 审阅后决定是否扩范围治理。
5. **店长别再点启停**：旧版本中店长能启停员工的能力已被收紧。如果您是店长，需要启停员工时请联系老板。
6. **头像着色仅在员工列表生效**：admin 后台的头像、商家"我的"页头像、订单页操作人头像等其他位置维持现状，未做角色着色。
7. **不能启停自己**：即使您是老板，启停按钮在自己那一行也不会出现。如需冻结自己账号，请联系平台运营在 admin 后台操作。

## 常见问题

**Q1：升级后我用核销员账号登录，发现角色变成了"店员"，是不是数据丢了？**
A：不是。核销员（`verifier`）已并入店员（`clerk`），这是治理治理性变更。您原先的所有权限、所属门店、订单核销能力都完整保留，只是中文名变了。

**Q2：店长能不能再启停员工？**
A：不能。本轮 R3 收紧后，**全平台只有老板可以启停员工**（无论 H5 PC 还是移动端、无论调接口还是点按钮）。如有业务必要请联系老板。

**Q3：员工头像颜色能改吗？**
A：本期不开放自定义。4 色按角色固定（金=老板、蓝=店长、橙红=财务、灰蓝=店员），便于团队管理快速识别角色。

**Q4：admin 后台员工列表的"账号"列去哪了？**
A：已合并到"手机号"列。账号本质就是手机号，两者一直完全相同，本次去除冗余。

**Q5：「我的 → 个人信息」页面之前一直空白，现在为什么能显示了？**
A：根因是后端两个同路径接口冲突，老接口覆盖了新接口。本版已删除老接口，您退出重新登录后即可看到完整 8 字段（姓名 / 手机号 / 角色 / 所属门店 / 商家名 / 账号状态 / 创建时间 / 最后登录时间）。
