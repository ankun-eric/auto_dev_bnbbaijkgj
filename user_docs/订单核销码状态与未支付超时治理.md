## 一句话结论

把「订单取消」**唯一收口**到 `cancel_order_with_items` 服务函数，凡走到取消必同步把订单下所有 `OrderItem.redemption_code_status` 刷为 `expired`（保护 used / redeemed / refunded 终态）；同时**用「未支付超时自动取消」彻底替代过时的「门店超时未确认自动取消」**，并把所有跟超时相关的口径（H5 容量保护、定时任务、下单提醒文案）统一收敛到全局配置 `PAYMENT_TIMEOUT_MINUTES`（默认 15 分钟，`.env` 可调）。一次性下线 `products.payment_timeout_minutes` / `unified_orders.payment_timeout_minutes` / `unified_orders.order_timeout_minutes` 三个废弃字段，并对历史脏数据（cancelled 订单 + active 核销码）做幂等迁移清洗。

## 功能简介

本次治理涵盖以下 7 项核心改造：

1. **统一取消出口** — 新增 `app/services/order_cancel.py::cancel_order_with_items`，所有取消路径强制走它。出口承诺：① 设置 `UnifiedOrder.status='cancelled'`、写入 `cancelled_at` / `cancel_reason`；② 遍历 `order.items` 把 `redemption_code_status='active'` / `'pending_use'` 全部刷为 `'expired'`；③ 已是 `used / redeemed / refunded / locked / expired` 的核销码**保持不动**（终态保护）；④ 同一订单重复调用幂等无副作用。

2. **三条取消路径完成接入** —
   - **路径 1（用户主动取消）** `POST /api/unified-orders/{id}/cancel` 改为调用统一出口；
   - **路径 2（管理员批准退款）** `POST /api/admin/orders/unified/{id}/refund/approve` 改为调用统一出口；
   - **路径 3（未支付超时）** 新增定时任务 `check_unpaid_order_timeout` 同样走统一出口。

3. **替换过时的「门店超时未确认」任务** — 自营业务模式下不再存在「门店确认」环节，旧任务 `check_order_confirm_timeout` 已被新任务 `check_unpaid_order_timeout` **完全替换**（每分钟扫一次 `pending_payment` 且 `created_at + PAYMENT_TIMEOUT_MINUTES < now` 的订单并取消）。

4. **全局支付超时配置 `PAYMENT_TIMEOUT_MINUTES`** —
   - 配置位置：`backend/app/core/config.py::Settings.PAYMENT_TIMEOUT_MINUTES`，默认 15；
   - 暴露方式：`.env` / `.env.example` 已加 `PAYMENT_TIMEOUT_MINUTES=15`，`docker-compose.yml` 与 `docker-compose.prod.yml` 后端服务 `environment` 段已透传；
   - 三处消费点全部接入：① 未支付超时任务窗口；② H5 下单容量保护回查窗口（替换 3 处硬编码 `timedelta(minutes=15)`）；③ 站内信「请在 X 分钟内完成支付」文案中的 X。

5. **删除 `Product.payment_timeout_minutes`** —
   - ORM 模型移除该列；
   - Pydantic Schema (`ProductCreate / ProductUpdate / ProductResponse`) 移除该字段；
   - Admin 商品编辑页（`admin-web/src/app/(admin)/product-system/products/page.tsx`）移除「支付超时(分)」表单项；
   - 数据库 `schema_sync` 自动 `DROP COLUMN`。

6. **下单提醒站内信引用全局配置** — `POST /api/unified-orders` 创建后给用户发的站内信「请在 X 分钟内完成支付」从硬编码切换为读取 `settings.PAYMENT_TIMEOUT_MINUTES`，与真实超时窗口完全一致。

7. **历史脏数据一次性清洗（幂等迁移）** — `schema_sync` 启动期间自动执行：
   - 删除 `products.payment_timeout_minutes` / `unified_orders.payment_timeout_minutes` / `unified_orders.order_timeout_minutes`（如存在）；
   - 把所有「父订单 = cancelled 但 OrderItem.redemption_code_status = active」的历史订单项一次性刷成 `expired`；
   - 反复重启不会重复改写，迁移幂等。

## 使用说明

### 场景 A：管理员发起退款 → 核销码自动失效

1. 进入 [管理后台](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/) → **退款审核**；
2. 选择一笔状态为「退款审核中」的订单，点击 **批准退款**；
3. 后端流转：退款成功 → 订单状态 → `cancelled` → 该订单**所有 OrderItem 的核销码** 一并刷为 `expired`；
4. 进入 **订单管理** → **核销码状态** 验证：之前是 `active` 的核销码现在应该全部显示 **已过期 (expired)**。

### 场景 B：用户取消未支付订单 → 核销码同步失效

1. H5 端 `/orders` 找到一笔 `pending_payment` 订单；
2. 点击「取消订单」；
3. 进入管理后台核对：订单 `cancelled`，且核销码 `expired`，无遗留 `active`。

### 场景 C：未支付订单 15 分钟自动超时

1. H5 下单后停留在收银台不付款；
2. 等待 ≥ `PAYMENT_TIMEOUT_MINUTES` 分钟（默认 15），定时任务每分钟扫一次；
3. 订单自动转为 `cancelled`，订单详情显示「未支付超时自动取消」，核销码 `expired`。

> 💡 如需短时间快速验证，可在远程 `.env` 把 `PAYMENT_TIMEOUT_MINUTES` 改成 1，重启 backend 容器即可。

### 场景 D：管理员编辑商品 — 「支付超时(分)」字段已下线

1. 进入 [商品编辑页](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/product-system/products)；
2. 新建或编辑任意商品，**不再出现「支付超时(分)」表单项**；
3. 后端 `GET/POST /api/products/*` 返回值与请求体里也不再包含 `payment_timeout_minutes`，老前端版本即便附带该字段也会被忽略。

### 场景 E：下单后站内信文案动态化

1. H5 下单后立即收到站内信「请在 **15** 分钟内完成支付」；
2. 当 `PAYMENT_TIMEOUT_MINUTES` 调整为 30，新订单的站内信会自动变为「请在 **30** 分钟内完成支付」，无需任何代码修改。

## 注意事项

1. **核销码终态保护** — 已是 `used` / `redeemed` / `refunded` / `locked` / `expired` 的核销码，取消订单时**绝不**会被回写为别的状态。
2. **统一取消出口幂等** — 对同一笔已 cancelled 的订单重复调用 `cancel_order_with_items` 不会产生副作用，也不会重复发通知。
3. **超时任务粒度** — 定时扫描间隔 60 秒，因此实际生效时间是「下单 + PAYMENT_TIMEOUT_MINUTES 分钟 ~ +1 分钟」之间，业务上属预期行为。
4. **配置覆盖优先级** — `Settings` 类按 Pydantic 默认行为读取顺序：环境变量 > `.env` > 类默认值。`docker-compose.yml` 的 `environment` 段优先级最高。
5. **schema_sync 幂等** — 容器启动重复执行不会重复 DROP 列、不会重复改写历史数据，可放心多次重启。
6. **历史脏数据清洗范围** — 仅修复「订单 = cancelled 且 核销码 = active」组合；其它组合（例如 partial_used / refund_pending 等）不动。
7. **H5 容量保护时间窗** — 现在与全局 `PAYMENT_TIMEOUT_MINUTES` 完全对齐，避免出现「订单还没真正超时但容量已经被释放」的边界穿透。
8. **生产部署提示** — 上线生产环境时，请在生产 `.env` 里也加 `PAYMENT_TIMEOUT_MINUTES=15`（或所需值），与产品策略一致。
9. **回归不破坏** — 已用 64 个旧订单关键测试 + 7 个新增专项测试覆盖，远程容器全部 PASS。
10. **数据观测** — 上线后建议观察 `unified_orders.status='cancelled'` 增长曲线 + 关联 `order_items.redemption_code_status` 分布，应只见 `expired / used / redeemed / refunded`，不应再出现 `cancelled 订单 + active 核销码` 的脏数据组合。
