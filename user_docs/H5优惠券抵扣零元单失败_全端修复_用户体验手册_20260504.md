## 一、Bug 现象（修复前）

用户在 H5 端选购服务，使用一张可全额抵扣订单金额的优惠券后，**实付金额变为 0 元**。点击"提交订单"时，**接口返回 500，订单创建失败**，前端报错"系统繁忙，请稍后再试"，导致用户**无法完成 0 元下单**。

### 根因

后端 `unified_orders` 创建订单时，仅允许 `payment_method` 为 `wechat` / `alipay`；当 H5 端在实付 0 元场景下仍然按"原选择的支付方式"提交时，**后端校验通过但后续支付链路找不到对应渠道而抛 500**；同时，即使前端绕过校验提交 `payment_method=coupon_deduction`，**Pydantic 也会拒绝该值**。问题需要前端先把"0 元单"判定为优惠券抵扣，并要求后端**强制兜底改写**+**扩列枚举**。

---

## 二、修复结果

修复后体验：

1. **下单成功**：H5 / 小程序 / Flutter App 在实付 0 元时，前端会自动把支付方式切换为"优惠券全额抵扣"，下单接口返回 200，跳到支付成功页。
2. **订单详情正确**：H5 / 小程序 / Flutter / 管理后台 的订单详情/列表页，对该订单**统一展示"优惠券全额抵扣"**，不再出现"微信支付"或空白。
3. **后端兜底加固**：即使有未升级的旧端绕过前端校验，**后端仍会自动把 `paid_amount=0` 的订单 `payment_method` 兜底改写为 `coupon_deduction`**，并把变更日志写入 `payment_logs`，永远不会再出现 500。
