## 功能简介

本次更新是对 **Bug410（订单改约弹窗自动关闭）** 一次修复后出现的「二次问题」的最终修复。

**修复前（v1 上线后用户反馈的二次 Bug 现象）**：

1. 顾客在订单详情页点【确认改约】
2. 出现「改约成功」轻提示
3. 改约抽屉**滑下去关闭一次** ✅
4. 但大约 0.5~1 秒后，**同一个抽屉又自动滑回来了** ❌
5. 弹回来的抽屉里**表单看上去是空的（或显示成默认明天）**
6. 必须再手动关一次才能正常浏览订单详情
7. 后台数据其实改约成功，订单已显示新时间

**修复后（v2 上线后预期体验）**：

> 顾客点【确认改约】→ 抽屉**只滑下去关闭一次**，不会再弹回来 → 「改约成功」轻提示自动消失 → 看到订单详情页时间已变 → 整个过程**只需要按一次确认按钮**。

## 使用说明（H5 顾客端验证步骤）

### 路径一：从订单列表的【改约】按钮进入（v2 重点修复路径）

1. **登录**：在浏览器打开 [H5 顾客端首页](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/)，使用真实顾客账号登录
2. **进入订单列表**：点击底部「我的」→「我的订单」（或直接访问 [我的订单](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/unified-orders)）
3. **找到一笔可改约的订单**：状态为「已预约」/「待核销」/「部分已核销」，且未达到改约次数上限
4. **点击订单卡片底部的【改约】按钮**：页面跳转到订单详情页，**改约抽屉自动从底部弹出**
5. **修改预约时间**：选择一个合法的新日期、新时段
6. **点击【确认改期】**：
   - ✅ 顶部出现「改约成功」绿色轻提示
   - ✅ 抽屉**滑下去关闭** —— **整个过程只发生一次**，不会再次弹回来
   - ✅ 订单详情页上的预约时间立即更新为新时间
   - ✅ 即使等待 1~2 秒观察，抽屉也不会再自动弹出
7. **返回订单列表**：列表里这笔订单显示的也是新时间（v1 已修复的列表联动刷新仍生效）

### 路径二：在订单详情页内点【修改改约】（既有路径，本次仅做加固）

1. 直接进入某笔订单详情页（不经过列表的【改约】按钮，URL 不带 `?action=appointment`）
2. 点击页面下方的【改约】按钮，抽屉弹出
3. 填表 → 点【确认改期】
4. 同样应观察到：抽屉只关闭一次，不会再弹回来

### 路径三：手动关闭抽屉（验证不会误触发）

1. 进入订单详情页 → 点【改约】 → 抽屉弹出
2. 点遮罩 / 点 X / 点「取消」/ 直接物理返回
3. 抽屉关闭后**不应再弹回**

## 修复原理（开发同学参考）

修复点位于 [`h5-web/src/app/unified-order/[id]/page.tsx`](https://github.com/ankun-eric/auto_dev_bnbbaijkgj/blob/master/h5-web/src/app/unified-order/%5Bid%5D/page.tsx)：

1. **新增 `actionAppointmentConsumedRef = useRef(false)`**：用于标记 `?action=appointment` 是否已被消费过
2. **修改 `useEffect([order, searchParams], ...)`**：判断 `actionAppointmentConsumedRef.current` 已为 true 则提前返回；只有首次自动打开抽屉时才设为 true
3. **`submitAppointment` 成功分支内增加 `router.replace()`**：把 URL 上的 `action=appointment` 移除，做双重保险

修复原则（已写入代码注释，固化下来防止再次回归）：

> **抽屉的 visible 变量，只允许在"用户主动点了打开按钮"和"首次响应 ?action=appointment URL 参数"两处被设为 true。其它任何 watcher / 提交回调 / 关闭回调里都只能设为 false，不能写 true。**
