## 三、功能 1：上游推送成功判定 — 兼容白名单

### 3.1 适用场景

管理后台 → "回调地址配置" Tab → 点击 🚀 推送到上游 后，与上游接口（如华为云）交互的成功判定。

### 3.2 新规则

满足以下"任一"条件即视为 ✅ 成功（之前只接受 `code=0`）：

| 维度 | 命中即成功 |
|---|---|
| HTTP 状态码 | 必须是 `200`（这一项是硬条件） |
| 业务 code | `0` / `200` / `"0"` / `"200"` |
| 业务 message | `ok` / `success` / `成功`（大小写不敏感） |

实际逻辑：`HTTP 200 AND (code 命中 OR message 命中)`。

### 3.3 操作步骤

1. 进入"回调地址配置" Tab
2. 确认"上游 URL""auth_token""org_id""回调域名"已填好
3. 点击 🚀 推送到上游 按钮
4. 在弹窗中查看自检结果（见下文功能 3），确认无问题后点击「自检通过，确认推送」
5. 推送完成后页面会显示：
   - **状态**：✅ 成功 / ❌ 失败（醒目色块）
   - **上游返回码**：原样透传上游返回的真实 code（不会被翻译/改写）
   - **判定依据**（新增，鼠标悬停查看）：例如 `HTTP 200 + code=200 命中成功白名单`、`HTTP 200 + message='success' 命中成功白名单`，让您一眼看清"为什么判成功（或失败）"

### 3.4 失败场景说明

| 场景 | 显示状态 | 显示消息 |
|---|---|---|
| 上游返回非 JSON（如 HTML） | ❌ 失败 | 上游响应解析失败（原始响应可在"原始响应"字段查看） |
| 网络超时（>10s） | ❌ 失败 | 请求超时 |
| HTTP 4xx/5xx | ❌ 失败 | 透传上游 message 或「上游服务异常」 |
| code/message 都未命中白名单 | ❌ 失败 | 原样透传上游 code/message |

## 四、功能 2：回调原始记录（新 Tab）

### 4.1 这个页面解决什么？

> 厂商说"我已经回调你们了"，但是"告警记录"却空空如也，到底是厂商没回？还是我们后端有 bug？

本页面就是回答这个问题的"审计黑匣子"。**只要请求到达 FastAPI 入口，无论解析成功失败、是否重复、是否未绑定，都会留下一条流水。**

### 4.2 入口

居家安全设备管理 → Tab「回调原始记录」

### 4.3 列表区

默认按"接收时间"倒序，每行展示：

| 列 | 说明 |
|---|---|
| 接收时间 | 厂商请求到达我方入口的 UTC 时间（Z 后缀） |
| 来源 IP | 厂商出口 IP（华为云为 `119.3.169.29`） |
| 解析状态 | ✅ ok / 🔁 duplicate / 🚫 unbound / ❓ unsupported_type / ❗ missing_field / 💥 internal_error / ⏳ pending |
| 设备 SN | 解析出的厂商设备号（devId） |
| 失败原因 | 仅当状态非 ok 时填写 |
| 操作 | 「查看」按钮 → 打开右侧详情抽屉 |

### 4.4 筛选区

| 筛选项 | 用法 |
|---|---|
| 时间范围 | 默认最近 7 天，可选择任意时间段 |
| 解析状态 | 全部 / ok / duplicate / unbound / unsupported_type / parse_fail / pending / fail |
| 设备 SN | 模糊匹配 device_sn 列 |
| 来源 IP | 精确匹配 source_ip 列 |
| 关键字 | 在请求体（JSON）中全文搜索（如搜 `msgId`、`devId`、`callType` 等） |

点击「查询」按钮即可应用筛选。

### 4.5 详情抽屉

点击列表中的「查看」按钮，右侧滑出抽屉，依次展示：

1. **基本信息**：ID / 接收时间 / 处理完成时间 / 来源 IP / 状态 / 失败原因 / 设备 SN / 厂商 msgId / 关联告警 ID
2. **请求方法**与**请求 URL**（含 query string）
3. **请求 Headers**（折叠展开）
4. **完整请求体（原始 JSON）**（高亮显示）
5. **解析后的字段映射对照**：厂商字段 → 我方字段 → 落库值（一目了然，便于排查映射差异）
6. **我方响应 HTTP 状态码 + 响应体**

### 4.6 状态枚举说明

| parse_status | 含义 | 是否需要关注 |
|---|---|---|
| `pending` | 入口刚记录，业务处理中（极短时间存在） | 不需要 |
| `ok` | 解析成功且已落告警表 | 正常 |
| `duplicate` | 同 `msgId` 已收到过（永久幂等） | 正常 |
| `unbound` | `devId` 未绑定到任何用户 | 排查"绑定关系" |
| `unsupported_type` | `dataType` 非 `call-msg` | 通知厂商按契约发送 |
| `missing_field` | 关键字段（如 devId）缺失 | 通知厂商修正报文 |
| `parse_fail` | JSON 解析失败 | 通知厂商发送合法 JSON |
| `internal_error` | 后端代码异常 | 立即排查后端日志 |
| `precheck` | 自检报文（dataType=`__precheck__`） | 自检产生的辅助流水 |
| `fail` | signature 校验失败等其它失败 | 视场景处理 |

### 4.7 防御性保证

后端入口已采用「**先写 pending → 业务后 update**」模式：

```
1. 入口最早期：立即 INSERT 一条 parse_status='pending' 流水（含完整 headers/body/source_ip）
2. 业务处理：解析、幂等、查绑定、落告警、发通知
3. 业务结束：UPDATE 流水的 parse_status / parse_fail_reason / linked_alarm_id
4. 即便业务代码抛异常，try/finally 也会把流水更新为 internal_error（不会永远 pending）
```

> **结论**：哪怕代码有 bug、DB 半路挂掉，只要请求进了 FastAPI 入口，**审计记录就一定存在**，您一定能在本 Tab 看到。

## 五、功能 3：回调地址自检

### 5.1 适用场景

避免"漏写 `/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27` 前缀"这类配置失误导致厂商回调被 nginx 404 拦截。

### 5.2 触发方式

在"回调地址配置" Tab 点击 🚀 推送到上游 按钮 → 弹窗中**先自动做一次本地自检**，再让您确认是否真的推送。

### 5.3 自检 5 项

| 检查项 | 通过条件 | 失败处理 |
|---|---|---|
| 1) URL 格式合法 | 正则匹配 `https?://...` | ❌ 阻止推送（fail） |
| 2) HTTPS 协议 | 以 `https://` 开头 | ⚠️ 警告（warn，可继续） |
| 3) 域名可解析 | DNS 能查到 IP | ⚠️ 警告（warn，可继续） |
| 4) 外网可达 | 后端 HEAD 请求得到 200/204/301/302/307/308/404/405/400/403 | ⚠️ 警告（warn，可继续） |
| 5) 路由命中本项目 | 自回环 POST `dataType=__precheck__`，对端返回 `matched_project: true` | ⚠️ 警告（warn，可继续） |

### 5.4 弹窗界面

```
即将向上游推送回调地址，请先核对自检结果：

✅ URL 格式合法
✅ HTTPS 协议
✅ 域名可解析
✅ 外网可达（HEAD HTTP 200）
✅ 路由命中本项目（自回环测试通过，已确认命中本项目）

完整推送内容：
- 上游 URL: http://119.3.169.29/treatment/api/setMsgCallBackUrl
- 回调 URL: https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/api/home_safety/callback/alarm

[取消]  [仍要推送]  [自检通过，确认推送]
```

### 5.5 按钮交互规则

| 自检结果 | 按钮状态 |
|---|---|
| 5 项全部 pass | 「自检通过，确认推送」高亮可点 |
| 有 warn（黄色）但无 fail | 「自检通过，确认推送」可点；同时显示「仍要推送」 |
| 有 fail（红色，blocked=true） | 「自检通过，确认推送」**置灰**；可点「仍要推送」（但极不推荐） |

> **建议**：除非您明确知道为什么 fail（如临时网络抖动），否则一定要把红色项修好再推送，避免"上游记错地址"的灾难。

## 六、推荐使用流程（首次对接 / 重新对接）

1. 在「回调地址配置」Tab 填写正确的：
   - 上游 URL（如 `http://119.3.169.29/treatment/api/setMsgCallBackUrl`）
   - auth_token / org_id（厂商提供）
   - 回调域名（**必须**包含 `https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27`）
2. 点击「保存」
3. 点击 🚀 推送到上游 → 等待自检结果
4. 自检 5 项全绿 → 点「自检通过，确认推送」
5. 等几秒，"最近一次推送结果"区域会显示 ✅ 成功 + 判定依据
6. 切换到「回调原始记录」Tab，让厂商触发一次真实告警，几秒后应该出现 `parse_status=ok` 的流水
7. 切换到「告警记录」Tab，应该出现新告警

## 七、常见问答

### Q1：上次推送显示 ❌ 失败，但厂商说他们 200 OK 了，怎么办？
本次修订后，已经把成功白名单扩展为 `code ∈ {0, 200, ...}` 或 `message ∈ {ok, success, 成功, ...}`。请重新推送一次，并在结果区悬停查看「判定依据」，可以看到我们用了哪条规则判定成功/失败。

### Q2：「回调原始记录」中所有记录都是 `unbound`，怎么办？
去「设备绑定」Tab 检查厂商上报的 `devId` 是否真的有绑定关系。如果没有，请用户先在 H5 / 小程序绑定该设备。

### Q3：「回调原始记录」一条记录都没有，怎么办？
99% 是厂商手里的回调地址错了（缺 `/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27` 前缀）。请：
1. 重新点 🚀 推送到上游 给厂商正确的地址
2. 或者直接联系厂商对接人，把回调地址改为带前缀的版本

### Q4：自检的「路由命中本项目」是 warn，但我能在「回调原始记录」看到自检流水（dataType=__precheck__）了，是为什么？
说明本机外网到 nginx 的路由是通的，但自检 POST 时未识别到 `matched_project=true`（可能是网络中转层做了改写）。这种情况通常可以忽略，可以放心推送。

### Q5：如何确认厂商真的回调了？
在「回调原始记录」筛选 `来源 IP = 119.3.169.29`（华为云出口 IP），最近的几条流水的接收时间应该和厂商触发的时间吻合。
