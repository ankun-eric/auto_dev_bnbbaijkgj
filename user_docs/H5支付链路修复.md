## 功能简介

本次修复彻底解决了 H5 端「假支付成功」的致命 Bug，并把「支付方式动态拉取」和「0 元订单免支付」机制铺到了 H5 / 微信小程序 / 安卓 App / iOS App 四个端：

1. **支付方式从后台动态拉取**：之前 H5 端硬编码显示"微信支付 + 支付宝"。现在四端均调用后端 `/api/pay/available-methods?platform=...` 接口，**仅展示后台启用且配置完整的通道**（如后台只启用支付宝 H5，则 H5 端只显示支付宝）。
2. **支付链路真正打通**：之前 H5 端点击「立即支付」只是创建订单后弹个 Toast「下单成功」就跳详情，**根本没真正发起支付**，订单却被错误标记为「已支付」。修复后：
   - 应付金额 > 0 → 调用 `/api/orders/unified/{id}/pay` 拉起对应通道（支付宝 H5 跳转沙盒收银台、微信小程序唤起 `wx.requestPayment`）
   - 应付金额 = 0（券或积分把订单抵扣到 0 元）→ 调用新接口 `/api/orders/unified/{id}/confirm-free` 直接将订单置为「已支付」
3. **订单详情页"已支付"判断修正**：以前用 `paid_amount > 0` 来判断是否已支付，导致 0 元免单订单永远显示「待付款」。修复后改为按 `订单状态` 或 `paid_at` 时间戳判断，**0 元订单也能正确显示为「已支付」**。
4. **后台支付配置真正生效**：后台「支付配置中心」启用 / 停用任一通道后，所有端立即生效；通道全部停用时：
   - 普通付费订单 → 立即支付按钮置灰，文案「暂未开通支付方式，请联系管理员」
   - 0 元订单 → 仍可下单成功（不受通道开关限制）
5. **历史「假已支付」订单可一键校准**：提供独立校准脚本 `backend/scripts/fix_paid_orders_without_paid_at.py`，扫描数据库中"状态已结算但 `paid_at` 为空"的订单，回滚为「待付款」。
6. **支付宝 H5 沙盒收银台**：本期为开发联调用，新增了一个 `/sandbox-pay` 模拟收银台页面。用户在 H5 端点击「立即支付（支付宝）」会跳到这个页面，点「确认支付」就会通知后端置订单为已支付（**待真实支付宝商户证书接入后会替换为真正的收银台**）。

## 使用说明

### 场景 1：商家在后台启用 / 停用支付通道

1. 用浏览器打开商家后台 [https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/)
2. 登录后进入「系统管理 → 支付配置中心」
3. 在通道列表中（如「支付宝 H5」「微信小程序」「微信 APP」「支付宝 APP」），点击对应通道的「启用 / 停用」开关
4. 保存后立即生效——无需重启服务、无需用户刷新页面，下次打开下单页时即可看到新的支付方式列表

### 场景 2：H5 用户结账（含支付宝沙盒）

1. 在手机浏览器打开 [H5 主站](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/)
2. 浏览商品 → 加入购物车 → 进入「下单页」
3. 在「支付方式」区块**只会看到后台启用的通道**（例如：仅启用支付宝 H5 时，只显示「支付宝」一行）
4. 选择支付方式 → 点击「立即支付」
5. **如果应付金额 > 0**：
   - 选择「支付宝」→ 跳转到沙盒收银台页面 → 点「确认支付」 → 自动跳回订单详情，状态变为「已支付」（**这是开发联调用的桩页面，未来接入真实支付宝商户证书后会替换为支付宝官方收银台**）
6. **如果应付金额 = 0**（用满减券免单 / 积分抵扣到 0）：
   - 直接弹出「支付成功」Toast → 跳订单详情 → 状态显示「已支付 / 待核销」

### 场景 3：H5 订单详情「立即支付」重试

1. 进入「我的订单」 → 选中一个「待付款」订单 → 点击「立即支付」
2. 系统会**实时拉取后台启用的支付通道**，若已无可用通道：
   - 应付金额 > 0 → 弹「暂未开通支付方式，请联系管理员」
   - 应付金额 = 0 → 直接走 `confirm-free`，立即结算成功

### 场景 4：微信小程序结账

1. 用「微信开发者工具」导入新版小程序 zip 并预览
2. 进入下单页 → 支付方式区块**动态渲染**（微信小程序场景下后台一般会启用「微信小程序支付」通道）
3. 点击「立即支付」 → 创建订单 → 调起 `wx.requestPayment` 微信支付弹窗 → 完成支付
4. 0 元订单同样直接走 `confirm-free`，无需弹支付窗

### 场景 5：安卓 / iOS App 结账

1. 安装最新版 APK / IPA（链接见上方表格）
2. 启动 App → 商品下单页：现在**多了一个「支付方式」选择卡片**（之前 Flutter 端没有），动态从后端拉取
3. 选定通道后点「提交订单」
4. 0 元订单 → 自动结算 → 跳详情
5. 普通付费订单 → 收到 SnackBar 提示「请在浏览器完成支付：{url}」（**本期未集成 url_launcher 自动跳浏览器**，待后续迭代），可手动复制链接到浏览器打开
6. 订单详情页和订单列表页的「立即支付」按钮也已联动改造，逻辑与下单页一致

### 场景 6：商家校准历史「假已支付」订单（一次性运维操作）

> 仅运维 / DBA 角色操作，普通用户无需关心。

1. SSH 登录服务器：`ssh ubuntu@newbb.test.bangbangvip.com`
2. 进入项目目录：`cd /home/ubuntu/6b099ed3-7175-4a78-91f4-44570c84ed27/`
3. **干跑（dry-run）**：先看哪些订单会被校准
   ```bash
   docker exec 6b099ed3-7175-4a78-91f4-44570c84ed27-backend python /app/scripts/fix_paid_orders_without_paid_at.py --dry-run
   ```
4. 确认列表无误后，**真正应用**：
   ```bash
   docker exec 6b099ed3-7175-4a78-91f4-44570c84ed27-backend python /app/scripts/fix_paid_orders_without_paid_at.py --apply
   ```
5. 脚本会把 `status` 已结算但 `paid_at IS NULL` 且 `paid_amount > 0` 的订单回滚为 `pending_payment`，并打印每条订单号

## 注意事项

1. **沙盒收银台是开发自测桩**：`/sandbox-pay` 页面是为了打通 H5 端到端联调流程。**实际生产上线前必须接入支付宝 H5 商户证书**，将后端 `pay_url` 替换为支付宝官方收银台 URL（具体改造点见 `backend/app/api/unified_orders.py` 中 `_build_sandbox_pay_url` 函数旁的 docstring）。
2. **iOS App 暂未自动跳浏览器**：本期 Flutter 端未集成 `url_launcher`，付费订单的 `pay_url` 用 SnackBar 文本提示。如需后续迭代加入 SDK 唤起 / WebView 内置，请在 `flutter_app/lib/screens/product/checkout_screen.dart::_submitOrder` 的 `pay_url` 分支处补充。
3. **支付通道全部停用 + 0 元订单不受影响**：这是设计行为，不要误以为「通道关了为什么还能下单」——0 元订单本质不需要走支付通道，符合财务对账规则 `paid_amount = 0 AND coupon_discount + points_deduction > 0` 即视为「用券 / 积分免单订单」。
4. **`paid_at` 字段语义**：`paid_at` = 完成支付（含 0 元免单确认）的 UTC 时间戳。**严禁**用 `paid_amount > 0` 判断「已支付」——这是本次 Bug 的核心根因。
5. **历史数据校准建议**：上线前先跑一次 `--dry-run` 看影响订单数量，再决定是否 `--apply`。校准后这些订单会回到「待付款」Tab，建议运营先短信通知用户。
6. **退款链路与 0 元订单**：0 元订单的退款走「券返还 + 积分返还」（已存在的业务逻辑），**不走**支付通道退款；后端 `request_refund` 暂未对 0 元订单做硬性拦截，建议后续迭代加 `if float(order.paid_amount) == 0: raise 400`。
