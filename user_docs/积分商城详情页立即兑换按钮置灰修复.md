## 一、本次修复了什么

### 1.1 Bug 现象

H5 端积分商城（路径 `/points/mall` → 点击商品卡片 → `/points/product-detail?id=xxx`）的**底部「立即兑换」按钮始终是灰色禁用态**，无论用户：

- 已登录、且**积分余额远大于商品所需积分**；
- 商品在架、库存充足；
- 也不在任何"已下架/已兑完/积分不足/已达上限"等业务规则拦截范围内。

按钮**点击 / 长按完全无响应**，没有 Toast、没有弹窗、没有跳转，用户无法通过详情页完成兑换。

> 兜底路径：列表页商品卡片上的「兑换」按钮一直是正常可点的，所以兑换流程并没有完全阻塞，但用户习惯先看详情再下单，详情页转化率几乎归零。

### 1.2 根因（一句话）

**后端详情接口的"按钮可兑换态"判定写错了**：

- 后端 `GET /api/points/mall/items/{id}` 的 `button_state` 默认值是 `"normal"`；
- 但 H5 详情页的按钮 disabled 属性严格绑定到 `button_state !== "exchangeable"`；
- 二者**字段口径不一致**，导致**正常情况下 button_state 永远不是 `"exchangeable"` → 按钮永远 disabled → 完全无响应**。

> 详情页和列表页用了两套不同的"是否可兑换"判定代码，是典型的"代码不一致"型前端 Bug。

### 1.3 修复点（覆盖全部 5 态）

修复后，「立即兑换」按钮的态完全统一，并与列表页一致：

| 业务场景 | 按钮状态 | 按钮文字 | 点击灰色按钮的反馈 |
| --- | --- | --- | --- |
| 商品在架 + 库存够 + 积分够 + 未达限兑 | **绿色高亮可点** | 立即兑换 | 进入兑换确认弹窗 |
| 商品已下架 | 灰色 | 已下架 | Toast「商品已下架」 |
| 商品库存为 0（physical / virtual / third_party） | 灰色 | 已兑完 | Toast「商品已兑完」 |
| 用户已达单人兑换上限 | 灰色 | 已达兑换上限 | Toast「已达兑换上限」 |
| 用户积分不足 | 灰色 | 积分不足，差 XXX 分 | Toast「积分不足，还差 XXX 分」 |

> 体验提升：以前灰色按钮"完全无响应"是体验黑洞，**修复后任意置灰场景下点击/长按灰色按钮都会弹出 Toast 提示原因**，不再让用户摸不着头脑。

## 二、操作步骤（用户视角）

### 步骤 1：进入积分商城

1. 在微信中点开 H5 链接，或在浏览器输入：
   - [https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/)
2. 完成登录后进入首页，从底部导航或「我的 → 积分」进入「积分商城」：
   - [https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/points/mall](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/points/mall)

### 步骤 2：选商品 → 进详情页

- 点击任意一个商品卡片本体，进入「商品详情页」。
- 详情页会显示商品轮播图、名称、所需积分、库存、详情富文本、使用须知。

### 步骤 3：底部「立即兑换」按钮（修复重点）

| 你看到的 | 含义 | 怎么处理 |
| --- | --- | --- |
| 绿色按钮「立即兑换」 | 可正常兑换 | 点击 → 弹出确认窗 → 确认后扣积分 |
| 灰色按钮「积分不足，差 XXX 分」 | 你的可用积分不够 | Toast 会提示还差多少分；可去做任务 / 签到补积分 |
| 灰色按钮「已兑完」 | 当前商品库存为 0 | 等待运营补货 |
| 灰色按钮「已下架」 | 商品已被运营下架 | 选其他商品 |
| 灰色按钮「已达兑换上限」 | 你对该商品的兑换次数已达单人上限 | 选其他商品 |

### 步骤 4：兑换流程

- 点击绿色「立即兑换」 → 弹出二次确认窗（体验服务类商品会额外提示"30 天有效，过期作废"）。
- 点「确认兑换」→ 接口扣减积分 → Toast「兑换成功」→ 自动跳转到「兑换记录」页。

## 三、注意事项

1. **登录态必须有效**：未登录访问会被网关 / 后端拦截。如长时间未操作，请重新登录。
2. **微信内置浏览器、iOS Safari、Android Chrome 三端表现一致**，本次修复完成后已在容器内通过 16/16 自动化测试覆盖核心场景。
3. **同一商品按钮态在列表页和详情页保持一致**：例如列表页"已兑完"则详情页也"已兑完"。如发现不一致，说明缓存了旧页面，刷新即可。
4. **体验服务类商品**特殊：库存字段为 0 时**视为无限**（由服务本体管控），不会被误判为"已兑完"，按钮可正常点击。
5. **积分余额计算口径**：以「累计获得 - 已消耗 - 已过期 - 已冻结」为权威，与「我的积分」页一致。

## 五、技术细节（运维 / 开发可读）

> 普通用户可跳过本节。

- 后端文件：`backend/app/api/points_exchange.py` → `GET /api/points/mall/items/{item_id}`
  - 默认 `btn_state="exchangeable"`（之前是 `"normal"`，本 Bug 根因）
  - 新增积分不足态 `"insufficient"` 判定 + 文案 `"积分不足，差 X 分"`
  - 新增 `user_available_points` / `user_exchanged_count` / `service_product` 三个响应字段，为前端渲染兜底
  - service 类型 stock=0 不再误判为 sold_out（与兑换接口的实际行为对齐）

- 前端文件：`h5-web/src/app/points/product-detail/page.tsx`
  - `disabled` 属性改为根据"是否可兑换"统一判定（兼容 `'exchangeable'` / `'normal'` 双值，防御性更强）
  - 灰色按钮**保留 onClick** + 弹出 Toast 说明原因，不再"完全无响应"
  - 按钮颜色仍随 disabled 状态变化（绿色 / 灰色），但点击不再被前端拦截

- 测试文件：`backend/tests/test_points_mall_detail_button_state.py`
  - 10 个用例覆盖 5 态 + 用户字段 + 优先级 + 服务类商品 + 字段完整性
  - 容器内 + 本地双跑全过
