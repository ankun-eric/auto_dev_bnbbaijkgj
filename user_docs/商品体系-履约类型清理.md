## 功能简介

本次更新是一次商品体系**履约类型清理 Bug 修复**，主要解决商品中台「履约类型」字段在管理后台下拉中**多出一项重复的"上门服务"**、**字典中残留 `to_store` 僵尸映射**、以及**"到店核销 / 线上服务"两个中文标签口径与业务期望不符**的问题。修复后：

- admin-web 后台「商品管理 → 新建/编辑商品」的「履约类型」下拉**有且仅有 4 项**，按 **快递配送 → 到店服务 → 上门服务 → 虚拟商品** 的固定顺序展示
- 5 端字典（admin-web / h5-web / 小程序 .js / 小程序 .wxs / Flutter）口径**完全统一**：
  - `delivery` → 快递配送
  - `in_store` → 到店服务（原"到店核销"）
  - `on_site` → 上门服务
  - `virtual` → 虚拟商品（原"线上服务"）
- 历史 `to_store` 僵尸映射（后端从未真正包含该枚举）已在 5 端字典中**彻底删除**
- admin-web 商品表单**禁止再手写本地履约下拉数组**，统一复用公共字典 `FULFILLMENT_OPTIONS`，从根上杜绝再次散落

> 数据库里**没有**任何 `fulfillment_type='to_store'` 的商品/订单/订单项数据，**无需任何 SQL 数据迁移**。修复纯属前端文案 + 字典清理 + UI 下拉重排。

## 使用说明

### 1. 商家管理后台 — 验证履约类型下拉清单

> 这是本次 Bug 的**正面验收路径**。

1. 用浏览器打开 [商家管理后台](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin)
2. 用商家账号登录后台
3. 左侧菜单进入 **「产品系统 → 商品管理」**
4. 点击右上角 **「+ 新建商品」** 按钮，弹出商品编辑表单
5. 找到 **「履约类型」下拉框**，点击展开
6. 期望看到的下拉选项：
   - **有且仅有 4 项**
   - 顺序：**快递配送 → 到店服务 → 上门服务 → 虚拟商品**
   - 「上门服务」**只出现一次**，**不再重复**
   - 不再出现「到店核销」「线上服务」字样
7. 任选其中一项保存，确认商品保存成功（商品库存、订单数据等业务流程不受影响）

### 2. H5 用户端 / 小程序 / App — 验证履约方式中文标签

> 这是本次 Bug 的**面向 C 端用户的影响验收路径**。

1. 在用户端打开任意商品列表 / 服务 Tab / 订单详情页
2. 找到带有「履约方式」角标或文案的位置
3. 期望看到的标签为以下 4 个之一：**快递配送 / 到店服务 / 上门服务 / 虚拟商品**
4. 不应再出现「到店核销」「线上服务」字样
5. 已存订单的履约类型字段值不变（业务数据未受影响），只是中文标签按新口径展示

### 3. 已存数据完整性确认（数据库层）

服务器后台已盘点确认：
- `SELECT COUNT(*) FROM products WHERE fulfillment_type = 'to_store'` → **0**
- `SELECT COUNT(*) FROM order_items WHERE fulfillment_type = 'to_store'` → **0**
- 后端 `FulfillmentType` 枚举仍为 4 值（in_store / delivery / virtual / on_site），未变化
- 各 `fulfillment_type` 值的分布与上线前**完全一致**

## 注意事项

1. **运营/客服公告**：业内对该字段在过去几周一直叫「到店核销 / 线上服务」，本次起改为「到店服务 / 虚拟商品」（与淘宝/京东/拼多多/抖音电商/有赞/微盟主流口径一致）。建议在内部群同步本次文案变更，避免新老叫法混用引起客户咨询时的歧义。
2. **数据完全无影响**：商品库存、订单数据、订单状态、字段枚举值均**100% 不变**，不存在数据迁移、不存在数据风险。
3. **历史 APK 兼容性**：旧版 APK（如 `app_orderstat_fulfillment_20260504_014200_3a07.apk` 等）仍可继续使用业务功能，只是其内部字典文案是旧口径。建议商家/测试人员替换为新 APK 以保持口径一致。
4. **后续防护**：今后所有端**禁止手写履约类型下拉数组**，必须 `import { FULFILLMENT_OPTIONS } from '@/utils/fulfillmentLabel'`（admin-web/h5-web）或对应端的等价方式。已在 PR Code Review Checklist 中加入此项约束。
5. **iOS 端**：本次 Flutter `lib/` 共享代码改动会同步影响 iOS。本次按修复方案约定不打单独 IPA，**iOS 包由下一次统一发版构建发布**，届时同步生效。
6. **回滚成本**：极低。所有改动均为前端文案 + 下拉重排，回滚仅需 `git revert` 对应提交并重新部署，**无任何数据库变更需要回滚**。
