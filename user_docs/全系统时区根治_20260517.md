## 功能简介

本次为**全系统时区根治**修复，解决长期存在的「历史对话/订单/消息列表里时间显示偏 8 小时（或显示「8 小时前」「1 天又 8 小时前」等明显错误）」的问题。

### 修复前

- 切换咨询人保存对话后，「历史对话」列表里刚保存的那条显示「**8 小时前**」（而非「刚刚」）
- 早于今天的对话显示「1 天又 8 小时前」之类的怪异时间
- 订单列表、消息中心、用药提醒等**所有展示相对时间的位置**都存在同样的 8 小时偏差
- 问题在 H5、微信小程序、安卓 App、苹果 App、Web 管理后台**所有端**都出现

### 修复后

- 所有时间列表按**您的设备本地时区**正确显示
- 刚保存的对话显示「**刚刚**」
- 真实 1 天前的对话显示「**1 天前**」
- 测试服在美国 / 生产服在中国的部署架构差异不再影响显示结果
- 未来即使出海、对接海外用户，时区会自动按用户本地显示

## 注意事项

### 给所有用户

- 本次修复**无需用户操作**，直接打开 App / H5 / 小程序即可体验
- 如果您之前看到的时间是错的（偏 8 小时），现在已经全部恢复正确
- 不需要清缓存、不需要重新登录、不需要重新安装 App

### 给开发者 / 内部团队

- **数据库不变**：所有 `datetime` 字段在数据库中**仍然是 UTC**，没有任何数据迁移
- **接口契约升级**：所有 API 出口的 `datetime` 字段现在**统一带 `+00:00` 后缀**（如 `"updated_at": "2026-05-17T02:30:00+00:00"`）
- **未来开发规范**：
  1. 后端写入数据库的 datetime **必须用 `datetime.utcnow()`**（禁用 `datetime.now()`）
  2. 后端 API 出口的 datetime 字段**无需手动 `.isoformat()`**，全局响应类已自动处理
  3. 前端各端**禁止散写** `new Date(iso).toLocaleString()` / 自实现 `formatRelativeTime`，必须通过统一工具
  4. 数据统计按北京时间用 `CONVERT_TZ(field, '+00:00', '+08:00')`

### 已知现象（非 Bug）

- 如果您的设备时区不是中国大陆（如海外用户），时间会按您当地的时区显示，这是预期行为
- 如果您手动修改了设备时区设置，时间显示会跟随您的设备设置
