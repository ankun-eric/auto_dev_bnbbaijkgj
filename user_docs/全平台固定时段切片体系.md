## 功能简介

本次升级落地 **PRD-01「全平台固定时段切片体系」v1.0**，把全平台预约时段的切片规则从「按门店营业时间动态生成」**统一为每段 2 小时、最早 06:00 起的固定 9 段**。这是后续所有按时段管理能力（看板、改期、统计、跨门店对账）的统一底座。

### 9 段固定切片一览

| 段号 | 时段 | 段号 | 时段 | 段号 | 时段 |
| ---- | ---- | ---- | ---- | ---- | ---- |
| 1 | 06:00-08:00 | 4 | 12:00-14:00 | 7 | 18:00-20:00 |
| 2 | 08:00-10:00 | 5 | 14:00-16:00 | 8 | 20:00-22:00 |
| 3 | 10:00-12:00 | 6 | 16:00-18:00 | 9 | 22:00-24:00 |

> 凌晨 **00:00-06:00 不开放预约**，与日视图 9 宫格 1:1 对应。

### 核心收益

- **数据口径统一**：跨门店、跨日期、跨视图完全可对比
- **看板格子化**：日视图直接 9 宫格强结构展示
- **客户体验一致**：所有门店时段粒度统一
- **改期通知准确**：改期短信/订阅消息中的时段标签按 9 段对齐，与商家看板看到的完全一致

## 使用说明

### 一、查看 9 段时段配置（公共接口，无需登录）

直接访问：

> 🔗 [https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/api/common/time-slots](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/api/common/time-slots)

预期响应：

```json
{
  "slots": [
    {"slot_no": 1, "start": "06:00", "end": "08:00"},
    {"slot_no": 2, "start": "08:00", "end": "10:00"},
    {"slot_no": 3, "start": "10:00", "end": "12:00"},
    {"slot_no": 4, "start": "12:00", "end": "14:00"},
    {"slot_no": 5, "start": "14:00", "end": "16:00"},
    {"slot_no": 6, "start": "16:00", "end": "18:00"},
    {"slot_no": 7, "start": "18:00", "end": "20:00"},
    {"slot_no": 8, "start": "20:00", "end": "22:00"},
    {"slot_no": 9, "start": "22:00", "end": "24:00"}
  ],
  "rule": "全平台固定 9 段时段（每段 2 小时，最早 06:00，最晚 24:00），凌晨 00:00-06:00 不开放预约"
}
```

### 二、商家端：9 宫格预约看板

1. 打开 [商家后台](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/admin/) 并登录
2. 进入「**预约管理 → 看板**」（或商家 H5 端「**门店看板**」）
3. **日视图**：直接看到 9 宫格（与上方 9 段一一对应）
   - 每格显示：`时段标签` / `预约总数` / `已核销数` / `已核销金额`
   - 点击任一格 → 弹出抽屉，列出该时段订单
4. **周视图 / 月视图**：按天聚合
5. **凌晨段订单（00:00-06:00）**：不在 9 宫格显示，但在订单列表中可见，标记为「凌晨异常单」

### 三、用户端：下单与改期

用户下单 / 改期入口在客户端无可见变化（仍按商品级 `time_slots` 渲染）。但订单后台已新增段号字段：

- 下单成功后，后端自动根据 `appointment_time` 计算 1-9 段号写入 `unified_orders.time_slot`
- 改期生效后，段号同步重算并写入
- 跨日订单（如 22:00-次日 00:00）按起始时间归段（计入第 9 段）

### 四、订单 `time_slot` 字段

订单详情接口响应中新增 `time_slot` 字段：

| 取值 | 含义 |
|------|------|
| 1-9 | 订单所属段号 |
| `null` | 凌晨段订单 / 无预约时间订单（先下单后预约） |

### 五、历史订单一次性回填

部署完成后，已自动执行 `backend/scripts/backfill_unified_orders_time_slot.py --apply`：

- 扫描 `time_slot IS NULL` 的历史订单 **32 单**
- 回填段号 1-9 共 **23 单**（段 2: 11 单 / 段 3: 3 单 / 段 4: 1 单 / 段 5: 3 单 / 段 6: 5 单）
- 凌晨段：**0 单**（保留 NULL）
- 无 appointment_time（先下单后预约 / 实物订单）：**9 单**（保留 NULL）

如需要重新执行（幂等，反复跑不会改写已有非 NULL 字段）：

```bash
docker exec 6b099ed3-7175-4a78-91f4-44570c84ed27-backend \
  python -m scripts.backfill_unified_orders_time_slot --apply
```

## 注意事项

### 1. 凌晨段订单（00:00-06:00）

- **R-01-02：不开放预约**——客户端时段选择器自动屏蔽 0-6 点
- **R-01-04：历史脏数据保留可见**——若历史确实存在凌晨预约订单：
  - `time_slot = NULL`
  - 商家看板 9 宫格不渲染
  - 订单列表仍可见，建议人工处理（改期或客服联系）

### 2. 跨日订单（如 22:00-次日 00:00）

- **R-01-03：按起始时间归段**——`appointment_time = 22:30` → `time_slot = 9`
- 第 9 段在看板抽屉中查询时窗口为 `[22:00, 次日 00:00)`，跨日订单不会丢点

### 3. 接口失败时的兜底

- 若客户端调用 `/api/common/time-slots` 失败（网络异常等），会自动使用本地常量兜底，确保下单/改期不阻塞
- 后端配置与客户端兜底值**完全一致**：每段 2 小时 / 最早 06:00 / 最晚 24:00

### 4. 与现有功能的兼容性

- ✅ 商品级 `time_slots`（每个商品配置自己的可预约时段 + capacity）**完全保留**，下单流程不变
- ✅ 改期上限 / 90 天范围 / 商家级 `allow_reschedule` 开关 **不变**
- ✅ 改期通知三通道（小程序订阅消息 + APP push + 短信）**不变**，仍按现有规则下发
- ✅ 历史订单状态机 / 核销码 / 退款链路 **不变**

### 5. 数据库变更

- `unified_orders` 表新增列 `time_slot INT NULL`，已索引（用于看板聚合性能）
- schema_sync 启动时增量 ALTER，**幂等可重复执行**
- 部署完成后已运行回填脚本
