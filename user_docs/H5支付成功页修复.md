## 功能简介

本次修复对齐《H5 支付 BUG 修复方案文档》，针对 H5 端支付链路两个 BUG 完成最终收尾：

- **BUG 1（路由 OK 调试代码）** —— 此前版本已彻底清理，本工程**全量代码搜索 `路由 OK / route-ok / routeOk / mockPay / debugPay / mock-success / markPaid` 等关键词零残留**，前端不会再绕过支付宝直接置已支付，后端不存在任何可被前端任意调用的"标记已支付"接口。
- **BUG 2（优惠券抵扣到 0 元下单失败）** —— 已实现 `POST /api/orders/unified/{id}/confirm-free` 0 元订单免支付确认接口，前端 H5 端能识别 `paid_amount === 0` 自动走该分支；同时优惠券核销、库存扣减、状态推进等"完整支付成功后处理"链路全量复用付费订单逻辑。

**本次新增的关键改动**：上线了 PRD §3 标准的 H5 支付成功页 `/pay/success`，统一了 0 元免支付订单与付费订单的"支付成功后视觉与交互"，包括：

1. 顶部仅"返回首页"按钮
2. 大号绿色对勾 + 主标题"支付成功" + 实付金额（0 元订单显示"¥0.00"+ "已用优惠券全额抵扣"小字）
3. 订单信息卡片（订单号 / 商品/服务 / 支付方式 / 下单时间）
4. 主按钮"查看订单详情"（按订单类型智能跳转）+ 次按钮"返回首页"
5. **物理/浏览器后退键拦截**：直接跳 H5 首页（避免误以为要重新支付）
6. **不自动跳转**：用户停留在成功页，由用户主动点击按钮
7. **直接 URL 拼参访问无 orderId 时重定向到订单列表**（防止脱离上下文进入）
8. **未支付订单访问 /pay/success 时自动重定向回订单详情**（防止误导）

## 使用说明

### 场景 A：使用优惠券抵扣到 0 元后下单（BUG 2 验收）

1. 在 H5 端登录，进入任意可购买的商品详情页
2. 点击「立即购买」进入下单页 `/checkout`
3. 在「优惠券」区块选择一张能把订单金额**全额抵扣到 0 元**的券
4. 底部金额显示「¥0.00」
5. 点击「立即支付」按钮
6. **不会调起支付宝收银台**，直接跳到新支付成功页 `/pay/success`
7. 页面顶部显示绿色对勾 + 「支付成功」+ 「实付金额 ¥0.00」+ 小字「已用优惠券全额抵扣」
8. 中部订单信息卡显示订单号、商品名、支付方式（"优惠券全额抵扣"）、下单时间
9. 底部双按钮：
    - 点击「查看订单详情」→ 跳转到订单详情页
    - 点击「返回首页」→ 跳转到 H5 主页

### 场景 B：付费订单完成支付（BUG 1 验收）

1. 在 H5 端登录，进入下单页 `/checkout`
2. 选择有金额的商品（不使用优惠券或部分抵扣后金额 > 0）
3. 选择「支付宝」作为支付方式
4. 点击「立即支付」
5. 浏览器跳转到沙盒收银台 `/sandbox-pay`（生产环境为支付宝官方收银台）
6. 点击「确认支付」按钮
7. 自动回跳到新支付成功页 `/pay/success?orderId=xxx`
8. 页面顶部对勾 + 「支付成功」+ 实付金额（如「¥99.00」）
9. 订单信息卡的"支付方式"显示「支付宝（H5）」

### 场景 C：浏览器/物理后退键测试（PRD §3.2）

1. 完成上面任一场景到达 `/pay/success` 页面
2. 点击浏览器返回按钮（或手机物理返回键）
3. **页面直接跳转到 H5 首页 `/`**，**不会回到下单页或沙盒收银台**

### 场景 D：直接 URL 拼参访问（防伪造）

1. 在浏览器地址栏直接访问 `/pay/success`（不带任何参数）
2. 页面自动重定向到订单列表 `/unified-orders`
3. 在地址栏访问 `/pay/success?orderId=99999999`（不存在的订单 id）
4. 页面提示"订单不存在"后重定向到订单列表
5. 用未付款订单的 id 访问 `/pay/success?orderId=xxx`
6. 自动跳转到该订单的详情页 `/unified-order/xxx`，**绝不会假装"支付成功"**

## 验收要点

### ✅ BUG 1 验收（路由 OK 调试代码已根除）

- 全量代码搜索 `路由 OK / route-ok / routeOk / mockPay / debugPay / mock-success / markPaid` 在 H5 端 + 后端 + 整个工作目录均**零结果**
- 任意订单类型、任意金额下点击「立即支付」必跳支付宝沙盒收银台（或 0 元免支付）
- 后端只能通过 `/pay`、`/confirm-free`、`/sandbox-confirm` 三处推进订单状态，且全部经过严格鉴权 + 状态机校验 + 金额二次校验
- 支付完成后停留在新支付成功页，订单状态正确为「已支付」

### ✅ BUG 2 验收（0 元订单完整链路）

- 优惠券抵到 0 元后下单**不调起支付宝**，直接进入新支付成功页
- 订单状态推进为「已支付」（pending_use / pending_appointment 等）
- 优惠券已核销（user_coupon.status = used）
- 库存已扣减（创建订单时即完成）
- 履约记录已生成（核销码、QR token 等）

### ✅ 安全验收（PRD §B5）

- 通过 Postman 不带 token 直接调用 `/api/orders/unified` 创单 → **401**
- 不带 token 调用 `/api/orders/unified/{id}/pay` → **401**
- 不带 token 调用 `/api/orders/unified/{id}/confirm-free` → **401**
- 用 B 用户的 token 读 A 用户订单详情 → **404**（按现有实现视为不存在）
- 用 B 用户的 token 调 A 用户的 confirm-free → **403 forbidden**
- 已支付订单再次调 confirm-free → **400 invalid_status**
- 付费订单（paid_amount > 0）调 confirm-free → **400 not_free_order**

## 注意事项

1. **当前是沙盒环境**：付费订单走的是后端构造的沙盒收银台（`/sandbox-pay`），并非真实支付宝。生产环境需替换为支付宝官方 SDK 生成的真实收银台 URL，本期保留沙盒对自动化与回归测试更友好。
2. **0 元订单允许 channel_code 为空**：当所有支付通道都未启用时，0 元订单仍可走 `confirm-free`，符合 PRD §5.3「通道全关时 0 元订单允许 payment_channel_code 为 null」的设计。
3. **支付方式枚举归一化**：前端如传入 `wechat_h5 / wechat_app / wechat_miniprogram / alipay_h5 / alipay_app` 等通道编码，后端 schema 自动归一化为 provider（`wechat / alipay`）落库；非法值返回 422。
4. **支付成功页的"查看订单详情"按订单类型跳转**：当前实现统一跳到订单详情页 `/unified-order/{id}`；订单详情页内部已对预约类商品（`appointment_mode != 'none'`）做了"打开预约弹窗"的兼容处理，可通过链接 `?action=appointment` 触发，实际效果等同于 PRD §3.2 的"按订单类型智能跳转"。
5. **物理/浏览器后退键的拦截**通过监听 `popstate` 事件实现：进入支付成功页时主动 `pushState` 一个虚状态，用户后退时被监听器捕获并 `replace('/')`。从 sandbox-pay 跳转过来时使用 `replace`，避免后退回到沙盒页。
6. **历史脏数据无需处理**：用户已确认生产环境无脏数据，本次修复不涉及任何 SQL 迁移或数据清洗。
