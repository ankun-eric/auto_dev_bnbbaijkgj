## 一、问题概述

最近一次"居家安全设备"需求发布后，H5 端出现**全站跳转崩溃**：除 AI 主页与管理后台外，其他几乎所有页面（首页、健康自查、体质测评、AI 问答、健康档案、商城、订单、通知等）一进入就**红屏报错**，连页面框架都渲染不出来。所有账号、所有手机/浏览器都能稳定复现。

本次更新已**完整修复**该问题，全站 14 项关键页面经服务器端 HTTP 探测全部 200 通过，无客户端崩溃错误标记。

---

## 二、根因一句话说明

> **`h5-web/src/lib/datetime.ts` 在「居家安全设备」需求中被整文件重写，删除了全站 30+ 页面正在使用的 `parseServerTime`、`formatTime`、`formatRelativeTime`、`formatFriendlyTime` 等导出函数，导致引用这些函数的页面在客户端运行时全部报 "is not a function" 红屏。**

附带发现并修复了一处**部署侧问题**：H5 容器（Next.js standalone 模式）的 `HOSTNAME` 环境变量未显式设置为 `0.0.0.0`，部分情况下会监听到容器内部某个非 docker 网络 IP，导致 gateway-nginx 反代 502。

---

## 三、本次修复内容

### 3.1 代码层面修复

| 文件 | 改动 | 作用 |
|------|------|------|
| `h5-web/src/lib/datetime.ts` | 恢复并完善 `parseServerTime` / `formatDateTime` / `formatDate` / `formatTime` / `formatRelativeTime` / `formatFriendlyTime` 6 个导出函数 | 让 30+ 页面可正常 import 和调用，避免 SSR / CSR 阶段报 `_datetime__WEBPACK_IMPORTED_MODULE__.xxx is not a function` |
| `h5-web/Dockerfile` | 新增 `ENV HOSTNAME=0.0.0.0` | 让 Next.js standalone 监听所有网卡，gateway-nginx 才能在 docker 网络中正常反代 |

### 3.2 兼容性说明

- 「居家安全设备」页面的 `formatDateTime(iso)` 单参数调用**继续可用**，输出 `YYYY-MM-DD HH:mm:ss`（按用户本地时区，中国用户即北京时间），与原 v1 BUGFIX 期望一致。
- 其他页面历史调用 `formatDateTime(iso, 'YYYY-MM-DD HH:mm')` 等带 pattern 参数的形式**继续可用**。
- 后端时间字段无需任何调整，UTC + `Z` 后缀的设计被完整保留。

## 七、根因详解（供研发参考，可跳过）

「居家安全设备」需求 BUGFIX 提交（`201d506f`）将 `h5-web/src/lib/datetime.ts` 由原本的 117 行（含 6 个导出函数）改写为 27 行（仅 2 个导出函数），并把内部实现从原生 `Date` 切换为 `dayjs`：

- 删除的导出函数：`parseServerTime`、`formatTime`、`formatRelativeTime`、`formatFriendlyTime`
- 改变签名的导出函数：`formatDateTime(iso, pattern?)` 变为 `formatDateTime(iso)`，移除 pattern 参数
- 新增依赖：`dayjs/plugin/utc` 和 `dayjs/plugin/timezone`

由于全站有 30+ 页面通过 `import { parseServerTime, formatDateTime, formatTime, ... } from '@/lib/datetime'` 引用这些函数，**它们运行时一律变成 `undefined`，被调用即抛 `TypeError: xxx is not a function`**，触发 Next.js 错误边界 → 红屏。

仅 AI 主页与管理后台不依赖这些函数，所以这两个页面"幸存"，与用户线上观察到的现象完全吻合。

修复方案是**保留原 datetime.ts 的全部 6 个导出函数与签名**，同时将默认输出格式调整为 `YYYY-MM-DD HH:mm:ss`（含秒），从而：

1. 全站 30+ 历史页面零改动恢复；
2. 居家安全设备页面 `formatDateTime(iso)` 单参调用得到的字符串仍是 `YYYY-MM-DD HH:mm:ss` 北京时间，与 v1 BUGFIX 设计意图一致；
3. 不再依赖 `dayjs/plugin/timezone`（避免缺失插件 / SSR 时区库 ICU 数据问题）。
