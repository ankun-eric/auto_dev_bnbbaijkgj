## 功能简介

本次紧急修复了两类阻塞性故障，恢复 AI 对话与拍照识药两条核心主线：

### Bug A：AI 对话所有入口"秒回 500 网络异常"

- **现象**：用户在任意 AI 对话场景（普通问诊 / 用药咨询 / 报告解读等）发送消息后，**瞬间**收到"网络异常，请稍后重试"
- **根因**：`backend/app/api/chat.py` 中 `_build_health_context()` 函数在拼接"既往病史 / 过敏史"时使用了 `"、".join(allergies)`，而线上数据库的 `health_profiles.allergies` 字段是 `list[dict]`（如 `[{"type": "custom", "value": "无"}]`），导致 `TypeError: sequence item 0: expected str instance, dict found`，进而 500
- **修复**：新增 `_normalize_tag_list()` 工具函数，统一把 `list[str]` / `list[dict]` / 不规则数据归一化为字符串数组后再拼接

### Bug B：拍照识药弹窗 4 个按钮"完全失灵" + 图标缺失 + `/ai-home/无/ 404`

- **现象**：拍照识药入口弹窗的"相册 / 拍照 / 本机 / 微信" 4 个按钮，**图标错位**、**点击毫无反应**，浏览器控制台立即出现 `/ai-home/%E6%97%A0/ 404`（`%E6%97%A0` 是中文"无"）
- **根因（三层）**：
  1. **后端脏数据**：`chat_function_buttons` 表中 id=2"拍照识药"按钮的 `icon_url` 字段被错误存为字面值 `"无"`
  2. **前端 `<img src="无">`**：`ChatCards.tsx` 直接把 `button.coverImage = "无"` 喂给 `<img src>`，浏览器把字符串"无"当成相对路径，解析为 `/ai-home/无` → 404
  3. **upload 卡片 4 按钮无 onAction 处理**：upload 类型下 onAction 只发 `autoUserMessage`（且为空时静默），未对 `'album'|'camera'|'local'|'wechat'` 子动作做任何实际处理 → 死按钮
- **修复（三层防护）**：
  1. **数据清理**：新增启动迁移 `_migrate_bug470_cleanup_placeholder`，把 `chat_function_buttons` 表中 `icon_url / card_cover_image / external_url` 字段里的"无 / 暂无 / null / N/A / 未设置"等占位词及一切非合法 URL **统一置 NULL**
  2. **Schema 验证兜底**：在 `ChatFunctionButtonResponse` Pydantic schema 加 `field_validator`，序列化时再过一遍 URL 合法性校验，未来即使有脏数据写入也不会泄漏到前端
  3. **前端 URL 校验**：`ChatCards.tsx` 新增 `isValidImageUrl()` 函数，仅当 coverImage 以 `http://` / `https://` / `/` / `./` / `data:image/` / `blob:` 开头才作为图片显示，否则降级为占位方块；同时 `<img onError>` 兜底
  4. **upload 4 按钮真实交互**：`ai-home/page.tsx` 给 upload 卡片的 `onAction(sub)` 4 种子动作各自实现：
     - `album` → 唤起系统相册（`<input type=file accept=image/*>`）
     - `camera` → 唤起后置摄像头拍照（`capture=environment`）
     - `local` → 唤起本机文件选择器（含 PDF）
     - `wechat` → Toast 友好提示"请先保存到相册再选择"并退化到相册

## 注意事项

1. **浏览器缓存**：如果用户在 5 月 15 日之前体验过页面，建议**强制刷新**（Ctrl+F5 或 Cmd+Shift+R）以加载最新的前端 chunk
2. **微信入口**：H5 端的"微信"按钮无法直接调用微信系统级"选图"能力（这是 H5 平台限制），现降级为"先保存到相册"提示，体验上微信内打开 H5 时仍可正常通过相册选图
3. **PDF 上传**：本机选择器现已支持 PDF，但服务端 OCR 是否支持 PDF 取决于后端 OCR 服务能力，超大 PDF 建议分页截图
4. **脏数据回查**：未来 admin 后台再有人误填 `icon_url='无'` 等占位值，**前端 + schema 会双重过滤**为 null，不会再触发 404；如需彻底防住录入侧，建议下次迭代在 admin 的"添加/编辑按钮"表单加 URL 格式校验
