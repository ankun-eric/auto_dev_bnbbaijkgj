## 修复背景

### 现象

商家兼顾客身份的用户（手机号既绑定商家角色、又是某门店的真实顾客）在客户端（H5 用户端 / 顾客微信小程序 / 顾客 APP）登录后，点击「修改预约」「申请退款」「取消订单」等顾客专属动作时，弹出 `403 无操作权限` 报错，导致**改期等核心功能完全无法使用**。

### 根因

后端 9 个订单顾客接口原本一刀切只看 `users.role` 字段，只要不是 `user`（即只要兼任商家）就直接 403 拦截。这与项目业务规则「同一手机号可同时承担两种角色」直接冲突。

### 修复方案

把鉴权依据从「**用户角色**」切换到「**当前会话来源**」（HTTP 请求 Header `Client-Type`）：

- 若当前是从客户端家族（`h5-user` / `miniprogram-user` / `app-user`）发起的请求 → **放行**
- 若当前是从商家端家族（`h5-mobile` / `pc-web` / `verify-miniprogram`）或来源不明 → **403 + 引导文案**

## 修复范围

### 后端

新增 `app/utils/client_source.py` 客户端家族常量、`CUSTOMER_CLIENTS` 集合、`require_customer_client_session` 依赖项；并在 `app/api/unified_orders.py` 9 个顾客操作接口集中接入：

| # | 接口 | 动作 |
|---|---|---|
| 1 | `POST /api/orders/unified` | 下单 |
| 2 | `POST /api/orders/unified/{id}/pay` | 支付 |
| 3 | `POST /api/orders/unified/{id}/confirm-free` | 0 元单确认 |
| 4 | `POST /api/orders/unified/{id}/appointment` | 改期（**Bug 主犯**） |
| 5 | `POST /api/orders/unified/{id}/cancel` | 取消订单 |
| 6 | `POST /api/orders/unified/{id}/confirm` | 确认收货 |
| 7 | `POST /api/orders/unified/{id}/review` | 评价订单 |
| 8 | `POST /api/orders/unified/{id}/refund` | 申请退款 |
| 9 | `POST /api/orders/unified/{id}/refund/cancel` | 撤回退款申请 |

### 前端三端

- **H5（h5-web）**：请求拦截器按路径三段式判定 Client-Type
  - `/merchant/m/*` → `h5-mobile`（商家移动）
  - `/merchant/*` → `pc-web`（商家 PC）
  - 其他 → `h5-user`（顾客 H5）
- **顾客微信小程序（miniprogram）**：`utils/request.js` 全局固定 `Client-Type: miniprogram-user`
- **顾客 Flutter APP（flutter_app）**：`ApiService` BaseOptions + 拦截器双保险固定 `Client-Type: app-user`

## 验收使用说明

### 场景 1：商家兼顾客用户在 H5 用户端改期（核心修复点）

1. 打开 [H5 用户端首页](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/)，用一个**已绑定商家身份的手机号**登录顾客端
2. 进入「我的订单 → 已预约订单详情」
3. 点击「修改预约」按钮
4. **预期**：可以正常打开日期/时段选择器，提交改期成功（修复前：`403 无操作权限`）

### 场景 2：商家兼顾客用户在顾客小程序 / Flutter APP 下单 / 退款 / 取消

- 在顾客小程序 / 顾客 APP（用 Bug-367 版本 APK / IPA）执行下单、申请退款、撤回退款、取消订单、确认收货、评价订单等操作
- **预期**：所有操作都不再误拦，全部正常通过（修复前：全部 403）

### 场景 3：商家通过商家端访问顾客接口（防越权）

1. 商家通过 H5 商家手机端 / 核销小程序 / PC 后台登录
2. 即使手动伪造请求路径调用顾客专属接口（例如 `/api/orders/unified/{id}/appointment`）
3. **预期**：返回 `403 该操作仅限客户端使用,请切换到顾客 APP / H5 用户端登录后再试`，并且文案明确**指导用户去客户端而不是再次重试**

### 场景 4：未携带 Client-Type Header 的兜底拦截

第三方脚本/旧版客户端如果忘记带 `Client-Type` Header，仍会被识别为 `unknown` 并拦截，避免悄悄绕过校验。

## 风险与回滚

- **风险**：第三方旧版本前端如果未升级到带 `Client-Type` Header 的版本，将被新依赖拦成 403；本项目内三端均已同步发布，**无外部第三方调用**，风险可控。
- **回滚**：`git revert da9d710` 即可回退到原一刀切 `users.role` 行为；后端单文件回滚耗时 < 30 秒。
