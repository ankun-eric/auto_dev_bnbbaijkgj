## 1. 功能简介

本次把「核销动作」**完全收归手机端**，PC 端任何位置不再允许直接发起核销，仅保留查看核销记录的能力。背景是 PC 端核销存在被冒用、未到店核销、流程脱节等风险；手机端（员工随身设备）才是核销动作发生的真实场景。

### 核心改造

| 改造点 | 说明 |
|---|---|
| **后端来源校验** | 所有核销类接口新增 `require_mobile_verify_client` 依赖，依据 `Client-Type` 头部 / `User-Agent` 识别来源，仅允许 `h5-mobile` 与 `verify-miniprogram`，PC 端调用直接 403 |
| **统一核销入口** | 新增 `POST /api/verifications/verify`（PRD §2.4 规定的对外接口名），与既有 `POST /api/verify/redeem` 行为完全等价，方便前端统一调用 |
| **PC 端入口移除** | 商家 H5（PC 浏览器访问）9 宫格弹窗 / 列表行操作菜单中的「核销」按钮强制置灰，文案改为「核销（请到手机端）」，鼠标悬浮提示「PC 端不允许发起核销，请到手机端 H5 / 核销小程序操作」 |
| **Admin 平台保持只读** | Admin 平台原本就没有「发起核销」按钮，本次额外为其 Axios 拦截器写死 `Client-Type: pc-web`，即使有人通过工具直接调接口也会被后端 403 |
| **核销记录页保留** | 任何端的「核销记录查询 / 列表 / 详情」页面**继续可用**，只是不允许在 PC 上执行 `POST /api/verifications/verify`、`POST /api/verify/redeem`、`POST /api/verify/member-qrcode`、`POST /api/verify/checkin`、`POST /api/staff/cards/redeem`、`POST /api/orders/{id}/verify` 这些动作类接口 |

## 3. 使用说明

### 3.1 顾客端

顾客侧无任何 UI 变化。在小程序 / APP / H5 用户端的「我的订单 → 出示核销码」、「会员二维码」继续正常使用，由商家用手机端扫码完成核销。

### 3.2 商家：用手机扫码 / 输码核销

#### 路径 A：商家 H5 移动端（推荐）

1. 用手机浏览器打开： [https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/m/verify/](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/m/verify/)
2. 登录商家账号。
3. 选择「扫码核销」或「手动输码核销」。
4. 系统识别请求来源为 `h5-mobile`，后端放行；核销成功提示「核销成功」。

#### 路径 B：核销小程序

1. 下载并解压：[verify_miniprogram_20260505_190014_1348.zip](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/downloads/verify_miniprogram_20260505_190014_1348.zip)
2. 微信开发者工具 → 导入项目 → 选择解压目录 → 选择测试号或填入 AppID。
3. 真机调试或模拟器中登录商家账号。
4. 系统识别请求来源为 `verify-miniprogram`，后端放行；核销成功提示「核销成功」。

### 3.3 商家：在 PC 浏览器看到什么？

- **预约日历 9 宫格抽屉**（`/merchant/calendar`）的订单卡片 Popover：「核销」按钮**置灰**，文案变为「核销（请到手机端）」；鼠标悬浮提示「PC 端不允许发起核销，请到手机端 H5 / 核销小程序操作」。
- **预约日历列表视图**：行末「⋯」下拉菜单中「核销」项**置灰**，文案变为「核销（请到手机端）」，悬浮提示「请到手机端 H5 / 核销小程序发起核销」。
- **如商家通过浏览器开发者工具或 Postman 直接调接口**：后端返回 `403 Forbidden`，错误文案：

  > 核销动作仅限手机端发起，请到手机端 H5 / 核销小程序操作

- **核销记录查询 / 列表 / 详情页**：完全可用，可正常翻页、筛选、查看明细，不会被拦截。

### 3.4 Admin 平台运营：什么变了？

- 原来 Admin 平台就没有「发起核销」按钮，本次保持不变。
- 增量保护：Axios 拦截器对所有请求注入 `Client-Type: pc-web`。即便有人通过 Admin 网络调用工具直接调用核销接口，后端也会立即返回 403。

## 4. 后端来源识别规则

后端按以下优先级识别请求来源：

1. **优先**：HTTP 头 `Client-Type` 或 `X-Client-Type`，已知值：
   - `h5-mobile` → 放行
   - `verify-miniprogram` → 放行
   - `pc-web` / 其他 / 缺失 → 进入步骤 2
2. **兜底**：`User-Agent` 兜底解析：
   - 含 `MicroMessenger` 与 `MiniProgram` → `verify-miniprogram` 放行
   - 含 `iPhone / iPad / Android / Mobile` → `h5-mobile` 放行
   - 含 `Windows / Macintosh / Linux X11` → `pc-web` → **403 拦截**
   - 其他 → `unknown` → **403 拦截**

> **设计意图**：默认从严，未知来源一律拒绝；前端通过显式 `Client-Type` 主动声明身份，确保后端识别准确率。

### 受保护的核销接口清单

| 接口 | 用途 |
|------|------|
| `POST /api/verifications/verify` | 统一核销入口（PRD §2.4 新增） |
| `POST /api/verify/redeem` | 订单核销（既有，等价别名） |
| `POST /api/verify/member-qrcode` | 会员码核销 |
| `POST /api/verify/checkin` | 到店核销 |
| `POST /api/staff/cards/redeem` | 卡核销（员工扫码） |
| `POST /api/merchant/orders/{order_id}/verify` | 商家端订单核销 |
| `POST /api/orders/{order_id}/verify` | 旧版核销（已注释禁用，仍加 403 防线） |

## 6. 注意事项

| 注意点 | 说明 |
|---|---|
| 移动端核销不受影响 | H5 移动端（`/merchant/m/*`）与核销小程序仍可正常发起核销；后端按 `Client-Type` 放行 |
| PC 端核销记录可查 | 「核销记录查询」「核销明细」等查询类页面在 PC 端继续可用，本次仅拦截写动作 |
| 错误码与文案 | PC 端调用核销接口将收到 `HTTP 403 {"detail": "核销动作仅限手机端发起，请到手机端 H5 / 核销小程序操作"}` |
| 401 优先于 403 | 未登录的 PC 请求会先被认证依赖 `require_identity` 拦下并返回 401（未授权），不会到达 403；这是 FastAPI 依赖注入的执行顺序，**不影响功能正确性** |
| 头部容错 | `Client-Type` 与 `X-Client-Type` 都被识别，写法不区分大小写、容忍 `_` / `-`；缺失头部时回退到 User-Agent 兜底 |
| 头部伪造风险 | 如果终端伪造 `Client-Type: h5-mobile` 头，仍可能绕过限制；这属于**业务层防护**，主要是防止 PC 用户**意外**发起核销。如需更强保护，建议结合 Token 中的 `client_id` 字段做额外校验（后续 PRD 可纳入） |
| 商家在 PC 看到核销请求被拒怎么办？ | 文案已直接给出指引：「请到手机端 H5 / 核销小程序操作」。引导员工使用本人手机即可 |
