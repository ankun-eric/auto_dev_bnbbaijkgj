## 功能简介

本次更新解决了一个**全端共通的图片/附件链接错位 Bug**，并对该类问题做了一次性根治：

### 修复了什么

- **商家 PC 后台 → 订单管理 → 附件**：图片缩略图与大图预览全部恢复正常（之前点开预览出现纯文本 `Gateway OK`）
- **管理后台**：用户头像、商品图、聊天图片预览、运营素材、Banner、菜单图标等所有依赖后端裸路径的图片，全部正常加载
- **用户 H5**：商品图、订单详情图、聊天图、报告封面、家庭授权图、点数兑换图、新闻封面、TCM 分享图、个人头像等全部正常
- **微信小程序**：聊天附件、检查报告封面、家庭邀请二维码、数字人物料等全部正常

### 工作原理（不必看也无妨）

后端给前端返回的图片地址是 `/uploads/xxx.jpg` 这种"裸路径"，浏览器会把它丢到域名根下，**绕过了部署子路径** `/autodev/<uuid>/`，最终被 nginx 网关兜底为纯文本 `Gateway OK`。
本次治理在每个前端工程引入一个 `resolveAssetUrl()` 工具函数，把所有由后端字段直出的图片地址统一补齐为带 basePath 的可访问 URL。后端代码**未做修改**，仅前端治理。

1. **强制刷新**：Web 端首次访问可能命中浏览器缓存，建议用户按 Ctrl+F5（Windows）或 Cmd+Shift+R（Mac）做一次强制刷新

3. **多环境兼容**：
   - 测试环境（带 `/autodev/<uuid>/` 子路径）：图片正常
   - 生产环境（根路径部署）：图片正常（工具函数透传，零副作用）
   - 本地开发（localhost）：图片正常
4. **新页面开发**：开发者后续新增图片展示位置时，请通过 `import { resolveAssetUrl } from '@/lib/asset-url'` 引入并使用，避免重新引入同类 Bug
5. **小程序需要在真机上完整体验**：开发者工具模拟器有时对域名校验更宽松，建议导入后用预览二维码扫码到真机做最终验证
