# 商家后台「预约看板」替换升级 用户体验手册（PRD-365）

> 版本：v1.0  
> 发布日期：2026-05-06  
> 适用环境：测试环境  
> 部署 ID：`6b099ed3-7175-4a78-91f4-44570c84ed27`

---

## 一、本次升级一句话概览

把商家 PC 后台原来的「预约日历」整体替换升级为与管理员同款的「预约看板」（9 宫格日视图 / 周视图 / 月视图 / 服务分组 / 资源视图 / 列表视图），并把管理员后台的「预约看板」入口收口到「商家管理 - 门店管理」每行的「预约看板」按钮，同时在订单首次进入「待核销」状态时通过微信模板消息实时通知该门店所有已绑定微信的员工。

## 三、商家端体验流程

### 3.1 入口变更
- 左侧菜单「预约日历」已替换为「预约看板」（图标保持日历图标）。
- 直接访问旧地址 `/merchant/calendar/` 时，会出现"正在为您跳转…"提示，1 秒内自动跳到 `/merchant/order-dashboard/`。

### 3.2 预约看板顶部 4 个 Tab
1. **预约看板（默认）**：3×3 = 9 宫格日视图、3×7 = 21 格周视图、月历月视图，可切换。
2. **服务分组日视图**：按服务/商品分组展示当日预约。
3. **资源视图**：按资源（房间 / 工位 / 床位）分组展示当日预约。
4. **列表视图**：跳转到「订单管理」列表页（沿用既有功能，避免重复实现）。

### 3.3 4 色状态色板
| 颜色 | 状态 | Ant Design Token |
|------|------|------------------|
| 蓝 #1677FF | 待到店（pending）| colorPrimary |
| 橙 #FA8C16 | 已到店（arrived）| colorWarning |
| 绿 #52C41A | 已核销（verified）| colorSuccess |
| 灰 #BFBFBF | 已取消（cancelled）| colorTextDisabled |

> 备注：PRD R07 中的"已退款（refunded）"状态不在订单看板的展示集合内，本期沿用代码现状的 4 色状态。退款订单走「订单管理」列表查看。

### 3.4 抽屉与操作
- 点击 9 宫格 / 周视图小格 / 月历某天 → 弹出 480 px 宽抽屉（PC）或全屏 Drawer（移动端）。
- 4 个操作按钮：
  - **核销** → PC 置灰，提示「请到手机端核销」。
  - **修改预约时间** → PC 置灰，提示「请到手机端修改时段」。
  - **取消预约** → PC 可用，二次确认后取消。
  - **联系客户** → PC 可用，`tel:` 协议 + 复制手机号。

### 3.5 门店多店切换
- 商家若关联多门店，看板默认展示当前选中门店；右上角切换门店后，看板自动按新门店刷新数据。
- 未选中任何门店时，看板会提示「尚未选择门店，请先在右上角切换门店」。

### 3.6 响应式
- 桌面（≥1024px）：9 宫格 3×3、抽屉 480 px。
- 平板（≥768px）：9 宫格 2×4 自适应。
- 移动浏览器：单列堆叠、全屏 Drawer，建议长流程改用 H5 小程序入口。

## 四、管理员端体验流程

### 4.1 入口变更
- 主菜单不再有独立「预约看板」入口（已在前序 PRD 完成）。
- 进入「商家管理 - 门店管理」列表，每行操作列新增 **「预约看板」** 文本按钮。
- 点击该按钮 → 直接进入该门店的预约看板，URL 形如 `/admin/product-system/orders/dashboard?storeId=12`。

### 4.2 头部展示
- 左侧标题：`预约看板 · {门店名称}`
- 右上角：`返回门店列表` 按钮，点击回到 `/admin/merchant/stores`。

### 4.3 兜底逻辑
- 直接访问 `/admin/product-system/orders/dashboard/`（不带 `storeId`）时：
  1. 弹出顶部 message 警告「请先选择门店再进入预约看板」。
  2. 自动 `router.replace` 到 `/merchant/stores`。

### 4.4 数据隔离
- admin 看板的所有后端请求都强制携带 `store_id` 参数，与商家端共用相同 `/api/merchant/dashboard/*` 接口；后端按 `store_id` 做数据隔离，admin 不再看到全量预约。

## 五、后端：新预约微信模板消息

### 5.1 触发时机（仅以下两种情况会触发推送）
1. **支付成功后订单进入"待核销"**：用户付款 → 订单状态由 `pending_payment` 推进到 `pending_use`（待核销）时自动触发。
2. **付款后首次设定预约时间**：订单状态为 `pending_appointment`（已付款待预约），客户首次设定预约时段后，状态推进到 `pending_use` 时触发。

### 5.2 不会触发推送的场景
- 已有预约的「改期」操作（走既有的改期通知三通道，避免重复推送）。
- 订单取消、退款、核销、过期等其他状态变化。

### 5.3 推送对象
- 接收人 = 订单所属门店的 `MerchantStoreMembership` 中所有 `wechat_openid` 不为空的员工（即已绑定微信的店员/店长）。
- 没有任何员工绑定微信时，本次推送跳过，但订单流程不受影响。

### 5.4 模板消息字段（PRD R09）
| 字段 | 内容示例 | 说明 |
|------|----------|------|
| 客户姓名 | 张** | 来自下单时填写的预约人 |
| 客户手机 | 138****1234 | 中间 4 位脱敏 |
| 服务名称 | 中医推拿 60 分钟 | 取自订单第一件商品 |
| 预约时间 | 2026-05-08 14:30 | 已格式化 |
| 门店名称 | 朝阳旗舰店 | 来自 `merchant_stores` |

> 模板 ID 通过环境变量 `WECHAT_NEW_APPOINTMENT_TEMPLATE_ID` 注入；access_token 优先使用公众号配置，缺失时回退小程序配置。

### 5.5 容错策略
- 微信模板消息发送失败、access_token 获取失败、模板 ID 未配置等，均仅写 `warning` 日志，**绝不抛异常打断订单主流程**。
- 部分员工推送失败不影响其他员工继续推送。

## 七、已知与遗留

1. **状态展示集合差异**：PRD R07 写的是「待核销 / 已核销 / 已取消 / 已退款」共 4 类，但代码现状的看板使用「待到店 / 已到店 / 已核销 / 已取消」。本期沿用代码现状，不破坏既有 admin 看板交互；若后续要把"已退款"纳入看板色板，需要单独提 PRD。
2. **真实微信模板消息发送**：测试环境 `WECHAT_NEW_APPOINTMENT_TEMPLATE_ID` 若未配置，函数会安静跳过，不会向微信发起真实请求；正式上线前需在生产环境 `.env` 中配置该模板 ID 并完成微信公众号模板申请。
3. **桌面与移动响应式**：基础响应式已通过 Ant Design 内置断点完成；如需复杂的横屏 Tablet 适配，可在下一期细化。

## 八、回归路径建议

请按以下路径回归：
1. 商家账号登录 PC 后台 → 左侧菜单「预约看板」 → 切换日 / 周 / 月 / 服务分组 / 资源 5 个视图。
2. 浏览器粘贴旧地址 `/merchant/calendar/` → 应自动跳转。
3. 管理员账号登录 → 「商家管理 - 门店管理」 → 任意一行点「预约看板」 → 看到「预约看板 · XX 门店」 + 「返回门店列表」按钮。
4. 管理员手动改 URL 去掉 `?storeId=xx` → 应回到门店列表并提示「请先选择门店」。
5. 触发一次"付款 → 待核销"或"付款后首次设定预约时间"，观察绑定微信的员工是否在公众号收到模板消息（生产环境必需，测试环境如未配模板 ID 则跳过）。
