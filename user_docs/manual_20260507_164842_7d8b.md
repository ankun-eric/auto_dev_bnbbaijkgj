## 功能简介

修复前，客户在 H5 / 小程序 / App 任一端下单（含预约时段与支付方式选择）后，**商家管理后台 Web、商家小程序、商家 App** 三端在「订单列表」与「订单详情」页查看同一笔订单时：

- 「预约时间」会丢失具体时段，回退到当天默认 09:00；
- 「支付方式」会出现错乱（例如客户选了「支付宝（H5）」，商家端却显示成「微信」）；

这些字段与客户端实际显示并不一致，导致商家备货、排程与对账困难。

修复后，**新下订单**在客户端与三端商家侧的「预约时间」「支付方式」展示**完全一致**，且无需任何客户端代码改动（仅后端接口字段补齐 + 商家 H5 渲染优先级调整）。

---

## 本次客户端变更

本次修复**未触动**任何独立客户端的代码（小程序、安卓、iOS、Windows、macOS、CLI 均未改动），因此**无需重新下载或重新安装任何客户端**。

具体来说：

- **后端**：补齐了商家端订单列表与详情接口的字段（`time_slot`、`appointment_date`、`payment_method_text`、`payment_channel_code`、`payment_display_name`），并在源头修复了支付方式被前端默认值污染的问题。
- **H5 商家页面**（`/merchant/orders` 与 `/merchant/m/orders/[id]`）：在 Web 服务器端构建并热更新，用户在浏览器刷新后自动获取最新版本，无需手动操作。

## 使用说明

### 一、客户端下单（无任何变化）

1. 打开 [H5 客户端首页](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/) 或商家小程序 / 商家 App。
2. 选择一款支持时段预约的商品，进入下单页面。
3. 在「预约时段」选择**非默认 09:00 时段**（例如 `14:00 - 15:00`）。
4. 选择支付方式（支付宝 H5、微信、其他），完成下单与支付。

### 二、商家端订单查看（修复重点）

1. 在浏览器中打开 [商家管理后台 PC Web](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/orders) 或 [商家移动端订单](https://newbb.test.bangbangvip.com/autodev/6b099ed3-7175-4a78-91f4-44570c84ed27/merchant/m/orders)。
2. 找到刚刚客户下的订单，点击查看详情。
3. **应当看到**：
   - 「预约时间」列：显示完整的「日期 + 起止时段」，例如 `2026-05-08 14:00-15:00`，与客户端显示一致；
   - 「支付方式」列：显示客户实际使用的支付通道，例如 `支付宝（H5）`，不再误显示成「微信」。

### 三、验收清单（在线上多笔订单上分别验证）

| 验收项 | 期望结果 |
|---|---|
| 客户端选择 `14:00-15:00`，商家管理后台 Web 显示 | `2026-05-08 14:00-15:00` |
| 客户端选择 `14:00-15:00`，商家小程序 / App 显示 | `2026-05-08 14:00-15:00` |
| 客户端使用 支付宝（H5），商家端三端显示 | `支付宝（H5）` |
| 客户端使用 微信支付，商家端三端显示 | `微信支付（小程序/APP）` 等通道文案 |
| 早 / 中 / 晚不同时段共 3 笔订单，三端商家侧显示 | 与客户端完全一致 |
| 5 笔覆盖不同支付方式订单，三端商家侧显示 | 与客户端完全一致 |
| 客户端订单详情 | 不受影响，继续正确 |

## 注意事项

1. **历史订单不会被自动回溯**：本次修复的核心策略是「修代码、不改库」。在本次修复**前**已存在的脏数据订单，数据库里仍可能保留 `payment_method=wechat` 但 `payment_channel_code=alipay_h5` 的不一致状态。但得益于商家端读取层新增的「以 `payment_channel_code` 反推 provider」逻辑，**这些历史脏单在商家端的展示也会被自动矫正为正确的「支付宝（H5）」**，不会再误显示成「微信」。
2. **客户端不需要任何动作**：所有改动仅在后端 + H5 商家管理页生效。客户原 App / 小程序无需更新，下完单后立即就能在商家端看到正确字段。
3. **新下订单立即生效**：修复部署完成后下的新订单，三端商家侧即时显示与客户端一致的预约时段与支付方式，无延迟。
4. **如果浏览器缓存了旧版商家页面**，请按 `Ctrl + F5`（Windows）或 `Cmd + Shift + R`（macOS）做一次强制刷新，确保加载的是新版前端代码。

## 修复改动概览（技术参考）

| 改动文件 | 说明 |
|---|---|
| `backend/app/api/merchant.py` | 商家端订单列表 / 详情接口补齐 `time_slot`、`appointment_date`、`payment_channel_code`、`payment_display_name`、`payment_method_text` 字段 |
| `backend/app/api/merchant_v1.py` + `backend/app/schemas/merchant_v1.py` | 商家端 v1 订单列表接口同步补齐上述字段 |
| `backend/app/api/unified_orders.py` | `_build_payment_method_text` 在 `payment_method` 与 `payment_channel_code` 不一致时以 channel_code 为准；`/pay` 端点根据 channel.provider 反推 payment_method，避免前端默认值污染 |
| `backend/app/api/alipay_notify.py` | 异步通知幂等条件下也强制矫正 payment_method / channel_code，杜绝脏数据残留 |
| `h5-web/src/app/merchant/orders/page.tsx` | 商家管理后台 PC 端订单列表「支付方式」与「预约时间」列优先用新字段渲染 |
| `h5-web/src/app/merchant/m/orders/[id]/page.tsx` | 商家移动端订单详情页同步使用新字段渲染 |
| `backend/tests/test_bugfix_uo_20260507_001.py` | 新增 10 个用例覆盖 payment_method ↔ channel_code 矫正与 _format_time_slot 的所有分支 |
